<task type="auto">
  <name>Task 1: Replace Tauri default menu with custom menu that emits undo/redo events to frontend</name>
  <files>Application/src-tauri/src/lib.rs</files>
  <action>
Modify lib.rs to set a custom application menu that replaces the default macOS menu. The default menu includes Edit > Undo (Cmd+Z) and Edit > Redo (Cmd+Shift+Z) accelerators that intercept keydown events at the Cocoa layer before they reach the WKWebView.

The fix:
1. Add `use tauri::menu::{MenuBuilder, SubmenuBuilder, MenuItem};` and `use tauri::Emitter;` to imports.
2. Convert the builder chain to use `.setup(|app| { ... })` for menu construction. Move the existing `.run()` call after setup.
3. Inside setup, create a custom menu with:
   - An app submenu (use the app name) containing Quit with Cmd+Q accelerator
   - An Edit submenu containing Undo and Redo menu items with Cmd+Z and Cmd+Shift+Z accelerators. Use `MenuItem::with_id()` to give them IDs "undo" and "redo". Include Cut/Copy/Paste/Select All items using `MenuItem::with_id()` with standard accelerators (Cmd+X, Cmd+C, Cmd+V, Cmd+A) -- these are needed for text input fields in the webview to work on macOS. Do NOT use PredefinedMenuItem for Undo/Redo (those would use the native implementation). PredefinedMenuItem is fine for Cut/Copy/Paste/Select All/Quit/Separator since those native operations work correctly in webview.
   - Set the menu with `app.set_menu(menu)?;`
4. Wire `app.on_menu_event()` to handle undo/redo clicks:
   - "undo" -> `app_handle.emit("menu:undo", ()).ok();`
   - "redo" -> `app_handle.emit("menu:redo", ()).ok();`
   Other menu items (cut, copy, paste, select_all) don't need event handling as PredefinedMenuItem handles them natively.

IMPORTANT: The `on_menu_event` closure needs to clone the app handle. Use `let handle = app.handle().clone();` before the closure, then `handle.emit(...)` inside.

The result: Cmd+Z and Cmd+Shift+Z accelerators now emit events to the frontend instead of being consumed by the native Cocoa undo system. The JS tinykeys bindings in shortcuts.ts will ALSO fire since the keydown event now reaches the webview. To avoid double-firing, the frontend listener (Task 2) will be the sole handler and the tinykeys bindings for Cmd+Z/Shift+Z should be kept as-is since tinykeys uses `event.key`/`event.code` matching while the menu accelerator fires the menu event -- they are separate event channels. Actually, since menu accelerators consume the keyboard event at the OS level AND fire the menu event, the keyboard event still won't reach tinykeys. The menu event handler in main.tsx will be the sole path for undo/redo. The tinykeys bindings can remain as a fallback for platforms where the native menu doesn't intercept.

<task type="auto">

  <files>Application/src/main.tsx, Application/src/lib/shortcuts.ts</files>
  <action>
Two changes in this task:

**A) Add menu event listeners in main.tsx:**

1. Import `{ listen } from '@tauri-apps/api/event'` and `{ undo, redo } from './lib/history'`.
2. In the existing app initialization (where `mountShortcuts()` is called), add two event listeners:
   ```typescript
   listen('menu:undo', () => { undo(); });
   listen('menu:redo', () => { redo(); });
   ```
   Place these calls near `mountShortcuts()`. No need to capture unlisten handles since these persist for app lifetime.

**B) Fix ? shortcut binding in shortcuts.ts:**

Change the binding key on line 171 from:
```typescript
'Shift+Slash': (e: KeyboardEvent) => {
```
to:
```typescript
'Shift+?': (e: KeyboardEvent) => {
```

This changes the matching from physical key code ('Slash') to character matching ('?'). tinykeys tries event.key first (case-insensitive), so 'Shift+?' will match event.key='?' with Shift modifier on ANY keyboard layout, regardless of which physical key produces the ? character.

Do NOT change any other shortcut bindings -- they use $mod+Key* format which is correct for key codes (Cmd+Z, Cmd+S, etc. are universal across layouts).
