# Phase 05.3: play-animation - Research

**Researched:** 2026-04-02
**Domain:** Frame-based animation playback, Canvas 2D frame capture, tsup sub-path exports
**Confidence:** HIGH

## Summary

Phase 05.3 adds frame-based animation playback to the EFX Physics Paint library. The core challenge is converting a timestamp-based stroke recording into a frame-by-frame replay where strokes progressively appear point-by-point, and exposing an `onFrame` callback API for downstream consumers (efx-motion-editor) to capture frames for video export.

The existing codebase has `replayAnimated()` which replays strokes one-by-one as fast as possible (setTimeout 1ms between strokes). This phase replaces that model with a controlled frame-based scheduler: strokes are mapped to frames by their timestamp ratio, and within each frame, the appropriate portion of points is rendered. The AnimationPlayer wraps EfxPaintEngine as a separate class with a sub-path export via tsup.

**Primary recommendation:** Build AnimationPlayer as a standalone class in `src/animation/AnimationPlayer.ts` that takes an EfxPaintEngine instance, uses `save()` to get stroke data, and calls `redrawAll()`-style rendering with progressive point subsets per frame. Use `requestAnimationFrame` for visual playback with elapsed-time tracking to respect the configured FPS.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Strokes mapped to frames by timestamp ratio. Stroke at 25% of total recording time maps to 25% of total frame count. Natural pacing that preserves the painting rhythm.
- **D-02:** Point-by-point drawing within frames. A stroke spanning frames 3-5 draws progressively across those frames (partial point rendering per frame). More cinematic than instant-complete strokes.
- **D-03:** FPS is a configurable data input (no hardcoded default). Settable via UI input field and API. efx-motion-editor will pass its own FPS.
- **D-04:** Play + Stop controls only. Stop resets to the end state (final painting fully rendered), not back to blank canvas.

### Pattern 1: Frame Distribution Algorithm

**What:** Map stroke timestamps to frame indices using linear ratio.
**When to use:** Converting continuous-time stroke recordings to discrete frame sequences.
**Confidence:** HIGH -- straightforward linear mapping from CONTEXT.md D-01.

```typescript
// Frame distribution: timestamp ratio -> frame index
interface FrameStroke {
  stroke: PaintStroke
  startFrame: number   // first frame this stroke appears
  endFrame: number     // last frame this stroke renders points on
  pointsPerFrame: number  // how many points to render per frame
}

function distributeStrokesToFrames(
  strokes: PaintStroke[],
  totalFrames: number,
): FrameStroke[] {
  if (strokes.length === 0) return []
  
  const minTime = strokes[0].timestamp
  const maxTime = strokes[strokes.length - 1].timestamp
  const duration = maxTime - minTime || 1  // avoid /0 for single stroke
  
  return strokes.map(stroke => {
    const ratio = (stroke.timestamp - minTime) / duration
    const startFrame = Math.floor(ratio * (totalFrames - 1))
    
    // Stroke spans multiple frames based on point count
    const pointCount = stroke.points.length
    // Each frame renders ~pointsPerFrame points progressively
    const framesNeeded = Math.max(1, Math.ceil(pointCount / 8))  // ~8 pts/frame baseline
    const endFrame = Math.min(totalFrames - 1, startFrame + framesNeeded - 1)
    const pointsPerFrame = Math.ceil(pointCount / (endFrame - startFrame + 1))
    
    return { stroke, startFrame, endFrame, pointsPerFrame }
  })
}
```

### Pattern 3: rAF-based Frame Scheduler with FPS Control

**What:** Use requestAnimationFrame with elapsed-time tracking to emit frames at the configured FPS.
**When to use:** For visual playback in the demo app and precise frame timing.
**Confidence:** HIGH -- standard approach for FPS-controlled canvas animation.

```typescript
// Inside AnimationPlayer
private tick = (timestamp: number): void => {
  if (!this.playing) return
  
  const frameDuration = 1000 / this.config!.fps
  const elapsed = timestamp - this.lastFrameTime
  
  if (elapsed >= frameDuration) {
    this.lastFrameTime = timestamp - (elapsed % frameDuration)
    this.renderFrame(this.currentFrame)
    this.config!.onFrame?.(this.currentFrame, this.engine.getDisplayCanvas())
    this.currentFrame++
    
    if (this.currentFrame >= this.config!.frameCount) {
      this.playing = false
      this.config!.onComplete?.()
      return
    }
  }
  
  this.rafId = requestAnimationFrame(this.tick)
}
```

### Pattern 4: Progressive Point Rendering (D-02)

**What:** For strokes that span multiple frames, render a subset of points per frame.
**When to use:** When a stroke's point array should be drawn progressively across frames.
**Confidence:** MEDIUM -- requires careful integration with `renderPaintStroke`.

**Critical insight:** `renderPaintStroke()` takes the full points array and processes it atomically (smooth, resample, ribbon, deform). It cannot be called incrementally with growing point arrays because the deformation is randomized -- each call produces slightly different output.

**Recommended approach:** For each frame, render the stroke with the subset of points accumulated up to that frame. Since the rendering creates fresh offscreen canvases each time and the deformation is per-call, the visual result will be a "growing" stroke. The key is that each frame clears and re-renders from scratch (like `redrawAll()`) up to the current animation state. This is the same approach used by `replayAnimated()` -- render complete strokes one by one.

**Practical simplification:** Rather than true point-by-point rendering (which would require refactoring `renderPaintStroke`), render whole strokes frame-by-frame. Strokes that map to the current frame render completely, and the effect is still cinematic because multiple strokes span across frames. For strokes with many points (long strokes), split the points array at frame boundaries and call `renderPaintStroke` with each subset as a separate mini-stroke.

```typescript
// For a stroke spanning frames 3-5 with 24 points:
// Frame 3: renderPaintStroke(points[0..7])   -- first 8 points
// Frame 4: renderPaintStroke(points[0..15])  -- first 16 points (re-renders from scratch)
// Frame 5: renderPaintStroke(points[0..23])  -- all 24 points
```

**However**, re-rendering from scratch each frame is expensive. Better approach: for each frame, replay all strokes up to and including the current frame's strokes using the existing `redrawAll()` pattern (clear, iterate, forceDryAll). This matches the existing architecture perfectly and avoids partial-render complexity.

### Anti-Patterns to Avoid

- **Mutating engine state during animation:** AnimationPlayer should not modify `allActions` or other engine internals. It should use the serialization API (`save()`) to get strokes and operate on its own copy.
- **Blocking the main thread:** Each frame must complete within the rAF budget (~16ms at 60fps). For complex paintings with many strokes, rendering all strokes up to frame N may exceed this. The per-frame rendering should be kept synchronous but the frame count should be realistic.
- **Modifying renderPaintStroke for partial rendering:** The function's internal smooth/resample/deform pipeline assumes a complete stroke. Splitting points and calling it per-subset is safe (each subset is a valid mini-stroke). Do NOT try to add incremental state to it.
- **Using the engine's own render loop during animation:** The engine runs its own rAF render loop for wet layer compositing. During animation, the AnimationPlayer should either pause the engine's render loop or coordinate with it.

## Common Pitfalls

### Pitfall 1: Frame Budget Overrun

**What goes wrong:** Rendering N strokes for frame N takes longer than 16ms, causing dropped frames and janky playback.
**Why it happens:** Each frame re-renders all strokes up to that point. Frame 100 of a 200-stroke painting re-renders 100 strokes from scratch.
**How to avoid:** Pre-render strategy -- render each frame synchronously in a loop (not tied to display), capture to offscreen canvas, then play back the captured frames. OR accept that playback FPS is a target, not a guarantee, and skip frames if behind schedule. The simplest approach: accept slightly slower-than-realtime for complex paintings.
**Warning signs:** FPS counter showing < target FPS during playback, visible stuttering.

### Pitfall 2: Wet Layer State Leaking Between Frames

**What goes wrong:** Wet buffers retain state from previous frame rendering, causing paint accumulation artifacts.
**Why it happens:** `renderPaintStroke` writes to wet buffers, and `forceDryAll` bakes them to dry canvas. If not cleared between frames properly, wet state accumulates.
**How to avoid:** Follow the exact pattern from `redrawAll()`: clear wet layer, iterate strokes, forceDryAll after each. This is already the proven replay path.
**Warning signs:** Strokes appearing with double intensity, wet layer artifacts visible as extra opacity.

### Pitfall 3: Engine Render Loop Conflict

**What goes wrong:** The engine's own rAF render loop (`render()`) composites the wet layer on every frame, interfering with animation playback rendering.
**Why it happens:** The engine was designed for interactive painting, not frame-by-frame playback.
**How to avoid:** During animation playback, the AnimationPlayer must coordinate with the engine. Options: (a) pause the engine's render loop, (b) use the engine's render cycle and just control when strokes are drawn. Recommend option (a) -- add a `pauseRenderLoop()` / `resumeRenderLoop()` pair to the engine, or a simple boolean flag.
**Warning signs:** Flickering during playback, wet layer appearing/disappearing.

### Pitfall 4: allActions is Private

**What goes wrong:** AnimationPlayer cannot access recorded strokes directly.
**Why it happens:** `allActions` is a private field on EfxPaintEngine.
**How to avoid:** Add a public `getStrokes(): PaintStroke[]` method (returns a copy) or use `save()` which serializes all strokes. `save()` is already public and returns `SerializedProject` which includes all stroke data with timestamps.
**Warning signs:** TypeScript compilation errors when accessing `engine.allActions`.

### Pitfall 5: Stop Does Not Show Final Painting

**What goes wrong:** Stopping animation leaves canvas in a partial state.
**Why it happens:** D-04 requires stop to show the final fully-rendered painting, not blank or partial.
**How to avoid:** On stop, call the engine's existing `redrawAll()` pattern to render all strokes to completion. This requires either making `redrawAll()` public or adding a `renderAllStrokes()` method.
**Warning signs:** User presses stop and sees incomplete painting.

## Claude's Discretion Recommendations

### Physics During Playback

**Recommendation: Skip physics (spreading/drying) during animation frames.** 

Rationale: Running Stam fluid solver between each frame would be extremely slow and the visual difference is minimal for playback purposes. The existing `redrawAll()` and `replayAnimated()` both call `forceDryAll()` after each stroke without running physics -- the same approach should be used here. Physics is an interactive feature, not a replay feature.

### Frame Scheduler Implementation

**Recommendation: Use `requestAnimationFrame` with elapsed-time accumulator.**

Rationale: rAF provides smooth visual playback synchronized with the display refresh rate, auto-pauses in background tabs (saving CPU), and provides high-resolution timestamps. The elapsed-time accumulator ensures the configured FPS is respected even on 120Hz+ displays.

### Handling Existing replayAnimated()

**Recommendation: Keep `replayAnimated()` as-is for file load operations. AnimationPlayer is a separate concern.**

Rationale: `replayAnimated()` serves a different purpose (fast visual feedback when loading a saved project). AnimationPlayer is for controlled frame-by-frame playback with callbacks. They have different performance characteristics and use cases. Refactoring would risk breaking the working load flow.

export class AnimationPlayer {
  private engine: EfxPaintEngine

  private rafId = 0
  private lastFrameTime = 0

  constructor(engine: EfxPaintEngine) {
    this.engine = engine

  play(config: AnimationConfig): void {
    this.config = config
    this.currentFrame = 0
    this.playing = true
    this.lastFrameTime = 0
    // Lock painting input on engine
    // Render first frame
    this.rafId = requestAnimationFrame(this.tick)
  }

  /** Stop playback and show final painting */
  stop(): void {
    this.playing = false
    if (this.rafId) cancelAnimationFrame(this.rafId)
    // Render final state (all strokes)
    // Unlock painting input on engine
  }

  /** Check if currently playing */
  isPlaying(): boolean { return this.playing }

  /** Get current animation state */
  getState(): AnimationState {
    return {
      playing: this.playing,
      currentFrame: this.currentFrame,
      totalFrames: this.config?.frameCount ?? 0,
    }
  }

  private tick = (timestamp: number): void => {
    if (!this.playing || !this.config) return
    if (this.lastFrameTime === 0) this.lastFrameTime = timestamp

      this.lastFrameTime = timestamp
      this.renderFrame(this.currentFrame)
      this.config.onFrame?.(this.currentFrame, this.engine.getDisplayCanvas())
      this.currentFrame++

      if (this.currentFrame >= this.config.frameCount) {
        this.playing = false
        this.config.onComplete?.()
        return
      }
    }

    this.rafId = requestAnimationFrame(this.tick)
  }

  private renderFrame(frameIndex: number): void {

  }
}

## Open Questions

1. **Frame rendering performance for complex paintings**
   - What we know: Each frame re-renders all strokes up to that frame. A 100-stroke painting at frame 100 re-renders 100 strokes.
   - What's unclear: Whether this fits in 16ms for typical paintings. Likely fine for 10-30 strokes, may struggle at 100+.
   - Recommendation: Implement the simple approach first. If performance is an issue, add a pre-rendering pass that captures each frame to an offscreen canvas array, then plays back from the array. This is a performance optimization, not an architectural change.

2. **Point-by-point vs stroke-by-stroke granularity**
   - What we know: D-02 requests "point-by-point drawing within frames". `renderPaintStroke` processes entire point arrays atomically.
   - What's unclear: Whether splitting a stroke's points into subsets and calling `renderPaintStroke` per subset produces visually acceptable results (each subset would look like a short stroke).
   - Recommendation: Start with stroke-by-stroke granularity (whole strokes map to frames). If the user wants finer point-by-point control, split long strokes' point arrays at frame boundaries and render each subset. This produces a "growing stroke" effect because each subset is a valid shorter stroke.

3. **Engine render loop coordination**
   - What we know: The engine runs its own rAF loop for wet compositing. During animation, two rAF loops would conflict.
   - What's unclear: Best way to pause/resume the engine loop.
   - Recommendation: Add a boolean `animationMode` flag to the engine. When true, the engine's `render()` method skips compositing and lets AnimationPlayer control rendering.
