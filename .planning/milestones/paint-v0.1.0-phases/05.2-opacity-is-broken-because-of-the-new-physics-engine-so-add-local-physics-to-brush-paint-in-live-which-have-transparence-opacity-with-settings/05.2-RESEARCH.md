# Phase 05.2: Opacity Fix + Local Physics Mode - Research

**Researched:** 2026-04-01
**Domain:** Canvas 2D per-pixel opacity compositing, Stam fluids subgrid solver, paint engine architecture
**Confidence:** HIGH

## Summary

Phase 05.2 addresses two coupled problems: (1) per-stroke opacity was lost when Phase 05.1 moved opacity from the wet-layer deposit to a global compositor multiplier (D-12), and (2) the Stam stable fluids solver currently only runs via manual hold-buttons on already-deposited paint, when users want paint to spread during the painting gesture itself.

The opacity regression is straightforward: `compositeWetLayer()` currently reads `userOpacity` as a scalar applied uniformly to every wet pixel (compositor.ts line 65). When the user changes the opacity slider, ALL existing wet paint changes brightness. The fix adds a `strokeOpacity` Float32Array to WetBuffers where each stroke deposits its intended opacity via Porter-Duff "over" accumulation. The compositor reads this per-pixel value instead of the scalar.

The local physics mode runs the Stam solver scoped to the active stroke's bounding box during painting (not after), using the same global fluid parameters. The existing `fluidPhysicsStep` already supports bounded processing via `lastStrokeBounds` -- the local mode reuses this mechanism but triggers it automatically during the stroke lifecycle.

**Primary recommendation:** Implement in three waves: (1) strokeOpacity buffer + compositor fix + directRender removal, (2) local physics mode in engine, (3) UI controls. Wave 1 is the critical regression fix; waves 2-3 are additive features.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Add a per-pixel opacity accumulator buffer (`wet.strokeOpacity`, Float32Array) that tracks intended visual opacity per pixel. Each stroke deposits its opacity value at the pixels it touches
- **D-02:** Overlap compositing uses Porter-Duff "over" operator: `result = existing + new * (1 - existing)`. Two 50% strokes overlap at 75%, three at 87.5%. Never exceeds 1.0
- **D-03:** Physics (Stam solver) still operates on full-intensity `wet.alpha` buffers -- D-12 principle preserved. The `strokeOpacity` buffer is display-only, read by the compositor
- **D-04:** Compositor reads `wet.strokeOpacity[i]` per-pixel instead of the global `userOpacity` parameter. `finalAlpha = displayAlpha * paperMod * strokeOpacity[i]`
- **D-05:** New physics mode: "local physics" -- Stam solver runs scoped to the active stroke's bounding box + margin, using the same global physics settings (viscosity, diffusion rate, etc.)
- **D-06:** Spread distance derived from global physics strength -- higher water amount = wider spread margin. Not a fixed pixel radius
- **D-07:** Both local and global physics modes exist. Local = physics runs on stroke footprint during painting. Global = Stam solver runs on entire wet layer (existing behavior). User toggles between them

## Architecture Patterns

### Current Code Flow (what needs changing)

```
Stroke input
  -> renderPaintStrokeSingleColor(curve, ..., directRender, ...)
    -> if directRender:
        -> post-multiply opacity on offscreen canvas
        -> draw to dry canvas directly (bypasses wet layer)
    -> else:
        -> transferToWetLayerClipped (full intensity, no opacity)
        -> compositor reads global userOpacity scalar
  -> forceDryAll (bakes wet to dry with /800 divisor)
```

### Target Code Flow (after phase 05.2)

```
Stroke input
  -> renderPaintStrokeSingleColor(curve, ..., opacity, ...)
    -> transferToWetLayerClipped (full intensity to wet.alpha -- D-03 preserved)
    -> write strokeOpacity via Porter-Duff: strokeOpacity[i] = old + opacity * (1 - old)
    -> if localPhysics: run fluidPhysicsStep on stroke bounding box
  -> compositor reads wet.strokeOpacity[i] per-pixel (D-04)
  -> forceDryAll (uses strokeOpacity to modulate dry transfer amount)
```

### Pattern 1: strokeOpacity Accumulator

**What:** A Float32Array (size width*height) added to WetBuffers that tracks the intended visual opacity for each pixel, accumulated across strokes via Porter-Duff "over".

**When to use:** Every time paint is deposited to the wet layer.

**How it works:**
```typescript
// In transferToWetLayerClipped, after writing to wet.alpha/r/g/b:
const existingOpacity = wetBuffers.strokeOpacity[i]
const strokeOpacity = userOpacity  // 0-1, from brush opts
wetBuffers.strokeOpacity[i] = existingOpacity + strokeOpacity * (1 - existingOpacity)
// Porter-Duff "over": 0.5 + 0.5*(1-0.5) = 0.75
```

**Critical detail:** Only write strokeOpacity for pixels that actually receive paint in this stroke (where `a >= 2` in the transfer loop). Don't retroactively modify opacity of previously painted pixels.

### Pattern 2: Local Physics Integration

**What:** Running the Stam solver during the stroke lifecycle (between pointer events or at stroke completion), scoped to the stroke's bounding box.

**When to use:** When `physicsMode === 'local'` (default).

**Integration point:** After `transferToWetLayerClipped` in `onPointerUp`, before `forceDryAll`. The engine runs N ticks of `fluidPhysicsStep` with the stroke bounding box, then bakes.

```typescript
// In onPointerUp, after transfer to wet layer:
if (this.state.physicsMode === 'local') {
  const margin = Math.ceil(30 + (this.state.brushOpts.waterAmount / 100) * 70)
  const localBounds = {
    x0: Math.max(0, strokeBounds.x0 - margin),
    y0: Math.max(0, strokeBounds.y0 - margin),
    x1: Math.min(this.width - 1, strokeBounds.x1 + margin),
    y1: Math.min(this.height - 1, strokeBounds.y1 + margin),
  }
  // Run N physics ticks for local spreading
  const ticks = Math.ceil(5 + (this.state.localSpreadStrength / 100) * 20)
  for (let t = 0; t < ticks; t++) {
    fluidPhysicsStep(this.wet, this.fluid, this.fluidConfig,
      this.width, this.height, this.blowDX, this.blowDY,
      localBounds, 'last', sampleHFn)
  }
}
```

### Pattern 3: Subgrid Stam Solver (Performance Optimization)

**What:** Instead of allocating full-canvas-sized Float32Arrays in `fluidPhysicsStep`, allocate grids sized to the local bounding box only.

**Current problem:** `fluidPhysicsStep` allocates 10 Float32Arrays of `(W+2)*(H+2)` size EVERY TICK (fluids.ts lines 387-398). For a 1000x650 canvas, that's 10 * 652002 * 4 = ~25 MB per tick. For local physics on a 200x200 bbox, that drops to 10 * 40804 * 4 = ~1.6 MB.

**Recommendation (Claude's discretion):** Use subset grid approach:
- Create a new function `localFluidPhysicsStep(wet, fluid, config, bbox, canvasW, canvasH, ...)` that:
  1. Computes local grid dimensions: `localW = bbox.x1 - bbox.x0 + 1`, `localH = bbox.y1 - bbox.y0 + 1`
  2. Allocates Stam grids at `(localW+2)*(localH+2)` size
  3. Copies wet data from bbox region into local grid
  4. Runs velStep + advect on local grid
  5. Copies results back to canvas-sized wet buffers

This avoids the full-grid allocation per tick while keeping the existing `fluidPhysicsStep` intact for global mode.

### Anti-Patterns to Avoid

- **Advecting strokeOpacity:** strokeOpacity is a display-only accumulator. Do NOT advect it through the Stam solver alongside R/G/B/alpha. Physics operates on full-intensity wet.alpha (D-03). If strokeOpacity were advected, it would smear/blur the intended per-stroke transparency, producing visual artifacts.

- **Modifying forceDryAll's /800 divisor:** The /800 is "sacred" (feedback). Instead, multiply the existing `sa = min(wet.alpha[i] / 800, 1)` by `strokeOpacity[i]` to get the final dry-layer transfer alpha. This preserves the calibrated density curve while incorporating per-pixel opacity.

- **Creating a separate physics loop for local mode:** Don't create a second `setInterval` for local physics. Run local physics synchronously in `onPointerUp` (blocking but brief for small bounding boxes). The Stam solver on a 200x200 grid runs in <2ms per tick.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Opacity compositing | Custom alpha blending formula | Porter-Duff "over" (`a + b*(1-a)`) | Industry standard, mathematically correct, handles any number of overlapping strokes |
| Fluid simulation | New solver for local physics | Existing Stam solver with bbox parameter | Already implemented and working; just needs local grid variant |
| Bounding box tracking | Manual point tracking | Existing `curveBounds()` from util/math.ts | Already used by renderPaintStroke for offscreen canvas clipping |

**Key insight:** This phase mostly rewires existing systems. The Stam solver works. The compositor works. The transfer functions work. The changes are: (1) add a buffer, (2) change where values come from, (3) scope the solver differently.

## Common Pitfalls

### Pitfall 1: strokeOpacity Not Cleared on Canvas Clear

**What goes wrong:** After `engine.clear()`, old strokeOpacity values remain, causing ghost transparency on new strokes.
**Why it happens:** `clearWetLayer()` doesn't know about the new buffer.
**How to avoid:** Add `strokeOpacity.fill(0)` to `clearWetLayer()`. Also add it to undo snapshot/restore.
**Warning signs:** Painting after clear produces unexpectedly opaque or transparent strokes.

### Pitfall 2: forceDryAll Ignoring Per-Pixel Opacity

**What goes wrong:** Paint dries at full opacity even when the user painted at 30%.
**Why it happens:** `forceDryAll` uses `sa = min(wet.alpha[i] / 800, 1)` without considering strokeOpacity.
**How to avoid:** Multiply: `const sa = Math.min(1, wet.alpha[i] / 800) * (strokeOpacity[i] || 1)`. Reset strokeOpacity[i] to 0 after drying (same as other wet buffers).
**Warning signs:** All strokes appear at 100% opacity after drying.

### Pitfall 3: dryStep Also Needs strokeOpacity

**What goes wrong:** During physics simulation, dryStep transfers paint at full opacity.
**Why it happens:** Same issue as forceDryAll but in the incremental path.
**How to avoid:** Same multiplication pattern in dryStep's transfer calculation (drying.ts line 95 area).
**Warning signs:** Transparency correct in wet composite but lost after physics + dry cycle.

### Pitfall 4: Local Physics Allocations Per Tick

**What goes wrong:** Running local physics during stroke produces frame drops / jank.
**Why it happens:** `fluidPhysicsStep` allocates 10 full-canvas Float32Arrays every tick (fluids.ts lines 387-398).
**How to avoid:** For local physics, either (a) create a subgrid variant that allocates smaller arrays, or (b) pre-allocate reusable scratch arrays in the engine constructor.
**Warning signs:** Frame rate drops during painting with local physics enabled.

### Pitfall 5: Replay/Redraw Not Updated

**What goes wrong:** Loading a saved project or changing background produces wrong opacity.
**Why it happens:** `redrawAll()` and `replayAnimated()` call `renderPaintStroke()` but may not track strokeOpacity accumulation between strokes correctly, since they forceDry between each stroke.
**How to avoid:** When replaying, either (a) reset strokeOpacity before each forceDry and let the dry transfer use the accumulated value, or (b) store per-stroke opacity in the serialized stroke data for faithful replay.
**Warning signs:** Loaded projects look different from when they were saved.

### Pitfall 6: Wet Composite Blending Under Unified Path

**What goes wrong:** With directRender removed, strokes that previously used wet composite blending (wetPaper=true) may look different.
**Why it happens:** The unified path always goes through `transferToWetLayerClipped` which deposits differently than direct canvas drawing with `applyWetCompositeClipped`.
**How to avoid:** Test with wetPaper=true and compare visual output. The wet composite effect (watercolor edge blending) should be recreated in the compositor path if needed.
**Warning signs:** Strokes look flat or lack watercolor blending when wetPaper is on.

### Pitfall 7: strokeOpacity in Pickup Path

**What goes wrong:** Strokes with pickup > 0 don't get correct per-pixel opacity.
**Why it happens:** The pickup path in `renderPaintStroke` (lines 401-446) uses segment-by-segment rendering with its own `transferToWetLayerClipped` calls. Each segment needs to write strokeOpacity.
**How to avoid:** Add strokeOpacity writing to ALL `transferToWetLayerClipped` calls, not just the single-color path.
**Warning signs:** Pickup strokes appear fully opaque regardless of opacity slider.

## Discretion Recommendations

### strokeOpacity During Physics Diffusion

**Recommendation: Keep strokeOpacity static (do NOT advect).**

Reasoning: strokeOpacity represents the user's intended visual opacity for that pixel. Physics moves paint (wet.alpha, wet.r/g/b) but should not dilute the user's opacity intent. If we advected strokeOpacity, a 50% opacity stroke that spreads would gradually become more transparent at edges -- undesirable. The physics spreads the paint mass (alpha), and wherever paint exists, it displays at the opacity the user intended for that area.

Edge case: When physics spreads paint to new pixels that have strokeOpacity = 0, those pixels would be invisible despite having wet.alpha > 0. Solution: When physics writes to a pixel with strokeOpacity = 0 and wet.alpha > threshold, set strokeOpacity to a default (e.g., the average of neighboring strokeOpacity values, or 1.0 for physics-spread paint).

### Performance: Subset Grid vs Full Grid with Mask

**Recommendation: Subset grid for local physics.**

The current `fluidPhysicsStep` allocates 10 Float32Arrays of `(canvasW+2)*(canvasH+2)` EVERY TICK (no reuse). For a 1000x650 canvas, that's ~26 MB of allocations per physics tick. For local physics running during painting, this is unacceptable.

Approach: Create `localFluidPhysicsStep(wet, localFluid, config, bbox, canvasW, canvasH, ...)`:
- Compute `localW = bbox.x1 - bbox.x0 + 1`, `localH = bbox.y1 - bbox.y0 + 1`
- Allocate Stam grids at `(localW+2)*(localH+2)` -- for a 200x200 bbox, this is ~1.6 MB total
- Map canvas coordinates to local coordinates with offset
- Run the standard velStep/advect pipeline on the local grid
- Copy results back to canvas wet buffers

Pre-allocate the local fluid buffers in the constructor at a "max local size" (e.g., 400x400) and reuse them. Only reallocate if a stroke exceeds this size.

### Bounding Box Margin Calculation

**Recommendation:** Dynamic margin based on water amount + spread strength:

```typescript
const baseMargin = 20  // minimum for numerical stability
const waterMargin = (waterAmount / 100) * 50  // 0-50px from water
const spreadMargin = (localSpreadStrength / 100) * 30  // 0-30px from spread slider
const margin = Math.ceil(baseMargin + waterMargin + spreadMargin)
```

This implements D-06 (spread from physics strength) with the new D-11 slider as an additional factor.

### wetPaper Under Unified Path

**Recommendation: No adjustment needed initially.**

With directRender removed, ALL paint goes through wet layer + compositor. The wetPaper toggle currently only matters in the `directRender` branches (paint.ts lines 438, 511). After removing those branches, wetPaper's only remaining effect should be on the wet composite blending behavior during physics (if at all).

If visual testing shows the wet blending is missed, it can be added as a post-compositor step that applies `applyWetComposite` blending between the wet composite and the dry canvas. But start without it and evaluate visually.

### Mode Switching Mid-Session

**Recommendation: Instant switch, no migration.**

When switching from local to global or vice versa, simply change the mode flag. Any in-progress physics (manual hold buttons in global mode) should stop first. Local physics only runs at stroke completion, so there's no "in-progress" local physics to interrupt.

## Open Questions

1. **strokeOpacity for physics-spread pixels**
   - What we know: Physics moves wet.alpha to new pixels. Those pixels start with strokeOpacity = 0.
   - What's unclear: Should physics-spread paint inherit the source pixel's strokeOpacity, or default to 1.0 (fully opaque)?
   - Recommendation: Default to 1.0 for physics-spread paint. Physics spreading is an intentional effect -- users expect spread paint to be visible. Setting to 0 would make it invisible, which is confusing.

2. **Per-stroke opacity in serialized project**
   - What we know: The `PaintStroke` type already stores `params.opacity`. During replay, each stroke's opacity is available.
   - What's unclear: Should strokeOpacity be serialized separately, or reconstructed during replay?
   - Recommendation: Reconstruct during replay. Each stroke already has its opacity parameter. Replay writes strokeOpacity the same way live painting does. No serialization format change needed.

3. **Interaction with undo**
   - What we know: Undo snapshots `wet.r/g/b/alpha/wetness + drying.dryPos`.
   - What's unclear: Does undo also need to snapshot strokeOpacity?
   - Recommendation: Yes. Add `strokeOpacity: new Float32Array(this.wet.strokeOpacity)` to the undo snapshot. Otherwise, undoing a stroke would leave stale strokeOpacity values.
