# Phase 05.3: play-animation - Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Frame-based stroke animation playback. User defines frame count and FPS, strokes distribute across frames by timestamp ratio, and the painting process plays back point-by-point. Simple UI controls in the demo toolbar (frame count, FPS, play/stop, frame counter). Separate AnimationPlayer class exported as sub-path for efx-motion-editor to consume via API for video export.

</domain>

<decisions>
## Implementation Decisions

### Frame distribution
- **D-01:** Strokes mapped to frames by timestamp ratio. Stroke at 25% of total recording time maps to 25% of total frame count. Natural pacing that preserves the painting rhythm.
- **D-02:** Point-by-point drawing within frames. A stroke spanning frames 3-5 draws progressively across those frames (partial point rendering per frame). More cinematic than instant-complete strokes.

### Playback model
- **D-03:** FPS is a configurable data input (no hardcoded default). Settable via UI input field and API. efx-motion-editor will pass its own FPS.
- **D-04:** Play + Stop controls only. Stop resets to the end state (final painting fully rendered), not back to blank canvas.
- **D-05:** Play once and stop at the last frame. No looping. efx-motion-editor can implement loop behavior on top of the API if needed.

### API surface
- **D-06:** onFrame(frameIndex, canvas) callback model. API emits per-frame events during playback. Consumer captures canvas data for export (efx-motion-editor video pipeline).
- **D-07:** Separate AnimationPlayer class that wraps EfxPaintEngine. `const player = new AnimationPlayer(engine)`. Clean separation from paint engine.
- **D-08:** Sub-path export: `import { AnimationPlayer } from '@efxlab/efx-physic-paint/animation'`. Tree-shakeable, keeps core bundle smaller for apps that don't need animation.

### Demo UI controls
- **D-09:** Animation controls in the existing Toolbar component. Frame count input, FPS input, play/stop button.
- **D-10:** Frame counter display showing 'current / total' next to play button during playback.
- **D-11:** Painting disabled during animation playback. Canvas locked until user stops animation.

### Claude's Discretion
- Whether physics (spreading/drying) ticks between frames during playback, or only stroke drawing animates. Decision based on performance constraints and codebase architecture.
- Internal frame scheduler implementation (requestAnimationFrame vs setTimeout timing)
- How to handle the existing `replayAnimated()` — refactor into AnimationPlayer or keep as separate internal method
- tsup configuration for sub-path export

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `EfxPaintEngine.replayAnimated()` — Already replays strokes sequentially with setTimeout(replayNext, 1). Can inform AnimationPlayer design but needs frame-based timing instead of as-fast-as-possible
- `EfxPaintEngine.redrawAll()` — Synchronous full replay. Useful for rendering a specific frame state
- `PaintStroke.timestamp` — Already captured per stroke, provides the timing data for frame distribution
- `SerializedProject` — JSON format already captures strokes with timing, settings. AnimationPlayer can consume this
- `renderPaintStroke()` / `applyEraseStroke()` — Existing stroke rendering functions that AnimationPlayer will call
- `forceDryAll()` — Dries wet layer between strokes during replay

### Established Patterns
- All physics modules are pure functions receiving buffers as arguments (Phase 05)
- EfxPaintEngine facade owns all typed array buffers
- Toolbar uses Preact functional component with props callbacks
- Package uses tsup for build (ESM-only per Phase 05 D-14)

### Integration Points
- AnimationPlayer wraps EfxPaintEngine — needs access to engine's internal replay methods or a public replay API
- Toolbar receives engine instance and calls methods on it — will also receive AnimationPlayer
- Sub-path export requires tsup multi-entry config and package.json exports field
- onFrame callback provides canvas element — consumer (efx-motion-editor) captures frame via canvas.toDataURL() or toBlob()

</code_context>

<specifics>
## Specific Ideas

- efx-motion-editor will extend with video export — this project provides the frame-by-frame API, efx-motion-editor handles encoding
- Stop button should show the final fully-rendered painting (not blank canvas) — user sees the result
- Point-by-point stroke drawing across frames creates a cinematic "painting being made" effect
- FPS as a data input (not preset) gives maximum flexibility for both demo and API consumers

</specifics>
