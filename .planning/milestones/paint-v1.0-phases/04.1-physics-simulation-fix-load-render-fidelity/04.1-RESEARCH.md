# Phase 04.1: Physics Simulation Fix & Load-Render Fidelity - Research

**Researched:** 2026-03-31
**Domain:** Watercolor diffusion physics (Canvas 2D, Float32Array typed arrays), FBM displacement, save/load replay fidelity
**Confidence:** HIGH

## Summary

Phase 04.1 replaces the non-functional Rebelle-derived flow physics (`flowStep()`) with a new FBM-displaced darken-flood diffusion algorithm adapted from the working `research/watercolor-canvas.html` prototype. The research prototype demonstrates a convincing watercolor bloom effect using greyscale ping-pong buffers, precomputed FBM displacement maps, `min()` flood (darken-only spreading), `*1.02` self-stabilization, and gentle drying. The key challenge is adapting this greyscale-only algorithm to work with the existing per-channel RGB wet layer arrays (`wetR`, `wetG`, `wetB`, `wetAlpha`, `wetness`).

Performance benchmarks on the target canvas (1000x650 = 650K pixels) show that a single greyscale diffuse pass takes ~19ms at full resolution and ~7ms at half resolution. Per-channel (4 separate diffuse passes for RGBA) at full resolution is ~79ms, which exceeds the 100ms physics tick budget. However, a wetness-gated approach (skipping dry pixels) reduces single-channel cost to ~10ms for 15% wet canvas. The recommended approach is per-channel RGB diffusion operating only on wet pixels, using the shared displacement field computed once per frame.

**Primary recommendation:** Implement per-channel RGB diffusion with wetness gating, using the same FBM + paper heightmap displacement field for all channels. One diffuse pass per 100ms physics tick, with the option to run 2-3 passes for faster visible spreading.

### Pattern 1: Ping-Pong Buffer Diffusion
**What:** Two float buffers swapped each frame; source read, destination written, then swap
**When to use:** Every diffusion pass inside `diffuseStep()`
**Example:**
```javascript
// Source: research/watercolor-canvas.html diffuse() method
// Adapted for per-channel RGB with shared displacement field

function diffuseStep() {
  const w = W, h = H;
  // Ping-pong: read from current wet arrays, write to temp buffers, then swap
  for (let y = 0; y < h; y++) {
    const yw = y * w;
    for (let x = 0; x < w; x++) {
      const idx = yw + x;
      if (wetAlpha[idx] < DIFF_THRESHOLD) continue; // wetness gating

      const dx = dispPxX[idx];
      const dy = dispPxY[idx];

      // Sample center + 4 displaced neighbors for each channel
      const prevR = wetR[idx], prevG = wetG[idx], prevB = wetB[idx];
      const prevA = wetAlpha[idx];

      // Bilinear sample at displaced positions
      const r1 = sampleChannel(wetR, x + dx, y);
      const r2 = sampleChannel(wetR, x - dx, y);
      const r3 = sampleChannel(wetR, x, y + dy);
      const r4 = sampleChannel(wetR, x, y - dy);

      // Flood: darkest of displaced neighbors (min)
      let floodR = Math.min(prevR, r1, r2, r3, r4) * 1.02;
      tmpR[idx] = Math.min(prevR, floodR) * BLEND_RATE + prevR * (1 - BLEND_RATE);
      // ... same for G, B, Alpha channels
    }
  }
  // Swap: copy tmp back to wet arrays
  wetR.set(tmpR);
  wetG.set(tmpG);
  // ...
}
```

### Pattern 2: Precomputed Displacement Map with Paper Heightmap Blend
**What:** FBM noise combined with paper grain heightmap, computed once at init or paper change
**When to use:** At initialization and whenever paper texture changes
**Example:**
```javascript
// Source: adapted from research/watercolor-canvas.html precomputeDisplacement()
function precomputeDisplacement() {
  const aspect = H / W;
  for (let y = 0; y < H; y++) {
    for (let x = 0; x < W; x++) {
      const u = x / W, v = y / H;
      const fbmVal = fbm(u * FBM_SCALE, v * FBM_SCALE, FBM_OCTAVES);

      // Blend with paper heightmap gradient for directional influence
      let paperInfluence = 0;
      if (paperHeight) {
        const idx = y * W + x;
        // Gradient = difference in height from neighbors
        const hL = x > 0 ? paperHeight[idx - 1] : paperHeight[idx];
        const hR = x < W-1 ? paperHeight[idx + 1] : paperHeight[idx];
        const hU = y > 0 ? paperHeight[idx - W] : paperHeight[idx];
        const hD = y < H-1 ? paperHeight[idx + W] : paperHeight[idx];
        // Paper gradient adds directional displacement
        paperInfluence = (hR - hL) * PAPER_INFLUENCE; // x component
      }

      const idx = y * W + x;
      dispPxX[idx] = (fbmVal * DISP_STRENGTH + paperInfluence) * W * 0.005;
      dispPxY[idx] = (fbmVal * DISP_STRENGTH) * H * 0.005 * aspect;
    }
  }
}
```

### Pattern 3: Diffusion Frame Counter for Replay Determinism
**What:** Track how many diffusion passes occur between strokes for deterministic replay
**When to use:** During live painting (increment counter on each physicsStep) and replay (run that many passes)
**Example:**
```javascript
// During live painting:
let diffusionFramesSinceLastStroke = 0;

function physicsStep() {
  diffuseStep();
  dryStep();
  diffusionFramesSinceLastStroke++;
}

// When stroke ends, record counter:
action.diffusionFrames = diffusionFramesSinceLastStroke;
diffusionFramesSinceLastStroke = 0;

// During replay:
for (const stroke of strokes) {
  // Run recorded number of diffusion passes
  for (let t = 0; t < stroke.diffusionFrames; t++) {
    diffuseStep();
    dryStep();
  }
  // Then apply the stroke
  dispatchStroke(stroke);
}
```

### Anti-Patterns to Avoid
- **Running diffusion on ALL pixels every frame:** At 1000x650, per-channel diffusion on all 650K pixels takes ~79ms. Always gate on wetness/wetAlpha threshold.
- **Separate displacement maps per channel:** The displacement field is position-based, not color-based. Compute once, apply to all channels.
- **Using Math.random() in diffusion:** The diffusion must be deterministic for replay. FBM noise from coordinates is deterministic. Never use Math.random() in the diffusion path.
- **Modifying wet arrays in-place during diffusion:** This creates order-dependent artifacts (pixels processed early affect pixels processed late). Use ping-pong buffers or temporary arrays.
- **Calling precomputeDisplacement() every frame:** The displacement map is static for a given paper texture. Compute once at init, recompute only when paper changes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Bilinear interpolation | Nearest-neighbor sampling | `sample()` function from research prototype | Nearest-neighbor creates jagged, pixelated spreading. Bilinear is essential for smooth organic watercolor look |
| FBM noise | Simple Math.random() displacement | Proper FBM with octaves and rotation | FBM creates coherent organic displacement; random noise creates stipple artifacts |
| Ping-pong buffer swap | In-place array modification | Float32Array.set() for buffer swap | In-place modification causes directional bias (processed pixels affect unprocessed neighbors) |

**Key insight:** The beauty of the watercolor effect comes from the specific combination of FBM displacement + darken flood + bilinear sampling. Removing any one of these degrades the result significantly. The `min()` flood creates characteristic edge darkening. The bilinear sampling creates smooth organic edges. The FBM displacement creates natural asymmetric spreading.

## Common Pitfalls

### Pitfall 1: Color Channel Inversion Problem
**What goes wrong:** The research prototype uses greyscale where 1.0 = white (no paint) and 0.0 = black (full paint). The existing codebase uses RGB 0-255 where 0 = no color and 255 = full color. Applying `min()` flood directly would darken toward 0 (no color) instead of spreading paint.
**Why it happens:** Direct algorithm port without considering value domain.
**How to avoid:** For RGB channels, the darken flood must work on *density* (paint amount), not raw color values. The `wetAlpha` array (0-200000) represents density. Diffuse `wetAlpha` using the darken flood, then blend RGB colors weighted by alpha flow. Alternatively, convert RGB to "inverse" (1.0 - normalized) for diffusion and convert back.
**Warning signs:** Paint diffusion produces white halos instead of colored watercolor bloom.

### Pitfall 2: Replay Divergence from Diffusion Count Mismatch
**What goes wrong:** Loaded paintings look different from live-painted because the number of diffusion passes differs.
**Why it happens:** Live painting runs physics at 10fps (setInterval 100ms) while the user paints at variable speed. A 2-second stroke = ~20 diffusion passes. If replay runs 0 passes between strokes, the organic spreading never happens.
**How to avoid:** Record `diffusionFrames` per stroke (count of physicsStep calls between end of previous stroke and end of this stroke). During replay, run exactly that many passes.
**Warning signs:** Loaded paintings have sharp brush stamps without any organic spreading/bloom.

### Pitfall 3: Displacement Map Invalidation on Paper Change
**What goes wrong:** User changes paper texture but displacement map still uses old paper heightmap.
**Why it happens:** `precomputeDisplacement()` is called once at init, but paper grain can change via UI.
**How to avoid:** Call `precomputeDisplacement()` in `setPaperPhysics()` after updating `paperHeight`. This is O(650K) iterations but only happens on UI click, not per-frame.
**Warning signs:** Paint spreading pattern doesn't change when user selects different paper grain.

### Pitfall 4: Self-Stabilization Constant Mismatch
**What goes wrong:** The `*1.02` self-stabilization factor from the research prototype creates infinite spreading or no spreading when applied to the different value ranges in the main codebase.
**Why it happens:** In the research prototype, values are 0.0-1.0 greyscale. In the main codebase, `wetAlpha` ranges 0-200000 and color channels are 0-255. The stabilization constant needs recalibration.
**How to avoid:** Normalize the diffusion domain. Either (a) work in 0.0-1.0 normalized space during diffusion and denormalize after, or (b) adjust the stabilization constant proportionally. Option (a) is cleaner.
**Warning signs:** Paint either spreads to fill entire canvas or doesn't spread at all.

### Pitfall 5: Drying + Diffusion Race Condition
**What goes wrong:** `dryStep()` transfers wet paint to dry canvas while `diffuseStep()` is spreading it. If drying is too aggressive, paint dries before it can diffuse visibly.
**Why it happens:** Both systems run in the same `physicsStep()` call and compete for `wetAlpha` values.
**How to avoid:** Run `diffuseStep()` first, then `dryStep()`. The diffusion needs wet paint to work with. Also, diffusion should respect the drying threshold -- already-dry pixels don't participate in diffusion.
**Warning signs:** Paint appears and instantly dries without any watercolor bloom. Reducing drySpeed slider helps (confirms the race).

### Pitfall 6: Blow Tool Direction Not Affecting Diffusion
**What goes wrong:** Blow tool adds wetness but doesn't push paint directionally.
**Why it happens:** The displacement map is static and doesn't account for user's stroke direction.
**How to avoid:** Blow tool should add a temporary directional bias to the displacement field in the affected area. Store per-pixel "blow velocity" that decays over time and adds to the displacement lookup during diffusion.
**Warning signs:** Blow tool makes paint spread uniformly instead of in the direction the user paints.

## Code Examples

### Bilinear Sample Function (from research prototype)
```javascript
// Source: research/watercolor-canvas.html sample() method
// Adapted for main codebase wet layer array sampling
function sampleChannel(buf, fx, fy) {
  fx = Math.max(0, Math.min(W - 1.001, fx));
  fy = Math.max(0, Math.min(H - 1.001, fy));
  const x0 = Math.floor(fx), y0 = Math.floor(fy);
  const x1 = Math.min(x0 + 1, W - 1);
  const y1 = Math.min(y0 + 1, H - 1);
  const tx = fx - x0, ty = fy - y0;
  const a = buf[y0 * W + x0];
  const b = buf[y0 * W + x1];
  const c = buf[y1 * W + x0];
  const d = buf[y1 * W + x1];
  return (a * (1 - tx) + b * tx) * (1 - ty) +
         (c * (1 - tx) + d * tx) * ty;
}
```

### FBM Noise with Rotation (from research prototype)
```javascript
// Source: research/watercolor-canvas.html fbm() method
// This is the higher-quality FBM used for displacement
function fbmDisplacement(x, y, octaves) {
  let v = 0.0, a = 0.5;
  let shiftX = 100, shiftY = 100;
  const cosR = Math.cos(0.5), sinR = Math.sin(0.5);
  for (let i = 0; i < octaves; i++) {
    v += a * noise(x, y);
    const nx = cosR * x + sinR * y;
    const ny = -sinR * x + cosR * y;
    x = nx * 2.0 + shiftX;
    y = ny * 2.0 + shiftY;
    a *= 0.5;
  }
  return v;
}
```
Note: The existing `fbm()` in v3.html (line 297) uses a simpler implementation without rotation. The research prototype's FBM includes rotation between octaves for more organic patterns. The research version should be used for the displacement map.

### Core Diffusion Algorithm Structure
```javascript
// Source: research/watercolor-canvas.html diffuse() method
// Adapted conceptual structure for per-channel wet layer diffusion

const DIFF_THRESHOLD = 1;     // minimum wetAlpha to participate in diffusion
const BLEND_RATE = 0.3;       // how much flood result blends vs original (0.3 from prototype)
const STABILIZE = 1.02;       // self-stabilization prevents infinite spread
const GENTLE_DRY = 0.5;       // gentle alpha drain per pass (scaled for wetAlpha range)

function diffuseStep() {
  // Phase 1: Diffuse wetAlpha (density) using darken flood
  for (let y = 0; y < H; y++) {
    for (let x = 0; x < W; x++) {
      const idx = y * W + x;
      if (wetAlpha[idx] < DIFF_THRESHOLD) { tmpAlpha[idx] = wetAlpha[idx]; continue; }

      const dx = dispPxX[idx], dy = dispPxY[idx];
      const prev = wetAlpha[idx];
      const s1 = sampleChannel(wetAlpha, x + dx, y);
      const s2 = sampleChannel(wetAlpha, x - dx, y);
      const s3 = sampleChannel(wetAlpha, x, y + dy);
      const s4 = sampleChannel(wetAlpha, x, y - dy);

      // Flood: maximum density of displaced neighbors
      // Note: wetAlpha is density where HIGHER = MORE paint, so use max() not min()
      let flood = Math.max(prev, s1, s2, s3, s4);
      flood = flood * (1 / STABILIZE); // stabilize: slightly reduce flood
      const brightened = Math.max(prev, flood);
      tmpAlpha[idx] = brightened * BLEND_RATE + prev * (1 - BLEND_RATE);
    }
  }

  // Phase 2: Move RGB color to follow alpha flow
  for (let y = 0; y < H; y++) {
    for (let x = 0; x < W; x++) {
      const idx = y * W + x;
      if (wetAlpha[idx] < DIFF_THRESHOLD) {
        tmpR[idx] = wetR[idx]; tmpG[idx] = wetG[idx]; tmpB[idx] = wetB[idx];
        continue;
      }
      const dx = dispPxX[idx], dy = dispPxY[idx];
      // Sample displaced neighbor colors weighted by their alpha
      // This carries color along with the density flow
      const prevR = wetR[idx], prevG = wetG[idx], prevB = wetB[idx];
      const r1 = sampleChannel(wetR, x+dx, y), g1 = sampleChannel(wetG, x+dx, y), b1 = sampleChannel(wetB, x+dx, y);
      // ... sample other 3 directions ...
      // Weighted average by alpha contribution
      tmpR[idx] = prevR * (1 - BLEND_RATE) + avgR * BLEND_RATE;
      // ...
    }
  }

  // Phase 3: Swap buffers
  wetAlpha.set(tmpAlpha);
  wetR.set(tmpR); wetG.set(tmpG); wetB.set(tmpB);
}
```

## Color Diffusion Strategy Decision (D-05 Discretion)

### Recommendation: Per-Channel RGB Diffusion with Alpha-Leading Density

**Rationale:** The research prototype uses greyscale density (1.0 = no paint, 0.0 = full paint). The existing codebase stores color as separate RGB channels with a density channel (`wetAlpha`). The cleanest integration is:

1. **Alpha channel leads:** `wetAlpha` is the paint density. Diffuse it using darken flood (adapted: `max()` since higher wetAlpha = more paint, inverse of the prototype's greyscale).
2. **RGB follows alpha:** Where alpha flows to a neighbor, carry the color with it. Use alpha-weighted blending at destination pixels.
3. **No greyscale conversion needed:** This avoids the complexity of converting RGB to density and back.

**Why not greyscale density + color map:** The existing brush deposit code writes directly to `wetR`, `wetG`, `wetB`. Converting to a greyscale density model would require rewriting all brush functions (D-04 explicitly says "minimal surgery"). Per-channel diffusion works with the existing arrays unchanged.

**Performance profile (benchmarked on 1000x650 canvas, Node.js):**

| Approach | Time per pass | Passes per 100ms tick | Notes |
|----------|--------------|----------------------|-------|
| Greyscale only (all pixels) | ~19ms | 5 | Research prototype approach |
| Greyscale only (half-res) | ~7ms | 14 | With upscale step |
| 4-channel all pixels | ~79ms | 1 | Too expensive |
| Wetness-gated single channel | ~10ms | 10 | 15% wet canvas |
| Wetness-gated 3-channel | ~8ms | 12 | Shared displacement lookup |

**Recommended approach: wetness-gated per-channel diffusion, 1-3 passes per physics tick.** At typical canvas coverage (5-25% wet), this takes 5-15ms leaving ample budget for `dryStep()`.

## Tool Adaptation Strategy (D-08, D-09)

### Water Tool
**Current:** Increases `wetness[i]` directly.
**New behavior:** Same -- increasing wetness makes pixels eligible for diffusion (diffusion gates on `wetAlpha > threshold`). Water tool should also slightly increase `wetAlpha` for pixels that have dried to zero, effectively "rehydrating" them into the diffusion system.
**Change level:** Minimal. Current implementation already works with the new system.

### Blow Tool
**Current:** Increases `wetness[i]` with directional movement.
**New behavior:** In addition to increasing wetness, add a temporary per-pixel directional displacement bias. Store as `blowDX[idx]`, `blowDY[idx]` arrays that decay each physics tick. During diffusion, add blow displacement to FBM displacement.
**Change level:** Medium. Needs new blow velocity arrays and diffusion integration.

### Smear Tool
**Current:** Captures paint at previous position, deposits at current position.
**New behavior:** Same capture/deposit mechanics. No change needed -- smear operates on wet arrays directly, and diffusion will spread the deposited paint naturally.
**Change level:** Minimal. Current implementation works as-is.

### Blend Tool
**Current:** Averages wet paint colors within brush radius.
**New behavior:** Same -- averaging colors within radius. Diffusion will then spread the blended result.
**Change level:** Minimal. Current implementation works as-is.

### Wet Tool
**Current:** Increases wetness + reverses dryPos for rehydration.
**New behavior:** Same. Increasing wetness + reversing dryPos makes pixels wet enough for diffusion to act on them.
**Change level:** Minimal. Current implementation works as-is.

### Dry Tool
**Current:** Advances dryPos for accelerated drying.
**New behavior:** Same. Dried pixels have wetAlpha = 0 and are skipped by diffusion.
**Change level:** None.

### Summary
Most tools need zero or minimal changes. The blow tool is the only one requiring significant new work (directional displacement bias). The existing tools already operate on `wetness` and `wetAlpha`, which are exactly the arrays the diffusion system reads.

## FBM Parameter Recommendations (Discretion Areas)

### FBM Octaves and Scale
**Recommendation:** 4 octaves, scale 22.0 (same as research prototype)
**Rationale:** The research prototype uses `fbm(u * 22.0, v * 22.0, 4)` and produces beautiful organic displacement. The scale of 22.0 at normalized coordinates (0-1) creates ~22 displacement features across the canvas, which at 1000px width means features ~45px apart -- appropriate for watercolor spread patterns.

### Paper Heightmap Blend
**Recommendation:** Additive blend with paper gradient, weighted 0.3 paper + 0.7 FBM
**Rationale:** Paper should modulate spreading direction (valleys catch paint) without dominating. A 30/70 split preserves the organic FBM character while making paper texture visible in the spreading pattern. The paper component uses the heightmap gradient (difference between neighbor heights) rather than raw height, since gradient indicates flow direction.

### Displacement Magnitude
**Recommendation:** Start with `0.005` in normalized UV space (same as research prototype), tune by visual test
**Rationale:** At 1000px width, `0.005 * 1000 = 5px` displacement per pass. This creates gentle spreading that builds up over many frames -- exactly the watercolor bloom effect.

### Diffusion Passes Per Tick
**Recommendation:** 1 pass per physics tick (100ms), with option for 2-3 during heavy painting
**Rationale:** 1 pass at ~10ms leaves 90ms for dryStep and other physics. The gentle spreading accumulates over seconds, matching real watercolor timing. More passes per tick would speed up spreading but risk making it look like instant flood fill rather than organic watercolor.

## Open Questions

1. **Blow tool displacement persistence**
   - What we know: Blow tool needs directional displacement bias that decays over time
   - What's unclear: Optimal decay rate and maximum displacement magnitude. Should blow displacement be additive to FBM map, or should it temporarily replace the FBM map in the affected area?
   - Recommendation: Start with additive displacement decaying at 5% per physics tick. Tune visually.

2. **Half-resolution diffusion**
   - What we know: Research prototype runs at half resolution for performance (0.5x scale). Our full-res wetness-gated approach is already fast enough (~10ms per pass).
   - What's unclear: Whether half-res would look better (smoother bilinear at coarser grid) or worse (lose detail of paper texture influence).
   - Recommendation: Start at full resolution since performance is acceptable. Keep half-res as a fallback if future canvas sizes grow.

3. **Diffusion interaction with undo**
   - What we know: Undo currently uses ImageData snapshots + wet array snapshots. Diffusion modifies wet arrays between strokes.
   - What's unclear: Should undo restore the diffusion state (wet arrays at stroke start) or just replay from scratch?
   - Recommendation: Undo restores wet array snapshots as it does now. The snapshot captures post-diffusion state naturally.
