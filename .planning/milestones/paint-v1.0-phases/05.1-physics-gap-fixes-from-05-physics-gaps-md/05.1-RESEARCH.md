**Researched:** 2026-04-01

## Summary

This phase replaces the current FBM-displaced height-gradient diffusion engine with Stam's stable fluids solver, fixes 6 physics gaps identified during Phase 05, and adds paper-height-influenced paint deposit, water-height equalization flow, edge darkening, and subtractive RYB color mixing.

The core technical challenge is implementing a 2D Navier-Stokes solver (Stam 2003) that operates on Float32Array grids at canvas resolution within a 16ms physics tick budget. The algorithm has three stages: advect (semi-Lagrangian transport), diffuse (implicit Gauss-Seidel relaxation), and project (pressure correction for incompressibility). All are per-pixel grid operations that map directly to the existing typed array architecture.

**Primary recommendation:** Implement Stam's solver as a new `src/core/fluids.ts` module that replaces `diffuseStep()` in the physics pipeline, keeping the existing `physicsStep()` orchestration pattern. Add water-height equalization as a velocity source term (Van Laerhoven omega_h=0.06), edge darkening as boundary pressure reduction (Curtis FlowOutward), and paper-height deposit modulation in `transferToWetLayerClipped()`. Split DENSITY_K into display/physics variants. Wire `mixSubtractive()` into the wet layer deposit path.

### Pattern 1: Stam's Stable Fluids Solver (CPU, 2D, Float32Array)
**What:** A complete 2D incompressible fluid solver operating on typed arrays at canvas resolution.
**When to use:** Every physics tick (16ms interval) when physics is running.

The solver has 5 core operations, all operating on `(N+2)*(N+2)` grids with 1-cell boundary padding:

```typescript
// Source: Stam 2003 "Real-Time Fluid Dynamics for Games"
// Port from: https://github.com/nornagon/stam-stable-fluids

// Grid indexing: 1-based interior cells, 0 and N+1 are boundary
function IX(N: number, i: number, j: number): number {
  return i + (N + 2) * j
}

// 1. Boundary conditions (Dirichlet for velocity, Neumann for scalars)
function setBnd(N: number, b: number, x: Float32Array): void {
  for (let i = 1; i <= N; i++) {
    x[IX(N, 0,   i)] = b === 1 ? -x[IX(N, 1, i)] : x[IX(N, 1, i)]
    x[IX(N, N+1, i)] = b === 1 ? -x[IX(N, N, i)] : x[IX(N, N, i)]
    x[IX(N, i,   0)] = b === 2 ? -x[IX(N, i, 1)] : x[IX(N, i, 1)]
    x[IX(N, i, N+1)] = b === 2 ? -x[IX(N, i, N)] : x[IX(N, i, N)]
  }
  x[IX(N, 0,   0  )] = 0.5 * (x[IX(N, 1, 0  )] + x[IX(N, 0,   1)])
  x[IX(N, 0,   N+1)] = 0.5 * (x[IX(N, 1, N+1)] + x[IX(N, 0,   N)])
  x[IX(N, N+1, 0  )] = 0.5 * (x[IX(N, N, 0  )] + x[IX(N, N+1, 1)])
  x[IX(N, N+1, N+1)] = 0.5 * (x[IX(N, N, N+1)] + x[IX(N, N+1, N)])
}

// 2. Gauss-Seidel linear solver (implicit diffusion)
function linSolve(N: number, b: number, x: Float32Array, x0: Float32Array,
                  a: number, c: number): void {
  for (let k = 0; k < 20; k++) {
    for (let i = 1; i <= N; i++) {
      for (let j = 1; j <= N; j++) {
        x[IX(N,i,j)] = (x0[IX(N,i,j)] +
          a * (x[IX(N,i-1,j)] + x[IX(N,i+1,j)] +
               x[IX(N,i,j-1)] + x[IX(N,i,j+1)])) / c
      }
    }
    setBnd(N, b, x)
  }
}

// 3. Diffusion (implicit, unconditionally stable)
function diffuse(N: number, b: number, x: Float32Array, x0: Float32Array,
                 diff: number, dt: number): void {
  const a = dt * diff * N * N
  linSolve(N, b, x, x0, a, 1 + 4 * a)
}

// 4. Semi-Lagrangian advection (backtrace + bilinear interpolation)
function advect(N: number, b: number, d: Float32Array, d0: Float32Array,
                u: Float32Array, v: Float32Array, dt: number): void {
  const dt0 = dt * N
  for (let i = 1; i <= N; i++) {
    for (let j = 1; j <= N; j++) {
      let x = i - dt0 * u[IX(N, i, j)]
      let y = j - dt0 * v[IX(N, i, j)]
      if (x < 0.5) x = 0.5; if (x > N + 0.5) x = N + 0.5
      const i0 = x | 0, i1 = i0 + 1
      if (y < 0.5) y = 0.5; if (y > N + 0.5) y = N + 0.5
      const j0 = y | 0, j1 = j0 + 1
      const s1 = x - i0, s0 = 1 - s1
      const t1 = y - j0, t0 = 1 - t1
      d[IX(N,i,j)] = s0 * (t0 * d0[IX(N,i0,j0)] + t1 * d0[IX(N,i0,j1)]) +
                      s1 * (t0 * d0[IX(N,i1,j0)] + t1 * d0[IX(N,i1,j1)])
    }
  }
  setBnd(N, b, d)
}

// 5. Pressure projection (enforce divergence-free / incompressible)
function project(N: number, u: Float32Array, v: Float32Array,
                 p: Float32Array, div: Float32Array): void {
  for (let i = 1; i <= N; i++) {
    for (let j = 1; j <= N; j++) {
      div[IX(N,i,j)] = -0.5 * (u[IX(N,i+1,j)] - u[IX(N,i-1,j)] +
                                 v[IX(N,i,j+1)] - v[IX(N,i,j-1)]) / N
      p[IX(N,i,j)] = 0
    }
  }
  setBnd(N, 0, div); setBnd(N, 0, p)
  linSolve(N, 0, p, div, 1, 4)
  for (let i = 1; i <= N; i++) {
    for (let j = 1; j <= N; j++) {
      u[IX(N,i,j)] -= 0.5 * N * (p[IX(N,i+1,j)] - p[IX(N,i-1,j)])
      v[IX(N,i,j)] -= 0.5 * N * (p[IX(N,i,j+1)] - p[IX(N,i,j-1)])
    }
  }
  setBnd(N, 1, u); setBnd(N, 2, v)
}
```

### Pattern 2: Velocity Step + Density Step (Full Pipeline)
**What:** The two high-level steps that compose the complete solver per tick.
**When to use:** Called from physicsStep every 16ms.

```typescript
// Source: Stam 2003

// Velocity update: add forces -> diffuse -> project -> advect -> project
function velStep(N: number, u: Float32Array, v: Float32Array,
                 u0: Float32Array, v0: Float32Array,
                 visc: number, dt: number): void {
  addSource(N, u, u0, dt)  // External forces (height gradient, blow tool)
  addSource(N, v, v0, dt)
  // Swap u<->u0, v<->v0 (or use ping-pong buffers)
  diffuse(N, 1, u0, u, visc, dt)
  diffuse(N, 2, v0, v, visc, dt)
  project(N, u0, v0, u, v)
  advect(N, 1, u, u0, u0, v0, dt)
  advect(N, 2, v, v0, u0, v0, dt)
  project(N, u, v, u0, v0)
}

// Density update: add source -> diffuse -> advect (per color channel + alpha)
function densStep(N: number, x: Float32Array, x0: Float32Array,
                  u: Float32Array, v: Float32Array,
                  diff: number, dt: number): void {
  addSource(N, x, x0, dt)
  diffuse(N, 0, x0, x, diff, dt)
  advect(N, 0, x, x0, u, v, dt)
}
```

### Pattern 3: Water-Height Equalization (Van Laerhoven)
**What:** Adds velocity contributions based on water height differences between neighboring cells.
**When to use:** Before `velStep`, as force sources in u0/v0.

```typescript
// Source: Van Laerhoven & Van Reeth 2005, omega_h = 0.06
// v_new = omega_i * v_old + omega_h * v_height
// where v_height = height_difference to neighbor

function addHeightEqualization(
  N: number,
  u0: Float32Array, v0: Float32Array,
  waterHeight: Float32Array,  // current wet alpha as proxy for water height
  omega_h: number,            // 0.06 per Van Laerhoven
): void {
  for (let i = 1; i <= N; i++) {
    for (let j = 1; j <= N; j++) {
      const idx = IX(N, i, j)
      const h = waterHeight[idx]
      if (h < 1) continue
      // Height gradient drives flow: water moves from high to low
      u0[idx] += omega_h * (waterHeight[IX(N, i-1, j)] - waterHeight[IX(N, i+1, j)])
      v0[idx] += omega_h * (waterHeight[IX(N, i, j-1)] - waterHeight[IX(N, i, j+1)])
    }
  }
}
```

### Pattern 4: Edge Darkening (Curtis FlowOutward)
**What:** Reduces pressure at wet-area boundaries, causing pigment to flow outward and concentrate at edges.
**When to use:** After velocity step, modulate pressure near boundaries.

```typescript
// Source: Curtis et al. SIGGRAPH 1997 / US Patent 6198489B1
// DarkenEdges procedure:
// 1. Create wet-area mask (binary: alpha > threshold)
// 2. Gaussian blur the mask
// 3. Reduce pressure where blur < mask (boundary cells)
// This pushes fluid outward, concentrating pigment at edges

function darkenEdges(
  N: number,
  wetMask: Float32Array,      // binary wet area
  blurredMask: Float32Array,  // gaussian-blurred wet area
  u: Float32Array, v: Float32Array,
  darkening: number,          // strength parameter ~0.1
): void {
  for (let i = 1; i <= N; i++) {
    for (let j = 1; j <= N; j++) {
      const idx = IX(N, i, j)
      if (wetMask[idx] < 0.5) continue
      // Interior cells have blur ~= mask; boundary cells have blur < mask
      const edgeFactor = Math.max(0, wetMask[idx] - blurredMask[idx])
      // Create outward velocity at edges
      const gradX = blurredMask[IX(N,i+1,j)] - blurredMask[IX(N,i-1,j)]
      const gradY = blurredMask[IX(N,i,j+1)] - blurredMask[IX(N,i,j-1)]
      u[idx] -= darkening * gradX * edgeFactor
      v[idx] -= darkening * gradY * edgeFactor
    }
  }
}
```

### Pattern 5: Paper-Height Deposit Modulation (Curtis/Van Laerhoven Adsorption)
**What:** Modulates paint deposit based on paper surface height, creating watercolor stipple texture.
**When to use:** In `transferToWetLayerClipped()` when deposit alpha is calculated.

```typescript
// Source: Curtis Eq. 7-8, adapted per D-09
// Delta_p_ad = dt * (p_water * (1 - w_frac) * (1 - h * gamma) * delta)
// gamma = granulation (how much paper height affects deposit, ~0.8)
// delta = density (deposit rate scaling, ~1.2)
// h = paper height at pixel (0-1, 0=valley, 1=peak)
// w_frac = wetness fraction (existing wetness / max)
// p_water = deposit alpha from rendered stroke

function modulateDeposit(
  depositAlpha: number,
  paperH: number,           // 0-1 paper height at this pixel
  existingWetness: number,  // current wetness at pixel
  gamma: number,            // granulation ~0.8
  delta: number,            // density ~1.2
  maxWetness: number,       // 1000 (current max)
): number {
  const wFrac = Math.min(1, existingWetness / maxWetness)
  // Valleys (h~0) get full deposit; peaks (h~1) get reduced deposit
  const adsorption = (1 - wFrac) * (1 - paperH * gamma) * delta
  return depositAlpha * Math.max(0.05, adsorption)  // 5% floor to avoid invisible strokes
}
```

### Anti-Patterns to Avoid
- **Running stable fluids at full canvas resolution with no bounds check:** The solver iterates over (N+2)^2 cells per operation. For 1000x650, this is 652K cells x 20 Gauss-Seidel iterations = 13M operations per `linSolve`. Use the existing `lastStrokeBounds` optimization to limit computation area when in 'last' mode.
- **Storing post-opacity values in savedWet:** D-12 is explicit -- savedWet must store full-intensity values. Opacity is applied only in the compositor.
- **Using separate grids for velocity vs. wet layer:** The stable fluids velocity field (u, v) must be spatially aligned with the wet layer buffers. Use the same width/height dimensions, with +2 boundary padding for Stam's solver.
- **Modifying the wet layer color arrays during advection:** Color channels (wetR, wetG, wetB) must be advected as separate density fields, each through `densStep`. Do not blend colors during transport -- only during deposit (where subtractive mixing applies).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| 2D fluid diffusion | Custom blur/averaging | Stam's implicit Gauss-Seidel solver | Unconditionally stable, never blows up, physically correct |
| Semi-Lagrangian advection | Custom displacement map sampling | Stam's `advect()` with bilinear interpolation | Handles CFL condition automatically, stable at any timestep |
| Pressure projection | Nothing (skip it) | Stam's `project()` | Without projection, velocity field diverges and paint accumulates/disappears non-physically |
| Edge darkening | Alpha fadeout at boundaries | Curtis FlowOutward via blurred wet mask | Produces realistic pigment concentration at edges, not just fade |
| Color mixing at overlaps | Linear RGB blending | Existing `mixSubtractive()` in util/color.ts | RYB subtractive mixing produces correct paint-like results (yellow + blue = green) |
| Paint deposit texture | Procedural noise patterns | Paper heightmap-based adsorption (D-08/D-09) | Uses actual paper surface to create watercolor stipple -- physically motivated, not arbitrary |

**Key insight:** The FBM-displaced height-gradient diffusion was an ad-hoc approximation. Stam's solver is the standard approach used by all serious paint simulations since 1999, and its implicit formulation guarantees stability regardless of viscosity or timestep. The current diffusion produces blocky spreading because it's a simple 4-neighbor conservative transport -- not a real fluid solver.

## Common Pitfalls

### Pitfall 1: Grid Indexing Mismatch Between Stam Solver and Wet Layer
**What goes wrong:** Stam's solver uses (N+2)*(N+2) grids with 1-based indexing (cells 0 and N+1 are boundary). The existing wet layer uses width*height with 0-based indexing (no boundary padding).
**Why it happens:** Two incompatible coordinate systems operating on the same data.
**How to avoid:** Either: (a) allocate separate Stam-sized buffers for velocity/pressure and copy wet layer data into/out of them per physics tick, or (b) adapt the Stam solver to work with 0-based indexing and explicit boundary clamping (saves memory but more error-prone). Recommendation: Option (a) -- use separate Stam grids for velocity/pressure (6 Float32Arrays of (W+2)*(H+2)), and advect the wet layer channels directly using the velocity field with 0-based adaptation.
**Warning signs:** Paint appearing/disappearing at canvas edges, velocity artifacts in first/last row/column.

### Pitfall 2: Performance Budget -- 16ms Tick with Gauss-Seidel
**What goes wrong:** 20 Gauss-Seidel iterations over 1000x650 grid = ~260M array accesses per velocity step. This may exceed 16ms on slower machines.
**Why it happens:** `linSolve` is the most expensive operation, called 4 times per physics tick (2 for velocity diffusion, 1 for pressure solve, and optionally 1 for density diffusion).
**How to avoid:** (1) Reduce Gauss-Seidel iterations from 20 to 4-10 (visual quality vs. speed tradeoff -- 4 iterations is often sufficient for paint). (2) Use `lastStrokeBounds` to limit solver area in 'last' mode. (3) Skip density diffusion for paint (pigment diffusion is very slow in real watercolor -- advection dominates). (4) Profile early and establish the iteration count that fits the budget.
**Warning signs:** Physics interval callbacks stacking up, jerky animation, browser dropping frames.

### Pitfall 3: forceDryAll /800 with Uniform Deposits
**What goes wrong:** The /800 divisor was calibrated for v3's sparse brush grain deposits (93% zeros). With uniform deposits, /800 makes any opacity >27% fully opaque.
**Why it happens:** Without paper-height modulation (D-08), deposits are uniform and /800 clips too aggressively.
**How to avoid:** Implement D-08 (paper-height deposit modulation) BEFORE restoring /800 in D-06. The paper-height modulation creates sparse deposits (valleys get paint, peaks don't) that match the sparsity v3 got from brush_texture.png. This is the correct ordering: D-08 first, then D-06.
**Warning signs:** All strokes appearing fully opaque after drying, no gradient visible in dried paint.

### Pitfall 4: Advecting Color Channels Independently Produces Ghosting
**What goes wrong:** Advecting wetR, wetG, wetB, wetAlpha as 4 independent density fields can cause color to separate from alpha, producing faint ghosting.
**Why it happens:** Each bilinear interpolation operates independently -- alpha at a cell may be nonzero while color is zero (or vice versa) after advection through different velocity-warped positions.
**How to avoid:** Use premultiplied alpha for advection: advect (R*A), (G*A), (B*A), A as four fields, then recover R = R_premul/A, etc. The existing `sampleColorPremul()` in diffusion.ts already demonstrates this pattern.
**Warning signs:** Faint colored halos around moving paint, white or black edges on flowing paint.

### Pitfall 5: Opacity Post-Multiply vs. Physics Interaction
**What goes wrong:** Currently, savedWet stores post-multiplied alpha. When physics restores savedWet to wet and runs diffusion, the wet alpha is already reduced by opacity, producing weaker effects.
**Why it happens:** The post-multiply happens in `renderPaintStrokeSingleColor()` at line 506-509, and the result is transferred to wet layer and then to savedWet.
**How to avoid:** Per D-12, change the rendering pipeline so the offscreen canvas used for wet layer transfer renders at full intensity (remove the post-multiply block for non-directRender path). Store full-intensity savedWet. Apply opacity multiplication only in the compositor (compositeWetLayer).
**Warning signs:** Physics spreading looks much weaker than the stroke itself, light/washed-out paint during diffusion.

### Pitfall 6: Subtractive Mixing Double-Application
**What goes wrong:** If mixSubtractive is applied both during deposit AND during diffusion/advection, colors darken excessively with each physics tick.
**Why it happens:** Subtractive mixing inherently darkens (absorbs light). Applying it during transport reapplies the darkening effect repeatedly.
**How to avoid:** Apply mixSubtractive ONLY during the initial deposit (in transferToWetLayerClipped when blending new paint with existing wet layer color). During advection, use linear RGB blending (the existing premultiplied alpha interpolation in advect).
**Warning signs:** Paint turning black after a few seconds of physics simulation.

## Code Examples

### Example 1: Modified physicsStep Pipeline

```typescript
// In diffusion.ts (or new fluids.ts), the physics step becomes:
export function physicsStep(
  wet: WetBuffers,
  drying: DryingLUT,
  ctx: CanvasRenderingContext2D,
  // ... existing params ...
  // NEW params:
  velU: Float32Array, velV: Float32Array,     // velocity field
  velU0: Float32Array, velV0: Float32Array,   // velocity scratch
  pressure: Float32Array, div: Float32Array,  // pressure solver scratch
  viscosity: number,                          // parameterizable per D-13
  paperHeight: Float32Array | null,
): void {
  const dt = 0.016  // 16ms tick

  // 1. Build velocity sources from water-height equalization (D-02)
  velU0.fill(0); velV0.fill(0)
  addHeightEqualization(width, velU0, velV0, wet.alpha, 0.06)

  // 2. Add edge darkening forces (D-03)
  // (compute wet mask and blurred mask, add outward velocity)

  // 3. Velocity step: diffuse + project + advect + project
  velStep(width, velU, velV, velU0, velV0, viscosity, dt)

  // 4. Advect wet layer channels using velocity field
  // (wetR, wetG, wetB, wetAlpha each through advect)

  // 5. Drying step (unchanged)
  dryStep(wet, drying, ctx, width, height, drySpeed, paperHeight)
}
```

### Example 2: Modified transferToWetLayerClipped with Paper-Height + Subtractive Mixing

```typescript
// In wet-layer.ts, deposit path with D-08, D-09, D-10
export function transferToWetLayerClipped(
  offCtx: CanvasRenderingContext2D,
  wetBuffers: WetBuffers,
  waterAmount: number,
  bounds: { x: number; y: number; w: number; h: number },
  width: number, height: number,
  paperHeight: Float32Array | null,  // NEW: paper heightmap
  gamma: number,                      // NEW: granulation ~0.8
  delta: number,                      // NEW: density ~1.2
): void {
  const offData = offCtx.getImageData(0, 0, bounds.w, bounds.h).data

  for (let ly = 0; ly < bounds.h; ly++) {
    const gy = bounds.y + ly
    if (gy < 0 || gy >= height) continue
    for (let lx = 0; lx < bounds.w; lx++) {
      const gx = bounds.x + lx
      if (gx < 0 || gx >= width) continue
      const pi = (ly * bounds.w + lx) * 4
      const a = offData[pi + 3]
      if (a < 2) continue

      const i = gy * width + gx
      let depositAlpha = (a / 255) * 3000

      // D-08/D-09: Paper-height modulation
      if (paperHeight) {
        const h = paperHeight[i]
        const wFrac = Math.min(1, wetBuffers.wetness[i] / 1000)
        const adsorption = (1 - wFrac) * (1 - h * gamma) * delta
        depositAlpha *= Math.max(0.05, adsorption)
      }

      const existingA = wetBuffers.alpha[i]
      const newA = Math.min(200000, existingA + depositAlpha)

      // D-10: Subtractive RYB mixing when blending with existing paint
      if (existingA > 1) {
        const blend = depositAlpha / (existingA + depositAlpha)
        const existing: [number, number, number] = [
          wetBuffers.r[i], wetBuffers.g[i], wetBuffers.b[i]
        ]
        const incoming: [number, number, number] = [
          offData[pi], offData[pi + 1], offData[pi + 2]
        ]
        const mixed = mixSubtractive(existing, incoming, blend)
        wetBuffers.r[i] = mixed[0]
        wetBuffers.g[i] = mixed[1]
        wetBuffers.b[i] = mixed[2]
      } else {
        wetBuffers.r[i] = offData[pi]
        wetBuffers.g[i] = offData[pi + 1]
        wetBuffers.b[i] = offData[pi + 2]
      }

      wetBuffers.alpha[i] = newA
      wetBuffers.wetness[i] = Math.min(1000,
        wetBuffers.wetness[i] + waterAmount * 800 * (a / 255))
    }
  }
}
```

## Open Questions

1. **Grid resolution vs. canvas resolution**
   - What we know: Canvas is 1000x650. Stam solver with 20 Gauss-Seidel iterations at this resolution requires ~260M array accesses per linSolve call.
   - What's unclear: Whether this fits in 16ms on target hardware. Quick profiling suggests ~4-8ms per linSolve on modern M-series Macs (Float32Array sequential access is fast), but this needs verification.
   - Recommendation: Start at canvas resolution with 4 Gauss-Seidel iterations. Profile. If over budget, try 10 iterations. If still over, consider half-resolution grid with upsampled velocity for advection.

2. **FBM as secondary perturbation**
   - What we know: The existing FBM displacement adds organic irregularity to paint flow. Stable fluids alone produces very smooth flow patterns.
   - What's unclear: Whether paper-height equalization (D-02) plus edge darkening (D-03) provide enough visual variation without FBM, or if the results look too "clean."
   - Recommendation: Keep the existing `precomputeDisplacement` FBM data and add it as a small perturbation to the velocity source (u0/v0) before velStep. Scale by ~0.02 for subtle organic variation. Can be tuned or removed after visual evaluation.

3. **Velocity field serialization**
   - What we know: Save/load currently serializes strokes + settings. Physics state (velocity, pressure) is ephemeral.
   - What's unclear: Whether velocity field needs to be saved/loaded for deterministic replay of physics runs.
   - Recommendation: Do NOT serialize velocity fields. Physics is triggered on-demand (hold button) and results are baked to dry canvas via forceDryAll. Replay re-runs physics steps, which is deterministic given the same paint state.

4. **Boundary conditions at canvas edges**
   - What we know: Stam's solver uses reflecting boundaries (velocity negated at edges, scalars copied).
   - What's unclear: Whether reflecting boundaries cause visible artifacts at canvas edges (paint "bouncing" off walls).
   - Recommendation: Use Stam's standard boundary conditions. If edge artifacts appear, switch to zero-velocity (no-slip) boundaries: set velocity to 0 at boundaries instead of negating.
