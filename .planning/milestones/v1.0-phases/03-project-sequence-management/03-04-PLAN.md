<task type="auto">
  <name>Task 1: Fix dragDrop.ts to only trigger overlay for external file drags</name>
  <files>Application/src/lib/dragDrop.ts</files>
  <action>
Fix the onDragDropEvent handler in dragDrop.ts so that isDraggingOver is only set to true when the drag event carries actual file paths from the OS (external file drag), not for SortableJS internal element drags.

The root cause: the current code sets `isDraggingOver.value = true` for ALL `enter` and `over` events without checking whether file paths exist. SortableJS drags generate native drag events that Tauri intercepts, but these events have empty paths arrays since no files are being dragged from the OS.

Changes to the handler logic:

1. For `type === 'enter'`: The DragDropEvent type guarantees `paths: string[]` exists. Only set `isDraggingOver.value = true` if `event.payload.paths.length > 0`. This distinguishes real file drags (OS sends file paths) from SortableJS internal drags (no file paths).

2. For `type === 'over'`: The DragDropEvent type does NOT include `paths` on `over` events. Do NOT set `isDraggingOver.value = true` on `over` events at all. The `enter` event already handles activation. Remove `over` from the `enter || over` condition entirely.

3. Leave the `drop` and `leave` handlers unchanged -- they correctly reset `isDraggingOver.value = false`.

The updated handler should look like:
```typescript
if (type === 'enter') {
  // Only show drop overlay for external file drags (paths > 0),
  // not for SortableJS internal element reorder drags
  if (event.payload.paths.length > 0) {
    isDraggingOver.value = true;
  }
} else if (type === 'drop') {
  // ... existing drop handling unchanged
} else {
  // 'leave' or 'over'
  if (type === 'leave') {
    isDraggingOver.value = false;
  }
}
```

This single fix resolves BOTH the key photo drag reorder issue (UAT test 4) and the sequence drag reorder issue (UAT test 5), since both SortableJS instances use internal HTML5 drag events that Tauri sees as path-less drag events.
