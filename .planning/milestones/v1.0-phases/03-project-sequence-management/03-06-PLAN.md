<task type="auto">
  <name>Task 1: Fix SequenceList SortableJS — add deps and DOM revert</name>
  <files>Application/src/components/sequence/SequenceList.tsx</files>
  <action>
Fix the SortableJS useEffect in SequenceList (lines 18-31) with two changes:

1. **Add `sequences.length` to useEffect dependency array** so the SortableJS instance is destroyed and recreated when sequences are added/removed. This prevents stale DOM references that block context menu actions (delete, duplicate) on the 3rd+ sequence.

2. **Revert SortableJS DOM mutation before updating store** in the onEnd callback. SortableJS physically moves DOM nodes during drag. If we update the Preact signal without reverting, Preact compares against the already-mutated DOM and sees no diff, so the reorder silently fails. Fix:

```typescript
onEnd(evt) {
  const { oldIndex, newIndex, item, from } = evt;
  if (oldIndex != null && newIndex != null && oldIndex !== newIndex) {
    // Revert SortableJS DOM mutation so Preact can re-render correctly
    from.removeChild(item);
    from.insertBefore(item, from.children[oldIndex] ?? null);
    sequenceStore.reorderSequences(oldIndex, newIndex);
  }
},
```

The key insight: `from.removeChild(item)` removes the item from where SortableJS placed it, then `from.insertBefore(item, from.children[oldIndex])` puts it back at its original position. Now when Preact re-renders from the updated signal, it sees the original DOM and correctly applies the new order.

Update the dependency array:
```typescript
}, [sequences.length]);
```

Note: Use `sequences.length` (not the full `sequences` array) to avoid unnecessary SortableJS recreation on every rename/hold-frame edit. We only need recreation when items are added or removed.
