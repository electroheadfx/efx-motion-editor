# Phase 12.5: Vertical Scroll Issues - Research

**Researched:** 2026-03-18
**Domain:** Timeline vertical scrollbar (DOM element) + auto vertical scroll during playback
**Confidence:** HIGH

## Summary

This phase adds a custom vertical scrollbar to the timeline panel and auto-scroll during playback. The codebase already has all the infrastructure needed: `timelineStore.scrollY` signal exists and works, the wheel handler in `TimelineInteraction` already computes `maxScrollY` and clamps scroll, and `SidebarScrollArea` provides a reference pattern for custom 4px scrollbar thumb math. The `ensureFrameVisible` / `ensureFrameVisiblePaged` methods in `timelineStore` provide the exact pattern to replicate for vertical auto-scroll.

The scrollbar will be a standalone Preact component (`TimelineScrollbar.tsx`) rendered as a DOM sibling to the canvas (not drawn on canvas, not overlaid). It reads `timelineStore.scrollY` for thumb position and writes to `timelineStore.setScrollY()` on drag/click. Auto vertical scroll hooks into the playback tick loop in `playbackEngine.ts`, calling a new `ensureTrackVisible()` method on `timelineStore` when the active sequence changes during playback.

**Primary recommendation:** Build a simple Preact component for the scrollbar, add `ensureTrackVisible(sequenceId)` to `timelineStore`, and call it from `playbackEngine.syncActiveSequence()`.

### Pattern 1: Signal-Driven Scrollbar
**What:** The scrollbar component reads from `timelineStore.scrollY` signal for thumb position and writes to `timelineStore.setScrollY()` on user interaction. No local state for scroll position -- the store is the single source of truth.
**When to use:** Always -- this matches how all timeline state is managed in this codebase.
**Example:**
```typescript
// TimelineScrollbar.tsx
import { useComputed } from '@preact/signals';
import { timelineStore } from '../../stores/timelineStore';

// Thumb position derived from signal
const thumbTop = useComputed(() => {
  const scrollY = timelineStore.scrollY.value;
  const maxScrollY = timelineStore.maxScrollY.value;
  const scrollRatio = maxScrollY > 0 ? scrollY / maxScrollY : 0;
  const thumbRange = containerHeight - thumbHeight;
  return scrollRatio * thumbRange;
});
```

### Pattern 2: ensureTrackVisible (Vertical Auto-Scroll)
**What:** Mirrors `ensureFrameVisible` (horizontal) but for the vertical axis. Given a sequence ID, computes the track's Y position and snaps `scrollY` if the track is outside the visible area.
**When to use:** During playback when the active sequence changes, and on explicit sequence selection.
**Example:**
```typescript
// In timelineStore.ts
ensureTrackVisible(sequenceId: string) {
  const tracks = trackLayoutsSignal.peek();
  const fxCount = fxTrackLayoutsSignal.peek().length;
  const trackIndex = tracks.findIndex(t => t.sequenceId === sequenceId);
  if (trackIndex < 0) return;

  const trackTop = fxCount * FX_TRACK_HEIGHT + trackIndex * TRACK_HEIGHT;
  const trackBottom = trackTop + TRACK_HEIGHT;
  const canvasH = /* from viewport signal or param */;
  const visibleTop = scrollY.value;
  const visibleBottom = scrollY.value + canvasH - RULER_HEIGHT;

  if (trackTop < visibleTop) {
    scrollY.value = trackTop;
  } else if (trackBottom > visibleBottom) {
    scrollY.value = trackBottom - (canvasH - RULER_HEIGHT);
  }
}
```

### Pattern 3: Pointer Event Drag (Not Mouse Events)
**What:** Use `pointerdown/pointermove/pointerup` with `setPointerCapture` for drag interactions on the scrollbar thumb.
**When to use:** All drag interactions in this codebase use pointer events (see TimelineInteraction.ts).
**Example:**
```typescript
const onPointerDown = (e: PointerEvent) => {
  (e.target as HTMLElement).setPointerCapture(e.pointerId);
  isDragging = true;
  dragStartY = e.clientY;
  dragStartScrollY = timelineStore.scrollY.peek();
};
```

### Anti-Patterns to Avoid
- **Drawing scrollbar on canvas:** The scrollbar is a DOM element, not a canvas drawing. Canvas redraws on every frame during playback -- the DOM scrollbar avoids this overhead.
- **Using local component state for scroll position:** The timelineStore.scrollY signal is the single source of truth. The scrollbar reads/writes it. This ensures the wheel handler and the scrollbar always agree.
- **Using mouse events instead of pointer events:** Pointer events with `setPointerCapture` are the project standard for all drag interactions.
- **Animating auto-scroll:** The user explicitly decided on instant snap, matching how horizontal `ensureFrameVisible` works.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Thumb ratio calculation | Complex scroll math from scratch | Simple ratio: `visibleHeight / totalContentHeight` | Same math as SidebarScrollArea, well-tested pattern |
| Scroll position clamping | Manual clamp logic everywhere | `timelineStore.setScrollY()` already clamps to `max(0, v)` | But need to also clamp to maxScrollY in the setter |
| Content height computation | Ad-hoc calculation in each consumer | Computed signal `totalContentHeight` in timelineStore | One source of truth -- RULER_HEIGHT + fxCount * FX_TRACK_HEIGHT + contentCount * TRACK_HEIGHT |

**Key insight:** The wheel handler in `TimelineInteraction.ts` lines 692-700 already computes `maxScrollY` inline. Extract this to a computed signal or shared helper so both the wheel handler and scrollbar use the same calculation.

## Common Pitfalls

### Pitfall 1: Canvas Height vs Visible Track Area
**What goes wrong:** Confusing canvas total height with the visible track area (below the ruler). The scrollable area is `canvasHeight - RULER_HEIGHT`, not `canvasHeight`.
**Why it happens:** RULER_HEIGHT (24px) is drawn on canvas but does not scroll. The `ctx.clip()` and `ctx.translate(0, -scrollY)` in TimelineRenderer only apply below RULER_HEIGHT.
**How to avoid:** Always subtract RULER_HEIGHT when computing visible track area for scroll math.
**Warning signs:** Scrollbar thumb overshoots or doesn't reach the bottom.

### Pitfall 2: maxScrollY Stale After Track Count Changes
**What goes wrong:** Adding/removing FX tracks or content sequences changes totalContentHeight but scrollY remains at the old value, potentially exceeding the new maxScrollY.
**Why it happens:** scrollY is not reactively clamped -- it's only clamped at write time.
**How to avoid:** When computing thumb position, clamp scrollY to [0, maxScrollY]. Also consider clamping in a signal effect when track counts change, or at least in the draw cycle.
**Warning signs:** scrollY > maxScrollY after removing tracks, causing empty space below content.

### Pitfall 3: Pointer Events Leaking During Drag
**What goes wrong:** If the user drags the scrollbar thumb quickly and the cursor leaves the thumb element, pointermove events stop firing.
**Why it happens:** Not using `setPointerCapture`, or attaching move/up listeners to the thumb element instead of the document/window.
**How to avoid:** Call `setPointerCapture(e.pointerId)` on pointerdown. This ensures all subsequent pointer events route to the capturing element regardless of cursor position.
**Warning signs:** Drag "sticks" or stops tracking when cursor moves fast.

### Pitfall 4: Flex Layout Disrupting Canvas Sizing
**What goes wrong:** Adding a 4px scrollbar sibling to the canvas in a flex-row container could change the canvas width calculation, breaking ResizeObserver or DPI scaling in TimelineRenderer.
**Why it happens:** The canvas currently uses `w-full h-full` inside a `flex-1 min-h-0 overflow-hidden` div. Adding a flex sibling changes how flex-1 distributes space.
**How to avoid:** The canvas container div (line 109 in TimelinePanel.tsx) becomes `flex flex-row`. The canvas wrapper keeps `flex-1 min-h-0`. The scrollbar is `w-1 shrink-0` (4px, no flex shrink). The ResizeObserver on the canvas parent will fire and TimelineRenderer.resize() will handle the new width automatically.
**Warning signs:** Canvas appears 4px narrower or DPI scaling breaks.

### Pitfall 5: Auto-Scroll Competing with Manual Scroll During Playback
**What goes wrong:** User manually scrolls to see a different track during playback, but auto-scroll immediately snaps back.
**Why it happens:** Auto-scroll fires on every sequence boundary crossing during playback.
**How to avoid:** This is actually acceptable per the user decision (matches horizontal behavior -- `ensureFrameVisiblePaged` also overrides manual scroll). The horizontal auto-scroll already exhibits this behavior and the user has not flagged it as a problem.
**Warning signs:** N/A -- this is expected behavior per the horizontal precedent.

## Code Examples

### Total Content Height Computation
```typescript
// Source: derived from TimelineInteraction.ts lines 694-697
// Extract to timelineStore or a shared computed signal

import { computed } from '@preact/signals';
import { trackLayouts, fxTrackLayouts } from '../lib/frameMap';
import { RULER_HEIGHT, TRACK_HEIGHT, FX_TRACK_HEIGHT } from '../components/timeline/TimelineRenderer';

// Computed signal for total content height
const totalContentHeight = computed(() => {
  const fxCount = fxTrackLayouts.value.length;
  const contentCount = trackLayouts.value.length;
  return RULER_HEIGHT + fxCount * FX_TRACK_HEIGHT + contentCount * TRACK_HEIGHT;
});
```

### Scrollbar Thumb Math
```typescript
// Source: adapted from SidebarScrollArea.tsx lines 20-32

function computeThumb(containerHeight: number, totalContentH: number, scrollY: number) {
  const viewableH = containerHeight; // full scrollbar height (spans full timeline including ruler)
  const trackAreaH = containerHeight - RULER_HEIGHT; // visible scrollable area
  const scrollableH = totalContentH - containerHeight; // max scroll range

  if (scrollableH <= 0) {
    // All content fits -- thumb fills entire track area
    return { thumbHeight: trackAreaH, thumbTop: RULER_HEIGHT, maxScrollY: 0 };
  }

  const ratio = trackAreaH / (totalContentH - RULER_HEIGHT);
  const thumbHeight = Math.max(24, ratio * trackAreaH); // 24px minimum
  const thumbRange = trackAreaH - thumbHeight;
  const scrollRatio = scrollY / scrollableH;
  const thumbTop = RULER_HEIGHT + scrollRatio * thumbRange;

  return { thumbHeight, thumbTop, maxScrollY: scrollableH };
}
```

### Scrollbar Track Click-to-Jump
```typescript
// Click on scrollbar track (not thumb) to jump
const onTrackClick = (e: MouseEvent) => {
  const rect = scrollbarRef.current!.getBoundingClientRect();
  const clickY = e.clientY - rect.top;
  // Convert click position to scroll ratio (excluding ruler area)
  const trackAreaTop = RULER_HEIGHT;
  const trackAreaH = rect.height - RULER_HEIGHT;
  const ratio = Math.max(0, Math.min(1, (clickY - trackAreaTop) / trackAreaH));
  const maxScrollY = totalContentHeight.peek() - rect.height;
  timelineStore.setScrollY(ratio * maxScrollY);
};
```

## Open Questions

1. **Canvas viewport height signal**
   - What we know: `timelineStore.viewportWidth` exists as a signal (set by ResizeObserver in TimelineCanvas). No equivalent `viewportHeight` signal exists.
   - What's unclear: Whether to add a `viewportHeight` signal or pass canvas height as a parameter to `ensureTrackVisible`.
   - Recommendation: Add `viewportHeight` signal to `timelineStore`, set it from the same ResizeObserver that sets `viewportWidth`. This keeps `ensureTrackVisible` purely signal-based (no DOM queries). The scrollbar component also needs this value for thumb computation.

2. **setScrollY clamping to maxScrollY**
   - What we know: Currently `setScrollY` only clamps to `max(0, v)`. The wheel handler in TimelineInteraction computes maxScrollY inline and clamps before calling setScrollY.
   - What's unclear: Whether setScrollY itself should enforce the upper bound, or callers remain responsible.
   - Recommendation: Keep setScrollY simple (lower bound only). The scrollbar component and wheel handler both know the canvas height and can compute maxScrollY. Adding upper-bound clamping to setScrollY would require it to know canvasHeight, creating a dependency.
