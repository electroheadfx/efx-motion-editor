# Phase 12.15: Sequence Playback - Research

**Researched:** 2026-03-20
**Domain:** Sequence isolation ("solo/mute") playback, loop toggle, Canvas 2D interaction, Preact signals, Tauri IPC persistence
**Confidence:** HIGH

## Summary

This phase implements a "solo" system for sequence playback -- users can isolate one or more content sequences so that only those sequences play during playback, skipping non-isolated frames. The feature touches four distinct subsystems: (1) a new isolation state signal (a `Set<string>` of sequence IDs), (2) playback engine modifications to skip non-isolated frames, (3) two visual entry points (timeline Canvas 2D name labels and sidebar accent bar), and (4) a global loop toggle persisted to `~/.config/efx-motion/builder-config.yaml` via Tauri IPC.

The codebase is well-structured for this work. The playback engine (`playbackEngine.ts`) has a clear tick loop with frame advancement that can be gated by isolation bounds. The timeline renderer (`TimelineRenderer.ts`) already draws sequence name overlays (lines 490-509 of `drawLinearTrack`) with exact positioning data needed for hit detection. The sidebar `SequenceList.tsx` already renders an accent bar (line 232) that just needs an onClick handler. The Tauri config persistence pattern (`config.rs` + `ipc.ts` + `appConfig.ts`) is well-established with 6 existing get/set command pairs.

**Primary recommendation:** Implement as a signal-driven isolation set in a new or existing store, wire it into `PlaybackEngine.tick()` frame gating, add Canvas 2D hit detection for name labels, and follow the established Tauri IPC pattern for loop persistence.

### Pattern 1: Isolation Store (new signal-based store)
**What:** A dedicated store holding the isolation set and loop toggle state.
**When to use:** This is the central state for the entire feature.
**Example:**
```typescript
// Application/src/stores/isolationStore.ts
import { signal, computed } from '@preact/signals';

const isolatedSequenceIds = signal<Set<string>>(new Set());
const loopEnabled = signal(false);

export const isolationStore = {
  isolatedSequenceIds,
  loopEnabled,

  /** Whether any sequences are currently isolated */
  hasIsolation: computed(() => isolatedSequenceIds.value.size > 0),

  /** Toggle a sequence in/out of the isolation set */
  toggleIsolation(sequenceId: string) {
    const current = new Set(isolatedSequenceIds.peek());
    if (current.has(sequenceId)) {
      current.delete(sequenceId);
    } else {
      current.add(sequenceId);
    }
    isolatedSequenceIds.value = current;
  },

  /** Check if a specific sequence is isolated */
  isIsolated(sequenceId: string): boolean {
    return isolatedSequenceIds.peek().has(sequenceId);
  },

  /** Clear all isolation */
  clearIsolation() {
    if (isolatedSequenceIds.peek().size > 0) {
      isolatedSequenceIds.value = new Set();
    }
  },

  /** Clean up: remove deleted sequence IDs from isolation set */
  removeSequence(sequenceId: string) {
    const current = isolatedSequenceIds.peek();
    if (current.has(sequenceId)) {
      const next = new Set(current);
      next.delete(sequenceId);
      isolatedSequenceIds.value = next;
    }
  },

  setLoopEnabled(v: boolean) {
    loopEnabled.value = v;
  },
  toggleLoop() {
    loopEnabled.value = !loopEnabled.value;
  },
};
```

### Pattern 2: Isolation-Aware Playback Tick
**What:** Modify `PlaybackEngine.tick()` to skip frames of non-isolated sequences during playback.
**When to use:** When `isolationStore.hasIsolation` is true.
**Key insight:** The `frameMap` already maps every global frame to a `sequenceId`. The tick loop can check if the current frame's sequence is in the isolation set and advance to the next isolated frame if not.

**Approach -- Precomputed isolated frame ranges:**
```typescript
// During playback with isolation, precompute sorted frame ranges from trackLayouts
// that belong to isolated sequences. Then the tick loop navigates only within those ranges.
function getIsolatedFrameRanges(
  tracks: TrackLayout[],
  isolatedIds: Set<string>,
): Array<{ start: number; end: number }> {
  return tracks
    .filter(t => isolatedIds.has(t.sequenceId))
    .map(t => ({ start: t.startFrame, end: t.endFrame }));
}

// In tick: when frame advances past an isolated range's end,
// jump to the next isolated range's start (or loop/stop).
```

### Anti-Patterns to Avoid
- **Don't filter frameMap itself:** The global `frameMap` computed signal must remain unchanged. Isolation is playback-only behavior; the timeline still shows all frames. Filtering frameMap would break the timeline layout.
- **Don't use effects for playback gating:** The tick loop must use `.peek()` to read isolation state, not `.value`, to avoid Preact subscription tracking in rAF.
- **Don't persist isolation set to disk:** Isolation is a transient editing state, not a project property. It should NOT be saved to .mce files.
- **Don't create a new signal for "isIsolated per sequence":** Use a single `Set<string>` signal. Individual per-sequence signals would create N subscriptions instead of 1.

## Common Pitfalls

### Pitfall 1: Signal Subscription in rAF Tick Loop
**What goes wrong:** Using `.value` instead of `.peek()` inside `PlaybackEngine.tick()` creates Preact signal subscriptions in the rAF callback, leading to memory leaks and unexpected re-triggers.
**Why it happens:** `.value` creates a tracking subscription. rAF callbacks are not Preact effects.
**How to avoid:** Always use `.peek()` inside the tick loop. This is an established pattern (see existing `shuttleSpeed.peek()`, `isFullSpeed.peek()` in current tick implementation).
**Warning signs:** Playback performance degradation, memory leak warnings.

### Pitfall 2: Stale Isolation Frame Ranges During Playback
**What goes wrong:** If the user modifies sequences while playback is running with isolation active, the precomputed frame ranges become stale.
**Why it happens:** Frame ranges are derived from `trackLayouts` which recompute on sequence changes.
**How to avoid:** Read `trackLayouts.peek()` fresh on each tick iteration (they're cached computed signals, so this is cheap). Or recompute ranges only when trackLayouts changes.
**Warning signs:** Playhead jumps to wrong positions after editing during playback.

### Pitfall 3: Name Label Hit Box Desync with Rendering
**What goes wrong:** The clickable area doesn't match the visual name label because geometry calculations differ between renderer and interaction handler.
**Why it happens:** Code duplication between `drawLinearTrack` (rendering) and `onPointerDown` (hit detection).
**How to avoid:** Extract name label geometry into a shared helper function or expose it from the renderer. At minimum, use the same constants and formulas.
**Warning signs:** Clicks miss the label or register on empty space.

### Pitfall 4: Escape Key Priority Chain
**What goes wrong:** Escape clears isolation when user intended to exit fullscreen, or vice versa.
**Why it happens:** Incorrect priority ordering in the Escape handler.
**How to avoid:** Follow the documented priority: fullscreen exit > isolation clear > layer deselect. The current handler already checks `isFullscreen.peek()` first. Add isolation clear as second priority.
**Warning signs:** User in fullscreen with isolation active presses Escape and isolation clears instead of exiting fullscreen.

### Pitfall 5: Loop + Isolation End-of-Range Boundary
**What goes wrong:** Off-by-one errors at the end of the last isolated sequence cause the playhead to overshoot or not loop correctly.
**Why it happens:** Frame ranges are `[startFrame, endFrame)` (exclusive end). The last playable frame is `endFrame - 1`.
**How to avoid:** Carefully handle the boundary: when `currentFrame >= lastIsolatedFrame`, loop to first isolated frame (loop ON) or stop (loop OFF). Use the same `endFrame - 1` pattern as `trackLayouts`.
**Warning signs:** Playhead hangs on last frame, or skips the last frame.

### Pitfall 6: Sidebar Accent Bar Reactivity During Playback
**What goes wrong:** The accent bar orange state triggers expensive re-renders during playback.
**Why it happens:** Toggling isolation state causes SequenceList to re-render all items.
**How to avoid:** The sidebar is already frozen during playback (displayFrame dual-signal pattern). Isolation toggle from the sidebar is only possible when playback is stopped (since the accent bar is only on the active sequence). No extra gating needed.
**Warning signs:** Sidebar jank during playback.

## Open Questions

1. **Should isolation set survive project switch?**
   - What we know: Isolation is transient editing state, not saved to .mce
   - What's unclear: Should it clear on project open/close?
   - Recommendation: Clear isolation on project reset (add `isolationStore.reset()` call in the project load flow). This is the safest default.

2. **Should isolation be undoable?**
   - What we know: It's a playback/preview feature, not a content edit
   - What's unclear: Whether users expect Cmd+Z to undo isolation changes
   - Recommendation: Do NOT make isolation undoable. It's analogous to the loop toggle or zoom level -- view state, not document state. Keep it out of the undo history.

3. **Orange color value**
   - What we know: The accent color in the app is typically `var(--color-accent)` which is an orange-ish brand color
   - Recommendation (Claude's discretion): Use `#E5841B` or `var(--color-accent)` for consistency. The existing accent bar uses `var(--sidebar-accent-bar)`. For isolation orange, use a dedicated hardcoded orange like `#E5841B` to distinguish from the general accent. On the Canvas 2D side, resolve via the theme color cache pattern.

4. **Should clicking a name label also seek the playhead to that sequence?**
   - What we know: The decision says "Clicking also makes it the active sequence" but doesn't mention seeking
   - Recommendation: Do set active sequence but do NOT auto-seek. The user is building an isolation set, not navigating. Seeking would be disruptive during multi-select.
