# Phase 12.1.1: Big UI Frontend Design Changes -- New Sidebar Layout - Research

**Researched:** 2026-03-17
**Domain:** Preact + Signals UI restructuring, resizable panels, CSS-variable-driven theming, sidebar-scoped design system
**Confidence:** HIGH

## Summary

This phase restructures the existing LeftPanel sidebar from a single-scroll collapsible layout into 3 independently resizable sub-windows (SEQUENCES, LAYERS, PROPERTIES) with pixel-perfect visual polish matching a .pen design spec. The codebase is a Tauri + Preact + Signals + TailwindCSS v4 desktop app with all UI state managed through `@preact/signals`. The existing sidebar code (`LeftPanel.tsx`, `CollapsibleSection.tsx`, `SequenceList.tsx`, `KeyPhotoStrip.tsx`, `LayerList.tsx`, `SidebarProperties.tsx`) will undergo major rewrites, and 5 new components are introduced (`PanelResizer`, `SidebarResizer`, `CollapseHandle`, `KeyframeNavBar`, `InlineInterpolation`).

The work spans 7 distinct concern areas: (1) sidebar structural layout with 3 resizable sub-windows, (2) sidebar-specific CSS variables stored in YAML config, (3) key photos moving inline under selected sequence cards, (4) keyframe navigation bar replacing the `+ Keyframe` button, (5) inline interpolation controls replacing the double-click popover, (6) visual polish (collapse handle, scrollbars, layer eye icon), and (7) removal of the old interpolation popover. The technical risk is moderate -- the pointer-drag resizer pattern already exists in `NumericInput` and the signal store infrastructure is mature. The main complexity is ensuring the 3-panel resizer system correctly handles collapse-to-zero, persistence, and independent scrolling without layout jank.

**Primary recommendation:** Implement in dependency order: CSS variables first, then structural layout (LeftPanel rewrite with PanelResizer + SidebarResizer), then component updates (CollapsibleSection, SequenceItem+KeyPhotoStrip inline, LayerRow with eye icon), then behavioral features (KeyframeNavBar, InlineInterpolation, CollapseHandle), then cleanup (remove old popover, old chevron).

### Pattern 1: Pointer Drag Resizer (reuse NumericInput pattern)
**What:** Panel resizer uses the same pointer capture pattern as NumericInput label-drag
**When to use:** For PanelResizer and SidebarResizer components
**Example:**
```typescript
// Same pattern as NumericInput.handleLabelPointerDown
function PanelResizer({ onResize, onResizeEnd }: Props) {
  const handlePointerDown = (e: PointerEvent) => {
    e.preventDefault();
    const target = e.currentTarget as HTMLElement;
    target.setPointerCapture(e.pointerId);
    let startY = e.clientY;

    const onMove = (ev: PointerEvent) => {
      const delta = ev.clientY - startY;
      startY = ev.clientY;
      onResize(delta);
    };

    const onUp = () => {
      onResizeEnd();
      target.removeEventListener('pointermove', onMove);
      target.removeEventListener('pointerup', onUp);
    };

    target.addEventListener('pointermove', onMove);
    target.addEventListener('pointerup', onUp);
  };

  return (
    <div
      class="h-4 flex items-center justify-center cursor-grab active:cursor-grabbing shrink-0"
      onPointerDown={handlePointerDown}
    >
      {/* Ellipsis icon */}
    </div>
  );
}
```

### Pattern 4: Config YAML Persistence via Rust IPC
**What:** Sidebar settings (width, panel heights) persist to `~/.config/efx-motion/builder-config.yaml` via existing Rust commands
**When to use:** For sidebar width and panel height persistence
**Note:** Requires extending the Rust `BuilderConfig` struct and adding new IPC commands

### Anti-Patterns to Avoid
- **Sidebar-level scrolling:** Each sub-window scrolls independently; the sidebar container itself MUST be `overflow: hidden`. Never put `overflow-y: auto` on the sidebar wrapper.
- **Hardcoded hex colors in components:** All sidebar colors MUST use `--sidebar-*` CSS variables, not hardcoded hex values. This enables future theme support.
- **Using existing `--color-*` variables for sidebar:** The sidebar has its own color system. Do not mix `--color-bg-selected` with `--sidebar-selected-row-bg` -- they are different palettes.
- **Debounced resize:** Panel resizing must be real-time with no debounce (smooth drag). Only persist on `pointerup`.
- **State in component for panel heights:** Panel heights must be in uiStore signals, not component state, because they need to persist and be shared.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Icon components | Custom SVG strings in each file | `lucide-preact` | Tree-shakeable, typed, consistent sizing; existing inline SVGs in the codebase are inconsistent |
| Drag-and-drop list reorder | Custom drag handler | `SortableJS` (already used) | Complex edge cases with DOM reversion for Preact compat, already solved |
| Config YAML read/write | JS-side YAML parsing | Rust `serde_yaml` via IPC (existing pattern) | Config is persisted by the Rust backend; the pattern for `config_get_theme`/`config_set_theme` already works |
| CSS variable injection | JS-side DOM style manipulation | `:root` CSS + `data-theme` attribute (existing pattern) | Sidebar variables go in `index.css` alongside existing theme variables |

**Key insight:** The config persistence pattern (Rust `BuilderConfig` struct + IPC commands) already handles theme and canvas_bg. Extending it for `sidebar_width` and `panel_heights` is the natural path -- don't try to use `LazyStore` for sidebar config since the YAML-based config via Rust is the established pattern for these settings.

## Common Pitfalls

### Pitfall 1: Panel Height Calculation with Flex Layout
**What goes wrong:** The 3-panel layout needs 2 panels with fixed heights and 1 panel (Properties) taking remaining space. If all 3 are fixed-height, the layout breaks when the sidebar resizes vertically.
**Why it happens:** Hard-coding all 3 panel heights doesn't account for window resize.
**How to avoid:** Only persist heights for SEQUENCES and LAYERS panels. PROPERTIES panel should use `flex: 1` to fill remaining space. The resizer between LAYERS and PROPERTIES adjusts LAYERS height and PROPERTIES auto-fills.
**Warning signs:** Properties panel content clipped or scrollbar appears on sidebar wrapper.

### Pitfall 2: Preact Signal Subscription in Resize Handlers
**What goes wrong:** Reading `signal.value` inside event handlers subscribes the component and causes unwanted re-renders during drag.
**Why it happens:** Preact signals auto-track when `.value` is accessed in render.
**How to avoid:** Use `.peek()` inside pointer event handlers (pointermove, pointerup). Only use `.value` in JSX to trigger reactive updates.
**Warning signs:** Jittery/laggy panel resizing, excessive re-renders during drag.

### Pitfall 3: SortableJS DOM Mutation vs. Preact VDOM
**What goes wrong:** SortableJS manipulates the DOM directly, then Preact tries to reconcile its VDOM, causing duplicated or missing elements.
**Why it happens:** Two systems (SortableJS and Preact) both trying to control the DOM.
**How to avoid:** In the `onEnd` handler, revert SortableJS's DOM mutation and let Preact re-render from signal state. This pattern is already established in `SequenceList.tsx` and `LayerList.tsx` -- maintain it for any refactored sortable lists.
**Warning signs:** Duplicate items after drag, items disappearing, console errors about missing DOM nodes.

### Pitfall 4: Collapse-to-Zero Panel State
**What goes wrong:** Panel collapsed to 0px height but content still renders, consuming CPU/memory.
**Why it happens:** Only hiding with `height: 0` but not removing content.
**How to avoid:** When panel height is 0, skip rendering children entirely (`height === 0 ? null : children`). Use the same pattern as `CollapsibleSection` which conditionally renders children.
**Warning signs:** Scrollbar flickers, CPU spike with collapsed panels.

### Pitfall 5: Key Photo Slide Animation Layout Shift
**What goes wrong:** Key photo strip expanding with `height` transition causes layout shift/jank in the sequence list, pushing other items down abruptly.
**Why it happens:** The expand area is inside the scroll container, and its height change affects all sibling layout.
**How to avoid:** Use `max-height` + `overflow: hidden` transition on the key photo container. Start from `max-height: 0` and animate to a calculated max-height. Use `transition: max-height 150ms ease-out`.
**Warning signs:** Sequence cards below the expanding one jitter or jump.

### Pitfall 6: CSS Variable Specificity with Tailwind v4
**What goes wrong:** Tailwind v4 utility classes may conflict with custom CSS variables defined in `index.css`.
**Why it happens:** Tailwind v4 uses `@layer` internally; custom styles may need to be in the right layer.
**How to avoid:** Define sidebar CSS variables in `:root` alongside existing theme variables. Use `var(--sidebar-*)` in inline styles or custom CSS classes, not in Tailwind arbitrary values like `bg-[var(--sidebar-panel-bg)]` (which works but is verbose). For frequently used values, consider adding utility classes in `index.css`.
**Warning signs:** Colors not applying, Tailwind classes overriding custom styles.

### Pitfall 7: Missing Keyframe Navigation Edge Cases
**What goes wrong:** < and > buttons navigate incorrectly when there are 0 or 1 keyframes, or when playhead is beyond the last keyframe.
**Why it happens:** Edge cases with empty arrays, finding "next" from a position after the last element.
**How to avoid:** Implement with explicit guards: (1) disable both arrows if keyframes.length < 2, (2) disable < if current frame <= first keyframe frame, (3) disable > if current frame >= last keyframe frame, (4) "next" means first keyframe whose frame > current, (5) "prev" means last keyframe whose frame < current.
**Warning signs:** Console errors when clicking arrows with no keyframes, playhead jumping to wrong position.

### Keyframe Navigation Logic
```typescript
function useKeyframeNav(layerKeyframes: Keyframe[], currentLocalFrame: number) {
  const sorted = [...layerKeyframes].sort((a, b) => a.frame - b.frame);
  const prevFrame = sorted.filter(k => k.frame < currentLocalFrame).pop()?.frame ?? null;
  const nextFrame = sorted.find(k => k.frame > currentLocalFrame)?.frame ?? null;
  const isOnKf = sorted.some(k => k.frame === currentLocalFrame);

  return { prevFrame, nextFrame, isOnKf, canPrev: prevFrame !== null, canNext: nextFrame !== null };
}
```

## Open Questions

1. **Rust IPC for panel heights**
   - What we know: Existing `BuilderConfig` struct in `config.rs` has `theme` and `canvas_bg` fields. New fields needed for `sidebar_width`, `panel_heights`.
   - What's unclear: Whether to add individual IPC commands (`config_get_sidebar_width`, `config_set_sidebar_width`, `config_get_panel_heights`, etc.) or a batch command.
   - Recommendation: Add individual commands following the existing pattern (`config_get_*` / `config_set_*`). This is consistent with theme and canvas_bg. The Rust struct gets new optional fields with serde defaults.

2. **EasingType "Cubic" discrepancy**
   - What we know: CONTEXT.md mentions "Cubic" as a 5th interpolation preset. The actual `EasingType` union only has 4: `'linear' | 'ease-in' | 'ease-out' | 'ease-in-out'`. The UI-SPEC lists the same 4 presets.
   - What's unclear: Whether a 5th easing type should be added.
   - Recommendation: Stick with the 4 existing easing types. The CONTEXT.md "Cubic" was likely referring to the mathematical implementation detail (all ease curves already use cubic polynomials). The UI-SPEC correctly lists 4 presets. Do NOT add a 5th type.

3. **Sidebar CSS variables in YAML config**
   - What we know: CONTEXT.md says new sidebar CSS variables are "stored in config YAML alongside existing theme variables." The existing YAML only stores `theme` (string) and `canvas_bg` (map).
   - What's unclear: Whether the user wants the hex color values themselves stored in YAML (so they can be customized per-user), or just the CSS variable definitions in `index.css` with a notation that they're "alongside" the theme system.
   - Recommendation: For this phase, define sidebar CSS variables directly in `index.css` (no YAML customization). The "stored in config YAML" note likely means "lives in the same theming system," not "user-editable via YAML." If customization is needed later, that's a separate phase.
