# Phase 15.2: Solid Sequence - Research

**Researched:** 2026-03-22
**Domain:** Data model extension, canvas rendering, UI components (Preact/Signals)
**Confidence:** HIGH

## Summary

This phase extends the existing key photo system to support solid color entries and transparent entries alongside imported images. The core challenge is extending the `KeyPhoto` type to become a discriminated union that can represent three variants (photo, solid, transparent), then propagating that type change through the entire rendering pipeline: `frameMap`, `FrameEntry`, `KeyPhotoRange`, `TimelineRenderer`, `PreviewRenderer`, `exportRenderer`, sidebar thumbnail, and project serialization.

The codebase is well-structured with clear data flow: `KeyPhoto[]` feeds into `frameMap` which produces `FrameEntry[]` for preview rendering and `TrackLayout`/`KeyPhotoRange` for timeline rendering. All mutation goes through `sequenceStore` with snapshot/restore undo. The existing `TransitionProperties` component already implements a solid/transparency toggle with conditional color picker -- the exact same UX pattern needed for key solid cells.

**Primary recommendation:** Use optional fields on `KeyPhoto` (not a discriminated union) to minimize structural change. A key photo with `imageId` is a photo; one with `solidColor` string is a solid; one with `isTransparent: true` is transparent. The `imageId` becomes optional. This approach requires the least change to serialization, undo, SortableJS reorder, and the 15+ call sites that touch `keyPhotos`.

## Architecture Patterns

### Data Flow (existing, to be extended)
```
KeyPhoto[] (sequenceStore)
  -> frameMap.ts -> FrameEntry[] (adds solidColor/isTransparent fields)
  -> trackLayouts -> KeyPhotoRange[] (adds solidColor/isTransparent fields)
  -> TimelineRenderer.drawLinearTrack() reads KeyPhotoRange to draw solid/checkerboard
  -> PreviewRenderer.resolveLayerSource() reads FrameEntry to render solid/transparent
  -> exportRenderer.renderGlobalFrame() reads FrameEntry for cross-dissolve dual-render
```

### Pattern 1: Optional Fields Extension (Recommended for KeyPhoto)
**What:** Add optional `solidColor?: string` and `isTransparent?: boolean` to `KeyPhoto`. A photo has `imageId` set; a solid has `solidColor` set (imageId is empty string or absent); transparent has `isTransparent: true`.
**When to use:** When extending an existing type used across 15+ callsites and serialized to disk.
**Example:**
```typescript
// types/sequence.ts
export interface KeyPhoto {
  id: string;
  imageId: string;          // '' for solids/transparents (not undefined, to avoid breaking existing .map/.filter)
  holdFrames: number;
  solidColor?: string;      // hex color, present for solid entries
  isTransparent?: boolean;  // true for transparent entries
}

// Helper discriminators
export function isKeySolid(kp: KeyPhoto): boolean {
  return !!kp.solidColor && !kp.isTransparent;
}
export function isKeyTransparent(kp: KeyPhoto): boolean {
  return !!kp.isTransparent;
}
export function isKeyPhoto(kp: KeyPhoto): boolean {
  return !!kp.imageId && !kp.solidColor && !kp.isTransparent;
}
```

**Why optional fields over discriminated union:** The existing code accesses `kp.imageId` in 15+ locations without type narrowing. A discriminated union would require adding type guards at every access point. Optional fields with empty-string `imageId` for non-photo entries keeps existing code paths safe (they just get empty imageId and handle gracefully -- image lookups return null, thumbnails show placeholder).

### Pattern 5: Checkerboard via Canvas (for TimelineRenderer)
**What:** Canvas-drawn checkerboard for transparent key photo ranges on the timeline.
```typescript
// In TimelineRenderer.drawLinearTrack()
function drawCheckerboard(ctx: CanvasRenderingContext2D, x: number, y: number, w: number, h: number, cellSize = 4) {
  const light = '#B0B0B0';
  const dark = '#808080';
  ctx.save();
  ctx.beginPath();
  ctx.rect(x, y, w, h);
  ctx.clip();
  ctx.fillStyle = light;
  ctx.fillRect(x, y, w, h);
  ctx.fillStyle = dark;
  for (let row = 0; row * cellSize < h; row++) {
    for (let col = 0; col * cellSize < w; col++) {
      if ((row + col) % 2 === 0) {
        ctx.fillRect(x + col * cellSize, y + row * cellSize, cellSize, cellSize);
      }
    }
  }
  ctx.restore();
}
```

### Anti-Patterns to Avoid
- **Separate keyPhotos and keySolids arrays:** Breaks reorder, hold frames, and the entire frameMap pipeline. D-01 explicitly forbids this.
- **New sequence kind for solids:** D-02 forbids this. Any content sequence mixes all entry types.
- **Breaking imageId as required:** Keep `imageId: string` (not optional `string | undefined`). Use `''` for solids/transparents. This avoids null checks at every imageId access site.
- **OS color dialog for every color change:** The user wants inline popover with live preview. Avoid triggering the OS dialog which blocks interaction.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Color picking | Custom HSL/RGB math + gradient canvas | `<input type="color">` or minimal wrapper | Browser provides full gradient UI; reinventing wastes time |
| Drag reorder with solids | Custom drag logic | SortableJS (already wired) | Solids are in same `keyPhotos` array; SortableJS handles them automatically |
| Checkerboard pattern (CSS) | Manual tiling logic | CSS `linear-gradient` pattern | 4-line CSS achieves Photoshop-standard checkerboard |
| Undo for solid operations | Custom undo stack | Existing `snapshot()/restore()/pushAction()` pattern | All key photo ops already use this pattern |

**Key insight:** Because key solids live in the same `keyPhotos` array, existing operations (reorder, delete, duplicate sequence, undo) work automatically. The only new operations are `addKeySolid`, `updateSolidColor`, and `toggleTransparent`.

## Common Pitfalls

### Pitfall 1: FrameEntry.imageId for Non-Photo Entries
**What goes wrong:** `frameMap.ts` builds FrameEntry with `imageId: kp.imageId`. For solids/transparents, imageId is `''`. Downstream code calling `imageStore.getById('')` returns undefined, which is safe. But `previewRenderer.resolveLayerSource()` uses `entry.imageId` to look up images -- it must NOT try to load an empty imageId.
**Why it happens:** The entire pipeline assumes every frame has an image.
**How to avoid:** In `resolveLayerSource()`, check for `entry.solidColor` or `entry.isTransparent` BEFORE attempting image lookup. For solid: fill canvas with color. For transparent: skip (canvas background shows through).
**Warning signs:** Black frames instead of colored solid, or 404 errors in console from attempting to load empty image paths.

### Pitfall 2: Cross-Dissolve with Solids
**What goes wrong:** `exportRenderer.ts` calls `buildSequenceFrames()` and `renderer.renderFrame()` for both outgoing and incoming sequences during cross-dissolve. If one sequence has a solid entry, the base layer's `resolveLayerSource()` returns null, and the frame renders empty.
**Why it happens:** `renderFrame()` checks `isGeneratorLayer`, `isAdjustmentLayer`, then falls through to content layer resolution. Solid entries don't match any branch.
**How to avoid:** Add a solid-color rendering path in `renderFrame()` or intercept at the `exportRenderer` level. The simplest approach: when `FrameEntry.solidColor` is set, fill the canvas with that color directly instead of trying to resolve a layer source.
**Warning signs:** Empty/black frames during cross-dissolve transitions between solids and photos.

### Pitfall 3: Sidebar Thumbnail for Solid-First Sequences
**What goes wrong:** `SequenceList.tsx` line 95-99 gets `firstKp.imageId` and looks up image for thumbnail URL. For a solid-first sequence, `imageId` is `''`, so `thumbUrl` is null. The thumbnail box renders empty (just the placeholder background).
**Why it happens:** Sidebar code assumes first key photo always has an image.
**How to avoid:** When first key entry is a solid, render the solid color as the thumbnail background instead of an image. When transparent, render checkerboard pattern.
**Warning signs:** Grey/empty thumbnail for sequences starting with a solid.

### Pitfall 4: Project Serialization Backward Compatibility
**What goes wrong:** Adding `solidColor` and `isTransparent` to MceKeyPhoto. Older versions of the app opening a v10 file would ignore these fields, turning solids/transparents into blank key photos with empty image references.
**Why it happens:** Optional fields are silently ignored by older parsers.
**How to avoid:** Bump version to 10. On save, write `solid_color` and `is_transparent` fields. On load, default to undefined (treated as regular photo). Add `serde(default)` equivalent in TypeScript hydration.
**Warning signs:** Saved projects losing solid entries when opened in older versions.

### Pitfall 5: KeyPhotoStrip SortableJS Recount
**What goes wrong:** SortableJS instance recreates on `[keyPhotos.length, sequenceId]` dependency. Adding a solid changes length, so SortableJS re-initializes correctly. No pitfall here -- existing code handles this.

### Pitfall 6: Asset Usage Scanner and Solids
**What goes wrong:** `assetUsage.ts` scans `kp.imageId` to find image references. A solid with `imageId: ''` would match if someone passes `''` as the scan target.
**Why it happens:** Empty string is a valid JavaScript equality match.
**How to avoid:** The scanner already skips `__placeholder__` sentinel. Add a similar guard: skip entries where `imageId === ''`.
**Warning signs:** Usage counts being inflated by solid entries.

### Pitfall 7: Export Renderer Frame Preloading
**What goes wrong:** `preloadExportImages()` collects `fm.map(f => f.imageId)` to preload. Empty imageIds for solids generate bogus preload requests.
**Why it happens:** Preloader doesn't filter empty IDs.
**How to avoid:** Filter: `[...new Set(fm.map(f => f.imageId).filter(id => id !== ''))]`.
**Warning signs:** Console errors during export, or export hanging waiting for never-loading images.

## Code Examples

### Adding a Key Solid (sequenceStore)
```typescript
// sequenceStore.ts - new method
addKeySolid(sequenceId: string, solidColor: string = '#000000', holdFrames: number = 4) {
  const before = snapshot();
  const kp: KeyPhoto = {
    id: genId(),
    imageId: '',           // no image for solids
    holdFrames,
    solidColor,
  };
  sequences.value = sequences.value.map((s) =>
    s.id === sequenceId ? {...s, keyPhotos: [...s.keyPhotos, kp]} : s,
  );
  markDirty();
  const after = snapshot();
  pushAction({
    id: crypto.randomUUID(),
    description: 'Add key solid',
    timestamp: Date.now(),
    undo: () => restore(before),
    redo: () => restore(after),
  });
},
```

### Toggling Solid/Transparent (sequenceStore)
```typescript
toggleKeyEntryTransparent(sequenceId: string, keyPhotoId: string) {
  const before = snapshot();
  sequences.value = sequences.value.map((s) =>
    s.id === sequenceId
      ? {
          ...s,
          keyPhotos: s.keyPhotos.map((kp) => {
            if (kp.id !== keyPhotoId) return kp;
            if (kp.isTransparent) {
              // Switch back to solid
              return {...kp, isTransparent: undefined, solidColor: kp.solidColor || '#000000'};
            } else {
              // Switch to transparent
              return {...kp, isTransparent: true};
            }
          }),
        }
      : s,
  );
  markDirty();
  const after = snapshot();
  pushAction({
    id: crypto.randomUUID(),
    description: 'Toggle solid/transparent',
    timestamp: Date.now(),
    undo: () => restore(before),
    redo: () => restore(after),
  });
},
```

## Open Questions

1. **Color picker popover UX vs native input**
   - What we know: User wants "gradient square + hex input" inline popover. The native `<input type="color">` provides a gradient square but opens as an OS dialog, not inline.
   - What's unclear: Whether the user considers the OS color dialog acceptable or wants a fully custom inline gradient picker.
   - Recommendation: Start with native `<input type="color">` styled inline (small, embedded in popover). The OS dialog provides the best UX. If user rejects, build a minimal HSL gradient canvas. This is Claude's discretion area.

2. **Canvas preview for transparent entries**
   - What we know: D-19 says "renders as canvas background color (currently black)". The canvas `clearRect` clears to transparent, which on a dark app background appears black.
   - What's unclear: Whether existing clear behavior already achieves this or if explicit black fill is needed.
   - Recommendation: Transparent entries should simply skip rendering (don't fill), letting the canvas background show through. This already works because `clearRect` in `renderFrame` clears the canvas, and no content layer draws anything.
