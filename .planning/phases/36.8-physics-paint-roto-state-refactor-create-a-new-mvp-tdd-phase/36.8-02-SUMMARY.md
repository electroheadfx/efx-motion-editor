**Dirty Roto save-before-navigation/action now lives in the session boundary with sticky failure retention and explicit save result transitions.**

## Accomplishments

- Added RED-first boundary tests for 36.8-REG-05 covering save-before-navigation, save-before-action, sticky save failure, explicit retry, and D-13 dirty exposure without close-window ownership.
- Implemented `savingFrame`, `pendingNavigationFrame`, `pendingKeyAction`, `failedSaveFeedback`, and `currentFrameIsDirty` Signals on the Roto session boundary.
- Added `onSaveSucceeded(frame)` and `onSaveFailed(frame, message)` transitions so save success resumes queued navigation/key effects and save failure keeps the source dirty while clearing stale queued work.
- Refactored save continuation queuing through a small helper while preserving explicit named methods, serializable descriptors, no XState, and no bridge/canvas/window imports in the boundary.

## Decisions Made

- Save orchestration remains decision-only in `physicsPaintRotoSession.ts`; actual frame save execution stays in Studio/adapter code for later wiring.
- Dirty navigation/key actions return a `saveFrame` effect as the only first effect and keep the requested target/action pending until `onSaveSucceeded`.
- Save failure does not auto-retry or resume stale queued work; it requires a later explicit user navigation/action to produce a fresh `saveFrame` retry.
