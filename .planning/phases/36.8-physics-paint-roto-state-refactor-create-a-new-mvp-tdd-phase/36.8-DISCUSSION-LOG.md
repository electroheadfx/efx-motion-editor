## Boundary ownership

| Option | Description | Selected |
|--------|-------------|----------|
| Key/session core | Own real keys, current frame, dirty/pending state, copied key, generated markers, blank/deleted cleanup, and derived action availability; leave bridge/canvas execution in Studio. | ✓ |
| Include overlays | Also own cached reference URL/playback/onion source selection and restore intent so Studio reads more derived view state from one place. | |
| Only key utilities | Keep existing save/navigation/cached reference state mostly in Studio and move only key utility state/transactions. | |

**User's choice:** Key/session core
**Notes:** The boundary should own Roto key/session state, not the external bridge/canvas execution layer.

| Option | Description | Selected |
|--------|-------------|----------|
| Bridge and canvas | Keep Tauri/browser bridge events, engine load/save/export calls, window close handling, and parent persistence apply-result handling in PhysicsPaintStudio. | ✓ |
| Only bridge events | Move some canvas restore/export coordination into the boundary, but keep external bridge/window events in Studio. | |
| Let boundary wrap it | Let the boundary own most Roto orchestration, including calling canvas/bridge adapter callbacks supplied by Studio. | |

**User's choice:** Bridge and canvas
**Notes:** Studio remains the adapter for true external systems.

| Option | Description | Selected |
|--------|-------------|----------|
| One session object | Expose a compact `rotoSession`-style object with derived state and actions, so Studio/WorkflowStrip avoid expanding prop lists further. | ✓ |
| Few grouped props | Expose grouped state/actions separately, such as `rotoState`, `rotoActions`, and `rotoEffects`, for more explicit wiring. | |
| Keep current props | Keep the current WorkflowStrip prop contract and only hide internal Studio state behind helper functions. | |

**User's choice:** One session object
**Notes:** The refactor should reduce prop/list sprawl rather than moving it around.

| Option | Description | Selected |
|--------|-------------|----------|
| Signals plus pure helpers | Use Signals for external reactive state and computed derived state, but keep frame-list transforms and transactions as pure tested helpers. | ✓ |
| Pure controller first | Make the first boundary mostly pure functions/classes and add Signals later only if component subscriptions need them. | |

**User's choice:** Signals plus pure helpers
**Notes:** This follows the project’s Preact guidance without mechanically replacing all hooks.

## Action API shape

| Option | Description | Selected |
|--------|-------------|----------|
| Named transactions | Expose explicit methods like `duplicateKey`, `insertBlankKey`, `deleteKey`, `copyKey`, `pasteKey`, `requestFrame`, and `markDirty`, each returning a transaction/effect description Studio executes. | ✓ |
| Single dispatch | Expose one `dispatch({ type, ... })` entry point with action types, closer to a future XState event model. | |
| Mixed surface | Expose named methods for UI callers plus an internal dispatch/event reducer under the hood. | |

**User's choice:** Named transactions
**Notes:** Explicit methods keep call sites readable while still allowing state-machine-shaped naming.

| Option | Description | Selected |
|--------|-------------|----------|
| Effect descriptors | Return plain descriptors such as `saveFrame`, `replaceKeys`, `restoreFrame`, `clearCanvas`, or `navigate`, and let Studio execute bridge/canvas work. | ✓ |
| Prepared payloads | Return full bridge payloads where possible, with Studio only sending them and applying results. | |
| Callbacks invoked | Boundary receives adapter callbacks and invokes them directly, hiding orchestration from Studio. | |

**User's choice:** Effect descriptors
**Notes:** The boundary chooses intent; Studio performs external work.

| Option | Description | Selected |
|--------|-------------|----------|
| Event-shaped names | Use explicit action/effect names and guarded results that can map cleanly to future state-machine events, without adding XState now. | ✓ |
| Reducer now | Implement a reducer/event log now so Phase 36.9 can swap internals more directly. | |
| No prep beyond tests | Keep the API ergonomic for this refactor and rely on tests if Phase 36.9 later changes internals. | |

**User's choice:** Event-shaped names
**Notes:** Phase 36.8 prepares for Phase 36.9 but does not adopt XState.

| Option | Description | Selected |
|--------|-------------|----------|
| Fold into boundary | Keep the proven pure transaction helpers, but make them implementation details of the new Roto session boundary and expand around them. | ✓ |
| Leave separate | Keep the key controller as a separate module and have the new session boundary call it as a peer dependency. | |
| Replace it | Use the phase to replace the partial controller with a new boundary module from scratch. | |

**User's choice:** Fold into boundary
**Notes:** Existing `physicsPaintRotoKeyController.ts` is a seed, not throwaway code.

## Save orchestration

| Option | Description | Selected |
|--------|-------------|----------|
| Boundary decides | Boundary decides when a dirty source requires save, what action/navigation is pending, and what happens on success/failure; Studio performs the save effect. | ✓ |
| Studio decides | Keep dirty save decision rules in Studio and use the boundary only for key/cache state after save succeeds. | |
| Split by flow | Boundary owns key action save-before rules; Studio keeps frame-navigation save-on-leave rules. | |

**User's choice:** Boundary decides
**Notes:** Save rules are part of Roto session coherence, but actual saving is still an external effect.

| Option | Description | Selected |
|--------|-------------|----------|
| Sticky failed state | Boundary keeps the source frame dirty, clears queued action/destination, records failed-save feedback, and requires the next explicit action/navigation to retry. | ✓ |
| Immediate retry intent | Boundary keeps a retryable pending action/destination so Studio can retry the same intent without the user reselecting it. | |
| Studio-only error | Studio handles save failure copy and dirty restoration; boundary only receives a reset/update call. | |

**User's choice:** Sticky failed state
**Notes:** This preserves Phase 36.6 and 36.7 failure semantics.

| Option | Description | Selected |
|--------|-------------|----------|
| Boundary owns intent | Boundary records restore intent such as load real key, blank key, clear blank, generated cleanup, or cached reference; Studio executes engine/canvas restoration. | ✓ |
| Studio infers intent | Boundary only returns updated key/cache data; Studio decides whether to load, clear, or show cached reference. | |
| Pure transaction only | Keep restore intent only inside transaction helpers and do not persist it as boundary state. | |

**User's choice:** Boundary owns intent
**Notes:** Restore intent is state/effect intent; Studio executes it.

| Option | Description | Selected |
|--------|-------------|----------|
| Preserve close flow | Keep explicit dirty-close behavior in Studio/window handling; boundary can expose whether current frame is dirty, but no close UX or semantics change. | ✓ |
| Boundary owns close | Boundary also models close requested/cancel/save/discard states, with Studio only executing window close. | |
| Defer entirely | Do not touch close handling in Phase 36.8 beyond ensuring tests still pass. | |

**User's choice:** Preserve close flow
**Notes:** Phase 36.4 dirty close behavior remains unchanged.

## Regression contracts

| Option | Description | Selected |

| Option | Description | Selected |

**User's choice:** All roadmap cases, with worst failures highest priority
**Notes:** The selected options are complementary: cover all roadmap cases and prioritize the known 36.7 pain points.

| Option | Description | Selected |

**Notes:** This refactor should prove UI stability without making visual changes part of scope.
