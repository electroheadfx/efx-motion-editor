## Implementation Decisions

### Boundary Ownership
- **D-01:** The new boundary owns the Roto key/session core only: real key ownership, current/selected Roto frame, dirty/pending frame status, copied key clipboard state, generated/render-only markers, blank inserted key state, deleted-frame cleanup state, restore intent, and derived action availability.
- **D-02:** `PhysicsPaintStudio` remains the adapter for bridge and canvas execution. Tauri/browser bridge events, engine load/save/export calls, window close handling, and parent persistence/apply-result handling stay outside the boundary.
- **D-03:** The component interface should expose one compact Roto session object with derived state and actions. Avoid expanding `PhysicsPaintStudio` / `PhysicsPaintWorkflowStrip` prop lists further.
- **D-04:** Use `@preact/signals` for external reactive state and computed derived state where they make the state boundary clearer. Keep frame-list transforms, key transactions, and validation helpers pure and directly unit-tested.
- **D-05:** Do not mechanically replace hooks with Signals. Only move state that belongs to the Roto session/key boundary.

### Action API Shape
- **D-06:** The Roto session object should expose explicit named transaction methods such as `duplicateKey`, `insertBlankKey`, `deleteKey`, `copyKey`, `pasteKey`, `requestFrame`, and `markDirty`, rather than a generic UI-facing dispatch function.
- **D-07:** When an action needs external work, the boundary returns plain effect descriptors such as `saveFrame`, `replaceKeys`, `restoreFrame`, `clearCanvas`, `showCachedReference`, or `navigate`. `PhysicsPaintStudio` executes those effects through the canvas engine, bridge, and window APIs.
- **D-08:** Action and effect names should be state-machine-shaped so Phase 36.9 can map them to future transition events without UI rewiring. Do not add XState in Phase 36.8.
- **D-09:** Preserve the existing proven pure transaction logic in `physicsPaintRotoKeyController.ts`, but fold it into the new boundary as implementation detail/helper logic instead of leaving it as a partial peer controller with orchestration still in `PhysicsPaintStudio`.

### Save Orchestration
- **D-10:** The boundary decides save-before-action and save-on-leave rules: when a dirty source frame requires saving, what action/navigation is pending, and how success/failure transitions change boundary state. `PhysicsPaintStudio` performs the actual save effect.
- **D-11:** Failed saves become sticky boundary state: keep the source frame dirty, clear queued action/destination, record failed-save feedback, and require the next explicit user action/navigation to retry.
- **D-12:** The boundary owns restore intent after key actions or navigation: load real key, blank real key, clear blank, generated cleanup, deleted-frame cleanup, or cached-reference display. `PhysicsPaintStudio` executes the engine/canvas restoration.
- **D-13:** Preserve Phase 36.4 explicit dirty-close behavior. The boundary may expose whether the current Roto frame is dirty, but close-window prompt state and window-close adapter handling remain in `PhysicsPaintStudio` and must not change visibly or semantically.

### Regression Contracts
- **D-14:** Start implementation with failing boundary/controller unit tests before UI rewiring.
- **D-15:** Boundary tests must cover all roadmap-listed regression cases: inserted blank keys remain clean, deleted keys do not reappear, paste onto empty/generated targets creates clean real keys and clears generated/render-only state, duplicate behavior remains unchanged, dirty save-before-action preserves visible edits and cancels safely on save failure, and UI labels/placement remain unchanged.
- **D-16:** Treat the known Phase 36.7 pain points as the highest-priority failures inside the full regression set: blank keys becoming dirty/stale, deleted keys reappearing, paste to empty/generated retaining stale generated/cache state, and yellow/translucent cached canvas/reference state.
- **D-17:** Add a source-contract test that `PhysicsPaintStudio` consumes the compact Roto session object/effect descriptors and does not gain new broad `useEffect` orchestration for Roto key/cache coherence.
- **D-18:** Prove unchanged UI through source-contract assertions for labels/placement/props plus focused `PhysicsPaintWorkflowStrip` / `PhysicsPaintStudio` tests. Live UAT still validates the visible 36.7 flow after implementation.

### Pattern 1: Factory-Created Session Boundary

**What:** Create a per-window/session factory that returns Signals, computed values, named actions, and plain effect descriptors rather than module-level singleton state. [CITED: https://preactjs.com/guide/v10/signals/]

**When to use:** Use it for Roto state that currently spans component refs/state and must be shared between `PhysicsPaintStudio` and workflow UI without expanding prop lists. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.8-physics-paint-roto-state-refactor-create-a-new-mvp-tdd-phase/36.8-CONTEXT.md]

**Example:**

```typescript
// Source: https://preactjs.com/guide/v10/signals/
import { batch, computed, signal } from '@preact/signals';

export function createRotoSession(input: RotoSessionInput): RotoSession {
  const currentFrame = signal(input.currentFrame);
  const realKeyFrames = signal<number[]>(input.realKeyFrames);
  const dirtyFrames = signal<Set<number>>(new Set());
  const copiedKey = signal<RotoCopiedKey | null>(null);

  const actionState = computed(() => deriveRotoKeyUtilityActionState({
    currentFrame: currentFrame.value,
    realKeyFrameNumbers: realKeyFrames.value,
    hasCopiedRotoKey: copiedKey.value !== null,
    dirtyFrameNumbers: dirtyFrames.value,
  }));

  function markDirty(frame = currentFrame.value): RotoSessionEffect[] {
    batch(() => {
      dirtyFrames.value = new Set([...dirtyFrames.value, frame]);
    });
    return [{ type: 'clearCachedReference', frame }];
  }

  return { currentFrame, realKeyFrames, dirtyFrames, copiedKey, actionState, markDirty };
}
```

## Common Pitfalls

### Pitfall 1: Boundary becomes another peer controller instead of the owner
**What goes wrong:** `PhysicsPaintStudio` keeps dirty refs, copied-key refs, restore decisions, and action gating while the new boundary only wraps transaction helpers. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/PhysicsPaintStudio.tsx]
**Why it happens:** The existing `physicsPaintRotoKeyController.ts` is pure and useful but does not own save-on-leave, clipboard, dirty, or restore state by itself. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/physicsPaintRotoKeyController.ts]
**How to avoid:** Plan an explicit ownership migration table from current state/ref to new session field/action/effect. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.8-physics-paint-roto-state-refactor-create-a-new-mvp-tdd-phase/36.8-CONTEXT.md]
**Warning signs:** `copiedRotoKeyRef`, `dirtyRotoFramesRef`, Roto key action state, and generated/deleted cleanup decisions remain in `PhysicsPaintStudio`. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/PhysicsPaintStudio.tsx]

### Pitfall 2: Effects used as internal control flow
**What goes wrong:** New `useEffect` branches synchronize Roto key/cache/canvas state after state changes, recreating the broad synchronization problem. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.8-physics-paint-roto-state-refactor-create-a-new-mvp-tdd-phase/36.8-CONTEXT.md]
**Why it happens:** Hooks feel convenient for observing state, but Preact docs distinguish effects from pure derived computation. [CITED: https://preactjs.com/guide/v10/hooks/]
**How to avoid:** Perform transitions inside named actions and return explicit effects; use `computed` for derivation. [CITED: https://preactjs.com/guide/v10/signals/]
**Warning signs:** A test diff shows added `useEffect` blocks whose dependency arrays mention Roto key/cache/session variables. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.8-physics-paint-roto-state-refactor-create-a-new-mvp-tdd-phase/36.8-CONTEXT.md]

### Pitfall 3: Save failure loses visible dirty edits
**What goes wrong:** Save-before-action or save-on-leave failure clears queued state and accidentally drops dirty in-memory canvas state. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.8-physics-paint-roto-state-refactor-create-a-new-mvp-tdd-phase/36.8-CONTEXT.md]
**Why it happens:** Current code uses multiple refs for `pendingRotoAdvanceRef`, `saveOnLeaveSourceFrameRef`, rendered payload refs, dirty set, and apply status. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/PhysicsPaintStudio.tsx]
**How to avoid:** Model save failure as a sticky boundary transition: source frame remains dirty, queued action/destination is cleared, feedback is recorded, and next explicit action retries. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.8-physics-paint-roto-state-refactor-create-a-new-mvp-tdd-phase/36.8-CONTEXT.md]
**Warning signs:** Tests must simulate adapter save failure and assert dirty/source frame retention. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.8-physics-paint-roto-state-refactor-create-a-new-mvp-tdd-phase/36.8-CONTEXT.md]

### Pitfall 4: Generated/cache reference pixels leak into real keys
**What goes wrong:** Paste/insert/delete keeps generated or cached-reference frames, causing yellow/translucent stale canvas state. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.8-physics-paint-roto-state-refactor-create-a-new-mvp-tdd-phase/36.8-CONTEXT.md]

[VERIFIED: /Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/physicsPaintRotoKeyController.ts]
**How to avoid:** Keep generated/render-only markers and restore intent in the boundary, and preserve existing transaction cleanup fields. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/physicsPaintRotoKeyController.ts]
**Warning signs:** A pasted frame still has `source: 'generated-interpolation'` or `nearestRealKeyFrame`. [VERIFIED: /Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/physicsPaintRotoKeyController.test.ts]
