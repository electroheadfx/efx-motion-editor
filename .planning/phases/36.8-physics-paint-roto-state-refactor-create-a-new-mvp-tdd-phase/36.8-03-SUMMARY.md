# Phase 36.8 Plan 03: Physics Paint Roto State Refactor Summary

Roto session state and key-action decisions are now consumed through the Signals-backed `createRotoSession` boundary, with `PhysicsPaintStudio` retained as the adapter for bridge, canvas engine, persistence, and window-close side effects.

| 3 | Stabilized the adapter boundary by removing redundant pending Roto frame component state and deriving pending markers from session `dirtyFrames`. | `ccf9e3f5` |

## Implementation Notes

- `PhysicsPaintStudio.tsx` imports `createRotoSession`, `RotoSessionActionResult`, and `RotoSessionEffect` from `physicsPaintRotoSession`.
- Studio creates one compact Roto session object with current frame, real/cached Roto frames, dirty frames, canvas dimensions, in-flight status, and blank-frame construction.
- Key utility handlers now call session actions (`duplicateKey`, `insertBlankKey`, `deleteKey`, `copyKey`, `pasteKey`) and pass the results to a Studio-side result runner.
- Roto navigation now calls `rotoSession.requestFrame(targetFrame)` and executes the returned effect descriptors.
- Studio executes descriptor side effects via an adapter runner covering `saveFrame`, `replaceKeys`, `restoreFrame`, `clearCanvas`, `showCachedReference`, `navigate`, and cleanup descriptors.
- Bridge calls, engine `save`/`load`/`clear`, parent persistence, apply-result handling, and close-window prompts remain in Studio/adapter code.
- `PhysicsPaintWorkflowStrip` receives `pendingRotoFrames={rotoSession.dirtyFrames.value}` and copied-key status from `rotoSession.copiedKey.value`.
