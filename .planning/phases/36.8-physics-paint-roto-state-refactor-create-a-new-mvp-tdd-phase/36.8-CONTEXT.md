# Phase 36.8: Physics Paint Roto State Refactor - Context

**Gathered:** 2026-06-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 36.8 delivers a narrow maintainability and correctness refactor for Physics Paint Roto key/session state after Phase 36.7. The phase must extract a small Preact-native Roto session/key state boundary so `PhysicsPaintStudio` stops directly owning all Roto key/cache/canvas coherence through scattered local state, refs, launch-context mutations, and broad synchronization branches.

This phase preserves the existing Phase 36.7 user-visible behavior and UI exactly: Duplicate, Insert, Delete, Copy, Paste, save-on-leave, dirty save-before-action, Roto cell semantics, cached-reference behavior, and explicit dirty-close behavior must continue to work. It must not add new controls, shortcuts, modals, toasts, interpolation UI, cached playback UI, Play/Roto conversion scope, or editor integration.

</domain>

<decisions>
## Implementation Decisions

### Boundary Ownership
- **D-01:** The new boundary owns the Roto key/session core only: real key ownership, current/selected Roto frame, dirty/pending frame status, copied key clipboard state, generated/render-only markers, blank inserted key state, deleted-frame cleanup state, restore intent, and derived action availability.
- **D-02:** `PhysicsPaintStudio` remains the adapter for bridge and canvas execution. Tauri/browser bridge events, engine load/save/export calls, window close handling, and parent persistence/apply-result handling stay outside the boundary.
- **D-03:** The component interface should expose one compact Roto session object with derived state and actions. Avoid expanding `PhysicsPaintStudio` / `PhysicsPaintWorkflowStrip` prop lists further.
- **D-04:** Use `@preact/signals` for external reactive state and computed derived state where they make the state boundary clearer. Keep frame-list transforms, key transactions, and validation helpers pure and directly unit-tested.
- **D-05:** Do not mechanically replace hooks with Signals. Only move state that belongs to the Roto session/key boundary.

### Action API Shape
- **D-06:** The Roto session object should expose explicit named transaction methods such as `duplicateKey`, `insertBlankKey`, `deleteKey`, `copyKey`, `pasteKey`, `requestFrame`, and `markDirty`, rather than a generic UI-facing dispatch function.
- **D-07:** When an action needs external work, the boundary returns plain effect descriptors such as `saveFrame`, `replaceKeys`, `restoreFrame`, `clearCanvas`, `showCachedReference`, or `navigate`. `PhysicsPaintStudio` executes those effects through the canvas engine, bridge, and window APIs.
- **D-08:** Action and effect names should be state-machine-shaped so Phase 36.9 can map them to future transition events without UI rewiring. Do not add XState in Phase 36.8.
- **D-09:** Preserve the existing proven pure transaction logic in `physicsPaintRotoKeyController.ts`, but fold it into the new boundary as implementation detail/helper logic instead of leaving it as a partial peer controller with orchestration still in `PhysicsPaintStudio`.

### Save Orchestration
- **D-10:** The boundary decides save-before-action and save-on-leave rules: when a dirty source frame requires saving, what action/navigation is pending, and how success/failure transitions change boundary state. `PhysicsPaintStudio` performs the actual save effect.
- **D-11:** Failed saves become sticky boundary state: keep the source frame dirty, clear queued action/destination, record failed-save feedback, and require the next explicit user action/navigation to retry.
- **D-12:** The boundary owns restore intent after key actions or navigation: load real key, blank real key, clear blank, generated cleanup, deleted-frame cleanup, or cached-reference display. `PhysicsPaintStudio` executes the engine/canvas restoration.
- **D-13:** Preserve Phase 36.4 explicit dirty-close behavior. The boundary may expose whether the current Roto frame is dirty, but close-window prompt state and window-close adapter handling remain in `PhysicsPaintStudio` and must not change visibly or semantically.

### Regression Contracts
- **D-14:** Start implementation with failing boundary/controller unit tests before UI rewiring.
- **D-15:** Boundary tests must cover all roadmap-listed regression cases: inserted blank keys remain clean, deleted keys do not reappear, paste onto empty/generated targets creates clean real keys and clears generated/render-only state, duplicate behavior remains unchanged, dirty save-before-action preserves visible edits and cancels safely on save failure, and UI labels/placement remain unchanged.
- **D-16:** Treat the known Phase 36.7 pain points as the highest-priority failures inside the full regression set: blank keys becoming dirty/stale, deleted keys reappearing, paste to empty/generated retaining stale generated/cache state, and yellow/translucent cached canvas/reference state.
- **D-17:** Add a source-contract test that `PhysicsPaintStudio` consumes the compact Roto session object/effect descriptors and does not gain new broad `useEffect` orchestration for Roto key/cache coherence.
- **D-18:** Prove unchanged UI through source-contract assertions for labels/placement/props plus focused `PhysicsPaintWorkflowStrip` / `PhysicsPaintStudio` tests. Live UAT still validates the visible 36.7 flow after implementation.

### Claude's Discretion
No user decisions were delegated to Claude. Exact TypeScript names are illustrative where shown; downstream agents may choose final names if they preserve the locked ownership, action/effect, and test contracts above.

</decisions>

## Existing Code Insights

### Reusable Assets
- `app/src/components/physic-paint/physicsPaintRotoKeyController.ts` — Existing partial extraction with pure key utility transaction helpers, action-state derivation, local editable/preview map application, cleanup, and restore intent types. Phase 36.8 should preserve and fold this logic into the broader Roto session boundary.
- `app/src/components/physic-paint/physicsPaintRotoKeyController.test.ts` — Current regression tests already cover insert blank cleanup, delete cleanup, paste to generated/empty, duplicate behavior, dirty-save exposure, canvas-size normalization, and local state cleanup. Extend these into boundary tests rather than discarding them.
- `app/src/components/physic-paint/physicsPaintWorkflowState.ts` — Existing helpers for Roto cell semantics, real/generated key utility eligibility, interpolation spans, pending labels, and Roto source labels.
- `app/src/components/physic-paint/PhysicsPaintStudio.tsx` — Main adapter seam. It currently owns many Roto refs/states: `rotoFrameStatesRef`, `rotoPreviewFramesRef`, `copiedRotoKeyRef`, `dirtyRotoFramesRef`, pending save/navigation refs, key-action state, restore/cache lists, save-on-leave, save-before-action, and engine/bridge execution.
- `app/src/components/physic-paint/PhysicsPaintWorkflowStrip.tsx` — Existing UI seam for Roto cells, key utility pill, disabled states, status copy, Save current, and navigation callbacks. UI placement and labels must stay unchanged.
- `app/src/stores/physicPaintStore.ts` — Parent cache/store surface for real Roto keys, generated interpolation cache, Roto metadata, replace-key-frame payload application, and project dirty/version signaling.
- `app/src/types/physicPaint.ts` and `app/src/lib/physicPaintBridge.ts` — Typed payload/result surfaces for apply-canvas, delete-roto-frame, replace-roto-key-frames, and bridge events.

### Established Patterns
- This is a Preact project. Prefer Signals for shared/reactive boundary state and computed derived state, but keep pure transform logic directly tested.
- Cached Roto PNG remains durable truth; cached references are visual references, not guaranteed editable stroke restoration.
- Generated/interpolated Roto frames are render-only. They may be paste targets but not source/editable targets for Duplicate, Insert, Delete, or Copy.
- Dirty source frames save first before navigation or key actions. Failed save cancels the requested action/navigation and preserves dirty in-memory state.
- The Physics Paint workflow strip owns a compact, physics-paint-specific timeline lane. Do not import or redesign the main editor timeline for this phase.
- Existing visible UI should remain stable; this is an internal state boundary refactor.

### Integration Points
- `PhysicsPaintStudio.tsx` should instantiate/consume the new compact Roto session object, execute returned effect descriptors, and keep bridge/canvas/window concerns.
- `PhysicsPaintWorkflowStrip.tsx` should receive derived state/actions from the compact session object, directly or via narrow adapter props, without changing labels or placement.
- Existing `flushRotoFrame`, `saveRotoFrame`, `requestRotoFrameNavigation`, `runRotoKeyAction`, `runRotoKeyCopyAction`, and restore/cache-update branches are the main extraction candidates.
- Tests to extend include `physicsPaintRotoKeyController.test.ts`, `PhysicsPaintStudio.test.ts`, `PhysicsPaintWorkflowStrip.test.ts`, `physicsPaintWorkflowState.test.ts`, `physicPaintStore.test.ts`, and bridge/persistence tests if payload/result contracts are touched.

</code_context>

<specifics>
## Specific Ideas

- Build a boundary that looks like a future state machine without introducing XState yet.
- Named transaction methods and effect descriptors are preferred over a UI-facing generic dispatch.
- The existing `physicsPaintRotoKeyController.ts` is not throwaway; it is the seed for the new boundary.
- `PhysicsPaintStudio` should become thinner specifically around Roto key/cache/session coherence, but it remains the only place that directly talks to the canvas engine, bridge, and window APIs.

</specifics>

<deferred>
## Deferred Ideas

- Introducing XState or another formal state machine is deferred to Phase 36.9.
- Any user-visible UI changes, new controls, shortcut systems, interpolation UI, cached playback UI, Play/Roto conversion scope, or editor integration remain out of scope for Phase 36.8.

</deferred>
