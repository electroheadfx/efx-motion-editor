# Phase 12.9: Fix Sidebar Add-Layer Dialogs - Research

**Researched:** 2026-03-18
**Domain:** Preact UI refactoring -- replacing popover dialogs with full Imported view flows
**Confidence:** HIGH

## Summary

This phase replaces the current popover-based add-layer dialogs in `AddLayerMenu.tsx` with proper Imported view flows that reuse and extend the existing `ImportedView.tsx` and `ImportGrid.tsx` components. The core challenge is adding asset filter modes (images-only, videos-only, all) and layer-creation callbacks to the existing Imported view infrastructure, which currently only handles key photo selection.

The existing codebase already has all the building blocks: `ImportedView` supports single-select and multi-select with Confirm/Cancel, `ImportGrid` renders both image and video thumbnails, `uiStore.setEditorMode('imported')` switches the view, and `layerStore.add()` creates layers. The primary work is introducing new signal state to communicate the "purpose" of opening the Imported view (key photo selection vs layer creation) and the asset filter mode.

**Primary recommendation:** Add new signals to uiStore (`importedViewMode` for asset filtering and `addLayerCallback` or similar for the creation purpose) then refactor ImportedView to read these signals and conditionally filter assets, adjust headers, and dispatch layer creation on selection.

## Architecture Patterns

### Signal-based View Mode Communication

The core architectural decision is how the AddLayerMenu communicates its intent to ImportedView. The existing pattern uses `uiStore.editorMode` to switch views and `uiStore.pendingNewSequenceId` to communicate multi-select mode for key photos.

**Recommended pattern: New signals on uiStore**

```typescript
// New signals to add to uiStore
export type ImportedAssetFilter = 'all' | 'images' | 'videos';
export type ImportedPurpose = 'browse' | 'key-photo' | 'add-static-image' | 'add-image-sequence' | 'add-video';

const importedAssetFilter = signal<ImportedAssetFilter>('all');
const importedPurpose = signal<ImportedPurpose>('browse');
```

**Why signals over parameter passing:** The Imported view is rendered by EditorShell based on `editorMode`, not passed as a child of AddLayerMenu. There is no direct parent-child prop path. Signals maintain the existing architectural pattern (see `pendingNewSequenceId` as precedent).

### Flow Dispatch Pattern

Each AddLayerMenu button sets the appropriate signals, then calls `setEditorMode('imported')`:

```typescript
// Static Image: single-select, images only
importedAssetFilter.value = 'images';
importedPurpose.value = 'add-static-image';
uiStore.setEditorMode('imported');

// Image Sequence: multi-select, images only
importedAssetFilter.value = 'images';
importedPurpose.value = 'add-image-sequence';
uiStore.setEditorMode('imported');

// Video: single-select, videos only
importedAssetFilter.value = 'videos';
importedPurpose.value = 'add-video';
uiStore.setEditorMode('imported');
```

ImportedView derives its behavior from the purpose signal:

| `key-photo` | `images` | single | add key photo | -- | "Select a key photo" |
| `add-static-image` | `images` | single | create layer, close | -- | "Select image for layer" |
| `add-image-sequence` | `images` | multi | toggle | create layer, close | "Select images for sequence (N)" |

## Common Pitfalls

### Pitfall 1: Cancel-delete collision with key photo flow
**What goes wrong:** The existing `handleClose` in ImportedView deletes a pending new sequence (`sequenceStore.remove(pending)`). If add-layer mode opens while a pendingNewSequenceId is set, closing could accidentally delete the sequence.
**Why it happens:** Both key-photo multi-select and add-layer flows share the Imported view.
**How to avoid:** Check `importedPurpose` in the close handler. Only do cancel-delete when purpose is `key-photo` (or `browse` with `pendingNewSequenceId`). For add-layer purposes, close simply resets signals without touching sequences.
**Warning signs:** Sequences disappearing when canceling an add-layer operation.

### Pitfall 2: Stale signals on toolbar toggle
**What goes wrong:** The toolbar "Imported" button toggles `editorMode` between 'imported' and 'editor'. If user opens Imported view via add-layer flow then clicks the toolbar button to close, signals may not reset.
**Why it happens:** Toolbar doesn't know about add-layer signals.
**How to avoid:** Either intercept the toolbar toggle to reset add-layer signals, or have ImportedView watch for mode changes and reset on unmount.
**Warning signs:** Reopening Imported view via toolbar shows filtered assets from a previous add-layer flow.

### Pitfall 3: Video assets not appearing in grid
**What goes wrong:** Video assets are stored in `imageStore.videoAssets` separately from `imageStore.images`. The filter logic must handle both collections.
**Why it happens:** Images and videos are separate signals with different data structures.
**How to avoid:** When `assetFilter === 'videos'`, render only from `videoAssets`. When `assetFilter === 'images'`, render only from `images`. When `assetFilter === 'all'`, render both (current behavior).
**Warning signs:** Empty grid in video mode despite having imported videos.

### Pitfall 4: Multi-select state leak
**What goes wrong:** `selectedIds` state persists between add-layer sessions if not properly reset.
**Why it happens:** `useState` in ImportedView doesn't reset when signals change.
**How to avoid:** Reset `selectedIds` whenever `importedPurpose` changes. Use a `useEffect` watching the purpose signal.
**Warning signs:** Pre-selected images appearing when opening a new add-image-sequence flow.

### Pitfall 5: Missing bidirectional layer selection
**What goes wrong:** New layer is created but not visually selected in sidebar/canvas.
**Why it happens:** Forgetting to call both `layerStore.setSelected(id)` AND `uiStore.selectLayer(id)`.
**How to avoid:** Always use the bidirectional pattern established in Phase 11-04.
**Warning signs:** Layer appears in sidebar but no transform handles on canvas.

  const videos = imageStore.videoAssets.value;

  const showVideos = assetFilter === 'all' || assetFilter === 'videos';

      {showImages && images.length > 0 && (/* image grid */)}
      {showVideos && videos.length > 0 && (/* video grid with click handler */)}
    </div>
  );
}

## Open Questions

1. **Toolbar "Imported" button interaction with add-layer mode**
   - What we know: Toolbar toggles editorMode. If user opens via add-layer then clicks toolbar button, it will close back to editor.
   - What's unclear: Should the toolbar button be aware of add-layer mode? Should it show differently?
   - Recommendation: No special treatment needed. Closing via toolbar should also reset add-layer signals (handle in ImportedView cleanup/unmount). The toolbar button already works as a toggle.

2. **Video import asset registration timing**
   - What we know: Current AddLayerMenu imports video, copies file, registers asset, then creates layer all in one handler.
   - What's unclear: When importing from within the Imported view video mode, should the newly imported video be auto-selected to create the layer, or just appear in the grid for manual selection?
   - Recommendation: Import adds to grid, user then clicks to create layer. This matches the image import pattern where "+ Import" adds to the grid and the user selects.
