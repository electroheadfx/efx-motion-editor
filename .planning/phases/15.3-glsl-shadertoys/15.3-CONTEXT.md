# Phase 15.3: GLSL Shadertoys - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Add GLSL shader support with a built-in library of Shadertoy-ported shaders. Two shader categories for this phase: **GLSL Generator** (procedural content — fog, sea, abstract, clouds) that acts like a solid layer on the timeline, and **GLSL FX Image** (image filter — blur, distortion, effects) that uses content below as input texture. A dedicated shader browser window with DaVinci Resolve-style tabs lets users browse, preview, configure, and apply shaders. Shaders are embedded in app source code; a markdown spec documents how to add new ones. GLSL Transition is deferred to Phase 15.4.

</domain>

<decisions>
## Implementation Decisions

### Shader browser window
- **D-01:** Shader browser is a new window-level panel (same tier as Importer and Settings), opened via a dedicated GLSL button next to the existing "+ Layer" button in the timeline area
- **D-02:** GLSL button uses a Lucide icon (shader/sparkles-related) and is visually distinct from the "+ Layer" button
- **D-03:** Browser displays shaders as a grid of cards with live animated preview thumbnails (mini WebGL renders)
- **D-04:** Grid is scrollable with a custom scrollbar for growing library
- **D-05:** Three tabs at top — Generator / FX Image / Transition — styled as DaVinci Resolve pill buttons with small icons + text labels, dark background. Transition tab is visible but disabled/greyed for Phase 15.4
- **D-06:** Click a card to expand it — expanded view shows full-size animated preview + full parameter controls (sliders, color pickers, checkboxes). User configures desired settings before applying
- **D-07:** "Apply" button in expanded view adds the shader as a new layer on the timeline with the configured parameters
- **D-08:** Browser window closes after apply (not persistent like Importer)

### Shader types and timeline behavior
- **D-09:** GLSL Generator layers act like solid content on the timeline — they have opacity, blend mode, and render as procedural content (not dependent on layers below)
- **D-10:** GLSL FX Image layers act like adjustments — they take the composited content below them as an input texture (iChannel0) and output a filtered result
- **D-11:** Both types appear on timeline with purple color (`#8B5CF6`) to distinguish from CPU-based FX (pink generators, orange adjustments)

### AddFxMenu integration
- **D-12:** AddFxMenu gets a new "GLSL" section (after ADJUSTMENTS) that opens the shader browser window — not individual shader entries in the menu

### Shader parameter controls
- **D-13:** Each shader has explicit metadata defining exposed parameters — name, type (float/vec3/bool), display label, min/max range, default value, step. Parameters are NOT auto-discovered from GLSL code
- **D-14:** Control types: sliders with numeric input (float), color pickers (vec3/vec4 color), checkboxes (bool)
- **D-15:** All numeric shader parameters are keyframe-animatable via the existing `sourceOverrides` system in the keyframe engine
- **D-16:** User will be consulted per-shader to decide which parameters to expose and what ranges to use
- **D-17:** Full parameter panel appears both in the browser expanded view (pre-apply) and in the sidebar properties panel (post-apply)

### Sidebar properties
- **D-18:** When a GLSL layer is selected, sidebar shows shader name as header text, then parameter controls below — no thumbnail in sidebar

### GLSL rendering architecture
- **D-19:** GLSL shaders render via WebGL2 offscreen canvas (extending the glBlur.ts pattern), then composite onto the main Canvas 2D PreviewRenderer
- **D-20:** GLSL FX Image layers capture the current canvas state as a texture input (iChannel0) before applying the shader
- **D-21:** Standard Shadertoy uniforms supported: iTime (frame-based), iResolution, iMouse (optional), iChannel0 (for FX Image input)

### Extensibility (developer-facing)
- **D-22:** All shaders are embedded in app source code — no user-facing import or file scanning
- **D-23:** A markdown specification documents the pattern for adding new shaders: GLSL source, metadata structure, registration in the shader library, and integration checklist
- **D-24:** Architecture supports easy addition of new shaders by following the documented spec — a developer (or Claude) can add a new shader without modifying core rendering logic

### Claude's Discretion
- WebGL2 context management strategy (shared vs per-layer)
- Shader compilation caching and error handling
- Animated preview thumbnail implementation (render loop, resolution, frame rate)
- Export pipeline integration details
- Exact Lucide icon choice for GLSL button

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `glBlur.ts` WebGL2 infrastructure: shader compilation (`compileShader`), program linking, texture creation, FBO management, offscreen canvas pattern — directly extensible for GLSL shader rendering
- `createDefaultFxSource()` in `layer.ts`: pattern for registering new layer types with default parameters
- `sourceOverrides` in keyframe system: already supports dynamic numeric property animation — GLSL params plug into this
- `ColorPickerModal` (shared component from Phase 15.2): reusable for vec3/vec4 color params in shader controls

### Established Patterns
- FX layers use discriminated union on `LayerSourceData.type` — new GLSL types follow this pattern
- Generator layers are identified by `layer.type.startsWith('generator-')` — GLSL generators could use `generator-glsl-*` prefix or a new prefix
- PreviewRenderer switch-cases on layer source type for rendering — new GLSL types need branches here
- AddFxMenu creates layers via `sequenceStore.createFxSequence()` — same flow for GLSL layers

### Integration Points
- `PreviewRenderer.renderFrame()` — needs GLSL generator rendering branch (WebGL2 offscreen → drawImage composite) and GLSL FX Image branch (capture canvas → process → composite)
- `exportRenderer.ts` — must handle GLSL layers for export (same rendering path as preview)
- `TimelineRenderer.ts` — needs purple color for GLSL track bars
- `SidebarFxProperties.tsx` / `LeftPanel.tsx` — needs GLSL parameter panel routing
- Project format — new layer types need serialization/deserialization (likely v11)

</code_context>

<deferred>
## Deferred Ideas

- **GLSL Transition shaders** — Phase 15.4 (third tab in browser, currently disabled)
- **User-facing shader import** — Not planned; extensibility is developer-facing via documented spec
- **iMouse uniform support** — May add if specific shaders need interactive mouse input, but not required for initial set
- **Multi-pass shaders (Buffer A/B/C/D)** — Some complex Shadertoy shaders use multiple passes; evaluate per-shader during implementation

</deferred>
