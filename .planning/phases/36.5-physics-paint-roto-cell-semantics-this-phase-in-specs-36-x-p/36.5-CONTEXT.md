<domain>
## Phase Boundary

This phase is about Roto timeline cell semantics for Phase 36.5. It is not about Phase 36.4 close behavior.

The MVP should make frame states visible and trustworthy in the timeline grid:
- current frame
- empty/missing frame
- cached/saved frame
- dirty/unsaved frame
- background-only frame, only if already supported by the existing implementation

Out of scope for this phase:
- key utilities
- copy/paste
- interpolation
- cached playback
- save-on-leave behavior
- final full timeline redesign

</domain>

<decisions>
## Implementation Decisions

### D-01: Scope is Roto timeline cell semantics only
- [locked] Phase 36.5 must focus on Roto timeline cell semantics and must not include Phase 36.4 close behavior.

### D-02: MVP state vocabulary
- [locked] The MVP must make these frame states visible and trustworthy: current, empty/missing, cached/saved, dirty/unsaved, and background-only only if already supported by the current code.

### D-03: Visual reference
- [locked] Use `SPECS/36.x-phases/timeline-ui/timeline-bottom-window.html` and `SPECS/36.x-phases/timeline-ui/TimelineBottomWindow.tsx` as visual references for the spreadsheet-like square frame cells with different colors per state.

### D-04: Selected frame affordance
- [locked] The selected frame must visibly show the selected state as demonstrated in the timeline UI references.

### D-05: Explicit exclusions
- [locked] Do not include key utilities, copy/paste, interpolation, cached playback, save-on-leave, or final full timeline redesign in this phase.

### Claude's Discretion
- Implementation details for mapping existing timeline data to visible cell states.
- Exact CSS tokens and component boundaries, as long as they preserve the visual contract from the referenced specs and existing UI-SPEC.

</decisions>
