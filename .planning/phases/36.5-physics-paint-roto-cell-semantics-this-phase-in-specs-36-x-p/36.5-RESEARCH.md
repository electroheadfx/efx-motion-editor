## Summary

Phase 36.5 should be planned as a narrow UI/state-semantics slice in the existing Physics Paint workflow strip, not as a timeline redesign or close/save behavior phase. [CITED: .planning/phases/36.5-physics-paint-roto-cell-semantics-this-phase-in-specs-36-x-p/36.5-CONTEXT.md] The authoritative behavior contract is that each Roto cell has one base meaning (`empty`, `cached`, `editable/current`, `generated`, `background-only`) and dirty/pending work is an overlay that must not replace the base meaning. [CITED: SPECS/36.x-phases/phase-36.5-cell-semantics/spec-36.5-cell-semantics.md]

The implementation seam already exists: `physicsPaintWorkflowState.ts` owns pure state helpers such as `getRotoCellFill`, and `PhysicsPaintWorkflowStrip.tsx` renders `physics-paint-roto-cell` buttons from `frameCells`, `cachedRotoFrames`, `editableRotoFrames`, `pendingRotoFrames`, and `currentFrame`. [VERIFIED: codebase grep] The current helper only returns `empty`, `cached-only`, and `editable-session`; Phase 36.5 should extend/rename this vocabulary to the approved user-facing semantics while preserving the principle that current and dirty/pending are overlays. [VERIFIED: codebase grep]

The best plan split is RED/GREEN/REFACTOR around the pure cell classification seam, followed by execute-style UI/CSS tasks for the visible cell palette, non-color cues, legend/status copy, and source-contract tests. [VERIFIED: codebase grep] Do not plan package installation; the existing stack already contains Preact, Vitest, TypeScript, CSS, `lucide-preact`, and app-local source-contract tests. [VERIFIED: npm registry]

**Primary recommendation:** Extend `physicsPaintWorkflowState.ts` with a single normalized Roto cell view-model helper, then render that view model in `PhysicsPaintWorkflowStrip.tsx` using UI-SPEC colors, non-color cues, legend/status copy, and tests that prove current, empty/missing, cached/saved, dirty/unsaved, and background-only states remain distinguishable. [CITED: .planning/phases/36.5-physics-paint-roto-cell-semantics-this-phase-in-specs-36-x-p/36.5-UI-SPEC.md]

### Pattern 1: Pure View-Model Before UI Rendering
**What:** Derive a frame's base meaning, overlays, CSS class names, and accessible/status copy in a pure helper before rendering. [VERIFIED: codebase grep]

## Common Pitfalls

### Pitfall 1: Treating `generated` as just an interpolation connector
**What goes wrong:** Generated output is not visible as a cell state, so users cannot tell render-only output from empty/missing frames. [CITED: SPECS/36.x-phases/phase-36.5-cell-semantics/spec-36.5-cell-semantics.md]
**Why it happens:** Existing code renders interpolation spans as connector elements and filters generated frames out of real square cells. [VERIFIED: codebase grep]
**How to avoid:** Plan a narrow decision: if generated cache entries are present in `cachedRotoFrames`, show generated/render-only semantics without enabling editing. [VERIFIED: codebase grep]
**Warning signs:** Tests only assert `.physics-paint-roto-interpolation-connector` and do not assert generated cell/status copy. [VERIFIED: codebase grep]

### Pitfall 2: Background-only silently looks empty
**What goes wrong:** Intentional background/reference output appears as missing content. [CITED: SPECS/36.x-phases/phase-36.5-cell-semantics/spec-36.5-cell-semantics.md]
**Why it happens:** `PhysicPaintRotoCacheFrame` has `backgroundOnly?: boolean`, but the current `getRotoCellFill` helper ignores it. [VERIFIED: codebase grep]
**How to avoid:** Add helper tests where a cached frame with `backgroundOnly: true` maps to `background-only` if already supported by current data. [VERIFIED: codebase grep]
**Warning signs:** CSS has no hatch cue and no `Background only on frame {frame}` copy. [VERIFIED: codebase grep]

### Pitfall 3: Dirty/pending hides base meaning
**What goes wrong:** A dirty cached frame becomes visually just amber/dashed, and the user loses whether it was cached, editable, generated, or background-only. [CITED: .planning/phases/36.5-physics-paint-roto-cell-semantics-this-phase-in-specs-36-x-p/36.5-UI-SPEC.md]
**Why it happens:** Existing `.pending` only changes border style; Phase 36.5 adds dirty wedge/status copy and must preserve fill. [VERIFIED: codebase grep]
**How to avoid:** Use CSS pseudo-elements or child spans for overlays and test that helper output separates `baseMeaning` from `overlays`. [CITED: .planning/phases/36.5-physics-paint-roto-cell-semantics-this-phase-in-specs-36-x-p/36.5-UI-SPEC.md]
**Warning signs:** New tests expect a single state string such as `dirty` instead of `baseMeaning + overlays`. [ASSUMED]

### Pitfall 4: Visible copy drifts back to legacy/internal terms
**What goes wrong:** Users see ambiguous labels like `cached-only` or `editable-session`. [CITED: .planning/phases/36.5-physics-paint-roto-cell-semantics-this-phase-in-specs-36-x-p/36.5-UI-SPEC.md]
**Why it happens:** Current TypeScript names include `cached-only` and `editable-session`. [VERIFIED: codebase grep]
**How to avoid:** Keep legacy names internal only during refactor, or rename to user-approved terms; visible labels must use `Empty`, `Cached`, `Current`, `Generated`, `Background only`, `Unsaved`, and `Saving`. [CITED: .planning/phases/36.5-physics-paint-roto-cell-semantics-this-phase-in-specs-36-x-p/36.5-UI-SPEC.md]
**Warning signs:** `aria-label`, tooltip/title, status text, or legend contains `occupied`, `cached-only`, or `editable-session`. [CITED: .planning/phases/36.5-physics-paint-roto-cell-semantics-this-phase-in-specs-36-x-p/36.5-UI-SPEC.md]

## Open Questions (RESOLVED)

1. **Should generated interpolation cache entries be rendered as actual cells or only as non-editable connectors with visible status/legend?**
   - What we know: Phase 36.5 requires generated as a base meaning; existing code currently renders interpolation connector lines and filters generated frames from real square cells. [CITED: SPECS/36.x-phases/phase-36.5-cell-semantics/spec-36.5-cell-semantics.md] [VERIFIED: codebase grep]
   - RESOLVED: Use the narrow Phase 36.5 plan. Generated/render-only cache entries may be represented visibly when present in existing metadata, but they are non-editable. Execution must not introduce interpolation UI, cached playback, editing utilities, or any new cache-generation behavior.
   - Recommendation: Plan a RED test around `cachedRotoFrames` entries with `source: 'generated-interpolation'`; render them as render-only generated state if present, and prove they cannot use the ordinary editable click path without a render-only guard. [VERIFIED: codebase grep]

2. **Is `backgroundOnly` actively emitted in current flows or only typed for future support?**
   - What we know: `PhysicPaintRotoCacheFrame` and `PhysicPaintApplyCanvasPayload` expose `backgroundOnly?: boolean`. [VERIFIED: codebase grep]
   - RESOLVED: Support visible background-only semantics only when existing metadata supplies `backgroundOnly`. Do not add persistence/save behavior, new metadata producers, or any migration path for creating background-only cache entries.
   - Recommendation: Plan background-only as conditional support: if metadata is present, render hatch/copy; do not add new persistence behavior. [CITED: .planning/phases/36.5-physics-paint-roto-cell-semantics-this-phase-in-specs-36-x-p/36.5-CONTEXT.md]
