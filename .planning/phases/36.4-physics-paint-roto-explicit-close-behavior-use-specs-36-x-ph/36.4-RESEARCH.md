## Summary

Phase 36.4 should be a small vertical change inside the existing `PhysicsPaintStudio` Roto close path. The current code already has the needed primitives: dirty Roto tracking via `dirtyRotoFramesRef` plus `pendingRotoFrames`, a proven `Save current` path via `saveRotoFrame()` -> `flushRotoFrame()` -> `sendPhysicPaintApplyPayload()`, and Tauri close interception via `getCurrentWindow().onCloseRequested()`. [VERIFIED: codebase]

Today, close interception snapshots the current Roto frame but does not prevent close, does not flush the dirty frame, and does not present choices. This matches Phase 36.3's deliberate "discard unsaved close edits" behavior but is insufficient for Phase 36.4's explicit three-choice contract. [VERIFIED: codebase] [VERIFIED: spec]

**Primary recommendation:** Add a small Roto-only close guard in `PhysicsPaintStudio.tsx` that detects whether the current Roto frame is dirty, prevents the native close only when dirty, shows an in-studio modal with the three exact choices, and reuses `saveRotoFrame(null)` for `Close saving` before closing the Physics Paint window. [VERIFIED: codebase] [VERIFIED: spec]

## Current Behavior Findings

### Close / Save / Discard / Cancel Patterns

- Project-level unsaved changes use `guardUnsavedChanges()` with a native Tauri dialog offering Save, Don't Save, and Cancel; it returns `saved`, `discarded`, or `cancelled`. [VERIFIED: app/src/lib/unsavedGuard.ts]
- Physics Paint Studio currently listens for browser `beforeunload` and Tauri `appWindow.onCloseRequested`; both paths call `snapshotCurrentRotoFrame()` for Roto mode but do not prevent close and do not call `flushRotoFrame()`. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- Phase 36.3 tests explicitly assert the close/unload block contains `snapshotCurrentRotoFrame()`, does not contain `flushRotoFrame(currentFrame`, does not call `event.preventDefault()`, and does not call `await appWindow.close()`. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.test.ts]
- The visible Roto save action is `Save current`; it is disabled when the app is not ready, there are no pending Roto frames, or a Roto save is in flight. [VERIFIED: app/src/components/physic-paint/PhysicsPaintWorkflowStrip.tsx]
- The current close path does not expose `Close without saving`, `Cancel`, or `Close saving`; those labels are new Phase 36.4 UI copy from the spec. [VERIFIED: codebase] [VERIFIED: spec]

### Dirty / Unsaved Roto State Representation Today

- Dirty Roto frames are tracked in `PhysicsPaintStudio` with `dirtyRotoFramesRef = useRef<Set<number>>(new Set())`. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- The visible pending-state list is `pendingRotoFrames`, synchronized from `dirtyRotoFramesRef` by `syncPendingRotoFrames()`. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- `beginRotoFrameEdit()` calls `stopRotoCachedPlayback()` and `markCurrentRotoFrameDirty()` when canvas input starts in Roto mode. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- `markCurrentRotoFrameDirty()` adds `currentFrame` to `dirtyRotoFramesRef`, clears the cached reference overlay, resets background if available, and updates pending UI state. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- `snapshotCurrentRotoFrame()` captures the engine state and transient preview for the current frame but does not persist to the editor store by itself. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- `getRotoPendingLabel(true, false)` returns `Unsaved changes — click Save current to cache`; `getRotoPendingLabel(false, true)` returns `Saving current frame…`. [VERIFIED: app/src/components/physic-paint/physicsPaintWorkflowState.ts]

### Persistence / Save Current Helpers

- `saveRotoFrame()` gates on `readyToApply`, snapshots the current frame, marks `currentFrame` dirty, syncs pending state, then calls `flushRotoFrame(currentFrame, { force: true, advanceToFrame })`. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- `flushRotoFrame()` sends an `apply-canvas` payload containing `editableState` and `renderedFrame` for a persistable Roto frame, or a `delete-roto-frame` payload when the frame should not persist. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- `flushRotoFrame()` sets `applyStatus` to `applying`, sets `applyMessage` to `Saving current frame…`, sends through `sendPhysicPaintApplyPayload()`, deletes the frame from `dirtyRotoFramesRef` after send, and starts an apply-result timeout. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- `handleApplyResult()` changes a successful Roto apply to `applyStatus: success`, clears errors, removes the saved frame from dirty state, updates saved markers, and sets `applyMessage` to `Saved current frame`. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- The main editor applies `apply-canvas` through `applyPhysicPaintPayload()` -> `physicPaintStore.applyCanvas()`. [VERIFIED: app/src/lib/physicPaintBridge.ts]
- `physicPaintStore.applyCanvas()` stores `editableState`, upserts a real Roto key frame, bumps `physicPaintVersion`, and marks the project dirty through `_notifyVisualChange()`. [VERIFIED: app/src/stores/physicPaintStore.ts]
- Project serialization includes Physics Paint frames, editable state, play script ranges, Roto cache metadata, interpolation settings, and workflow metadata in `toMceOutputs()`. [VERIFIED: app/src/stores/physicPaintStore.ts]

### Lifecycle Dependencies for Safe Close

- Do not close during an active apply operation; `missingConditions` already treats `applyStatus === 'applying'` as not ready, and the Save current button uses `rotoSaveInFlight` to prevent duplicate save. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- Saving before close must wait for the main editor apply result, not merely for `sendPhysicPaintApplyPayload()` to resolve, because the send resolves before `handleApplyResult()` receives confirmation. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- The existing apply timeout is 5 seconds and sets an error if the main editor does not return a result. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
- The close guard should be Roto-only; Play close behavior and Play cache status are out of this phase. [VERIFIED: spec] [VERIFIED: user prompt]
- Dirty-close detection for the MVP should focus on the current Roto frame because the spec says "current Roto frame" and the phase goal is explicit close behavior for an unsaved Roto frame. [VERIFIED: spec]

## Recommended MVP Implementation

### Vertical Slice

1. Add a tiny close-decision state machine in `PhysicsPaintStudio.tsx`: `idle`, `prompt`, `saving`, `error`. [VERIFIED: codebase]
2. Compute `isCurrentRotoFrameDirty` from `workflowMode === 'roto' && dirtyRotoFramesRef.current.has(currentFrame)`, after calling `snapshotCurrentRotoFrame()` in the close request handler to catch current live canvas state. [VERIFIED: codebase]
3. In `appWindow.onCloseRequested`, allow default close when the current Roto frame is clean. [VERIFIED: spec]
4. When dirty, prevent the close request and show an in-studio modal with exactly three actions: `Close without saving`, `Cancel`, `Close saving`. [VERIFIED: spec]
5. `Close without saving`: clear the prompt guard and close the window without sending any apply payload; this preserves the prior cache unchanged. [VERIFIED: codebase]
6. `Cancel`: dismiss the prompt and leave the canvas/window untouched. [VERIFIED: spec]
7. `Close saving`: set saving feedback, call the existing Save current path, wait for successful apply result, then close the window. [VERIFIED: codebase] [VERIFIED: spec]
8. On apply error or timeout, keep the window open and show error feedback; do not get stuck in a disabled saving state. [VERIFIED: spec]

## Common Pitfalls

### Pitfall 1: Closing Immediately After Sending Save

**What goes wrong:** The window closes after `sendPhysicPaintApplyPayload()` resolves but before the main editor confirms the apply result. [VERIFIED: codebase]
**Why it happens:** The current `flushRotoFrame()` starts an async result timeout and receives success later in `handleApplyResult()`. [VERIFIED: codebase]
**How to avoid:** Close only after the matching successful apply result. [VERIFIED: codebase]
**Warning signs:** Tests assert only payload send, not apply-result-driven close. [ASSUMED]

### Pitfall 2: Prompting on Clean Close

**What goes wrong:** A clean Roto frame prompts unnecessarily. [VERIFIED: spec]
**Why it happens:** Checking `pendingRotoFrames.length` may include non-current frames; the spec targets the current Roto frame. [VERIFIED: spec] [VERIFIED: codebase]
**How to avoid:** Check `dirtyRotoFramesRef.current.has(currentFrame)` after snapshotting the current frame. [VERIFIED: codebase]
**Warning signs:** Closing a reopened cached frame shows the dialog before any edit. [VERIFIED: spec]

### Pitfall 3: Discard Accidentally Deletes Previous Cache

**What goes wrong:** `Close without saving` sends a delete or apply payload and changes the previous saved cache. [VERIFIED: spec]
**Why it happens:** Treating discard like clearing the active source instead of simply bypassing persistence. [VERIFIED: codebase]
**How to avoid:** Discard must close without calling `saveRotoFrame()`, `flushRotoFrame()`, `applyCanvas`, or `deleteRotoFrame`. [VERIFIED: codebase]
**Warning signs:** Reopening shows the dirty canvas result instead of the previous cached frame. [VERIFIED: spec]

### Pitfall 4: Getting Stuck in Saving State

**What goes wrong:** After save failure, timeout, or cancel, the module remains blocked. [VERIFIED: spec]
**Why it happens:** Close prompt state and apply state can diverge if the save result is not handled centrally. [ASSUMED]
**How to avoid:** Reset prompt/saving state on cancel, apply error, timeout, and unmount cleanup. [ASSUMED]
**Warning signs:** `Close saving` disables all choices permanently after a failed apply result. [VERIFIED: spec]
