# Phase 36.4: Code Review Report

**Reviewed:** 2026-06-20T08:36:39Z
**Depth:** standard
**Files Reviewed:** 4

## Summary

Reviewed the Phase 36.4 Physics Paint close-behavior changes across the Tauri capability, studio implementation, tests, and CSS. The explicit close flow is wired in the UI, but the Tauri capability grants the wrong window permission for the API being called, so the primary close path will fail in production Tauri contexts. The accompanying test codifies that wrong permission and will not catch the regression.

## Narrative Findings (AI reviewer)

## Critical Issues

### CR-01: Tauri close API is missing the required permission

**File:** `app/src-tauri/capabilities/default.json:10`
**Issue:** `PhysicsPaintStudio.closePhysicsPaintWindow()` calls `appWindow.close()` (`app/src/components/physic-paint/PhysicsPaintStudio.tsx:1167-1168`), but the capability adds `core:window:allow-destroy` instead of the permission for the `close` command. Tauri's generated ACL lists `allow-close` for the `close` command and `allow-destroy` for the separate `destroy` command. In a Tauri window this means the explicit “Close without saving” and post-save close path can be denied by capabilities; the browser fallback `window.close()` is not a reliable substitute for a Tauri app window.
**Fix:** Grant the permission matching the API call, or change the API call to match the granted permission. Prefer keeping the graceful close API and adding `core:window:allow-close`:
```json
"core:window:allow-set-fullscreen",
"core:window:allow-close",
"core:app:default",
```
If `allow-destroy` is not otherwise needed, remove it to avoid granting an unnecessary force-destroy capability.

## Warnings

### WR-01: Test asserts the wrong permission and masks the close regression

**File:** `app/src/components/physic-paint/PhysicsPaintStudio.test.ts:500`
**Issue:** The close-permission contract test expects `core:window:allow-destroy` while the implementation calls `appWindow.close()`. This makes the test pass for a capability that does not authorize the command under test, so CI will not catch the production close failure described in CR-01.
**Fix:** Update the assertion to require `core:window:allow-close` and, if desired, assert that `core:window:allow-destroy` is absent unless a `destroy()` call is intentionally introduced:
```ts
expect(capability.permissions).toContain('core:window:allow-close');
expect(capability.permissions).not.toContain('core:window:allow-destroy');
```
