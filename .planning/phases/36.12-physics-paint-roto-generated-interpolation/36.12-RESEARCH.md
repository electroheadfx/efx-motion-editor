### Locked Decisions
## Implementation Decisions

### Frame Placement
- **D-01:** When interpolation is enabled, generate every integer frame strictly between each adjacent pair of real Roto keys. Do not use a partial in-between count for the visible MVP.
- **D-02:** Generate interpolation for the whole Roto layer at once: every interior gap between every adjacent pair of real keys should be filled while interpolation is enabled.
- **D-03:** Generated interpolation may replace derived background-only support state for interior gap frames, but must keep paper/background composition separate. Generated frames draw interpolated paint alpha over paper/background; they must not become real editable keys and must not bake background into alpha caches.
- **D-04:** Real keys always win. If a generated frame would collide with an existing real key after rounding, layout changes, or regeneration, skip generated output for that frame and treat the real key as authoritative.

### Generated Look
- **D-05:** The MVP generated look is proportional alpha blending between neighboring real-key paint alpha caches.
- **D-06:** At each generated gap frame, blend the previous and next real-key alpha by normalized position within the gap. Do not use stepped hold behavior for this phase.
- **D-07:** Phase 36.12 exposes only the proportional alpha-blend interpolation behavior. Do not expose duplicate/hold, Move, Deform, or multiple interpolation modes in the visible MVP.
- **D-08:** Blend alpha only. Paper/background composition follows the existing Roto background and missing-frame rules from Phases 36.10/36.11, and generated cache output must not contain baked paper/background pixels.

### Refresh Timing
- **D-09:** While interpolation is enabled, source real-key changes auto-refresh generated frames immediately. Saves, repaint merges, paste, duplicate, insert, delete, replace, or any real-key layout/cache change should clear and regenerate dependent generated output from the current real-key layout.
- **D-10:** Interpolation is a persistent toggle. Once enabled, generated frames keep auto-refreshing until the user disables interpolation.
- **D-11:** If generated regeneration fails after a real-key change, preserve the real-key change, remove or ignore generated frames, and show compact status that interpolation could not regenerate. Never roll back or risk real-key edits because generated output failed.
- **D-12:** When interpolation is disabled, remove generated interpolation frames immediately and leave only real keys plus existing background/missing-frame rules.

### Controls and Feedback
- **D-13:** Use a simple visible Generate/Interpolation toggle in the existing Physics Paint workflow strip for UAT. Do not expose count, mode, or one-shot Generate controls in this phase.
- **D-14:** Generated cells must remain visually distinct and explicitly labeled as generated/render-only, with copy such as `Generated frame N — render-only.` Reuse existing generated-cell semantics where possible.
- **D-15:** Clicking a generated frame cell should preview/select that generated render-only frame without opening it for editing and without converting it into a real key.
- **D-16:** While a generated frame is preview-selected, allow navigation/timeline/playback only. Save current, paint editing, and key utilities must be disabled with render-only status copy.
- **D-17:** Keep UI changes minimal and inside the existing workflow strip/status surfaces. Final Pencil timeline layout and visual polish remain Phase 36.13.

## Common Pitfalls

### Pitfall 1: Rounded fractional spans overwrite or duplicate frames
**What goes wrong:** Count-based spans such as `1.5` are rounded to integer targets, producing collisions or skipped gaps. [VERIFIED: codebase]
**Why it happens:** Existing `getRotoInterpolationSpanFrames` uses `inBetweenCount` and fractional placement. [VERIFIED: codebase]
**How to avoid:** Replace with integer interior gap iteration: for real keys `a` and `b`, generate `a + 1` through `b - 1`; compute `t = (frame - a) / (b - a)`. [VERIFIED: CONTEXT.md]
**Warning signs:** Tests still expect frames like `1.5`, `1 + 1 / 3`, or `Math.round(span.frame)`. [VERIFIED: codebase]

### Pitfall 2: Generated cache bakes paper/background
**What goes wrong:** Generated PNG includes paper/background and later preview/export composes paper again. [VERIFIED: CONTEXT.md]
**Why it happens:** It is tempting to blend final visible composites instead of alpha paint caches. [VERIFIED: CONTEXT.md]
**How to avoid:** Decode/blend only real-key `dataUrl` alpha caches, leave background composition to existing Roto missing/background renderer rules. [VERIFIED: CONTEXT.md] [VERIFIED: codebase]
**Warning signs:** Generated data URL changes when only Roto background metadata changes. [VERIFIED: codebase]

### Pitfall 3: Real-key mutation clears generated frames but forgets regeneration
**What goes wrong:** Interpolation toggle stays enabled but generated frames disappear or become stale after Save current, repaint merge, paste, duplicate, insert, delete, or replace. [VERIFIED: CONTEXT.md]
**Why it happens:** Existing cleanup effects remove generated frames during real-key transactions. [VERIFIED: codebase]
**How to avoid:** Plan store-level regeneration after every authoritative real-key mutation path when settings are enabled, and update launch context/status after regeneration. [VERIFIED: codebase]
**Warning signs:** Tests only assert generated cleanup, not regenerated output after mutation. [VERIFIED: codebase]

### Pitfall 4: Generated selection opens an editable engine session
**What goes wrong:** Clicking a generated frame redirects to a nearest real key or opens generated data as editable without clear render-only feedback. [VERIFIED: codebase]
**Why it happens:** Prior generated semantics were launch redirects for generated-only frames rather than preview selection. [VERIFIED: STATE.md]
**How to avoid:** Separate preview-selected generated frame from editable current real key; disable Save current, paint editing, and key utilities while selected. [VERIFIED: CONTEXT.md]
**Warning signs:** `Save current` remains enabled or `apply-canvas` can target a generated selected frame. [VERIFIED: codebase]

### Pitfall 5: Version invalidation is missed
**What goes wrong:** Preview/export or project dirty state does not repaint after generated cache changes. [VERIFIED: memory]
**Why it happens:** Store mutations that bypass `_notifyVisualChange()` do not bump `physicPaintVersion`. [VERIFIED: codebase]
**How to avoid:** Route generation/removal through store methods that call `_notifyVisualChange()` exactly when cache/settings changed. [VERIFIED: codebase]
**Warning signs:** `physicPaintVersion.value` does not change after enabling/disabling interpolation or regenerating after a real-key edit. [VERIFIED: codebase]

### Integer interior span derivation
```typescript
// Source: Phase 36.12 context D-01/D-06 and app/src/components/physic-paint/physicsPaintWorkflowState.ts
function getIntegerRotoInterpolationSpanFrames(realKeys: number[]) {
  const keys = Array.from(new Set(realKeys.filter((frame) => Number.isInteger(frame) && frame >= 0))).sort((a, b) => a - b);
  const spans = [];
  for (let i = 0; i < keys.length - 1; i += 1) {
    const fromFrame = keys[i];
    const toFrame = keys[i + 1];
    for (let frame = fromFrame + 1; frame < toFrame; frame += 1) {
      spans.push({ fromFrame, toFrame, frame, t: (frame - fromFrame) / (toFrame - fromFrame) });
    }
  }
  return spans;
}
```

### Canvas alpha-cache blend
```typescript
// Source: MDN globalAlpha and drawImage docs
async function blendAlphaCaches(previous: CanvasImageSource, next: CanvasImageSource, width: number, height: number, t: number): Promise<string> {
  const canvas = document.createElement('canvas');
  canvas.width = width;
  canvas.height = height;
  const ctx = canvas.getContext('2d');
  if (!ctx) throw new Error('Could not create interpolation canvas.');
  ctx.clearRect(0, 0, width, height);
  ctx.globalAlpha = 1 - t;
  ctx.drawImage(previous, 0, 0, width, height);
  ctx.globalAlpha = t;
  ctx.drawImage(next, 0, 0, width, height);
  ctx.globalAlpha = 1;
  return canvas.toDataURL('image/png');
}
```
