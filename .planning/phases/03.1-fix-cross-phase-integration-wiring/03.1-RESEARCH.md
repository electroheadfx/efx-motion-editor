# Phase 3.1: Fix Cross-Phase Integration Wiring - Research

**Researched:** 2026-03-03
**Domain:** Rust/TypeScript cross-phase integration bug fixes (Tauri 2.0 + Preact Signals)
**Confidence:** HIGH

## Summary

Phase 3.1 addresses three integration gaps discovered by the v1.0 milestone audit. All three are wiring issues between code written in different phases -- not architectural problems. The fixes are small, surgical, and well-understood.

**Gap 1 (P0):** `image_pool.rs` creates thumbnails in `images/thumbs/` but `project_io.rs` expects `images/.thumbs/`. After temp-to-real project migration, thumbnails are orphaned. **Important discovery:** The committed HEAD code in `image_pool.rs` already uses `.thumbs` (matching `project_io.rs`), but there is an uncommitted working-copy change that reverted it to `thumbs`. The fix is to revert the uncommitted change (or ensure the committed `.thumbs` convention is preserved).

**Gap 2 (P1):** `autoSave.ts` subscribes to `projectStore` and `sequenceStore.sequences` signals but not `imageStore.images`. Image imports silently skip auto-save.

**Gap 3 (P1):** After opening a project, `hydrateFromMce()` sets `sequenceStore.activeSequenceId` but never syncs `uiStore.selectedSequenceId`, causing the SequenceList visual highlight to point to a stale ID.

**Primary recommendation:** Fix all three in a single plan with three focused tasks. Each fix is 1-5 lines of code. Revert the uncommitted `image_pool.rs` change, add `imageStore.images.value` to the auto-save effect, and add `uiStore.selectSequence()` calls in `hydrateFromMce()` and `closeProject()`.

### Pattern 1: Preact Signals Effect Subscription (Auto-Save)

**What:** `effect()` from `@preact/signals` automatically subscribes to any signal whose `.value` is read inside the callback. Adding `imageStore.images.value;` inside the effect body is sufficient to subscribe.

**When to use:** When you need reactive side-effects triggered by signal changes.

**Current code (autoSave.ts, line 26-35):**
```typescript
disposeEffect = effect(() => {
  // Access signals to subscribe
  projectStore.name.value;
  projectStore.fps.value;
  projectStore.width.value;
  projectStore.height.value;
  sequenceStore.sequences.value;
  // Missing: imageStore.images.value;
  scheduleSave();
});
```

**Fix pattern:** Add `imageStore.images.value;` to the effect body. The effect auto-subscribes.

### Pattern 2: Cross-Store Sync via batch()

**What:** `batch()` from `@preact/signals` groups multiple signal writes into a single update cycle. When syncing `sequenceStore.activeSequenceId` and `uiStore.selectedSequenceId`, both should be set inside the same `batch()` to prevent intermediate render states.

**When to use:** When updating related signals across stores that must remain consistent.

**Current code (projectStore.ts hydrateFromMce, lines 104-107):**
```typescript
// Set first sequence as active if any exist
if (sortedSeqs.length > 0) {
  sequenceStore.setActive(sortedSeqs[0].id);
  // Missing: uiStore.selectSequence(sortedSeqs[0].id);
}
```

**Fix pattern:** Add `uiStore.selectSequence()` call alongside `sequenceStore.setActive()`. Both are already inside a `batch()` block in `hydrateFromMce()`.

### Pattern 3: markDirty Callback (Avoiding Circular Imports)

**What:** `sequenceStore` uses a `_setMarkDirtyCallback` pattern to call `projectStore.markDirty()` without importing projectStore directly (which would create a circular dependency). `imageStore` does NOT have this pattern and does NOT call `markDirty()` on import.

**Relevance:** The auto-save fix (Gap 2) uses the signal subscription approach (reading `imageStore.images.value` in the effect) rather than the `markDirty` callback approach. Both would work, but the signal subscription is simpler and doesn't require adding a callback to imageStore. However, for completeness, the planner should also consider whether `imageStore.importFiles()` should call `markDirty()` to set `isDirty = true` immediately (not just via auto-save). Currently, `isDirty` is never set true after image import, so the periodic 60-second safety net also misses it.

### Anti-Patterns to Avoid
- **Fixing only the auto-save effect without setting isDirty:** If `imageStore.images.value` is added to the effect but `isDirty` is never set true on import, the 60-second interval save also won't trigger. Both the effect subscription AND `isDirty = true` on import are needed.
- **Adding markDirty callback to imageStore when simpler approach exists:** The effect subscription approach is preferred because imageStore already has no markDirty wiring and adding one creates coupling. But isDirty DOES need to be set.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Signal change detection | Custom diffing logic | `effect()` auto-subscription | Preact Signals handles dependency tracking automatically |
| Batched signal updates | Sequential `.value =` writes | `batch()` wrapper | Prevents intermediate renders and computed recalculations |

**Key insight:** Preact Signals' `effect()` auto-subscription model means adding a signal read is the entire fix for auto-save. No manual wiring needed.

## Common Pitfalls

### Pitfall 1: Fixing the Thumbnail Dir in Wrong Direction
**What goes wrong:** The audit says `image_pool.rs` uses `thumbs` and recommends changing to `.thumbs`. But in the committed HEAD, `image_pool.rs` already uses `.thumbs`. There is an uncommitted working-copy change that introduced the mismatch (changed `.thumbs` -> `thumbs`).
**Why it happens:** The audit was run against the working tree, not HEAD. The uncommitted change IS the bug.
**How to avoid:** Revert the uncommitted change to `image_pool.rs` (restore `.thumbs`). Do NOT change `project_io.rs` to match `thumbs`.
**Warning signs:** `git diff` shows `image_pool.rs` is modified.

### Pitfall 2: Auto-Save Effect Without isDirty Flag
**What goes wrong:** Adding `imageStore.images.value` to the effect triggers `scheduleSave()`, which checks `projectStore.isDirty.value`. If isDirty is false, the save is skipped even though images changed.
**Why it happens:** `scheduleSave()` has a guard: `if (projectStore.filePath.value && projectStore.isDirty.value)`. Image import never sets isDirty.
**How to avoid:** Either (a) call `projectStore.markDirty()` in `imageStore.importFiles()` after images are added, or (b) add a `_setMarkDirtyCallback` pattern to imageStore like sequenceStore has. Option (a) is simpler if circular import is not an issue. Option (b) mirrors the existing sequenceStore pattern.
**Warning signs:** Images import successfully but project doesn't auto-save and isDirty stays false.

### Pitfall 3: uiStore Not Reset on closeProject
**What goes wrong:** `closeProject()` resets `sequenceStore` and `imageStore` but never resets `uiStore`. The `selectedSequenceId` points to a stale ID after close.
**Why it happens:** `uiStore.reset()` is never called in `closeProject()`.
**How to avoid:** Add `uiStore.reset()` call in `projectStore.closeProject()`.
**Warning signs:** After closing a project and opening another, sequence highlight is wrong or stale.

### Pitfall 4: Circular Import Between imageStore and projectStore
**What goes wrong:** If imageStore imports projectStore to call `markDirty()`, and projectStore already imports imageStore, you get a circular dependency.
**Why it happens:** `projectStore.ts` already imports `imageStore` (line 5). If `imageStore.ts` also imports `projectStore`, the circular import may cause initialization order issues.
**How to avoid:** Use the same `_setMarkDirtyCallback` pattern that `sequenceStore` uses. Or rely solely on the auto-save effect subscription (but still need to set isDirty somehow).
**Warning signs:** Runtime error "Cannot access X before initialization" or undefined exports.

## Open Questions

1. **Should `stopAutoSave()` be called on project close?**
   - What we know: `stopAutoSave()` is exported from `autoSave.ts` but never called. The audit flagged this as tech debt (effect subscriptions accumulate).
   - What's unclear: Whether this causes real issues or is just wasteful. Multiple effect subscriptions to the same signals may trigger redundant saves.
   - Recommendation: Out of scope for this phase (it's a tech debt item, not a gap closure requirement). But the planner should note it for future cleanup. If fixing it is trivial (call `stopAutoSave()` in `closeProject()` and `startAutoSave()` in `openProject()`), it could be included as a bonus task.

2. **Are there existing projects with the wrong thumbnail directory?**
   - What we know: The uncommitted change suggests someone tested with `thumbs/` instead of `.thumbs/`. Any temp projects created during that session may have thumbnails in `thumbs/`.
   - What's unclear: Whether any persisted .mce files reference `images/thumbs/` paths.
   - Recommendation: The fix should be forward-looking (ensure `.thumbs` convention). Existing broken thumbnails would need manual re-import. This edge case is acceptable since no users have the app yet.
