# Phase 12.2: Auto-seek Timeline to Sequence Start on Sidebar Selection - Research

**Researched:** 2026-03-17
**Domain:** Preact signals wiring, timeline playhead control, sidebar UX
**Confidence:** HIGH

## Summary

This phase is a small, well-scoped wiring change. When the user clicks a sequence in the SEQUENCES sidebar panel, the timeline playhead should automatically move to that sequence's first global frame. All the infrastructure already exists: `trackLayouts` computed signal provides `startFrame` for each sequence, `playbackEngine.seekToFrame()` handles seeking with full sync (displayFrame, activeSequence, player), and `SequenceList.tsx` has a clear `handleSelect` callback where the seek call should be added.

The change is a single line addition to the `handleSelect` callback in `SequenceItem` component (`SequenceList.tsx`), plus a helper function to resolve a sequence ID to its global start frame using `trackLayouts`. No new stores, signals, types, or components are needed.

**Primary recommendation:** Add a `playbackEngine.seekToFrame(startFrame)` call inside `SequenceItem.handleSelect` in `SequenceList.tsx`, computing `startFrame` from `trackLayouts`.

## Architecture Patterns

### Existing Pattern: Seek-on-Selection

The project already has the exact pattern needed. When clicking a sequence track header on the timeline, `TimelineInteraction.ts` calls both `sequenceStore.setActive()` + `uiStore.selectSequence()` and then `playbackEngine.seekToFrame()`. The sidebar `handleSelect` does the same store calls but currently lacks the seek call.

**Reference implementation** (TimelineInteraction.ts, line 410-422):
```typescript
// Click-to-seek on track area + select content sequence
const trackIndex = this.trackIndexFromY(e.clientY);
const tracks = trackLayouts.peek();
if (trackIndex >= 0 && trackIndex < tracks.length) {
  const track = tracks[trackIndex];
  sequenceStore.setActive(track.sequenceId);
  uiStore.selectSequence(track.sequenceId);
  this.clearFxLayerSelection();
}
const frame = this.getFrame(e.clientX);
playbackEngine.seekToFrame(frame);
```

### Pattern: Sequence ID to Global Start Frame Resolution

The project has two existing approaches for this:

**Approach 1: `trackLayouts` lookup (preferred)**
```typescript
import { trackLayouts } from '../../lib/frameMap';

function getSequenceStartFrame(sequenceId: string): number {
  const layouts = trackLayouts.peek();
  const track = layouts.find(t => t.sequenceId === sequenceId);
  return track?.startFrame ?? 0;
}
```

This pattern is already used in `KeyframeNavBar.tsx` (line 30-41, but keyed by layerId) and in `frameMap.ts` `activeSequenceStartFrame` computed signal (line 39-45, but only for the active sequence). Neither does exactly what we need (lookup by arbitrary sequenceId), but both demonstrate the same underlying `trackLayouts.find()` approach.

**Approach 2: `activeSequenceStartFrame` computed signal**
```typescript
// Already exists in frameMap.ts - but only works AFTER setActive() has run
export const activeSequenceStartFrame = computed<number>(() => {
  const activeId = sequenceStore.activeSequenceId.value;
  if (!activeId) return 0;
  const tracks = trackLayouts.value;
  const track = tracks.find(t => t.sequenceId === activeId);
  return track?.startFrame ?? 0;
});
```

This won't work directly because `setActive()` must be called first, and the computed value updates reactively -- calling `.value` immediately after `setActive()` in the same synchronous block may or may not reflect the new value depending on signal batching. The direct `trackLayouts.peek()` lookup is safer and more explicit.

### Implementation Location

The change goes in `SequenceList.tsx`, inside the `handleSelect` callback of `SequenceItem` (line 118-126):

```typescript
const handleSelect = useCallback(() => {
  if (!editing) {
    layerStore.setSelected(null);
    uiStore.selectLayer(null);
    uiStore.selectSequence(seq.id);
    sequenceStore.setActive(seq.id);
    sequenceStore.clearKeyPhotoSelection();
    // Auto-seek timeline to sequence start
    const layouts = trackLayouts.peek();
    const track = layouts.find(t => t.sequenceId === seq.id);
    if (track) {
      playbackEngine.seekToFrame(track.startFrame);
    }
  }
}, [seq.id, editing]);
```

### Anti-Patterns to Avoid
- **Using `.value` instead of `.peek()` inside callbacks:** `.value` creates signal subscriptions in render context, `.peek()` reads without subscribing -- always use `.peek()` in event handlers
- **Using `activeSequenceStartFrame.value` after `setActive()`:** Signal computed values may not update synchronously in the same callback; use direct `trackLayouts.peek()` lookup instead
- **Seeking during playback:** The `playbackEngine.seekToFrame()` method already handles this correctly (it calls `timelineStore.seek()` which clamps), but consider whether to skip auto-seek if playback is running

## Common Pitfalls

### Pitfall 1: Re-selecting the already-active sequence
**What goes wrong:** User clicks the already-selected sequence; playhead jumps to start unexpectedly
**Why it happens:** `handleSelect` fires on every click regardless of current state
**How to avoid:** Guard with `if (seq.id !== sequenceStore.activeSequenceId.peek())` before seeking
**Warning signs:** Playhead jumps to start when user clicks the active sequence row for any reason

### Pitfall 2: Seeking during active playback
**What goes wrong:** User selects a sequence while playback is running, causing a jarring jump
**Why it happens:** `seekToFrame` works during playback but creates a discontinuity
**How to avoid:** Either: (a) accept this as expected behavior (matching timeline click-to-seek which also works during playback), or (b) gate with `if (!timelineStore.isPlaying.peek())`
**Warning signs:** Playhead teleports mid-playback

### Pitfall 3: Empty sequence (no key photos)
**What goes wrong:** Sequence with 0 key photos has no entry in `trackLayouts`, start frame lookup returns undefined
**Why it happens:** `trackLayouts` only includes sequences with frames; empty sequences have zero-length spans
**How to avoid:** The `if (track)` guard handles this -- if no track layout exists, skip the seek. This is correct behavior: an empty sequence has no meaningful start frame.
**Warning signs:** Error when clicking a newly created but still empty sequence

### Pitfall 4: FX sequence selection path
**What goes wrong:** This feature should only apply to content sequences, not FX sequences
**Why it happens:** FX sequences are already filtered out in `SequenceList` (`sequences.filter(s => s.kind !== 'fx')`), so this is not actually a risk. The `SequenceList` component only renders content sequences.
**How to avoid:** No action needed -- already handled by existing filtering.

## Code Examples

### Complete Implementation (verified from codebase analysis)

In `SequenceList.tsx`, add imports and modify `handleSelect`:

```typescript
// Add to existing imports:
import { trackLayouts } from '../../lib/frameMap';
import { playbackEngine } from '../../lib/playbackEngine';

// Inside SequenceItem, modify handleSelect:
const handleSelect = useCallback(() => {
  if (!editing) {
    layerStore.setSelected(null);
    uiStore.selectLayer(null);
    uiStore.selectSequence(seq.id);
    sequenceStore.setActive(seq.id);
    sequenceStore.clearKeyPhotoSelection();
    // Auto-seek timeline playhead to this sequence's first frame
    if (seq.id !== sequenceStore.activeSequenceId.peek()) {
      const layouts = trackLayouts.peek();
      const track = layouts.find(t => t.sequenceId === seq.id);
      if (track) {
        playbackEngine.seekToFrame(track.startFrame);
      }
    }
  }
}, [seq.id, editing]);
```

**Important ordering note:** The `activeSequenceId.peek()` check must happen BEFORE `sequenceStore.setActive(seq.id)` is called, since `setActive` changes the value. Alternatively, capture the old value first:

```typescript
const handleSelect = useCallback(() => {
  if (!editing) {
    const wasActive = sequenceStore.activeSequenceId.peek();
    layerStore.setSelected(null);
    uiStore.selectLayer(null);
    uiStore.selectSequence(seq.id);
    sequenceStore.setActive(seq.id);
    sequenceStore.clearKeyPhotoSelection();
    // Auto-seek only when switching to a different sequence
    if (seq.id !== wasActive) {
      const layouts = trackLayouts.peek();
      const track = layouts.find(t => t.sequenceId === seq.id);
      if (track) {
        playbackEngine.seekToFrame(track.startFrame);
      }
    }
  }
}, [seq.id, editing]);
```

### playbackEngine.seekToFrame internals (for reference)

```typescript
// From playbackEngine.ts - already handles all sync:
seekToFrame(frame: number) {
  timelineStore.seek(frame);        // clamp + set currentFrame
  timelineStore.syncDisplayFrame(); // sync displayFrame = currentFrame
  this.syncActiveSequence();        // update active sequence if frame is in different seq
  this.syncPlayer();                // sync motion-canvas player
}
```

Note: `syncActiveSequence()` inside `seekToFrame` will call `sequenceStore.setActive()` again if the frame is in a different sequence, but since we already called `setActive(seq.id)` with the same ID, this is a no-op. No circular issue.

## Open Questions

1. **Should re-selecting the already-active sequence re-seek to start?**
   - What we know: Timeline header clicks don't have this distinction (they always seek to click position)
   - What's unclear: User preference -- is re-seeking useful (quick "go to start") or annoying (accidental jumps)?
   - Recommendation: Skip seek on re-select (guard with `seq.id !== wasActive`). The user can always use the timeline to navigate. This avoids surprising jumps.

2. **Should auto-seek work during playback?**
   - What we know: Timeline click-to-seek works during playback (it overrides current position)
   - What's unclear: Whether switching sequences via sidebar during playback is a common workflow
   - Recommendation: Allow it -- match the timeline behavior. `seekToFrame` already handles this correctly.
