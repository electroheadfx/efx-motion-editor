# Phase 36.7 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope

Phase 36.7 adds the minimum visible Roto key utility controls required for real UAT: `Insert`, `Dup`, `Copy`, `Paste`, and `Delete` for real Roto keys. These controls must live in the existing Physics Paint workflow strip, visually matching the provided bottom timeline mock references.

This phase must not add interpolation UI, cached Roto playback UI, Play/Roto conversion UI, a new keyboard shortcut system, a new modal/toast surface, or a broad timeline redesign. The existing Roto cell semantics from Phase 36.5 and save-on-leave/save-before-action feedback from Phase 36.6 remain the visual foundation.

Source of truth: `SPECS/36.x-phases/phase-36.7-key-utilities/spec-36.7-key-utilities.md` and the referenced timeline mock files under `SPECS/36.x-phases/timeline-ui/`.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Key button gaps, inline control gaps, cell-to-pill alignment nudges |
| sm | 8px | Pill padding, status message spacing, compact group separation |
| md | 16px | Header/control group spacing where existing strip has room |
| lg | 24px | Roto cell height, playhead/current cell marker height, compact lane height |
| xl | 32px | Major workflow-strip grouping only; avoid inside the key utility pill |
| 2xl | 48px | Section-level spacing outside the compact timeline strip; not used by this phase |
| 3xl | 64px | Page-level spacing; not used by this phase |

Fixed component dimensions, not spacing tokens:

- Preserve existing Roto timeline cell density and mock-matching control proportions as component dimensions sourced from the referenced UI files. These dimensions are fixed visual measurements for timeline cells, the contextual lane, the pill shell, and the utility buttons; they are not part of the spacing scale.
- Implementation spacing inside and around the key utility group must use only the declared scale above: `4px`, `8px`, `16px`, `24px`, `32px`, `48px`, or `64px`.
- Use `4px` for the tightest button-to-button gap in the existing dense strip; use `8px` only where the current workflow strip has enough room.
- Use `8px` horizontal pill/button padding as the default. If the existing compact strip needs tighter alignment, reduce to `4px`; do not introduce intermediate spacing values.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Save current |
| Empty state heading | No Roto key selected |
| Empty state body | Select a real Roto key to insert, duplicate, copy, or delete. Paste is available after copying a key. |
| Error state | Roto key action could not finish. The unsaved frame is still open; save again or retry the action. |
| Destructive confirmation | Delete Roto key: Delete real Roto key at frame {frame}? Later real keys shift back one frame. This cannot be undone. |

Required visible labels and status copy:

| Scenario | Copy |
|----------|------|
| Context label | `Key {frame}` |
| Insert action label | `Insert` |
| Duplicate action label | `Dup` |
| Copy action label | `Copy` |
| Paste action label | `Paste` |
| Delete action label | `Delete` |
| Duplicate success | `Duplicated to frame {frame}.` |
| Insert success | `Inserted blank key before frame {frame}.` |
| Copy success | `Copied key {frame}.` |
| Paste success | `Pasted key to frame {frame}.` |
| Delete success | `Deleted key {frame}.` |
| Save-before-action pending | `Saving frame {frame} before {action}...` |
| Save-before-action failed | `Could not save frame {frame}; {action} was cancelled.` |
| Real key required | `Select a real Roto key to {action}.` |
| Paste unavailable | `Copy a real Roto key before pasting.` |
| Generated source blocked | `Generated frame {frame} is render-only.` |
| Action disabled by save | `Finish saving frame {frame} before using key tools.` |

Copy rules:

- Use `Dup` for the contextual button to match the UI mock, but use `Duplicated` in success feedback for clarity.
- Use `real Roto key` in unavailable/error copy when source eligibility matters.
- Keep status feedback in the existing workflow-strip status stack; do not add a toast, tutorial panel, or new modal for normal success/error feedback.
- Delete is the only destructive action in Phase 36.7. Use the existing compact confirmation pattern only if a confirmation is already established for destructive Roto actions; otherwise require a guarded destructive button state plus clear destructive copy in the workflow-strip status area before execution. Do not create a broad new modal system.

## Interaction Contract

### Placement and Layout

- Render the five controls as one contextual pill under or immediately adjacent to the Roto frame cells, matching the mock row that shows `Key 0`, `Insert`, `Dup`, `Copy`, `Paste`, `Delete`.
- The group appears only in Roto mode.
- The group must be horizontally aligned to the current/selected real key when practical. If the existing strip cannot position per-cell without layout churn, place the pill directly below the Roto cell lane at the current frame side of the visible lane.
- Do not add these actions to the top header action cluster as a second duplicate set. The top mock's `Duplicate` button is not the Phase 36.7 contextual contract; the required MVP controls are the contextual pill buttons.
- Preserve the Phase 36.5 legend/status stack. If vertical space is constrained, the key utility pill takes priority over adding any new legend variants because this phase is buttons-only UAT.

### Availability Rules

| Action | Enabled When | Disabled Feedback |
|--------|--------------|-------------------|
| Insert | Current selected frame is a real Roto key, app is ready, no save/action is in flight | `Select a real Roto key to insert.` |
| Dup | Current selected frame is a real Roto key, app is ready, no save/action is in flight | `Select a real Roto key to duplicate.` |
| Copy | Current selected frame is a real Roto key, app is ready, no save/action is in flight | `Select a real Roto key to copy.` |
| Paste | A real Roto key has been copied, app is ready, no save/action is in flight; target may be empty, generated, or real | `Copy a real Roto key before pasting.` |
| Delete | Current selected frame is a real Roto key, app is ready, no save/action is in flight | `Select a real Roto key to delete.` |

Generated/interpolated frames may be paste targets, but must never become editable sources for Insert, Dup, Copy, or Delete. Empty frames may be paste targets only after a copied key exists.

### Action Flow

| Action | Required Behavior | Required Feedback |
|--------|-------------------|-------------------|
| Insert | Save dirty current key first if needed, create a blank real key before the current selected real key, shift current/later real keys forward | Timeline cell layout updates immediately; status `Inserted blank key before frame {frame}.` |
| Dup | Save dirty current key first if needed, duplicate current real key to the next frame, shift later real keys forward | New real key cell appears at destination; status `Duplicated to frame {frame}.` |
| Copy | Save dirty current key first if needed, store current real key as clipboard source without changing timeline layout | Status `Copied key {frame}.`; Paste becomes enabled. |
| Paste | Create or replace a real key at the current target frame from copied source | Target cell becomes a real key; generated target loses render-only state; status `Pasted key to frame {frame}.` |
| Delete | Remove current real key; later real keys shift backward; deleting the last real key is allowed | Removed key cell no longer shows as real; status `Deleted key {frame}.` |

### Pending and Failure States

- During save-before-action, disable all five key utility buttons and show `Saving frame {frame} before {action}...` in the existing compact status area.
- During the actual key utility mutation after save success, keep all five buttons disabled until cache, timeline cell state, and preview state have completed the same visible update cycle.
- If save-before-action fails, cancel the requested key utility action, keep the dirty source frame visible as the unsaved truth, and show `Could not save frame {frame}; {action} was cancelled.`
- On action failure after a successful save, keep the latest durable cache visible and show `Roto key action could not finish. The unsaved frame is still open; save again or retry the action.`
- Do not clear the copied key on unrelated disabled states. Clear or replace copy state only when implementation source truth requires it, and surface Paste availability accordingly.

### Accessibility and Hit Targets

- Each button must be a real `<button type="button">` with an accessible label matching its visible label plus frame context where useful, for example `Duplicate Roto key at frame {frame}`.
- Disabled buttons must use the native `disabled` attribute and keep a visible disabled style.
- The contextual group must have an accessible label such as `Roto key utilities for frame {frame}`.
- Focus order inside the pill: context label skipped if non-interactive, then `Insert`, `Dup`, `Copy`, `Paste`, `Delete`.
- Keyboard activation is normal button activation only. Do not introduce new global shortcuts in this phase.
