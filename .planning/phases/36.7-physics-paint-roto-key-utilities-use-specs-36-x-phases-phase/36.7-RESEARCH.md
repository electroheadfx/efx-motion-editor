## Implementation Decisions

### Placement Rules
- **D-01:** Duplicate creates the new real Roto key on the next frame and shifts later real Roto keys forward. This is a splice-like timeline operation, not a “next empty only” operation.
- **D-02:** Insert creates a blank real Roto key before the current selected real key and shifts the current/later real Roto keys forward.
- **D-03:** Delete removes the current real Roto key and shifts later real Roto keys backward to close the gap.
- **D-04:** Splice actions shift only real Roto keys. Generated/interpolated frames must not be treated as editable keys; they should be regenerated or refreshed from the updated real-key layout.

### Target Eligibility
- **D-05:** Duplicate, Copy, Delete, and Insert require the current selected frame to be a real Roto key. Generated in-betweens and empty frames are not valid sources/splice anchors for those actions.
- **D-06:** Paste may create or replace a real Roto key at the current target frame, including an empty frame or generated frame. This makes generated frames valid paste targets but not editable/generated sources.
- **D-07:** Delete may remove the last remaining real Roto key for the layer. The layer may return to having no real Roto keys if the animator chooses that action.

### Dirty State Handling
- **D-08:** If the current real key is dirty and the animator triggers Duplicate, Copy, Insert, or Delete, save the current frame first through the existing Roto save path, then run the requested key utility action using the saved result.
- **D-09:** If the save-before-action fails, cancel the requested key utility action. Leave the dirty frame visible/in memory as the unsaved truth and surface failure feedback for retry.
- **D-10:** After a successful Duplicate, Insert, Delete, or Paste, the affected key layout is already saved/clean in the parent cache. The action itself must update cache, timeline cell state, and preview state immediately.

### UI and Feedback
- **D-11:** The five key utility controls must use the placement defined by `SPECS/36.x-phases/phase-36.7-key-utilities/spec-36.7-key-utilities.md` and its referenced UI assets/specs. Do not invent a new placement. If an implementation fallback is needed, use the spec-defined contextual Roto key utility group near the Roto timeline controls.
- **D-12:** The referenced UI assets show a contextual pill under the frame cells labeled like `Key 0` with `Insert`, `Dup`, `Copy`, `Paste`, and destructive `Delete`. Treat that as the visual source for placement and grouping unless the Phase 36.7 spec says otherwise.
- **D-13:** Disabled-state feedback must follow the Phase 36.7 spec and referenced UI assets/specs. If they do not define a more specific pattern, keep the five buttons visible but disabled and reuse the existing compact workflow-strip status/message area to explain why the action is unavailable, such as real key required, no copied key, save in progress, or save failed.
- **D-14:** Phase 36.7 is buttons-only for the five utilities. Do not add new keyboard shortcuts unless the source spec explicitly requires them.
- **D-15:** Success feedback must follow the Phase 36.7 spec and referenced UI assets/specs. If they do not define a more specific pattern, update timeline cell state and preview immediately and show compact workflow-strip status copy naming the action and affected frame, such as “Duplicated to frame 12” or “Deleted frame 12”. Do not add a new modal or unrelated toast surface for MVP UAT.

## Common Pitfalls

### Pitfall 1: Paste Is Broader Than Other Actions
**What goes wrong:** Paste gets incorrectly gated by `requireCurrentRealRotoKey()`, preventing paste onto empty/generated frames. [VERIFIED: codebase grep]
**Why it happens:** Existing provisional `pasteRotoFrame` currently calls `requireCurrentRealRotoKey()`. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
**How to avoid:** Plan a separate Paste eligibility path: copied key required, app ready, no save/action in flight, target frame may be empty/generated/real. [VERIFIED: 36.7-CONTEXT.md]
**Warning signs:** UAT cannot paste onto a generated in-between or empty frame after copying a key. [VERIFIED: 36.7-UI-SPEC.md]

### Pitfall 2: Save-Before-Action Missing From Utility Buttons
**What goes wrong:** Duplicate/Copy/Insert/Delete operate on stale cached state while the visible source has unsaved edits. [VERIFIED: 36.7-CONTEXT.md]
**Why it happens:** Existing utility callbacks are synchronous and directly mutate refs/store after eligibility checks. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
**How to avoid:** Plan an async action wrapper that snapshots/saves dirty current source first, disables all five tools during the save/action, then mutates only after save success. [VERIFIED: 36.7-CONTEXT.md]
**Warning signs:** Status never shows `Saving frame {frame} before {action}...`, or save failure still shifts/deletes keys. [VERIFIED: 36.7-UI-SPEC.md]

### Pitfall 3: Deleting/Splicing Store Data Inconsistently
**What goes wrong:** Frame markers shift but cached frame payloads or editable states remain at old frame numbers. [VERIFIED: codebase grep]
**Why it happens:** Existing callbacks shift `rotoFrameStatesRef` but only remove a frame range on delete; duplicate/insert do not obviously shift store frames in the same visible way. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
**How to avoid:** Plan explicit cache/store mutation tests or assertions for unrelated real keys, shifted real keys, generated cache regeneration, and preview state after each action. [VERIFIED: 36.7-CONTEXT.md]
**Warning signs:** Cell states update but preview shows the previous key image, or unrelated real keys disappear after a splice. [VERIFIED: SPECS/36.x-phases/phase-36.7-key-utilities/spec-36.7-key-utilities.md]

### Pitfall 4: UI Placement Drift
**What goes wrong:** Buttons are added to the header, duplicated in multiple places, or styled like a broad toolbar rather than the contextual `Key {frame}` pill. [VERIFIED: 36.7-UI-SPEC.md]
**Why it happens:** The mock contains a top `Duplicate` button, but the UI contract says the contextual pill is the Phase 36.7 MVP control surface. [VERIFIED: SPECS/36.x-phases/timeline-ui/TimelineBottomWindow.tsx] [VERIFIED: 36.7-UI-SPEC.md]
**How to avoid:** Plan UI work against the contextual row under/immediately adjacent to Roto cells, using existing CSS classes and fixed labels. [VERIFIED: 36.7-UI-SPEC.md]
**Warning signs:** A new top-header action cluster appears, labels differ from `Insert/Dup/Copy/Paste/Delete`, or delete loses destructive styling. [VERIFIED: 36.7-UI-SPEC.md]

1. **RESOLVED — Delete does not require a new confirmation flow in this codebase for Phase 36.7.**
   - Evidence: `PhysicsPaintWorkflowStrip.tsx` has an existing `confirmation` state and `.physics-paint-confirmation` dialog only for `convert-play-to-roto` and `convert-roto-to-play`; `PhysicsPaintStudio.tsx` has a separate `.physics-paint-roto-close-confirmation` only for closing an unsaved Roto frame.    - Resolution: Phase 36.7 Delete should use the UI-SPEC's fallback: a guarded destructive button with real-key eligibility, native disabled state, destructive styling, compact status feedback, and immediate execution when enabled. Do not add a new modal/confirmation system for Delete. [VERIFIED: 36.7-UI-SPEC.md]

2. **RESOLVED — Store shifts must be composed from existing real-key APIs; there is no single existing shift helper.**
   - Evidence: `physicPaintStore` exposes `getFrame`, `getFrames`, `getRotoCacheFrames`, `getRealRotoKeyFrames`, `upsertRealRotoKeyFrame`, `removeRealRotoKeyFrame`, `removeFrameRange`, `setFrame`, and `regenerateRotoInterpolationCache`, but no shift/splice helper for cached real-key payloads. [VERIFIED: app/src/stores/physicPaintStore.ts]
   - Evidence: `setFrame` updates rendered frame storage only; it does not update Roto cache metadata. `upsertRealRotoKeyFrame` updates both rendered frame storage and real-key metadata, while `removeRealRotoKeyFrame` removes both. [VERIFIED: app/src/stores/physicPaintStore.ts]
   - Resolution: Studio implementation should shift cached real-key payloads by reading existing payloads with `getFrame`, moving them with `upsertRealRotoKeyFrame` in overwrite-safe order, removing vacated keys with `removeRealRotoKeyFrame`, then calling `regenerateRotoInterpolationCache`. Duplicate/Insert should shift later real keys in descending order; Delete should shift later real keys in ascending order. Paste should use `upsertRealRotoKeyFrame`, not plain `setFrame`, so the target becomes real-key cache truth. No store helper/file modification is required unless implementation finds the composition too brittle during execution. [VERIFIED: app/src/stores/physicPaintStore.ts] [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
