<objective>
Close the Phase 36.7 failed-UAT gap by repairing the Roto key utility state model so Insert, Delete, and Paste keep active frame, real-key cache, generated/render-only frames, background/reference overlays, canvas contents, and parent persistence coherent.

Purpose: Live UAT proved automated helper/source-contract tests were insufficient: Insert/Delete/Paste can leave stale yellow/translucent reference or generated cache paths visible even when timeline keys should be clean. This plan requires a failing regression proof first, then a small controller/store boundary so Studio no longer owns key utility cache truth through scattered ad-hoc mutations, refs, or new broad effects.
Output: A tested `physicsPaintRotoKeyController.ts` controller/store boundary and narrow Studio adapter integration that preserve the existing UI while making key utility transactions coherent end-to-end.

<architecture_constraints>
- Do not add new broad `useEffect` branches inside `PhysicsPaintStudio.tsx` for this repair.
- Do not solve Insert/Delete/Paste coherence with more scattered local `useState`/ref mutations inside `PhysicsPaintStudio.tsx`.
- Move Roto key utility state/cache ownership into the controller/store boundary so `PhysicsPaintStudio.tsx` keeps only an adapter role.
- The boundary must expose derived action state and/or transaction results for Insert, Delete, Paste, Duplicate, Copy, dirty-save-before-action, restore intent, generated-target cleanup, and deleted-frame cleanup.
- Use a Preact-native reactive boundary, preferably `@preact/signals`, if state must live outside `PhysicsPaintStudio.tsx`. If Signals are not used, the executor must explicitly justify in `36.7-05-SUMMARY.md` why the controller remains pure/stateless and how that still prevents new `useEffect`/ref sprawl.
- Keep bridge/store persistence, engine canvas restore, and UI prop wiring in `PhysicsPaintStudio.tsx` only where they are true boundary concerns.
- Keep UI unchanged and keep scope limited to the Phase 36.7 key utility/cache coherence gap.
</architecture_constraints>
</objective>
