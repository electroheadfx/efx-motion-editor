<task type="auto">
  <name>Task 1: Make Rust FFmpeg encoding async with spawn_blocking</name>
  <files>Application/src-tauri/src/commands/export.rs, Application/src-tauri/src/services/ffmpeg.rs</files>
  <action>
    **Root cause:** `export_encode_video` is `pub fn` (synchronous Tauri command). When it calls `Command::new().output()` via `ffmpeg::encode_video`, it blocks the Tauri main thread. The WebView cannot process IPC responses while the main thread is blocked, causing the app to freeze indefinitely.

    **Fix in `Application/src-tauri/src/commands/export.rs`:**
    1. Change `export_encode_video` from `pub fn` to `pub async fn`
    2. Clone all String parameters before moving into the spawn_blocking closure
    3. Use `tokio::task::spawn_blocking` to run the FFmpeg encode on a background thread
    4. Await the result and flatten the nested Result (spawn_blocking returns Result<Result<(), String>, JoinError>)

    The updated signature:
    ```rust
    #[command]
    pub async fn export_encode_video(
        png_dir: String,
        glob_pattern: String,
        output_path: String,
        codec: String,
        fps: u32,
        h264_crf: u32,
        av1_crf: u32,
        prores_profile: String,
        audio_path: Option<String>,
    ) -> Result<(), String> {
        let quality = ffmpeg::VideoQualityArgs {
            h264_crf,
            av1_crf,
            prores_profile,
        };
        tokio::task::spawn_blocking(move || {
            ffmpeg::encode_video(
                &png_dir, &glob_pattern, &output_path, &codec, fps, &quality,
                audio_path.as_deref(),
            )
        })
        .await
        .map_err(|e| format!("FFmpeg task panicked: {e}"))?
    }
    ```

    **No changes needed to ffmpeg.rs** — the encode_video function itself stays synchronous (it runs blocking I/O which is correct for a spawn_blocking context). The key fix is moving the blocking call off the main thread.

    **No changes needed to TypeScript IPC** — `safeInvoke` already awaits the Tauri invoke promise. Making the Rust command async means Tauri returns the IPC response when the background task completes instead of blocking the main thread.

<task type="auto">
  <name>Task 2: Add cancel check and timeout to audio pre-render in exportEngine</name>
  <files>Application/src/lib/exportEngine.ts, Application/src/lib/audioExportMixer.ts</files>
  <action>
    **Problem 1:** exportEngine.ts has no cancel check between the frame rendering loop (line 116-164) and the audio pre-render block (line 182-203). If the user clicks Cancel during frame rendering, the cancel flag is set but audio pre-render starts anyway and blocks forever.

    **Fix in `Application/src/lib/exportEngine.ts`:**
    1. Add a cancel check immediately before the audio pre-render block (before line 186). If cancelled, set status to 'cancelled' and return early:
    ```typescript
    // Check cancel before audio pre-render
    if (exportStore.isCancelled()) {
      exportStore.updateProgress({
        status: 'cancelled',
        resumeFromFrame: total, // frames done, audio step skipped
      });
      return;
    }
    ```
    2. Pass an AbortSignal to renderMixedAudio so it can be aborted mid-render:
    ```typescript
    const abortController = new AbortController();
    // Wire cancel to abort
    const cancelCheckInterval = setInterval(() => {
      if (exportStore.isCancelled()) abortController.abort();
    }, 200);
    try {
      const wavData = await renderMixedAudio(
        audioStore.tracks.peek(),
        projectStore.fps.peek(),
        totalDurationSec,
        abortController.signal,
      );
      audioWavPath = `${exportDir}/audio_mix.wav`;
      await writeFile(audioWavPath, new Uint8Array(wavData));
    } catch (err) {
      if (abortController.signal.aborted) {
        exportStore.updateProgress({ status: 'cancelled' });
        return;
      }
      console.error('Audio pre-render failed:', err);
      audioWavPath = null;
    } finally {
      clearInterval(cancelCheckInterval);
    }
    ```

    **Fix in `Application/src/lib/audioExportMixer.ts`:**
    1. Add optional `signal?: AbortSignal` parameter to `renderMixedAudio`
    2. After creating the OfflineAudioContext and scheduling all sources but before calling `offline.startRendering()`, check if already aborted:
    ```typescript
    if (signal?.aborted) throw new DOMException('Aborted', 'AbortError');
    ```
    3. Wrap `offline.startRendering()` in a `Promise.race` with an abort listener that rejects on abort:
    ```typescript
    const renderedBuffer = await Promise.race([
      offline.startRendering(),
      new Promise<never>((_, reject) => {
        if (!signal) return; // no signal = no timeout, just render
        const onAbort = () => reject(new DOMException('Aborted', 'AbortError'));
        signal.addEventListener('abort', onAbort, { once: true });
      }),
    ]);
    ```
    4. Also add a 60-second timeout safety net in case OfflineAudioContext hangs for unknown reasons:
    ```typescript
    const timeoutMs = 60_000;
    const renderedBuffer = await Promise.race([
      offline.startRendering(),
      new Promise<never>((_, reject) => {
        const timer = setTimeout(() => reject(new Error('Audio pre-render timed out (60s)')), timeoutMs);
        if (signal) {
          signal.addEventListener('abort', () => {
            clearTimeout(timer);
            reject(new DOMException('Aborted', 'AbortError'));
          }, { once: true });
        }
        // Keep the timeout alive even without signal
      }),
    ]);
    ```

    The updated renderMixedAudio signature:

    ```typescript
      tracks: AudioTrack[],
      fps: number,
      totalDurationSec: number,
      signal?: AbortSignal,
    ): Promise<ArrayBuffer>
    ```
