# Phase 15.4: GL Transition - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Add GLSL-powered transition effects between content sequences. Enable the disabled "Transition" tab in the shader browser (built in Phase 15.3), implement a dual-texture GPU rendering pipeline using the gl-transitions.com convention, and ship 15-20 curated transition shaders. GL transitions replace the existing cross-dissolve between a given sequence pair. Preview and export both supported from day one.

</domain>

<decisions>
## Implementation Decisions

### Transition application flow
- **D-01:** The Transition tab in the shader browser (disabled since Phase 15.3) becomes the entry point for applying GL transitions. User browses, configures, and clicks Apply
- **D-02:** GL transition replaces cross-dissolve — a sequence pair has ONE transition type: either a cross-dissolve or a GL shader transition, not both simultaneously
- **D-03:** Apply targets the currently active sequence and the next adjacent sequence. Apply button is disabled if there is no next sequence
- **D-04:** TransitionProperties sidebar panel handles post-apply editing: shows shader name, parameter controls, duration, and a Remove button. Clicking the shader name reopens the browser to swap the transition

### GL rendering pipeline
- **D-05:** Dual offscreen capture — render outgoing sequence to offscreen canvas A, incoming sequence to offscreen canvas B, upload both as WebGL2 textures (`from`/`to`), run the GL transition shader, composite the result onto the main canvas
- **D-06:** Use the gl-transitions.com GLSL convention: `vec4 transition(vec2 uv)` function with `uniform float progress` (0-1), `uniform sampler2D from`, `uniform sampler2D to`. This matches the open-source library format directly
- **D-07:** Progress is frame-based: `progress = framesIntoOverlap / overlapDuration`, same timing mechanism as current cross-dissolve. The existing easing curve is applied to the progress value before passing to the shader
- **D-08:** GL transitions render identically in preview and export — exportRenderer captures both sequences to offscreen canvases and runs the GL shader, same path as preview

### Transition parameters
- **D-09:** GL transitions reuse the existing crossDissolve duration/overlap mechanism — the overlap region on the timeline defines where the transition plays
- **D-10:** Shader parameters (direction, color, segments, etc.) are static, not keyframe-animatable. The `progress` uniform already provides the animation over time
- **D-11:** Parameters configurable in two places: browser expanded card view (pre-apply, same as GLSL FX pattern from Phase 15.3) and TransitionProperties sidebar panel (post-apply editing)

### Shader library
- **D-12:** Primary source is gl-transitions.com — MIT licensed, already in the correct GLSL format (`transition()` + `progress` uniform). Minimal porting needed
- **D-13:** Ship 15-20 curated transition shaders covering variety: directional wipes, dissolve variants, slides, zooms, and creative/artistic effects
- **D-14:** Browser Transition tab shows animated looping previews using two built-in sample images (e.g., a gradient and a photo), showing exactly what each transition looks like

### Claude's Discretion
- Specific transition selection from gl-transitions.com (15-20 covering wipes, dissolves, slides, zooms, creative effects)
- WebGL2 context management strategy for dual-texture rendering (extend glslRuntime.ts or separate module)
- Built-in sample image selection for browser preview thumbnails
- Data model changes for storing GL transition reference on sequences (extend existing Transition type or new field)
- Timeline visual indicator distinguishing GL transition zones from cross-dissolve zones
- Integration details with exportRenderer's existing cross-dissolve code path
- Shader compilation caching for transition shaders

</decisions>

## Existing Code Insights

### Reusable Assets
- `glslRuntime.ts` WebGL2 runtime: shared offscreen context, shader compilation/caching, fullscreen quad VAO, texture upload — extend for dual-texture transition rendering
- `shaderLibrary.ts` registry: `ShaderCategory` already includes `'transition'`, `ShaderDefinition` and `ShaderParamDef` types ready for transition shaders
- `ShaderBrowser` component (Phase 15.3): 3-tab layout with disabled Transition tab — enable it
- `computeCrossDissolveOpacity()` in `transitionEngine.ts`: timing/easing logic reusable for GL progress calculation
- `ColorPickerPopover` (Phase 15.2): reusable for color-type transition params

### Established Patterns
- GLSL shaders follow the `ShaderDefinition` metadata pattern: id, name, category, fragmentSource, params array
- Shader browser expanded card view: live preview + parameter controls + Apply button (Phase 15.3 D-06/D-07)
- Browser closes after apply (Phase 15.3 D-08)
- FX layers use discriminated union on `LayerSourceData.type` — transitions use a different model (on Sequence, not Layer)
- Cross-dissolve stores config on `Sequence.crossDissolve` field

### Integration Points
- `exportRenderer.renderGlobalFrame()` — needs new GL transition rendering branch alongside existing cross-dissolve opacity branch
- `frameMap` computed signal — needs to expose GL transition overlap data
- `TransitionProperties.tsx` — needs GL shader parameter controls (sliders, color pickers)
- `ShaderBrowser` Transition tab — enable, populate with transition shaders, wire Apply to sequence transition
- Timeline renderer — visual indicator for GL transition zones
- Project format — GL transition references need serialization (likely new fields on Sequence)

</code_context>
