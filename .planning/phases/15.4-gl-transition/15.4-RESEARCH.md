# Phase 15.4: GL Transition - Research

**Researched:** 2026-03-23
**Domain:** WebGL2 dual-texture transition rendering, gl-transitions.com shader convention, Preact UI integration
**Confidence:** HIGH

## Summary

Phase 15.4 adds GLSL-powered transitions between content sequences. The architecture involves three major workstreams: (1) a new `renderGlslTransition()` function in `glslRuntime.ts` that accepts two textures (`from`/`to`) and a `progress` uniform, (2) integration into the existing cross-dissolve rendering pipeline in `exportRenderer.ts` where GL transitions replace the opacity-blending approach, and (3) UI work to enable the disabled Transition tab in ShaderBrowser, wire up the Apply flow to sequence data, and extend TransitionProperties for shader parameter editing.

The gl-transitions.com convention is well-defined and maps cleanly to the existing `ShaderDefinition` + `glslRuntime.ts` infrastructure. The convention uses `vec4 transition(vec2 uv)` as the entry point with `progress`, `ratio`, `getFromColor(vec2 uv)`, and `getToColor(vec2 uv)` as contextual variables. Custom parameters use `uniform float foo; // = 42.0` syntax. The project's existing shader wrapping pattern (build fragment source around user GLSL code) extends naturally to this convention -- the wrapper provides `getFromColor`/`getToColor` via `sampler2D from`/`sampler2D to` textures, and calls `transition(uv)` from `main()`.

The rendering pipeline change is the most architecturally significant piece: during a GL transition overlap zone, instead of rendering outgoing and incoming sequences at different opacities (the cross-dissolve approach), the renderer must capture each sequence to a separate offscreen canvas, upload both as WebGL2 textures, run the transition shader with the computed `progress` value, and composite the result onto the main canvas. This dual-capture approach is already conceptually present in the cross-dissolve code path (which renders outgoing then incoming), so the change is an evolution of existing logic.

**Primary recommendation:** Extend `glslRuntime.ts` with a `renderGlslTransition(shader, fromCanvas, toCanvas, progress, ratio, params)` function, add a `glTransition` field to the `Sequence` type alongside `crossDissolve`, and wire the rendering through the existing `crossDissolveOverlaps` computed signal (renaming/extending it to handle both transition types).

### Pattern 1: GL Transition Fragment Shader Wrapper
**What:** Wrap gl-transitions.com GLSL code in a WebGL2-compatible fragment shader with the required uniform bridge
**When to use:** Every transition shader compilation
**Example:**
```typescript
// Build fragment source for a transition shader
function buildTransitionFragmentSource(shader: ShaderDefinition): string {
  const paramUniforms = shader.params
    .map(p => `uniform float u_${p.key};`)
    .join('\n');

  return `#version 300 es
precision highp float;

uniform sampler2D u_from;
uniform sampler2D u_to;
uniform float progress;
uniform float ratio;

${paramUniforms}

out vec4 out_fragColor;

// gl-transitions convention: getFromColor / getToColor wrappers
vec4 getFromColor(vec2 uv) {
  return texture(u_from, uv);
}
vec4 getToColor(vec2 uv) {
  return texture(u_to, uv);
}

// --- Transition code (from gl-transitions.com) ---
${shader.fragmentSource}

void main() {
  vec2 uv = gl_FragCoord.xy / vec2(ratio * float(textureSize(u_from, 0).y), float(textureSize(u_from, 0).y));
  // Simpler: use gl_FragCoord / iResolution-style approach
  out_fragColor = transition(gl_FragCoord.xy / vec2(textureSize(u_from, 0)));
}
`;
}
```

**Note:** The actual UV calculation needs care. Since we use a fullscreen quad, `gl_FragCoord.xy` spans `[0, width] x [0, height]`. The UV should be normalized to `[0,1]` using the viewport dimensions. The cleanest approach:

```glsl
void main() {
  vec2 uv = gl_FragCoord.xy / iResolution.xy;
  out_fragColor = transition(uv);
}
```

Where `iResolution` is passed as a uniform (consistent with existing glslRuntime pattern).

### Pattern 2: Dual-Texture Capture and Render
**What:** Render outgoing and incoming sequences to offscreen canvases, upload as WebGL textures, run transition shader
**When to use:** During GL transition overlap zones in renderGlobalFrame
**Example:**
```typescript
// In renderGlobalFrame, inside the overlap loop:
if (overlap.glTransition) {
  // Render outgoing sequence to offscreen canvas A
  const offA = getOrCreateOffscreenCanvas(width, height);
  const rendererA = new PreviewRenderer(offA);
  rendererA.renderFrame(outLayers, outLocalFrame, outFrames, seq.fps, true, 1.0);

  // Render incoming sequence to offscreen canvas B
  const offB = getOrCreateOffscreenCanvas(width, height);
  const rendererB = new PreviewRenderer(offB);
  rendererB.renderFrame(inLayers, inLocalFrame, inFrames, seq.fps, true, 1.0);

  // Compute eased progress
  const t = framesIntoOverlap / overlap.duration;
  const easedProgress = applyEasing(t, overlap.curve);

  // Run GL transition shader
  const glResult = renderGlslTransition(
    shaderDef, offA, offB,
    easedProgress, width / height,
    overlap.params, width, height
  );

  // Composite onto main canvas
  if (glResult) {
    ctx.drawImage(glResult, 0, 0, width, height);
  }
}
```

**Important optimization:** Creating new PreviewRenderer instances per frame is expensive. Instead, use two reusable offscreen canvases and render with the existing renderer pattern (similar to how `blurOffscreen` works in `previewRenderer.ts`).

### Pattern 3: Transition Shader Definition
**What:** TypeScript module exporting a ShaderDefinition with `category: 'transition'` and gl-transitions.com GLSL source
**When to use:** Each transition shader file
**Example:**
```typescript
// src/lib/shaders/transitions/directional.ts
import type { ShaderDefinition } from '../../shaderLibrary';

const SOURCE = `// Author: Gaetan Renaudeau
// License: MIT

uniform vec2 direction; // = vec2(0.0, 1.0)

vec4 transition(vec2 uv) {
  vec2 p = uv + progress * sign(direction);
  vec2 f = fract(p);
  return mix(
    getToColor(f),
    getFromColor(f),
    step(0.0, p.y) * step(p.y, 1.0) * step(0.0, p.x) * step(p.x, 1.0)
  );
}`;

export const directional: ShaderDefinition = {
  id: 'transition-directional',
  name: 'Directional',
  category: 'transition',
  description: 'Slide the scene in a configurable direction',
  author: 'Gaetan Renaudeau',
  license: 'MIT',
  url: 'https://gl-transitions.com/editor/Directional',
  fragmentSource: SOURCE,
  params: [
    { key: 'directionX', label: 'Dir X', type: 'float', default: 0.0, min: -1, max: 1, step: 0.1 },
    { key: 'directionY', label: 'Dir Y', type: 'float', default: 1.0, min: -1, max: 1, step: 0.1 },
  ],
};
```

**Important porting note:** gl-transitions uses `uniform vec2 direction; // = vec2(0.0, 1.0)` but the project's param system uses individual float uniforms. Two approaches:
1. **Convert vec2 params to paired floats** (directionX, directionY) and reconstruct in the wrapper: `vec2 direction = vec2(u_directionX, u_directionY);`
2. **Add vec2 param support to ShaderParamDef** (more invasive but cleaner)

Recommend approach 1 (paired floats) to avoid modifying the entire param system. The wrapper can reconstruct vec2/vec3 types.

### Anti-Patterns to Avoid
- **Creating WebGL contexts per transition:** The project already has a shared GL context in glslRuntime.ts. Creating additional contexts risks hitting the browser limit (typically 8-16 contexts). Always reuse the shared context.
- **Uploading textures every frame without reuse:** Use `texSubImage2D` for updating existing textures rather than `texImage2D` which may reallocate. The existing `renderGlslFxImage` pattern shows the correct approach.
- **Rendering preview thumbnails at full resolution:** Transition browser previews should render at `PREVIEW_WIDTH x PREVIEW_HEIGHT` (200x140), not at canvas resolution.
- **Modifying the existing cross-dissolve path for all transitions:** Keep cross-dissolve working exactly as-is. The GL transition should be a parallel code path, not a replacement of the cross-dissolve logic.

## Common Pitfalls

### Pitfall 1: WebGL Context Loss During Dual-Texture Render
**What goes wrong:** Uploading two full-resolution textures in quick succession can trigger context loss on memory-constrained devices.
**Why it happens:** Each texture upload allocates GPU memory. Two full-resolution RGBA textures at 1920x1080 = ~16MB of GPU memory per frame.
**How to avoid:** Reuse two pre-allocated textures (`_fromTexture`, `_toTexture`) rather than creating new ones per frame. Use `texSubImage2D` for updates, `texStorage2D` for initial allocation (as glBlur.ts does).
**Warning signs:** Console warnings about WebGL context lost, blank frames during transitions.

### Pitfall 2: UV Coordinate Mismatch Between Canvas 2D and WebGL
**What goes wrong:** Canvas 2D has origin at top-left (Y down), WebGL has origin at bottom-left (Y up). Transition appears upside down or with inverted progress direction.
**Why it happens:** When capturing Canvas 2D content to WebGL texture, Y axis needs flipping.
**How to avoid:** Use `gl.pixelStorei(gl.UNPACK_FLIP_Y_WEBGL, true)` before `texImage2D` (already established pattern in glslRuntime.ts). Reset to false after upload.
**Warning signs:** Transition appears mirrored vertically, "from" and "to" swapped.

### Pitfall 3: gl-transitions.com Shaders Use GLSL 1.0/1.2 Syntax
**What goes wrong:** `texture2D()` calls in gl-transitions source fail in WebGL2 (GLSL ES 3.0 uses `texture()`).
**Why it happens:** gl-transitions.com was created for WebGL1. The project uses WebGL2 (`#version 300 es`).
**How to avoid:** The `getFromColor`/`getToColor` wrapper functions hide this difference. Since the transition shader calls `getFromColor(uv)` and `getToColor(uv)` instead of `texture2D()` directly, the wrapper provides the ES 3.0 `texture()` call. However, some shaders may use `texture2D()` directly in helper functions -- these need manual conversion.
**Warning signs:** Shader compilation errors mentioning `texture2D` not found.

### Pitfall 4: Offscreen Canvas Rendering Without Image Preloading
**What goes wrong:** When rendering outgoing/incoming sequences to offscreen canvases, images may not be loaded in the offscreen renderer's cache, producing blank frames.
**Why it happens:** The offscreen PreviewRenderer has a separate image cache from the main renderer.
**How to avoid:** Don't create separate PreviewRenderer instances. Instead, render to offscreen Canvas 2D contexts using the SAME PreviewRenderer (which already has all images cached). The key insight: `PreviewRenderer.renderFrame()` draws to `this.canvas` -- so temporarily swap the canvas reference or use the existing `blurOffscreen` pattern.
**Warning signs:** Transition shows one or both textures as blank/transparent.

### Pitfall 5: DPR Scaling in Offscreen Canvas Capture
**What goes wrong:** On Retina displays (DPR=2), the offscreen canvas captures at 2x resolution while the GL canvas expects 1x, causing size mismatch.
**Why it happens:** `PreviewRenderer.renderFrame()` applies DPR scaling internally. Offscreen canvases have `clientWidth=0`, which triggers the fallback path (DPR=1).
**How to avoid:** Create offscreen canvases at the same physical pixel dimensions as the main canvas (`canvas.width x canvas.height`), not logical dimensions. The GL texture upload will then match.
**Warning signs:** Transition appears cropped or has black borders.

### Pitfall 6: Mutually Exclusive Transition Types
**What goes wrong:** User applies a GL transition while a cross-dissolve already exists on the same sequence pair, or vice versa.
**Why it happens:** D-02 says they're mutually exclusive but the code paths are separate.
**How to avoid:** When applying a GL transition, automatically remove `crossDissolve`. When applying a cross-dissolve, automatically remove `glTransition`. Enforce at the store level.
**Warning signs:** Both transition indicators visible on timeline, rendering artifacts.

### Pitfall 7: Progress Value Out of Range
**What goes wrong:** Some gl-transitions shaders produce artifacts when progress is exactly 0.0 or 1.0 (division by zero in smoothstep, etc.).
**Why it happens:** Edge cases in shader math.
**How to avoid:** Clamp progress to `[0.001, 0.999]` before passing to shader. This is a common practice in video editing tools.
**Warning signs:** NaN or Inf in shader output, flickering at first/last frame of transition.

   - Recommendation: Add `_transitionOffA` and `_transitionOffB` as module-level canvases in a new helper or within the exportRenderer, since the dual-capture is renderer logic, not GL runtime logic.

2. **vec2/vec3 uniform parameters from gl-transitions**
   - What we know: gl-transitions uses `uniform vec2 direction` but the ShaderParamDef system only supports `type: 'float'`
   - What's unclear: Whether to add vec2/vec3 param types or decompose into individual floats
   - Recommendation: Decompose into individual floats (e.g., `directionX`, `directionY`) and reconstruct in the shader wrapper via auto-generated `vec2 direction = vec2(u_directionX, u_directionY);` lines. This avoids modifying the entire param system UI.

3. **Project format version bump**
   - What we know: Currently at v10. Adding `gl_transition` field to MceSequence requires a v11 bump.
   - What's unclear: Whether the Rust MceTransition struct should be extended or a new MceGlTransition struct created
   - Recommendation: Create a new `MceGlTransition` struct with `shader_id: String`, `params: HashMap<String, f64>`, `duration: u32`, `curve: String`. Add as `gl_transition: Option<MceGlTransition>` on MceSequence with `serde(default)` for backward compat.

4. **Sample images for transition browser previews**
   - What we know: D-14 says two built-in sample images (gradient + photo), looping animation
   - What's unclear: Image source, optimal dimensions, and bundling approach
   - Recommendation: Use small (400x280) JPEG images bundled as static imports. One color gradient, one photograph-like image. Import as base64 data URLs to avoid asset path complexity.
