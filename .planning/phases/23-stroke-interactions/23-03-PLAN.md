The CanvasArea outer container (line 367 of CanvasArea.tsx) has class `p-4 pt-0` which creates asymmetric vertical padding: top=0px, right=16px, bottom=16px, left=16px. The CSS flexbox `items-center justify-center` centers the canvas within the content area, but `getBoundingClientRect()` returns the border-box. The content center is offset from border-box center by `(paddingTop - paddingBottom) / 2 = (0 - 16) / 2 = -8px` vertically.

Modify `clientToCanvas` to accept optional `paddingTop` and `paddingBottom` parameters (default 0) so the center calculation accounts for asymmetric padding:

```typescript
export function clientToCanvas(
  clientX: number, clientY: number,
  containerRect: DOMRect,
  zoom: number, panX: number, panY: number,
  projectWidth: number, projectHeight: number,
  paddingTop = 0, paddingBottom = 0,
): Point {
  const containerCenterX = containerRect.left + containerRect.width / 2;
  // Adjust for asymmetric vertical padding: content center is shifted from border-box center
  const paddingOffsetY = (paddingTop - paddingBottom) / 2;
  const containerCenterY = containerRect.top + containerRect.height / 2 + paddingOffsetY;
  // ... rest unchanged
}
```

Three independent fixes in PaintOverlay.tsx:

**Fix B ã Stale bounding box after undo/redo (UAT tests 1, 6):**
At ~line 1669, the useEffect that re-renders the selection overlay subscribes only to `selectedStrokeIds`. After undo/redo, the element positions change (via `restoreElementSnapshot` in undo closures that call `paintVersion++`), but since selectedStrokeIds doesn't change, the overlay never re-draws. Fix by adding `paintStore.paintVersion.value` to the dependency array:

```typescript
useEffect(() => {
  void paintStore.selectedStrokeIds.value;
  void paintStore.paintVersion.value;  // ADD: re-render overlay when undo/redo changes element positions
  renderLivePreview();
}, [paintStore.selectedStrokeIds.value, paintStore.paintVersion.value]);
```

**Fix C ã Exponential edge scale (UAT test 8):**
At ~lines 1034-1100, edge scale applies an absolute ratio (`newW / origW`) to already-mutated coordinates each frame. This compounds exponentially. Corner scale (lines 1104-1137) avoids this by using incremental deltas (`scale = newDist / startDist` then updating `startDist = newDist`).

Fix: Before applying the absolute scale ratio, restore element coordinates from `transformSnapshot`. Add a restore step at the start of the edge-scale block (just inside `if (transformCorner.current.length === 1)`):

```typescript
if (transformCorner.current.length === 1) {
  // Non-uniform edge scale (D-04, D-05, D-06)
  const edge = transformCorner.current;

  // FIX: Restore from snapshot before applying absolute scale to prevent exponential compounding
  if (transformSnapshot.current) {
    restoreElementSnapshot(paintFrame.elements, transformSnapshot.current);
  }

  for (const el of paintFrame.elements) {
    // ... existing scale logic unchanged ...
  }
}
```

This works because `transformSnapshot` captures the pre-gesture state on pointerdown, and `restoreElementSnapshot` resets elements to that state before each frame applies the absolute ratio. The absolute ratio `newW / origW` is computed from the current mouse position vs the original dimension, so it's always correct relative to the snapshot.

Do NOT change the corner scale logic ã it uses incremental deltas and works correctly.
