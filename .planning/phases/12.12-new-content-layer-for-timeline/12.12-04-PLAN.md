**Preview.tsx** -- Modify BOTH the `renderFromFrameMap` function and the `disposeRender` effect to include content overlay sequences in the overlay compositing pass. The key change: replace the FX-only filter with an overlay filter that includes BOTH FX and content-overlay sequences.

In BOTH render paths (the `renderFromFrameMap` function used during playback AND the `disposeRender` effect used during scrub):

Change:
```typescript
const fxSeqs = allSeqs.filter(s => s.kind === 'fx' && s.visible !== false);
for (let i = fxSeqs.length - 1; i >= 0; i--) {
  const fxSeq = fxSeqs[i];
  if (fxSeq.inFrame != null && globalFrame < fxSeq.inFrame) continue;
  if (fxSeq.outFrame != null && globalFrame >= fxSeq.outFrame) continue;
  const fxLayers = fxSeq.layers.filter((l) => l.visible);
  if (fxLayers.length > 0) {
    renderer.renderFrame(fxLayers, localFrame, seqFrames, seq.fps, false);
  }
}
```

To:
```typescript
// Composite overlay sequences (FX + content overlays): reverse order so top-of-timeline renders last
const overlaySeqs = allSeqs.filter(s => s.kind !== 'content' && s.visible !== false);
for (let i = overlaySeqs.length - 1; i >= 0; i--) {
  const overlaySeq = overlaySeqs[i];
  if (overlaySeq.inFrame != null && globalFrame < overlaySeq.inFrame) continue;
  if (overlaySeq.outFrame != null && globalFrame >= overlaySeq.outFrame) continue;

  if (overlaySeq.kind === 'content-overlay') {
    // Content overlay: compute local frame relative to inFrame, apply keyframe interpolation
    const overlayLocalFrame = globalFrame - (overlaySeq.inFrame ?? 0);
    const overlayLayers = overlaySeq.layers.filter(l => l.visible).map(layer => {
      if (!layer.keyframes || layer.keyframes.length === 0) return layer;
      const values = interpolateAt(layer.keyframes, overlayLocalFrame);
      if (!values) return layer;
      return {
        ...layer,
        opacity: values.opacity,
        transform: {
          ...layer.transform,
          x: values.x,
          y: values.y,
          scaleX: values.scaleX,
          scaleY: values.scaleY,
          rotation: values.rotation,
        },
        blur: values.blur,
      };
    });
    if (overlayLayers.length > 0) {
      renderer.renderFrame(overlayLayers, overlayLocalFrame, seqFrames, seq.fps, false);
    }
  } else {
    // FX sequence (existing behavior)
    const fxLayers = overlaySeq.layers.filter((l) => l.visible);
    if (fxLayers.length > 0) {
      renderer.renderFrame(fxLayers, localFrame, seqFrames, seq.fps, false);
    }
  }
}
```

IMPORTANT: This change must be applied in BOTH places where the FX loop exists:
1. The `renderFromFrameMap` function (used during rAF playback)
2. The `disposeRender` effect (used during scrub/seek)

**previewRenderer.ts** -- Add video looping for content overlay sequences:

Find the video source resolution code (likely in a method like `resolveVideoSource` or within the layer rendering switch statement). Currently it sets `video.currentTime = frame / fps` without modding by video duration.

For content overlay layers, the `frame` parameter passed to `renderFrame` is already the local frame (relative to `inFrame`). The video should loop by modding the time:

Find the line:
```typescript
video.currentTime = frame / fps;
```

Change to:
```typescript
const targetTime = frame / fps;
video.currentTime = video.duration > 0 ? targetTime % video.duration : targetTime;
```

This applies the mod universally (for ALL video layers, not just content overlays). Content sequence video layers won't be affected since their hold frames are typically shorter than the video. But the mod is safe for all cases and prevents video freeze at end for any video layer type.

Also ensure that for image-sequence content overlay layers, the frame cycling uses `frame % imageIds.length` which is already the existing behavior (since `resolveLayerSource` for image-sequence already indexes as `imageIds[frame % imageIds.length]`). No change needed for image-sequence looping.

For static-image content overlay layers, the same image renders at every frame, which is the existing behavior. No change needed.
