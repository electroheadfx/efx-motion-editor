# Phase 12.12: New Content Layer for Timeline - Research

**Researched:** 2026-03-19
**Domain:** Timeline content overlay sequences (static-image, image-sequence, video) with FX-like range bars, interleaved compositing, property controls, and keyframe animation
**Confidence:** HIGH

## Summary

This phase adds a new category of timeline-level sequences: content overlay layers (static-image, image-sequence, video) that render above content sequences and interleave with FX sequences. The architecture already supports the `kind` discriminator on `Sequence`, `inFrame`/`outFrame` range semantics, and FX track layout/rendering/interaction patterns. The primary work is extending these systems to support a third kind of sequence (or reusing `kind: 'fx'` with content layer types), updating the menu, wiring sidebar properties, enabling keyframes on these new sequences, and rendering content overlay sources in the compositing pipeline.

The codebase is mature with well-established patterns. Every required feature (range bars, in/out handles, drag reorder, visibility toggle, sidebar properties, keyframe animation, preview compositing, serialization) has an existing FX-sequence precedent. The strategy is to extend, not rebuild.

**Primary recommendation:** Introduce `kind: 'content-overlay'` as a new sequence kind to clearly distinguish timeline-level content layers from both content sequences (key photos) and FX sequences (generators/adjustments). This avoids ambiguity in the `kind === 'fx'` checks scattered throughout the codebase and keeps the type system self-documenting.

## Architecture Patterns

### Sequence Kind Strategy (Claude's Discretion)

**Recommendation: Introduce `kind: 'content-overlay'`**

| Option | Pros | Cons |
|--------|------|------|
| `kind: 'content-overlay'` (NEW) | Clean type discrimination, no risk of breaking FX logic, self-documenting | One more union member to handle |
| Reuse `kind: 'fx'` | Less type-level changes | Every `kind === 'fx'` check in the codebase would need auditing; FX-specific logic (generator rendering, adjustment effects) would get confused by content sources |

Current `kind === 'fx'` checks that would break if reused:
- `frameMap.ts` line 11: `if (seq.kind === 'fx') continue` -- skips FX from frame map. Content overlays also should be skipped, so this works either way.
- `keyframeStore.ts` line 27: `if (seq.kind === 'fx') continue` -- currently blocks keyframes on FX sequences. Content overlays need keyframes, so this would break.
- `sequenceStore.ts` line 269: `getContentSequences()` filters `kind === 'content'`. Content overlays are NOT content sequences.
- `sequenceStore.ts` line 274: `getFxSequences()` filters `kind === 'fx'`. Content overlays should NOT appear in FX-only lists.
- `Preview.tsx` line 35: `if (seq.kind === 'fx') return` -- skips FX from content rendering. Content overlays need their own render pass.
- `Preview.tsx` line 70: `allSeqs.filter(s => s.kind === 'fx')` -- collects FX for overlay pass. Content overlays need to be included here too.

**Verdict:** A new `kind: 'content-overlay'` is cleaner. The type union becomes `'content' | 'fx' | 'content-overlay'`. The Sequence interface already has optional `inFrame`/`outFrame`/`visible` fields that content overlays will use.

### FxTrackLayout Extension (Claude's Discretion)

The `FxTrackLayout` interface currently has `kind: 'fx'`. Since content overlay tracks interleave with FX tracks, extend to support both:

```typescript
// types/timeline.ts
export interface FxTrackLayout {
  sequenceId: string;
  sequenceName: string;
  kind: 'fx' | 'content-overlay';  // extended
  inFrame: number;
  outFrame: number;
  color: string;
  visible: boolean;
  thumbnailImageId?: string;  // for content overlay thumbnail rendering
  layerType?: LayerType;      // to distinguish static-image/image-sequence/video
}
```

### Compositing Order in fxTrackLayouts

The existing `fxTrackLayouts` computed signal filters `kind !== 'fx'` and skips everything else. This must be extended to include `kind === 'content-overlay'` sequences, preserving their interleaved order from `sequenceStore.sequences`.

```typescript
// frameMap.ts - extend fxTrackLayouts
export const fxTrackLayouts = computed<FxTrackLayout[]>(() => {
  const layouts: FxTrackLayout[] = [];
  for (const seq of sequenceStore.sequences.value) {
    if (seq.kind === 'content') continue; // skip key-photo sequences
    // Both 'fx' and 'content-overlay' get track layouts
    const primaryLayer = seq.layers[0];
    let color: string;
    if (seq.kind === 'content-overlay') {
      // Color by content type
      color = primaryLayer?.type === 'static-image' ? 'var(--sidebar-dot-green)'
            : primaryLayer?.type === 'image-sequence' ? 'var(--sidebar-dot-blue)'
            : '#8B5CF6'; // video
    } else {
      color = primaryLayer ? fxColorForLayerType(primaryLayer.type) : FX_DEFAULT_COLOR;
    }
    layouts.push({
      sequenceId: seq.id,
      sequenceName: seq.name,
      kind: seq.kind as 'fx' | 'content-overlay',
      inFrame: seq.inFrame ?? 0,
      outFrame: seq.outFrame ?? 100,
      color,
      visible: seq.visible !== false,
      thumbnailImageId: seq.kind === 'content-overlay' ? getThumbnailId(primaryLayer) : undefined,
      layerType: primaryLayer?.type,
    });
  }
  return layouts;
});
```

**Important CSS variable note:** The sidebar dot colors use CSS variables (`--sidebar-dot-green`, `--sidebar-dot-blue`). For Canvas 2D rendering in TimelineRenderer, these need to be resolved at render time via `getComputedStyle`. Add them to the theme color cache in TimelineRenderer.

### Thumbnail Strategy for Range Bars (Claude's Discretion)

**Recommendation: Small icon thumbnail at left edge of range bar**

For content overlay range bars (24px height), rendering a full thumbnail pattern like content tracks is not feasible. Instead:

1. **Static Image:** Reuse the `ThumbnailCache` already in TimelineRenderer. Load a small thumbnail of the source image. Draw it as a 16x16 icon (with aspect-ratio fitting) at the left end of the range bar, inside the bar fill.

2. **Image Sequence:** Use the first image in the `imageIds` array as the thumbnail source. Same 16x16 icon approach.

3. **Video:** Use a solid color fill (purple) with the layer name text. Video thumbnails require frame extraction which is complex and deferred.

### Keyframe Frame Offsets for Content Overlays (Claude's Discretion)

**Recommendation: Use in-frame-relative offsets (same as content layers)**

Current keyframe system uses sequence-local frame offsets:
- Content layer keyframes: `frame` is offset from the sequence's start frame in the global timeline
- The `keyframeStore.findLayerContext()` function finds the sequence start frame and computes local frame as `globalFrame - startFrame`

For content overlay sequences, keyframe offsets should be relative to `inFrame`:
- A content overlay with `inFrame=10, outFrame=50` has local frame range `[0, 40)`
- A keyframe at local frame 5 means global frame 15
- This matches how FX sequences would work if they had keyframes

The `findLayerContext` function currently skips `kind === 'fx'`. It needs to be extended to handle `kind === 'content-overlay'`:

```typescript
function findLayerContext(layerId: string): { sequenceId: string; startFrame: number } | null {
  const seqs = sequenceStore.sequences.peek();
  for (const seq of seqs) {
    if (seq.kind === 'fx') continue; // FX sequences still skip
    if (seq.layers.some(l => l.id === layerId)) {
      if (seq.kind === 'content-overlay') {
        // Content overlay: startFrame is inFrame
        return { sequenceId: seq.id, startFrame: seq.inFrame ?? 0 };
      }
      // Content sequence: use trackLayouts
      const layouts = trackLayouts.peek();
      const layout = layouts.find(t => t.sequenceId === seq.id);
      return { sequenceId: seq.id, startFrame: layout?.startFrame ?? 0 };
    }
  }
  return null;
}
```

Similarly, `getSelectedContentLayer()` needs to allow `kind === 'content-overlay'`:
```typescript
function getSelectedContentLayer() {
  const layerId = layerStore.selectedLayerId.value;
  if (!layerId) return null;
  const seqs = sequenceStore.sequences.value;
  for (const seq of seqs) {
    if (seq.kind === 'fx') continue;
    const layer = seq.layers.find(l => l.id === layerId);
    if (layer && !layer.isBase && !isFxLayer(layer)) return { layer, sequenceId: seq.id };
  }
  return null;
}
```

### Reorder Drag with Interleaved Tracks (Claude's Discretion)

Currently `reorderFxSequences` works on FX-only indices within the full sequences array. With content overlays interleaved, the reorder must work on ALL non-content sequences (both `fx` and `content-overlay`).

**Recommendation:** Rename `reorderFxSequences` to `reorderOverlaySequences` (or keep the name and change its behavior) so it operates on `seq.kind !== 'content'` indices. The existing algorithm of extracting indices and splicing works unchanged -- just broaden the filter:

```typescript
reorderOverlaySequences(fromIndex: number, toIndex: number) {
  const all = [...sequences.value];
  const overlayIndices = all.reduce<number[]>((acc, s, i) => {
    if (s.kind !== 'content') acc.push(i); // both 'fx' and 'content-overlay'
    return acc;
  }, []);
  // ... same splice logic
}
```

### Content Overlay Rendering in Preview

The Preview component's render pipeline is:
1. `renderer.renderFrame(layers, ...)` for content sequence layers (clearCanvas=true)
2. Loop through FX sequences in reverse order, `renderer.renderFrame(fxLayers, ..., false)` (overlay mode)

Content overlay sequences must be inserted into step 2. Since timeline ordering determines compositing order and content overlays interleave with FX, the loop should process ALL non-content sequences in order:

```typescript
// In Preview.tsx, replace the FX-only loop:
const overlaySeqs = allSeqs.filter(s =>
  (s.kind === 'fx' || s.kind === 'content-overlay') && s.visible !== false
);
for (let i = overlaySeqs.length - 1; i >= 0; i--) {
  const overlaySeq = overlaySeqs[i];
  if (overlaySeq.inFrame != null && globalFrame < overlaySeq.inFrame) continue;
  if (overlaySeq.outFrame != null && globalFrame >= overlaySeq.outFrame) continue;

  if (overlaySeq.kind === 'content-overlay') {
    // Content overlay: resolve source, apply keyframe interpolation
    const localFrame = globalFrame - (overlaySeq.inFrame ?? 0);
    const interpolatedLayers = overlaySeq.layers.map(layer => {
      if (!layer.keyframes || layer.keyframes.length === 0) return layer;
      const values = interpolateAt(layer.keyframes, localFrame);
      if (!values) return layer;
      return { ...layer, opacity: values.opacity, transform: { ...layer.transform, x: values.x, y: values.y, scaleX: values.scaleX, scaleY: values.scaleY, rotation: values.rotation }, blur: values.blur };
    });
    renderer.renderFrame(interpolatedLayers, localFrame, seqFrames, seq.fps, false);
  } else {
    // FX sequence (existing behavior)
    const fxLayers = overlaySeq.layers.filter(l => l.visible);
    if (fxLayers.length > 0) {
      renderer.renderFrame(fxLayers, localFrame, seqFrames, seq.fps, false);
    }
  }
}
```

### Content Source Looping in PreviewRenderer

The `resolveLayerSource` method in PreviewRenderer already handles `static-image`, `image-sequence`, and `video` source types. For content overlays:

- **Static Image:** Works as-is. The image renders at every frame within the in/out range.
- **Image Sequence looping:** Currently uses `frame % imageIds.length`. For content overlays, `frame` will be the local frame (relative to inFrame), which cycles naturally. Works as-is.
- **Video looping:** The video `currentTime = frame / fps`. For looping, need to mod by video duration: `currentTime = (frame / fps) % video.duration`. The existing implementation sets `video.currentTime = frame / fps` without modding. Need to add the mod for content overlay video layers.

### Anti-Patterns to Avoid
- **DO NOT add content overlay layers to the active content sequence.** They are standalone sequences with `kind: 'content-overlay'`, not layers within a content sequence. The ImportedView handler must create sequences, not layers.
- **DO NOT duplicate FX rendering logic.** Content overlay rendering should follow the same `renderFrame(layers, frame, frames, fps, false)` pattern.
- **DO NOT hardcode CSS variable colors in Canvas code.** Resolve them via `getComputedStyle` and cache like existing theme colors.
- **DO NOT break the existing `kind === 'fx'` FX-only filtering.** Always use explicit checks: `seq.kind === 'fx'` for FX-only, `seq.kind !== 'content'` for all overlay types.

## Common Pitfalls

### Pitfall 1: Confusing Content Overlays with Content Layers in Content Sequences
**What goes wrong:** Adding content overlay layers to the active content sequence instead of creating standalone sequences
**Why it happens:** Phase 12.9 established the pattern of `layerStore.add()` for content layers. Phase 12.12 needs `sequenceStore` operations.
**How to avoid:** Content overlay menu items must call `sequenceStore.createContentOverlaySequence()`, NOT `layerStore.add()`
**Warning signs:** Content overlay appears in sidebar layer list of a content sequence rather than as its own track in the timeline

### Pitfall 2: FX-Only Filters Missing Content Overlays
**What goes wrong:** Content overlay sequences not appearing in timeline tracks or preview compositing
**Why it happens:** Many places filter `seq.kind === 'fx'` and would miss `'content-overlay'`
**How to avoid:** Audit all `kind` filters. Use `seq.kind !== 'content'` when you want both FX and content overlay sequences.
**Warning signs:** Content overlay tracks visible in timeline but not rendered in preview (or vice versa)

### Pitfall 3: Keyframe System Blocking Content Overlays
**What goes wrong:** Keyframes not working on content overlay layers
**Why it happens:** `keyframeStore.findLayerContext()` skips `kind === 'fx'`. If `kind: 'content-overlay'` is added but not handled, it falls through to the content sequence path which looks in `trackLayouts` (which doesn't contain content overlays).
**How to avoid:** Explicitly handle `kind === 'content-overlay'` in `findLayerContext()` and `getSelectedContentLayer()`, using `seq.inFrame` as the startFrame instead of `trackLayouts`.
**Warning signs:** "I" key to add keyframe does nothing when content overlay layer is selected

### Pitfall 4: CSS Variables in Canvas 2D
**What goes wrong:** Content overlay track colors render as literal strings "var(--sidebar-dot-green)" instead of resolved colors
**Why it happens:** Canvas 2D doesn't resolve CSS variables -- they must be resolved via `getComputedStyle()`
**How to avoid:** Add content overlay colors to the `getThemeColors()` cache in TimelineRenderer, resolved at init and on theme change
**Warning signs:** Tracks render as black or don't render at all

### Pitfall 5: Video Looping Not Working
**What goes wrong:** Video content overlays play once and freeze at last frame
**Why it happens:** `resolveVideoSource` sets `video.currentTime = frame / fps` without modding by video duration
**How to avoid:** For content overlay video layers, compute `targetTime = (localFrame / fps) % video.duration`
**Warning signs:** Video shows black after its natural duration within the overlay range

### Pitfall 6: Interleaved Reorder Breaking FX-Only Logic
**What goes wrong:** Reordering a content overlay track breaks FX track ordering
**Why it happens:** `reorderFxSequences()` uses FX-only indices. If content overlays are interleaved, the index mapping is wrong.
**How to avoid:** Change the reorder method to operate on all non-content sequences (both `fx` and `content-overlay`)
**Warning signs:** Dragging a track moves the wrong track in the timeline

### Pitfall 7: Selection State Confusion
**What goes wrong:** Selecting a content overlay track doesn't show properties, or selecting it deselects the wrong thing
**Why it happens:** The selection logic in TimelineInteraction has separate paths for FX area and content area. Content overlays occupy the FX area but are not FX.
**How to avoid:** When clicking a track in the FX area, check the track's `kind` and route to appropriate selection logic. Content overlay selection should populate sidebar properties (same as content layer selection).
**Warning signs:** Clicking a content overlay track shows FX properties or shows nothing

### Pitfall 8: Version Bump for .mce Format
**What goes wrong:** Old projects with `kind: 'content'` or `kind: 'fx'` fail to load when new code expects `'content-overlay'`
**Why it happens:** New `kind` value added without backward compat handling
**How to avoid:** The hydration already uses `mceSeq.kind ?? 'content'` fallback. Version 7 is not strictly needed since `'content-overlay'` is a new value that old files won't have. But bumping to v7 is good practice.
**Warning signs:** Projects crash on open after update

### Creating a Content Overlay Sequence

```typescript
// sequenceStore.ts
createContentOverlaySequence(name: string, layer: Layer, totalFrames: number): Sequence {
  const before = snapshot();
  const seq: Sequence = {
    id: genId(),
    kind: 'content-overlay',
    name,
    fps: 24,
    width: 1920,
    height: 1080,
    keyPhotos: [],
    layers: [layer],
    inFrame: 0,
    outFrame: totalFrames > 0 ? totalFrames : 100,
  };
  sequences.value = [...sequences.value, seq];
  markDirty();
  const after = snapshot();
  pushAction({
    id: crypto.randomUUID(),
    description: `Add content overlay "${name}"`,
    timestamp: Date.now(),
    undo: () => restore(before),
    redo: () => restore(after),
  });
  return seq;
}
```

## Open Questions

1. **Content overlay deletion behavior**
   - What we know: FX sequences delete the entire sequence when the last layer is removed (sequenceStore line 561). Content overlays have a single layer each.
   - What's unclear: Should Delete key on a content overlay layer delete just the layer (leaving empty sequence) or the entire sequence?
   - Recommendation: Delete entire sequence (same as FX behavior since there's only one layer per content overlay sequence). The `removeLayerFromSequence` already handles this.

2. **Content overlay interaction with playback seek**
   - What we know: Content sequences use `sequenceStore.setActive(track.sequenceId)` on click. FX tracks use `selectFxSequenceLayer`.
   - What's unclear: Should clicking a content overlay track set it as the active sequence? Content overlays don't contribute to the frame map.
   - Recommendation: No -- clicking a content overlay track selects its layer for property editing but does NOT set activeSequenceId. This is the same behavior as FX tracks.

3. **How imageStore URL is passed to TimelineRenderer for thumbnail lookup**
   - What we know: TimelineRenderer receives `imageStore` in DrawState. Content overlay thumbnails need image URLs.
   - What's unclear: The current ThumbnailCache takes an `imageId` and `url`. For content overlays, the imageId comes from the layer source, and the URL from `imageStore.getDisplayUrl()`.
   - Recommendation: Pass thumbnail data (imageId + url) in FxTrackLayout. The renderer already has access to imageStore via DrawState -- use it to resolve URLs at draw time.
