## Summary

Phase 36.9 should be planned as a small visible Play/Stop automation slice over the existing Roto cached-frame preview path, not as a cache renderer, export path, interpolation phase, or timeline redesign. [VERIFIED: SPECS/36.x-phases/phase-36.9-cached-autoplay/spec-36.9-cached-autoplay.md] The spec explicitly says playback must use existing cached-frame display, must be preview-only, and must not rewrite, regenerate, or mutate cached frames. [VERIFIED: SPECS/36.x-phases/phase-36.9-cached-autoplay/spec-36.9-cached-autoplay.md]

The codebase already contains a dormant cached Roto playback implementation in `PhysicsPaintStudio.tsx`: it tracks `isRotoCachedPlaybackActive`, `cachedRotoPlaybackFrame`, `rotoCachedPlaybackStatus`, builds cached playback frames, starts/stops an interval at `previewFps`, overlays the selected cached frame in `PhysicsPaintCanvasStack`, and wires props into `PhysicsPaintWorkflowStrip`. [VERIFIED: codebase grep/read] The current workflow strip intentionally keeps cached playback props in its interface while hiding the transport in the strict Phase 36.3 Roto strip; tests assert this hidden state. [VERIFIED: app/src/components/physic-paint/PhysicsPaintWorkflowStrip.test.ts] Therefore the likely planning center is: (1) necessity check/UAT gate, (2) expose minimal Play/Stop controls in the Roto workflow header using the provided timeline references, (3) harden/verify the existing Studio cached-playback behavior, and (4) update tests that previously required the transport to be hidden. [VERIFIED: codebase grep/read]

**Primary recommendation:** Plan one narrow implementation wave that exposes Play/Stop in `PhysicsPaintWorkflowStrip.tsx`, reuses `toggleRotoCachedPlayback`/`stopPreview`/cached overlay state in `PhysicsPaintStudio.tsx`, and adds focused tests proving no cache mutation/regeneration while manual previous/next stepping remains unchanged. [VERIFIED: codebase grep/read]

## Architecture Patterns

### System Architecture Diagram

```text
User presses Play in Roto workflow header
  -> PhysicsPaintWorkflowStrip calls onToggleRotoPlayback
  -> PhysicsPaintStudio.toggleRotoCachedPlayback()
      -> collect playback frame numbers from store + launchContext + local preview refs
      -> map each frame to existing cached reference or transient background/missing frame
      -> start interval at previewFps
          -> set cachedRotoPlaybackFrame for current playback index
          -> PhysicsPaintCanvasStack overlays cachedRotoPlaybackUrl
          -> status/animFrame update visible feedback
      -> when sequence ends or user presses Stop
          -> clear interval
          -> clear cachedRotoPlaybackFrame
          -> set isPlaying/isRotoCachedPlaybackActive false
          -> leave current frame context coherent

Manual previous/next path remains separate:
User presses nav/cell
  -> requestRotoFrameNavigation(frame)
      -> snapshot current frame if needed
      -> rotoSession.requestFrame(frame)
      -> execute navigate/save effects
      -> loadCachedRotoReferenceFrame(frame)
```

All arrows above reflect current code paths except the visible Roto header Play/Stop controls, which are currently hidden by tests. [VERIFIED: codebase grep/read]

## Common Pitfalls

### Pitfall 1: Treating Play as Render/Regenerate
**What goes wrong:** Play rewrites cached PNGs or calls store regeneration/apply paths. [VERIFIED: spec risk]  
**Why it happens:** Existing Play-mode preview/render code lives near Roto playback code and does generate/apply frames for Play canvas. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]  
**How to avoid:** Keep Roto cached playback in the cached overlay branch; add tests that the Roto playback block does not call `AnimationPlayer`, `flushRotoFrame`, `sendPhysicPaintApplyPayload`, or `regenerateRotoInterpolationCache`. [VERIFIED: codebase/tests]  
**Warning signs:** Status copy says “Render,” “Save all,” or “regenerate”; Play button is wired to `onSavePlay` instead of `onToggleRotoPlayback`. [VERIFIED: codebase patterns]

### Pitfall 2: Reopening Broad Studio State Orchestration
**What goes wrong:** Implementers add effects or scattered state to coordinate playback, navigation, cache refs, and dirty frames. [VERIFIED: Phase 36.8 tests]  
**Why it happens:** `PhysicsPaintStudio.tsx` is large and already has many refs/state fields. [VERIFIED: codebase read]  
**How to avoid:** Reuse existing `rotoCachedPlaybackTimerRef`, `stopRotoCachedPlayback`, `toggleRotoCachedPlayback`, and workflow-strip props; if logic grows, extend a small controller/helper rather than broad `useEffect` branches. [VERIFIED: codebase]

### Pitfall 3: Breaking Manual Previous/Next
**What goes wrong:** Play changes `currentFrame` or frame-sync behavior in a way that manual previous/next no longer loads cached references. [VERIFIED: spec acceptance]  
**Why it happens:** Playback frame index and app `currentFrame` are different concepts; existing playback currently sets overlay/anim frame rather than using navigation for every tick. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]  
**How to avoid:** Keep manual navigation through `requestRotoFrameNavigation`; use playback overlay state for timed preview; Stop/manual edit/navigation should clear playback overlay. [VERIFIED: codebase]

### Pitfall 4: Copying the React/Tailwind Mock Directly
**What goes wrong:** React imports, Tailwind-only classes, or `lucide-react` leak into the Preact app. [VERIFIED: timeline mock + CLAUDE.md]  
**Why it happens:** The provided TSX mock is a React/Tailwind design reference, not app source. [VERIFIED: TimelineBottomWindow.tsx]  
**How to avoid:** Translate only placement/visual intent into existing Preact/CSS conventions. [VERIFIED: CLAUDE.md + codebase]
