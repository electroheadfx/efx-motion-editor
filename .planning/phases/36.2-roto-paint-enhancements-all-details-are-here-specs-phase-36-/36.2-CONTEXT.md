# Phase 36.2: Roto paint enhancements - Context

**Gathered:** 2026-06-16T20:05:25Z
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 36.2 turns Physics Paint Roto into a cache-first animation workflow. Roto frames are real key frames; cached rendered PNGs are the durable playback/export truth in EFX Motion, while editable per-frame stroke state is only guaranteed during the currently open standalone session.

The core slice is: show cached Roto frames in the standalone spreadsheet, navigate with immediate cached reference previews, auto-render/apply dirty Roto keys when leaving a frame or closing the standalone, expose only a lightweight flush-pending action instead of a costly manual Save all, and make EFX Motion playback/export use cached Roto frames quickly.

A final tail plan is also in scope after the cache/autosave foundation: Roto interpolation and key edit utilities. Roto interpolation generates render-only cached in-between frames for EFX Motion from real Roto key squares; Physics Paint shows only key squares and interpolation connector lines, not editable generated boxes. Roto key utilities include duplicate, insert, delete, copy, and paste behavior for real Roto keys.

This phase does not add guaranteed durable per-frame editable stroke persistence, stroke selection/editing, a Play-like Roto script module, editable interpolation frames, or a broader source-lane system.

</domain>

<decisions>
## Implementation Decisions

### Cache-First Editable Model
- **D-01:** Phase 36.2 must ship Roto as cache-first only. Cached PNG frames are the durable Roto playback/export truth; do not add guaranteed per-frame editable stroke persistence in this phase.
- **D-02:** When reopening or navigating to a cached Roto frame with no editable stroke state, show the cached PNG immediately as a visual reference/background. If the user paints, start from a clean editable state.
- **D-03:** Cached-only frames are repaintable, not stroke-editable. Communicate this with subtle status text, not a modal or noisy warning.
- **D-04:** If the user paints over a cached-only reference frame, the next save replaces that frame's cached PNG. Do not composite over the old cache by default and do not ask every time.

### Auto-Save and Flush Rules
- **D-05:** Roto auto-save happens on frame leave. Painting marks the current Roto key dirty; when the user navigates from frame A to frame B, the standalone saves/renders dirty frame A once, then moves to frame B.
- **D-06:** No auto-save should run during active brush movement. Avoid repeated apply/render requests while staying on the same frame.
- **D-07:** Closing the standalone with a dirty current Roto key must attempt one final save/render/apply so closing preserves visible cached output in EFX Motion.
- **D-08:** While a dirty frame is being saved during navigation, block briefly with a saving state and prevent second navigation until the previous frame apply is queued/settled enough to avoid races.
- **D-09:** Do not implement a default costly Save all frames re-render. With save-on-leave working, already cached frames should not be re-rendered by default.
- **D-10:** Replace the current `Save roto frame` primary action with a modest `Save pending` / `Save current` action that flushes only unsaved dirty frame(s), disabled when nothing is dirty.

### Roto Spreadsheet Cell UX
- **D-11:** The Roto timeline/spreadsheet uses the same square cell model, with exactly three semantic fills: gray, green, and pink.
- **D-12:** Gray means empty/no cached output and no editable session Roto key.
- **D-13:** Green means cached PNG output exists but no editable Roto state exists in the current standalone session.
- **D-14:** Pink means editable Roto state exists in the currently open standalone session. A pink frame may already have cache after frame-leave save; pink is not limited to unsaved/dirty.
- **D-15:** Do not add a separate dirty color. Pending save may be shown with subtle status or a small non-semantic indicator if needed.
- **D-16:** The current frame overlays the gray/green/pink fill with a bright outline/active border. It must not obscure the cell's semantic fill.

### Missing, Empty, Deleted, and Background Frames
- **D-17:** Empty or deleted real Roto keys remove their cached output. Do not persist empty/deleted keys as green cached cells.
- **D-18:** During EFX Motion playback/export, frames with no real Roto key and no generated interpolation cache render transparent when the Roto background is transparent.
- **D-19:** During EFX Motion playback/export, frames with no real Roto key and no generated interpolation cache render the paper/background only when a paper/background is active.
- **D-20:** Paper/background-only output for missing frames is playback/export behavior only. It should not create green cached cells for empty frames.
- **D-21:** Interpolation only generates between two real Roto key squares. If one side of a span is missing, no interpolation is generated and missing-frame background rules apply.

### Roto Interpolation Tail Plan
- **D-22:** Include Roto interpolation as the final/tail plan of phase 36.2, after cache-first cells, cached preview, and auto-save are working.
- **D-23:** Default Roto remains frame-by-frame. When interpolation is enabled, a global in-between frame value controls spacing: `1` gives `[1]-[2]-[3]`, `2` gives `[1]--[2]--[3]`, and so on.
- **D-24:** Interpolation has two modes: frame duplication and frame blending.
- **D-25:** Frame duplication duplicates the source key frame into in-between cached frames and applies motion options: deform and position.
- **D-26:** Frame blending blends between the first and second real key frame in the in-between cached frames with transparency and applies motion options: deform and position.
- **D-27:** Interpolation generates real cached PNG frames in EFX Motion for preview/export. It is not only a virtual standalone preview.
- **D-28:** In Physics Paint, interpolation is represented only as connector lines between real Roto key squares, e.g. `[]-[]`. Generated interpolation frames are not shown as editable boxes.
- **D-29:** The user can move, edit, and erase only real Roto key squares in Physics Paint. Generated interpolation frames are render-only cache.
- **D-30:** Interpolation settings are global for the Roto layer/session, not per gap.
- **D-31:** When global interpolation settings change, regenerate all generated in-between cached frames between existing Roto key pairs.
- **D-32:** If the standalone has been closed and editable session strokes are gone, interpolation changes later work from the cached transparent key images.
- **D-33:** Opening Physics Paint from an EFX Motion frame that contains only a generated interpolation cache frame should open the nearest real Roto key frame. Do not turn interpolation-only frames into editable targets.

### Roto Key Edit Utilities Tail Plan
- **D-34:** Include Roto key edit utilities in the same final/tail plan as interpolation: duplicate key, insert frame, delete frame, copy frame, and paste frame.
- **D-35:** Duplicate Roto key defaults to duplicating the selected key to the next frame, useful for creating a pause in animation.
- **D-36:** If the next frame already has a real Roto key, duplicate performs a ripple insert: push all later real Roto keys one frame to the right to make room for the duplicated key.
- **D-37:** Insert frame also shifts later real Roto keys one frame to the right.
- **D-38:** Delete frame removes the selected real Roto key and shifts later real Roto keys one frame to the left.
- **D-39:** Copy/paste is replace-style: paste replaces the selected/current real Roto key without changing timing.
- **D-40:** Ripple insert/duplicate/delete affect only real Roto key frames. Generated interpolation cache is regenerated from the updated key sequence; it is not independently shifted as editable data.
- **D-41:** Duplicated keys and ripple-edited key sequences participate in interpolation like any other Roto keys, including global motion options.

### Claude's Discretion
Planner/researcher may choose exact storage names, helper boundaries, CSS class names, subtle status copy, and implementation sequencing inside the core cache/autosave plans, but must preserve the cache-first model, the gray/green/pink cell semantics, save-on-leave/save-on-close behavior, and the final tail-plan scope for interpolation/key utilities.

</decisions>

### Established Patterns
- Physics paint remains additive and must not replace perfect-freehand Basic paint or p5.brush FX paint.
- The rejected headless/batch adapter/editor-driven replay direction must not return. Preserve interactive standalone behavior and rendered-frame/cache output.
- Rendered output crosses the app/standalone bridge; editable engine internals do not.
- Non-reactive Map storage must bump explicit version signals (`physicPaintVersion`) and mark the project dirty on mutations.
- Play already has cached/stale/missing concepts and cached preview playback that can inform Roto cache UI, but Roto must remain key-frame/cache-first, not Play-script-first.
- Standalone Physics Paint UI should stay polished and match EFX Motion style; avoid demo-like or noisy visual states.

### Integration Points
- Roto navigation in `PhysicsPaintStudio.tsx` currently snapshots session state and loads/clears editable state when moving frames. Phase 36.2 should add save-on-leave/save-on-close, immediate cached reference preview, and dirty-frame tracking here.
- `saveRotoFrame()` currently builds an `apply-canvas` payload with `editableState` and one `renderedFrame`; phase 36.2 should reuse this path for dirty-frame flushes while changing the user-facing save behavior.
- `physicPaintStore.applyCanvas()` currently stores one layer-level editable state and one rendered frame; phase 36.2 should not require guaranteed per-frame editable persistence, but must support cache status/metadata enough to distinguish green cached-only from pink session-editable cells in the standalone UI.
- `physicPaintStore.getFrames(layerId)` and MCE serialization already preserve rendered frames; use this as the source of durable Roto cache truth.
- Roto interpolation should generate/update real cached PNG frames for EFX Motion using the existing rendered-frame cache path, while separately tracking real Roto keys so generated interpolation frames do not become editable squares.
- Ripple key utilities must operate on real Roto keys only, then regenerate interpolation cache from the updated key sequence.

<specifics>
## Specific Ideas

- User corrected “dirty” terminology: dirty means changed but not cached, but the UI should still use only gray/green/pink square cells.
- Desired Roto spreadsheet colors: gray = empty, green = cached/no editable paint, pink = editable session Roto paint.
- Current frame should be a bright outline over the square cell.
- Roto interpolation display examples: default `[1][2][3]`; interpolation frame count `1` → `[1]-[2]-[3]`; interpolation frame count `2` → `[1]--[2]--[3]`.
- In Physics Paint, real Roto keys are squares and interpolation is a line between squares. Generated in-between cache frames are visible only in EFX Motion playback/export.
- Frame duplication interpolation is for making extra frames from the first key with motion deform/position options.
- Frame blending interpolation is for transparency blending between first and second key with motion deform/position options.
- Duplicate Roto key is useful for creating an animation pause.
- Copy/paste frame should replace; duplicate/insert/delete should ripple real Roto keys.

</specifics>
