# Phase 36.2 Plan 03: Roto Cache Lifecycle Summary

**Cache-first Roto lifecycle with dirty-frame save boundaries, cached-only reference repainting, pending-save controls, and best-effort close flushing.**

## Accomplishments

- Added dirty Roto tracking so paint/edit intent marks frames pending without continuously sending apply payloads during brush movement.
- Refactored Roto saving through `flushRotoFrame(frame, options)` so navigation, Save pending/current, and close/unload all share the same guarded `apply-canvas` path.
- Added cached-only Roto reference lookup/display from session previews, launch `cachedRotoFrames`, and `physicPaintStore.getFrame` fallback.
- Ensured cached references are reset before replacement export so repainting a cached-only frame replaces the previous cache instead of compositing over it.
- Wired `pendingRotoFrames`, `rotoSaveInFlight`, and `onSavePendingRotoFrames` into the workflow strip.
- Preserved bridge operation ID matching and existing apply result handling.

## Decisions Made

- Dirty Roto state is tracked separately from bridge apply status so brush input can mark local pending state without triggering expensive apply traffic.
- `flushRotoFrame(frame, options)` is the single save boundary for navigation, manual Save pending/current, and close/unload, which keeps operation ID matching centralized.
- Cached-only Roto frames are visual references/backgrounds only; repaint saves reset the cached reference path before rendering the replacement frame.
- Navigation returns early while Roto save/apply work is in flight, rather than queueing multiple competing frame syncs.
