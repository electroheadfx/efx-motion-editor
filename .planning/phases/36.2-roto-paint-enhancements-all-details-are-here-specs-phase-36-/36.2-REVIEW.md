# Phase 36.2: Code Review Report

Reviewed the listed Physics Paint studio, workflow, bridge, renderer, store, and type files at standard depth. The review found correctness and security issues that can lose Roto frame data, make visible UI actions fail to persist, or allow untrusted cross-origin frame-sync messages.

## Critical Issues

### CR-01: Roto interpolation can overwrite real adjacent key frames

**File:** `app/src/stores/physicPaintStore.ts`

`_regenerateGeneratedRotoCache` rounds fractional in-between frame numbers with `Math.round(span.frame)` and writes the generated frame into `layerFrames`. Adjacent real keys such as `[1, 2]` with one in-between produce `1.5`, which rounds to frame `2` and can overwrite the real key with a generated interpolation frame.

**Fix:** Only generate interpolation frames for free integer frames strictly between real keys, and skip targets that collide with real keys.

### CR-02: Clearing a saved Roto frame never removes it from the main editor output

**File:** `app/src/components/physic-paint/PhysicsPaintStudio.tsx`

`clearActiveSource` clears the local engine and marks the current Roto frame dirty, but the later flush path returns `null` when the editable state should not persist. No apply/delete payload is sent to the main editor, so a previously saved Roto frame can remain in the parent editor/store even though the standalone UI says it was cleared.

**Fix:** Add an explicit removal payload path for cleared Roto frames, or make `flushRotoFrame` send a removal operation when a forced dirty frame has no persistable content.

### CR-03: Roto key utility actions update UI/local refs without persisting affected frames

**File:** `app/src/components/physic-paint/PhysicsPaintStudio.tsx`

Duplicate, insert, delete, and paste mutate `rotoFrameStatesRef`, `savedRotoFrames`, and `occupiedRotoFrames`, but they do not reliably mark target/shifted frames dirty or send corresponding apply/remove operations to the parent editor. User-visible key utility actions can be lost on close/reopen, and delete/insert shifts can leave stale rendered frames in the parent output.

**Fix:** Treat key utilities as persisted mutations: mark every affected frame dirty and flush them through existing apply/removal paths, or send a dedicated batch key-edit payload to the main store.

### CR-04: Frame-sync listener accepts unauthenticated cross-origin messages

**File:** `app/src/lib/physicPaintBridge.ts`

`installPhysicPaintFrameSyncListener` listens for all `message` events and passes `event.data` directly to `handlePhysicPaintFrameSyncMessage`. Unlike the apply listener, it does not check `event.origin`. Any embedded/child/cross-origin window that can post to the editor window can seek the timeline and force `ensureFrameVisible` in the browser fallback path.

**Fix:** Match the origin validation used by the apply listener, or pass an explicit allowed origin into `installPhysicPaintFrameSyncListener` and compare against it.

## Warnings

### WR-01: Launch-context parsing silently fails for encoded browser fallback contexts

**Files:** `app/src/components/physic-paint/PhysicsPaintStudio.tsx`, `app/src/lib/physicPaintBridge.ts`

`buildPhysicsPaintUrl` encodes JSON context with `encodeURIComponent` before passing it to `URLSearchParams.set`, which encodes it again. The receiving side decodes only one layer via `URLSearchParams`; `parseLaunchContext` calls `JSON.parse(encodedContext)` without `decodeURIComponent`. Browser fallback launch contexts can fail to parse and fall through to flat query parsing without most Play/Roto cache metadata.

**Fix:** Stop pre-encoding before `URLSearchParams.set`, or decode once before JSON parsing.

### WR-02: Store accepts invalid loaded editable states from persisted project data

**File:** `app/src/stores/physicPaintStore.ts`

`loadFromMceOutputs` stores `output.editable_state` with `structuredClone` without validating it via `isSerializedProject`. Other payload paths validate editable states before mutation, so malformed project data can hydrate into the runtime store and later crash launch flows.

**Fix:** Validate persisted editable state before storing it.
