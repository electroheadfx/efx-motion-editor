## User Constraints (from CONTEXT.md)

### Locked Decisions

### Cache-First Editable Model
- **D-01:** Phase 36.2 must ship Roto as cache-first only. Cached PNG frames are the durable Roto playback/export truth; do not add guaranteed per-frame editable stroke persistence in this phase.
- **D-02:** When reopening or navigating to a cached Roto frame with no editable stroke state, show the cached PNG immediately as a visual reference/background. If the user paints, start from a clean editable state.
- **D-03:** Cached-only frames are repaintable, not stroke-editable. Communicate this with subtle status text, not a modal or noisy warning.
- **D-04:** If the user paints over a cached-only reference frame, the next save replaces that frame's cached PNG. Do not composite over the old cache by default and do not ask every time.

### Auto-Save and Flush Rules
- **D-05:** Roto auto-save happens on frame leave. Painting marks the current Roto key dirty; when the user navigates from frame A to frame B, the standalone saves/renders dirty frame A once, then moves to frame B.
- **D-06:** No auto-save should run during active brush movement. Avoid repeated apply/render requests while staying on the same frame.
- **D-07:** Closing the standalone with a dirty current Roto key must attempt one final save/render/apply so closing preserves visible cached output in EFX Motion.
- **D-08:** While a dirty frame is being saved during navigation, block briefly with a saving state and prevent second navigation until the previous frame apply is queued/settled enough to avoid races.
- **D-09:** Do not implement a default costly Save all frames re-render. With save-on-leave working, already cached frames should not be re-rendered by default.
- **D-10:** Replace the current `Save roto frame` primary action with a modest `Save pending` / `Save current` action that flushes only unsaved dirty frame(s), disabled when nothing is dirty.

### Roto Spreadsheet Cell UX
- **D-11:** The Roto timeline/spreadsheet uses the same square cell model, with exactly three semantic fills: gray, green, and pink.
- **D-12:** Gray means empty/no cached output and no editable session Roto key.
- **D-13:** Green means cached PNG output exists but no editable Roto state exists in the current standalone session.
- **D-14:** Pink means editable Roto state exists in the currently open standalone session. A pink frame may already have cache after frame-leave save; pink is not limited to unsaved/dirty.
- **D-15:** Do not add a separate dirty color. Pending save may be shown with subtle status or a small non-semantic indicator if needed.
- **D-16:** The current frame overlays the gray/green/pink fill with a bright outline/active border. It must not obscure the cell's semantic fill.

### Missing, Empty, Deleted, and Background Frames
- **D-17:** Empty or deleted real Roto keys remove their cached output. Do not persist empty/deleted keys as green cached cells.
- **D-18:** During EFX Motion playback/export, frames with no real Roto key and no generated interpolation cache render transparent when the Roto background is transparent.
- **D-19:** During EFX Motion playback/export, frames with no real Roto key and no generated interpolation cache render the paper/background only when a paper/background is active.
- **D-20:** Paper/background-only output for missing frames is playback/export behavior only. It should not create green cached cells for empty frames.
- **D-21:** Interpolation only generates between two real Roto key squares. If one side of a span is missing, no interpolation is generated and missing-frame background rules apply.

### Roto Interpolation Tail Plan
- **D-22:** Include Roto interpolation as the final/tail plan of phase 36.2, after cache-first cells, cached preview, and auto-save are working.
- **D-23:** Default Roto remains frame-by-frame. When interpolation is enabled, a global in-between frame value controls spacing: `1` gives `[1]-[2]-[3]`, `2` gives `[1]--[2]--[3]`, and so on.
- **D-24:** Interpolation has two modes: frame duplication and frame blending.
- **D-25:** Frame duplication duplicates the source key frame into in-between cached frames and applies motion options: deform and position.
- **D-26:** Frame blending blends between the first and second real key frame in the in-between cached frames with transparency and applies motion options: deform and position.
- **D-27:** Interpolation generates real cached PNG frames in EFX Motion for preview/export. It is not only a virtual standalone preview.
- **D-28:** In Physics Paint, interpolation is represented only as connector lines between real Roto key squares, e.g. `[]-[]`. Generated interpolation frames are not shown as editable boxes.
- **D-29:** The user can move, edit, and erase only real Roto key squares in Physics Paint. Generated interpolation frames are render-only cache.
- **D-30:** Interpolation settings are global for the Roto layer/session, not per gap.
- **D-31:** When global interpolation settings change, regenerate all generated in-between cached frames between existing Roto key pairs.
- **D-32:** If the standalone has been closed and editable session strokes are gone, interpolation changes later work from the cached transparent key images.
- **D-33:** Opening Physics Paint from an EFX Motion frame that contains only a generated interpolation cache frame should open the nearest real Roto key frame. Do not turn interpolation-only frames into editable targets.

### Roto Key Edit Utilities Tail Plan
- **D-34:** Include Roto key edit utilities in the same final/tail plan as interpolation: duplicate key, insert frame, delete frame, copy frame, and paste frame.
- **D-35:** Duplicate Roto key defaults to duplicating the selected key to the next frame, useful for creating a pause in animation.
- **D-36:** If the next frame already has a real Roto key, duplicate performs a ripple insert: push all later real Roto keys one frame to the right to make room for the duplicated key.
- **D-37:** Insert frame also shifts later real Roto keys one frame to the right.
- **D-38:** Delete frame removes the selected real Roto key and shifts later real Roto keys one frame to the left.
- **D-39:** Copy/paste is replace-style: paste replaces the selected/current real Roto key without changing timing.
- **D-40:** Ripple insert/duplicate/delete affect only real Roto key frames. Generated interpolation cache is regenerated from the updated key sequence; it is not independently shifted as editable data.
- **D-41:** Duplicated keys and ripple-edited key sequences participate in interpolation like any other Roto keys, including global motion options.

### Claude's Discretion
Planner/researcher may choose exact storage names, helper boundaries, CSS class names, subtle status copy, and implementation sequencing inside the core cache/autosave plans, but must preserve the cache-first model, the gray/green/pink cell semantics, save-on-leave/save-on-close behavior, and the final tail-plan scope for interpolation/key utilities.

### Deferred Ideas (OUT OF SCOPE)
- Durable per-frame editable Roto stroke persistence remains deferred. 36.2 should be honest that cached-only frames are repaintable, not fully stroke-editable.
- Stroke selection/editing for Roto frames remains out of scope for this phase.
- A broader persisted source-lane model, hybrid Roto/Play source mixing, and editable generated interpolation frames remain future work.

The existing app already stores rendered Physics Paint PNGs in `physicPaintStore._frames`, serializes them through `toMceOutputs()`, hydrates them through `loadFromMceOutputs()`, and draws them in preview/export through the existing renderer path. The risk is preserving the user's mental model: Roto key squares are real editable-session keys, green cached cells are durable PNGs, generated interpolation frames are render-only cache, and navigation/close flushing must not race active brush input or double navigation. [VERIFIED: SPECS/phase-36.2-physics-paint-roto-cache.md; VERIFIED: 36.2-CONTEXT.md]

**Primary recommendation:** Reuse the existing `apply-canvas` rendered-frame pipeline and store `_frames` as the durable Roto cache, adding only the minimum Roto metadata/state needed to distinguish real session-editable keys, cached-only frames, pending dirty keys, and generated interpolation cache. ## Architecture Patterns

### System Architecture Diagram

```text
User paints/edits Roto key in standalone
  -> PhysicsPaintStudio marks current real key dirty after paint/edit intent
  -> User navigates/closes/clicks Save pending
     -> if dirty and not active brush movement: engine.save() + exportCompositeCanvas()
     -> send apply-canvas payload with fresh operationId
     -> physicPaintBridge validates and routes payload
     -> physicPaintStore.applyCanvas stores renderedFrame in _frames and bumps physicPaintVersion
     -> PreviewRenderer/exportRenderer read cached PNG by layer/frame
     -> EFX Motion playback/export displays durable Roto cache

User opens/navigates to Roto frame
  -> launch context/store cache identifies current frame
  -> if session editable state exists: pink cell and editable state can load
  -> else if cached PNG exists: green cell and cached reference/background displays immediately
  -> else: gray empty cell and transparent/clean engine state

Tail interpolation
  -> real Roto key sequence + global interpolation settings
  -> generate render-only cached in-between PNGs between adjacent real keys
  -> store generated cache for preview/export
  -> standalone strip shows only real key squares plus connector lines
```

### Pattern 1: Store rendered output, not engine internals

**What:** Use `PhysicPaintRenderedFrame` PNG data URLs as the durable playback/export artifact, and avoid transporting engine internals over the bridge. [VERIFIED: app/src/types/physicPaint.ts]

**When to use:** Every Roto auto-save, save-on-close, cached-only repaint, interpolation output, and key utility regeneration should ultimately update rendered cache frames in `physicPaintStore`. [VERIFIED: 36.2-CONTEXT.md]

**Example:**

```typescript
// Source: app/src/components/physic-paint/PhysicsPaintStudio.tsx + app/src/types/physicPaint.ts
const editableState = engine.save();
const renderedFrame = buildRotoOutputFrame(engine, currentFrame);
const payload: PhysicPaintApplyPayload = {
  operationId,
  kind: 'apply-canvas',
  layerId: launchContext.layerId,
  startFrame: currentFrame,
  editableState,
  renderedFrame,
};
await sendPhysicPaintApplyPayload(payload, bridgeMode);
```

### Pattern 3: Derive Roto cell semantics from two sources

**What:** Pink is session editable state in the open standalone; green is durable cached PNG without editable session state; gray is neither. Dirty state is not a semantic fill. [VERIFIED: 36.2-CONTEXT.md]

**When to use:** `PhysicsPaintWorkflowStrip` currently has `occupiedRotoFrames` and `savedRotoFrames`; the plan should add a small cell-state model that can produce gray/green/pink/current/pending without conflating saved cache with editable state. [VERIFIED: PhysicsPaintWorkflowStrip.tsx]

**Example:**

```typescript
// Source: recommended helper shape based on 36.2-CONTEXT.md and existing props
function getRotoCellState(frame: number, sessionEditableFrames: Set<number>, cachedFrames: Set<number>) {
  if (sessionEditableFrames.has(frame)) return 'editable'; // pink
  if (cachedFrames.has(frame)) return 'cached-only';       // green
  return 'empty';                                         // gray
}
```

### Pattern 4: Block navigation during a pending Roto flush

**What:** Save-on-leave should use a single in-flight operation guard and avoid a second navigation until the previous apply is queued/settled enough to prevent state races. [VERIFIED: 36.2-CONTEXT.md]

**When to use:** Wrap `navigateToSyncedFrame`, timeline cell clicks, nav buttons, and keyboard frame navigation with the same guard. [VERIFIED: PhysicsPaintStudio.tsx; PhysicsPaintWorkflowStrip.tsx]

**Example:**

```typescript
// Source: recommended pattern based on existing activeOperationIdRef/applyStatus/pendingRotoAdvanceRef
if (rotoFlushInFlightRef.current) return false;
rotoFlushInFlightRef.current = true;
try {
  await flushDirtyRotoFrame(previousFrame, { advanceToFrame: nextFrame });

}
```

### Anti-Patterns to Avoid

- **Re-render all Roto frames by default:** A costly default Save all conflicts with D-09 and undermines cache-first scrubbing. [VERIFIED: 36.2-CONTEXT.md]
- **Dirty as a fourth color:** The user locked exactly gray/green/pink semantic fills; dirty belongs in subtle text or a small non-semantic indicator. [VERIFIED: 36.2-CONTEXT.md]
- **Treat cached-only frames as stroke-editable:** Cached-only frames are repaintable references, not editable stroke sources. [VERIFIED: 36.2-CONTEXT.md]
- **Persist empty/deleted keys as green cells:** Empty/deleted real keys must remove cached output; background-only behavior for missing frames is playback/export-only and must not create green cells. [VERIFIED: 36.2-CONTEXT.md]
- **Confuse Play ranges with Roto interpolation:** Play script ranges are editable script sources; generated Roto interpolation cache is render-only and should not become a Play-like source module. [VERIFIED: 36.1-CONTEXT.md; 36.2-CONTEXT.md]
- **Batch `renderFromStrokes` integration:** The milestone explicitly excludes the failed headless/batch replay path because it damaged physics quality and scaling. [VERIFIED: .planning/REQUIREMENTS.md; .planning/STATE.md]

## Common Pitfalls

### Pitfall 1: Saving during brush movement

**What goes wrong:** Repeated `engine.save()`/render/apply calls during active drawing can lag the brush and race stroke finalization. [VERIFIED: SPECS/phase-36.2-physics-paint-roto-cache.md]
**Why it happens:** Dirty tracking is attached to every pointer mutation rather than frame-leave/close/save-pending boundaries. [VERIFIED: 36.2-CONTEXT.md]
**How to avoid:** Mark dirty on paint/edit intent, but flush only when leaving a frame, closing, or clicking Save pending. [VERIFIED: 36.2-CONTEXT.md]
**Warning signs:** Multiple apply payloads for one continuous stroke while the frame number has not changed. [VERIFIED: SPECS/phase-36.2-physics-paint-roto-cache.md]

### Pitfall 2: Current-frame border destroys semantic fill

**What goes wrong:** Existing CSS sets `.physics-paint-roto-cell.current` background to orange, which would obscure gray/green/pink semantics if reused unchanged. [VERIFIED: physicsPaintStudio.css]
**Why it happens:** Current state is currently encoded as a fill, not an overlay/outline. [VERIFIED: physicsPaintStudio.css]
**How to avoid:** Change current to a bright border/outline/box-shadow only, preserving the cell background color. [VERIFIED: 36.2-CONTEXT.md]
**Warning signs:** Current cached-only frame appears orange instead of green, or current editable frame appears orange instead of pink. [VERIFIED: 36.2-CONTEXT.md]

### Pitfall 3: Store cache updated without version bump

**What goes wrong:** Preview/export or project serialization may not reflect new/removed/generated Roto frames. [VERIFIED: physicPaintStore.ts; previewRenderer.ts]
**Why it happens:** Direct mutation of Maps bypasses `_notifyVisualChange()`. [VERIFIED: physicPaintStore.ts]
**How to avoid:** Add store methods for remove/upsert/generated-cache operations and route all mutations through notification. [VERIFIED: physicPaintStore.ts]
**Warning signs:** UI cell changes but main preview/export remains stale until unrelated edits. [VERIFIED: project memory]

### Pitfall 4: Deleting a Roto key leaves a stale green cache

**What goes wrong:** The strip shows output for a deleted/empty frame, or EFX Motion still plays the old cached PNG. [VERIFIED: 36.2-CONTEXT.md]
**Why it happens:** Clearing only `rotoFrameStatesRef` and local UI markers does not remove the durable store frame. [VERIFIED: PhysicsPaintStudio.tsx; physicPaintStore.ts]
**How to avoid:** Clear/delete must remove the store frame for transparent output and clear local session/cached markers; background-only missing-frame rendering must remain virtual playback/export behavior. [VERIFIED: 36.2-CONTEXT.md]
**Warning signs:** Clear current Roto frame reports success but playback still displays the old drawing. [VERIFIED: SPECS/phase-36.2-physics-paint-roto-cache.md]

### Pitfall 5: Launch context only exposes the current Roto frame

**What goes wrong:** Reopening standalone cannot color all cached Roto frames green or navigate quickly to cached-only frames. [VERIFIED: physicPaintBridge.ts]
**Why it happens:** `createPhysicPaintLaunchContext()` currently checks `physicPaintStore.getFrame(layerId, currentFrame)` for current-frame editable state, but the launch context type has cached Play frames and not an equivalent Roto cached-frame list. [VERIFIED: physicPaintBridge.ts; types/physicPaint.ts]
**How to avoid:** Add a Roto cache summary/list to launch context or provide a store-backed lookup path available in browser fallback, then hydrate `savedRotoFrames`/cached-only cell states on launch. [VERIFIED: SPECS/phase-36.2-physics-paint-roto-cache.md]
**Warning signs:** Only the opened frame is green after reopen, even though project save/load contains multiple cached frames. [VERIFIED: SPECS/phase-36.2-physics-paint-roto-cache.md]

### Pitfall 6: Interpolation frames become editable targets

**What goes wrong:** Generated in-between cache frames appear as editable boxes, accept erasing/moving, or open as real Roto keys. [VERIFIED: 36.2-CONTEXT.md]
**Why it happens:** Generated cache is stored in the same rendered-frame map as real keys without metadata distinguishing source. [VERIFIED: 36.2-CONTEXT.md; physicPaintStore.ts]
**How to avoid:** Track real Roto key frames separately from generated interpolation frames; opening a generated-only frame should redirect to the nearest real Roto key. [VERIFIED: 36.2-CONTEXT.md]
**Warning signs:** Ripple delete shifts generated frames as if they were editable keys. [VERIFIED: 36.2-CONTEXT.md]

## Open Questions (RESOLVED)

1. **How should generated interpolation cache be represented in serialized project data?**
   - What we know: real Roto keys and generated interpolation frames must be distinguishable, and generated frames are render-only cache. [VERIFIED: 36.2-CONTEXT.md]
   - **RESOLVED:** Generated interpolation PNGs live in the existing `_frames` rendered-frame map so preview/export keep using the established cache path, while a separate minimal Roto metadata map in `physicPaintStore` records each cached frame source as `real-key` or `generated-interpolation`. MCE output persists the existing PNG frames plus only the metadata required to distinguish real keys from generated cache and restore interpolation settings; it does not add durable per-frame editable stroke state. Plans 01, 02, 04, and 06 must treat typed `PhysicPaintRotoCacheFrame` metadata as the contract: real keys can become pink/editable-session in the open standalone, generated interpolation frames are filtered out of editable cell rendering and used only as render-only cache/connector data.

2. **What exact close event path should flush dirty Roto frames in Tauri/browser fallback?**
   - What we know: D-07 requires a final save/render/apply on close. [VERIFIED: 36.2-CONTEXT.md]
   - **RESOLVED:** Plan 03 implements a best-effort close/unload path that calls the same dirty-frame flush helper used by Save pending and save-on-leave. Tauri/browser lifecycle limitations do not weaken the requirement: dirty current Roto state must attempt one final render/apply, visible status must report the attempt/failure, and automated tests/source assertions must prove the close handler shares the named flush helper instead of introducing a second persistence path.

3. **Should Roto cached reference use engine background image or a separate overlay?**
   - What we know: Play already uses `setBackgroundImageUrl` and `cachedPlayPreviewUrl` patterns, and cached Roto references must not be composited into saved output by default. [VERIFIED: PhysicsPaintStudio.test.ts; 36.2-CONTEXT.md]
   - **RESOLVED:** Prefer a separate cached-reference overlay for Roto when painting over cached-only frames, because it is visibly a reference and cannot be captured by `engine.exportCompositeCanvas()`. If implementation reuses an engine background hook for visual fidelity, the save helper must clear/reset the reference immediately before `buildRotoOutputFrame()` and tests must prove D-04 replacement does not composite over the old cache.
