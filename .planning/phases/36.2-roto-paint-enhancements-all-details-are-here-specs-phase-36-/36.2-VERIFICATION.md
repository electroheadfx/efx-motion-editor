## Gaps Found

### 1. Requirement IDs are not accounted for in REQUIREMENTS.md

**Truth:** Phase requirement IDs `PH36.2-CACHE`, `PH36.2-MISSING`, `PH36.2-PLAYBACK`, and `PH36.2-INTERP` must be traceable in `.planning/REQUIREMENTS.md`.

**Status:** failed

**Evidence:** PLAN frontmatter declares all four IDs across Phase 36.2 plans, but `.planning/REQUIREMENTS.md` does not define or trace any `PH36.2-*` IDs.

**Missing:**

- Add explicit definitions and traceability rows for `PH36.2-CACHE`, `PH36.2-MISSING`, `PH36.2-PLAYBACK`, and `PH36.2-INTERP`.

### 2. Blend/position/deform interpolation is not implemented as real generated PNG behavior

**Truth:** Global Roto interpolation should generate render-only cached PNG in-betweens between real Roto keys, and visible controls for blend/deform/position should affect generated output.

**Status:** failed

**Evidence:** `app/src/stores/physicPaintStore.ts` exposes interpolation controls, but `renderDuplicateRotoInterpolationFrame` only copies the source frame and consumes `position`/`deform` through no-op arithmetic. `renderBlendedRotoInterpolationFrame` appends blend metadata to a `dataUrl` string instead of rendering a real alpha-blended image.

**Missing:**

- Implement real offscreen/canvas generation for blend mode.
- Apply `position` and `deform` to generated image pixels or disable/remove those controls until implemented.
- Add behavioral tests that inspect generated output semantics rather than only source-string consumption.

### 3. Roto cached playback does not actually synthesize gap frames across the playback range

**Truth:** Roto cached playback should synthesize transient background-only frames for missing gaps and leave transparent gaps transparent, without creating green cached cells.

**Status:** failed

**Evidence:** `app/src/components/physic-paint/PhysicsPaintStudio.tsx` defines transient missing-background helpers, but `getRotoCachedPlaybackFrames()` only assembles frames from known frame numbers. It does not enumerate missing frame numbers between cached/real keys or across a defined playback range. A gap such as cached frames 0 and 2 will not synthesize frame 1 unless frame 1 is already in one of the known sets.

**Missing:**

- Define the Roto playback range.
- Enumerate every frame in that range.
- For each missing frame, return either `null` for transparent or a transient background-only frame when background metadata is active.
- Add a behavioral/unit test for cached frames with an actual missing in-between gap.

### 4. Close/save and UI behavior are present but not behaviorally proven end-to-end

**Truth:** Closing standalone after painting a dirty Roto frame awaits a final dirty-frame flush before close is allowed, and user-visible Roto UI semantics work as intended.

**Status:** behavior_unverified

**Evidence present:** `app/src/components/physic-paint/PhysicsPaintStudio.tsx` includes `flushCurrentRotoFrameBeforeClose`, Tauri `onCloseRequested` handling that awaits the flush before `appWindow.close()`, a `beforeunload` fallback, and dirty/in-flight Roto refs. Focused tests pass.

**Why not fully verified:** This is runtime behavior involving Tauri/browser close lifecycle and bridge apply-result ordering. Static code and source-contract tests cannot prove the close event is delayed in the live app under all supported close paths. Visual semantics also require human confirmation because project instructions say not to run the server/app.

**Human verification needed:**

1. Paint a Roto frame, do not navigate, close the standalone window.
2. Reopen the project/layer.
3. Confirm the frame’s cached output persists and appears as a full-strength repaintable reference.
4. Confirm empty/background-only visited frames remain gray, cached-only frames are green, editable session frames are pink, and generated frames are connector-only.

## Verified Must-Haves

1. **Roto cache metadata/types exist and validate.** Evidence: `app/src/types/physicPaint.ts` defines Roto cache frame/source/interpolation settings and validators.
2. **Roto cache/store helpers exist and mutate existing frame store.** Evidence: `app/src/stores/physicPaintStore.ts` includes cache accessors, real-key upsert/remove, generated replacement, settings, and regeneration helpers.
3. **Gray/green/pink cell classification exists and excludes generated frames from square cells.** Evidence: `physicsPaintWorkflowState.ts` defines `RotoCellFill`, and `PhysicsPaintWorkflowStrip.tsx` filters generated frames out of editable square cells.
4. **Save pending/current visible action exists and disables when no pending Roto frames exist.** Evidence: `PhysicsPaintWorkflowStrip.tsx` renders `Save pending` or `Save current` and disables it when there are no pending Roto frames.
5. **Cached Roto launch hydration and generated-only redirect exist.** Evidence: `physicPaintBridge.ts` reads Roto cache frames and redirects generated-only frames to nearest real key.
6. **Visible interpolation controls exist.** Evidence: `PhysicsPaintWorkflowStrip.tsx` renders `Interpolation`, `In-betweens`, `Duplicate`, `Blend`, `Deform`, and `Position` controls.
7. **Visible real-key utility controls exist and wire to Studio handlers.** Evidence: `PhysicsPaintWorkflowStrip.tsx` renders `Duplicate key`, `Insert frame`, `Delete frame`, `Copy frame`, and `Paste frame`; `PhysicsPaintStudio.tsx` passes corresponding callbacks.
