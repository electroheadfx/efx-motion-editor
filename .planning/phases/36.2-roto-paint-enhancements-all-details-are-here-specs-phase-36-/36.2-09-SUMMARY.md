# Phase 36.2 Plan 09: Visible Real-Key Utility Controls Summary

Visible standalone Roto Duplicate/Insert/Delete/Copy/Paste controls now call the existing real-key utility handlers with disabled/status affordances that preserve generated interpolation frames as render-only cache.

## What Was Built

- Added a compact `physics-paint-roto-key-utilities` toolbar to the Roto workflow strip with visible `Duplicate key`, `Insert frame`, `Delete frame`, `Copy frame`, and `Paste frame` buttons.
- Added real-key target and copied-key disabled logic so empty/generated-only frames cannot be presented as editable key utility targets.
- Threaded Studio callbacks into the strip for all five utilities, reusing existing `duplicateRotoKey`, `insertRotoFrame`, `deleteRotoFrame`, `copyRotoFrame`, and `pasteRotoFrame` handlers.
- Added a copied-key state flag to enable Paste only after Copy and only on real Roto key targets.
- Added handler-side rejection copy for non-real-key targets and kept interpolation regeneration after real-key sequence mutations.
- Added colocated status copy clarifying real-key-only scope, render-only generated in-betweens, replace-style Paste, and ripple-only real-key semantics.
- Styled the toolbar and subtle status text to match the existing compact Physics Paint control surface.

## Decisions Made

1. Roto key utility controls are inline in the standalone workflow strip rather than hidden behind keyboard shortcuts or another panel.
2. The strip owns button visibility/disabled affordances, while Studio remains responsible for mutation and regeneration logic.
3. Non-real-key rejection exists in both UI disabled state and Studio handler guards so generated-only frames cannot be mutated via accidental callbacks.
4. Status copy explicitly says Paste replaces the current real key and generated in-betweens are render-only.
