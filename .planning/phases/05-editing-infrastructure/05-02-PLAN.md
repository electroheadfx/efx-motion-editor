**1. Create `Application/src/lib/history.ts`:**

Build the undo/redo engine following the command pattern from 05-RESEARCH.md:

```typescript
import { batch } from '@preact/signals';
import { historyStore } from '../stores/historyStore';
import type { HistoryEntry } from '../types/history';

const MAX_STACK_SIZE = 200;
let coalescing = false;
let coalesceEntry: HistoryEntry | null = null;
```

Implement these exported functions:

- `pushAction(entry: HistoryEntry): void`
  - If coalescing AND coalesceEntry exists: update the redo closure of coalesceEntry (keep original undo), return early
  - Truncate stack at pointer+1 to discard stale redo entries (CRITICAL — see Pitfall 2 in research)
  - Push entry to stack
  - If stack exceeds MAX_STACK_SIZE, shift oldest entry
  - Use batch() to update stack.value and pointer.value atomically
  - If coalescing and no coalesceEntry yet, set coalesceEntry to the pushed entry

- `undo(): void`
  - If pointer < 0, return (nothing to undo)
  - Get entry at pointer position
  - Call entry.undo() inside batch() (atomic signal restoration)
  - Decrement pointer

- `redo(): void`
  - If pointer >= stack.length - 1, return (nothing to redo)
  - Increment pointer
  - Get entry at new pointer position
  - Call entry.redo() inside batch()

- `startCoalescing(): void`
  - Set coalescing = true, coalesceEntry = null

- `stopCoalescing(): void`
  - Set coalescing = false, coalesceEntry = null

- `resetHistory(): void`
  - Use batch() to set stack.value = [] and pointer.value = -1

- `canUndo(): boolean` (computed-like helper)
  - Returns historyStore.pointer.value >= 0

- `canRedo(): boolean`
  - Returns historyStore.pointer.value < historyStore.stack.value.length - 1

**CRITICAL IMPLEMENTATION DETAILS:**
- Always use `structuredClone()` when capturing before/after signal snapshots (Pitfall 3: shallow copy trap)
- The undo() closure must restore state inside `batch()` to prevent intermediate renders
- Do NOT store full project snapshots — only the specific signal(s) that changed (sequences signal)
- Coalescing pattern: first pushAction during coalesce sets coalesceEntry, subsequent pushActions update only the redo closure (keeping original undo = before state from mousedown)

<task type="auto">
  <name>Task 2: Wrap all sequenceStore mutations with undo support</name>
  <files>Application/src/stores/sequenceStore.ts</files>
  <action>
Wrap every undoable mutation in `sequenceStore` to capture before/after state snapshots and push to the history engine.

**Pattern for each mutation:**
1. Capture `before` state: `const before = structuredClone(sequences.peek())` and `const beforeActive = activeSequenceId.peek()`
2. Execute the existing mutation logic (keep it inline, don't call the method recursively)
3. Capture `after` state: `const after = structuredClone(sequences.peek())` and `const afterActive = activeSequenceId.peek()`
4. Call `pushAction({ id: crypto.randomUUID(), description: '...', timestamp: Date.now(), undo: () => { batch(() => { sequences.value = before; activeSequenceId.value = beforeActive; }); markDirty(); }, redo: () => { batch(() => { sequences.value = after; activeSequenceId.value = afterActive; }); markDirty(); } })`

Import `pushAction` from `../lib/history`.

**Mutations to wrap (per user decision — sequence edits only):**

1. `createSequence(name)` — description: `'Create sequence "${name}"'`
2. `remove(id)` — description: `'Delete sequence'`
3. `duplicate(id)` — description: `'Duplicate sequence'`
4. `reorderSequences(oldIndex, newIndex)` — description: `'Reorder sequences'`
5. `rename(id, name)` — description: `'Rename sequence'`
6. `setSequenceFps(id, fps)` — description: `'Change sequence FPS'`
7. `setSequenceResolution(id, width, height)` — description: `'Change sequence resolution'`
8. `addKeyPhoto(sequenceId, imageId, holdFrames)` — description: `'Add key photo'`
9. `removeKeyPhoto(sequenceId, keyPhotoId)` — description: `'Remove key photo'`
10. `reorderKeyPhotos(sequenceId, oldIndex, newIndex)` — description: `'Reorder key photos'`
11. `updateHoldFrames(sequenceId, keyPhotoId, holdFrames)` — description: `'Update hold frames'`

**NOT wrapped (per user decision):**
- `add(seq)` — hydration from .mce file, not a user edit
- `setActive(id)` — UI navigation, not data
- `reset()` — lifecycle, not editing

**IMPORTANT implementation notes:**
- Each method still calls `markDirty()` as before (for auto-save). The undo/redo closures also call `markDirty()` because restoring state is itself a data change that should trigger auto-save.
- Use `sequences.peek()` (not `.value`) for the snapshot capture to avoid signal subscriptions in non-reactive context.
- Use `structuredClone()` for ALL snapshots — the Sequence type has nested `keyPhotos` arrays (Pitfall 3 from research).
- The `createSequence` method returns the created Sequence. After wrapping, it must still return the new sequence object. Capture the return value before taking the `after` snapshot.
- The `duplicate` method returns the copy or null. Same — capture return before snapshot.
