**Researched:** 2026-03-19

## Summary

This phase replaces all Unicode text-based button characters across three areas (TimelinePanel, CanvasArea, Toolbar) with lucide-preact SVG icons, making all buttons icon-only with native HTML `title` tooltips that include keyboard shortcuts. No layout rearrangement is needed -- only character-to-icon swaps within existing button elements.

The project already uses lucide-preact v0.577.0 extensively in sidebar components (KeyframeNavBar, LayerList, SequenceList, etc.), establishing a clear pattern: `<IconName size={N} />` with `currentColor` inheritance. Every icon specified in the user's decision list has been verified to exist in the installed version.

**Primary recommendation:** Perform icon swaps file-by-file (TimelinePanel, CanvasArea, Toolbar) using the established `<IconName size={N} />` pattern, keeping all existing button wrappers, click handlers, and conditional styling intact. Use native HTML `title` attributes for tooltips (already used throughout the app).

### Pattern 1: Icon Button (established in KeyframeNavBar.tsx)
**What:** Replace `<span>Unicode</span>` with `<IconName size={N} />` inside existing button elements
**When to use:** Every button replacement in this phase
**Example:**
```tsx
// BEFORE (TimelinePanel.tsx line 47-49):
<span class="text-[11px] text-white">
  {timelineStore.isPlaying.value ? '\u23F8' : '\u25B6'}
</span>

// AFTER:
{timelineStore.isPlaying.value ? <Pause size={14} /> : <Play size={14} />}
```

### Pattern 2: Conditional Icon Toggle (for fit-lock)
**What:** Switch between two different icon components based on state
**When to use:** Canvas fit-lock button (Maximize2 vs Minimize2)
**Example:**
```tsx
// Source: CONTEXT.md decision + established pattern in KeyframeNavBar.tsx
<button
  class={`rounded px-2.5 py-1.5 cursor-pointer ${
    canvasStore.fitLocked.value ? 'bg-[var(--color-accent)]' : 'bg-[var(--color-bg-settings)]'
  }`}
  onClick={() => canvasStore.toggleFitLock()}
  title={canvasStore.fitLocked.value ? 'Fit to window -- locked (F)' : 'Fit to window (F)'}
>
  {canvasStore.fitLocked.value
    ? <Minimize2 size={16} class="text-white" />
    : <Maximize2 size={16} class="text-[var(--color-text-secondary)]" />}
</button>
```

### Pattern 3: Native title Tooltip with Shortcut
**What:** Use HTML `title` attribute with "Label (Shortcut)" format
**When to use:** Every button in this phase
**Example:**
```tsx
// Already used in CanvasArea.tsx and Toolbar.tsx
title="Zoom in (=)"
title="Save (Cmd+S)"
title="Bypass All Blur (Shift+B)"
```

### Anti-Patterns to Avoid
- **Wrapping icons in `<span>` tags:** Icons are already valid JSX elements; no wrapper needed. Remove the `<span class="text-[11px]...">` wrapper when replacing.
- **Setting explicit color on icons:** Use `currentColor` (the default) -- color should come from the parent button's `text-[var(--color-*)]` class or `style` prop.
- **Changing button dimensions:** Keep existing `px-*`, `py-*`, `w-*`, `h-*` classes. Only adjust if the icon visually doesn't fit the existing box.

## Common Pitfalls

### Pitfall 1: Breaking the canvas play/pause emphasis styling
**What goes wrong:** The canvas play/pause button has special treatment (rounded-full, accent bg, white text, larger size) that must be preserved while swapping the icon.
**Why it happens:** Copy-pasting the simpler icon swap pattern from other buttons and forgetting this button's unique classes.
**How to avoid:** Keep the existing `class="... w-9 h-9 rounded-full bg-[var(--color-accent)] ..."` intact. Only replace the inner `<span>` content with `<Play size={18} />` / `<Pause size={18} />` and set `class="text-white"` on the icon.
**Warning signs:** Play button looks like all other buttons after the change.

### Pitfall 2: Removing the `<span>` wrapper but keeping its text color class
**What goes wrong:** The `<span class="text-[11px] text-[var(--color-text-secondary)]">` wrapper carries the text color. Removing the span without transferring the color to the icon or parent button leaves icons inheriting wrong color.
**Why it happens:** The color was on the `<span>`, not the `<button>`.
**How to avoid:** Either (a) add the color class to the icon element itself, or (b) add it to the parent `<button>` so `currentColor` inheritance works. Option (b) is cleaner.
**Warning signs:** Icons appear in wrong color or invisible after replacement.

### Pitfall 3: Forgetting to update the AddLayerMenu button text
**What goes wrong:** The AddLayerMenu in TimelinePanel has a `+ Layer` text button that isn't in the CONTEXT.md icon mapping (it's part of the existing "Add Layer" popover trigger).
**Why it happens:** It lives in the same control bar as the other timeline buttons.
**How to avoid:** Leave `AddLayerMenu` unchanged -- it's a separate component (`AddFxMenu.tsx`) and not in scope. The CONTEXT.md explicitly says no new controls, and this is a popover trigger, not a simple action button.
**Warning signs:** Scope creep into AddLayerMenu modification.

### Pitfall 4: Losing the "Saving..." state text on the Save button
**What goes wrong:** Toolbar Save button currently shows "Saving..." during save operations. Making it icon-only loses this feedback.
**Why it happens:** Switching to icon-only removes the text that carried state.
**How to avoid:** Keep the Save icon always, but add a subtle visual indicator for save-in-progress state (e.g., opacity pulse, spinner, or changing the title to "Saving..."). The simplest approach: update the `title` attribute dynamically and optionally add an `opacity-60` class during save.
**Warning signs:** No visual feedback when save is in progress.

### Pitfall 5: Dirty indicator dot displacement
**What goes wrong:** The unsaved changes dot (orange circle) currently sits after the Save text button. With icon-only buttons, its position relative to the Save icon needs to look intentional.
**Why it happens:** Layout shift when button width changes from text to icon.
**How to avoid:** Position the dot adjacent to or overlapping the Save icon (e.g., as a small superscript indicator). The user gave Claude discretion on this.
**Warning signs:** Orphaned orange dot floating between buttons.

### Pitfall 6: ThemeSwitcher uses Unicode circles
**What goes wrong:** The ThemeSwitcher in the toolbar uses Unicode circles for light/medium/dark theme. These are not buttons listed in CONTEXT.md scope.
**Why it happens:** ThemeSwitcher lives inside Toolbar but is a separate component.
**How to avoid:** Leave ThemeSwitcher unchanged -- it's not listed in the icon mapping decisions and is a separate component (`ThemeSwitcher.tsx`).
**Warning signs:** Unnecessary changes to ThemeSwitcher.

## Discretion Decisions (Research Recommendations)

### Timeline "Fit All" Icon: `Shrink`
**Recommendation:** Use `Shrink` -- it visually represents "compress to fit" with inward-pointing arrows, which matches the "fit all frames into view" concept. Alternative `Scan` also works but `Shrink` is more intuitive for "fit into viewport."

### Icon Size Fine-Tuning
- Timeline: 14px for all icons (compact bar, h-9 = 36px height, matching existing text at 11px)
- Canvas: 16px standard, 18px for the emphasis play/pause button (inside 32px/36px hit targets)
- Toolbar: 16px for all icons (inside ~32px square buttons)

### Hover State Transitions
- Keep existing hover classes (already defined with `hover:bg-[var(--color-bg-hover-item)]` etc.)
- No additional transitions needed -- the existing `transition-colors` class handles it

### Tooltip Delay/Position
- Use native `title` attribute -- browser handles delay (~400ms) and positioning automatically
- Consistent with the 19+ existing `title` usages across the app

### Unsaved Indicator
- Keep the orange dot immediately after the Save icon button (same gap-1 flow)
- The dot is small (w-2 h-2) and already has `shrink-0` -- it will look fine adjacent to a 32px icon button
