### Locked Decisions
## Implementation Decisions

### Mode Control
- **D-01:** Transparent vs background-only missing-frame rendering must reuse existing app settings. Do not introduce a new missing-frame mode system for Phase 36.10.
- **D-02:** The primary source is existing Roto paper/background metadata when available. If no saved Roto paper/background metadata exists yet, fall back to the existing layer paint background/color rule; otherwise missing frames render transparent.
- **D-03:** The chosen missing-frame mode should be visible through existing Roto cell/status surfaces only. Use cell state labels such as `Background only` and compact workflow-strip status copy; do not add a new persistent toggle or control.

### Leading, Interior, and Trailing Frame Rules
- **D-04:** An interior missing frame means a missing frame strictly between two real Roto keys. Only this between-real-keys span is eligible for bounded background-only cache materialization.
- **D-05:** Leading frames before the first real cached Roto key use the same dynamic rule as trailing frames after the last real key. They render on demand from existing settings and must not create cache entries or metadata.
- **D-06:** Trailing frames after the last real cached Roto key render dynamically for the requested preview/export frame. If paper/background mode is enabled, they show background-only; otherwise they are transparent. They must not create unbounded per-frame cache entries.

### Cache Bounds and Replacement
- **D-07:** Background-only cache materialization is interior-only. Persist/cache background-only support frames only for missing frames strictly between real keys; never for leading or trailing spans.
- **D-08:** Background-only interior cache entries are render support, not editable painted Roto keys. They must not be treated as old physics paint stroke state or as alpha paint content.
- **D-09:** If the animator opens and paints a background-only interior frame, that exact frame becomes a real Roto key and replaces/removes only that frame's derived background-only marker/cache.
- **D-10:** When key utilities shift, delete, replace, or paste real keys, derived background-only interior caches must be recomputed from the current real-key layout. Remove/regenerate only bounded interior derived entries as needed.
- **D-11:** Existing real cached frames continue to render normally beside missing/background-only frames. Phase 36.9 cached playback remains real-key-only and must not start playing empty/background fallback frames.

### Preview/Export Parity and Alpha Separation
- **D-12:** TDD should prioritize a shared missing-frame resolver/path used by both preview and export, covering transparent, background-only, leading, interior, and trailing cases.
- **D-13:** Preview and export must produce the same visual rule for the same frame and settings. Human UAT should verify representative transparent and background-only missing frames in both live preview and exported PNG sequence.
- **D-14:** Preserving alpha-only real-key paint cache separation is a hard test gate. Background-only rendering must never mutate, replace, or bake paper/background pixels into real-key paint alpha caches.
- **D-15:** Phase 36.10 must keep paper/background rendering separate from paint alpha cache data so Phase 36.11 can reopen cached real keys as alpha-only additive repaint bases.

## Summary

Phase 36.10 should be planned as a narrow, shared missing-frame rendering rule for Physics Paint Roto frames, not as a new renderer, playback feature, interpolation feature, or UI redesign. The source of truth requires missing Roto frames to resolve to either `transparent` or `background-only`; preview and export must use the same rule for the same frame/settings; existing real cached frames must continue rendering normally beside missing/background-only frames. [VERIFIED: SPECS/36.x-phases/phase-36.10-missing-background/spec-36.10-missing-background.md]

The current code already has the right seed: `app/src/lib/rotoFrameDraw.ts` exposes `resolveMissingRotoFrameDraw` and `drawMissingRotoBackground`; `PreviewRenderer` imports them; `exportRenderer.renderGlobalFrame` delegates rendering through `PreviewRenderer.renderFrame`, so parity should be preserved by deepening the shared resolver/path rather than duplicating export logic. [VERIFIED: codebase Read]

**Primary recommendation:** Extend `rotoFrameDraw` into the single source of truth for missing-frame span classification and draw instruction, then update store/session/UI seams only enough to maintain bounded interior `backgroundOnly` support state and compact `Background only` feedback without changing cached playback or real-key alpha cache semantics. [VERIFIED: codebase Read]

## Common Pitfalls

### Pitfall 1: Treating every missing frame as materializable
**What goes wrong:** Leading/trailing frames after the first/last real key create unbounded cache entries during preview/export. [VERIFIED: CONTEXT.md]
**Why it happens:** A resolver sees `background-only` and store code blindly persists every requested frame. [VERIFIED: CONTEXT.md]
**How to avoid:** Classify span before materialization and only allow strict interior frames between two real keys. [VERIFIED: CONTEXT.md]
**Warning signs:** `toMceOutputs()` grows when scrubbing/exporting frames after the last real key. [VERIFIED: codebase Read]

### Pitfall 2: Baking paper into real-key paint cache
**What goes wrong:** A real painted Roto key contains paper/background pixels, breaking alpha-only repaint assumptions for Phase 36.11. [VERIFIED: CONTEXT.md]
**Why it happens:** Background-only support entries are stored through the same path as real painted keys without provenance separation. [VERIFIED: codebase Read]
**How to avoid:** Keep background rendering as draw-time support or explicitly bounded support metadata, and assert real-key `dataUrl` is not replaced by background-only frame generation. [VERIFIED: CONTEXT.md]
**Warning signs:** Saving/replacing one missing frame changes existing real-key `dataUrl` values. [VERIFIED: codebase Read]

### Pitfall 3: Preview/export parity tested only visually
**What goes wrong:** Manual UAT catches a mismatch late because preview and export use subtly different missing-frame branches. [VERIFIED: CONTEXT.md]
**Why it happens:** Export-specific logic bypasses `PreviewRenderer` or resolver source contract. [VERIFIED: app/src/lib/exportRenderer.test.ts]
**How to avoid:** Add source-contract and helper tests proving export remains delegated through `PreviewRenderer` and both paths call the same resolver instruction. [VERIFIED: codebase Read]
**Warning signs:** `exportRenderer.ts` imports `physicPaintStore`/physics engine directly or duplicates `getMissingRotoBackgroundState`. [VERIFIED: codebase Read]

### Pitfall 4: Background-only cell becomes an editable stroke state
**What goes wrong:** The workflow strip presents derived support as actual restored physics paint content. [VERIFIED: CONTEXT.md]
**Why it happens:** `backgroundOnly` is currently represented on `real-key` cache frames, which can blur derived support and no-stroke real-key semantics if not planned carefully. [VERIFIED: codebase Read]
**How to avoid:** Planner should explicitly decide how derived interior support is distinguished from a user-saved no-stroke/background-only real key, while preserving existing labels and replacement behavior. [VERIFIED: codebase Read]
**Warning signs:** Key utilities copy/paste/delete background-only support as if it were a real painted key. [VERIFIED: CONTEXT.md]

### Pitfall 5: Missing frame repaint does not clean derived support
**What goes wrong:** Painting an interior background-only frame leaves stale support metadata/cache alongside the new real key. [VERIFIED: CONTEXT.md]
**Why it happens:** `applyCanvas`/`upsertRealRotoKeyFrame` does not explicitly remove derived support for the same frame before writing the real key. [VERIFIED: codebase Read]
**How to avoid:** Make replacement single-frame and exact: remove derived marker/cache for that frame, then upsert the real key and recompute bounded interiors. [VERIFIED: CONTEXT.md]
**Warning signs:** `getRotoCacheFrames(layerId)` returns duplicate/colliding entries for one frame or workflow state prefers support over real key. [VERIFIED: codebase Read]
