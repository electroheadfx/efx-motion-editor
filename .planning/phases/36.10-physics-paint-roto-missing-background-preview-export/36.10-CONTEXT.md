# Phase 36.10: Physics Paint Roto Missing Background Preview Export - Context

**Gathered:** 2026-06-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 36.10 delivers predictable visual rendering for missing Physics Paint Roto frames in EFX Motion preview and export. Missing frames must render as either transparent or background-only according to existing app settings, with no accidental paint content, no infinite cache generation, and no paper/background pixels baked into real-key paint alpha caches.

This phase covers missing-frame rendering behavior only: interior gaps between real cached Roto keys, leading frames before the first real key, trailing frames after the last real key, replacement when a missing/background-only frame is painted, and preview/export parity proof. It must preserve Phase 36.9 real-key-only cached playback behavior and protect Phase 36.11's alpha-only repaint foundation.

This phase must not add generated interpolation, cached playback changes, timeline redesign, Play/Roto conversion behavior, state-machine/XState work, or final Pencil UI polish.

</domain>

<decisions>
## Implementation Decisions

### Mode Control
- **D-01:** Transparent vs background-only missing-frame rendering must reuse existing app settings. Do not introduce a new missing-frame mode system for Phase 36.10.
- **D-02:** The primary source is existing Roto paper/background metadata when available. If no saved Roto paper/background metadata exists yet, fall back to the existing layer paint background/color rule; otherwise missing frames render transparent.
- **D-03:** The chosen missing-frame mode should be visible through existing Roto cell/status surfaces only. Use cell state labels such as `Background only` and compact workflow-strip status copy; do not add a new persistent toggle or control.

### Leading, Interior, and Trailing Frame Rules
- **D-04:** An interior missing frame means a missing frame strictly between two real Roto keys. Only this between-real-keys span is eligible for bounded background-only cache materialization.
- **D-05:** Leading frames before the first real cached Roto key use the same dynamic rule as trailing frames after the last real key. They render on demand from existing settings and must not create cache entries or metadata.
- **D-06:** Trailing frames after the last real cached Roto key render dynamically for the requested preview/export frame. If paper/background mode is enabled, they show background-only; otherwise they are transparent. They must not create unbounded per-frame cache entries.

### Cache Bounds and Replacement
- **D-07:** Background-only cache materialization is interior-only. Persist/cache background-only support frames only for missing frames strictly between real keys; never for leading or trailing spans.
- **D-08:** Background-only interior cache entries are render support, not editable painted Roto keys. They must not be treated as old physics paint stroke state or as alpha paint content.
- **D-09:** If the animator opens and paints a background-only interior frame, that exact frame becomes a real Roto key and replaces/removes only that frame's derived background-only marker/cache.
- **D-10:** When key utilities shift, delete, replace, or paste real keys, derived background-only interior caches must be recomputed from the current real-key layout. Remove/regenerate only bounded interior derived entries as needed.
- **D-11:** Existing real cached frames continue to render normally beside missing/background-only frames. Phase 36.9 cached playback remains real-key-only and must not start playing empty/background fallback frames.

### Preview/Export Parity and Alpha Separation
- **D-12:** TDD should prioritize a shared missing-frame resolver/path used by both preview and export, covering transparent, background-only, leading, interior, and trailing cases.
- **D-13:** Preview and export must produce the same visual rule for the same frame and settings. Human UAT should verify representative transparent and background-only missing frames in both live preview and exported PNG sequence.
- **D-14:** Preserving alpha-only real-key paint cache separation is a hard test gate. Background-only rendering must never mutate, replace, or bake paper/background pixels into real-key paint alpha caches.
- **D-15:** Phase 36.10 must keep paper/background rendering separate from paint alpha cache data so Phase 36.11 can reopen cached real keys as alpha-only additive repaint bases.

### Claude's Discretion
No decisions were delegated to Claude. The user selected all discussed areas and locked concrete choices for mode source, leading-frame behavior, cache bounds, replacement, parity proof, UAT, and alpha separation.

</decisions>

## Existing Code Insights

### Reusable Assets
- `app/src/lib/rotoFrameDraw.ts` — Existing missing-frame draw helper with `resolveMissingRotoFrameDraw` and `drawMissingRotoBackground`. This is the likely seed for the shared resolver/path that preview and export must use.
- `app/src/lib/previewRenderer.ts` — Existing preview render path already imports `rotoFrameDraw`, derives missing-frame background from Roto metadata or layer background fallback, and draws background-only when a physics-paint frame is missing.
- `app/src/lib/exportRenderer.ts` — Export rendering uses `renderGlobalFrame` and `PreviewRenderer.renderFrame`, so parity should be proven by keeping missing-frame rule resolution shared rather than duplicating export-only logic.
- `app/src/stores/physicPaintStore.ts` — Owns Roto frames, `_rotoCacheMetadata`, `backgroundOnly` metadata, real-key upsert/removal/replacement, interpolation regeneration, serialization, and `physicPaintVersion` invalidation.
- `app/src/types/physicPaint.ts` — Defines `PhysicPaintRotoBackgroundMetadata`, `PhysicPaintRotoCacheFrame`, `backgroundOnly`, apply payloads, and validation. The current source model only distinguishes `real-key` and `generated-interpolation`, so planners must avoid modeling derived background-only support as editable paint content.
- `app/src/components/physic-paint/physicsPaintWorkflowState.ts` — Already has Roto cell states including `background-only`, labels such as `Background only on frame N`, and editable-target semantics for background-only cells.
- `app/src/components/physic-paint/physicsPaintRotoSession.ts` — Phase 36.8 boundary for current frame, real keys, cached/generated frames, dirty state, copied key, save continuations, restore intent, and effect descriptors. Frame ownership decisions should extend this boundary incrementally.
- `app/src/components/physic-paint/PhysicsPaintStudio.tsx` — Main adapter for launch context, Roto save/apply, cached references, frame navigation, Roto session effects, and parent store/bridge communication.
- `app/src/components/physic-paint/PhysicsPaintWorkflowStrip.tsx` — Existing compact UI surface for Roto cells, navigator/status feedback, playback controls, key utilities, and `Background only` cell/status display.

### Established Patterns
- Real Roto keys are the only editable painted frames. Generated/interpolated frames are render-only, and background-only missing-frame caches must also remain render support rather than editable stroke state.
- Roto cache changes must bump `physicPaintVersion` / notify visual change so preview and export observe current cached/missing state.
- Preact/Signals and pure helper/controller tests are preferred over new broad `useEffect` orchestration inside `PhysicsPaintStudio`.
- Existing Roto UI feedback should stay compact in the workflow strip. Phase 36.10 should not add modals, tutorials, or a separate mode toggle.
- Preview/export parity should be source-contract and behavior tested through shared resolver usage before manual UAT.

### Integration Points
- Extend or wrap `resolveMissingRotoFrameDraw` so it can distinguish leading, interior, and trailing missing frames using the current real-key layout and background settings.
- Store/cache logic must support bounded interior background-only materialization without confusing derived background-only support with real painted key alpha data.
- Key-utility and real-key replacement paths must recompute derived interior background-only support after real-key layout changes.
- PreviewRenderer and exportRenderer should keep using the same missing-frame resolver and draw path for the same frame/settings.
- Tests should cover `rotoFrameDraw`, `physicPaintStore`, `physicsPaintWorkflowState`, `physicsPaintRotoSession`, `PreviewRenderer`/`exportRenderer` source contracts, and any affected Studio/WorkflowStrip wiring.

</code_context>

<specifics>
## Specific Ideas

- Keep the phase MVP/TDD. The key design risks are mode selection, leading-frame behavior, bounded vs dynamic cache rules, and tests proving preview/export parity without baking paper/background into real-key paint alpha caches.
- Leading and trailing frames are both dynamic edge spans: render on demand, no cache, no metadata.
- Interior background-only means strictly between two real keys; no timeline-start or export-window interpretation.
- Existing layer background fallback is acceptable when no Roto-specific background metadata has been saved yet.
- Human UAT should include both preview and exported PNG sequence for representative missing transparent/background-only frames.

</specifics>
