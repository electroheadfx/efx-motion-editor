## User Constraints (from CONTEXT.md)

### Locked Decisions
## Implementation Decisions

### Override Creation
- **D-01:** Custom segment spacing overrides must be created from both sources: existing distant source-key gaps discovered during hydration/regeneration, and new far-ahead saves while interpolation is enabled.
- **D-02:** Existing source-key gaps become custom overrides only when the source gap is larger than the current global `inBetweenCount`. Adjacent/normal segments remain controlled by the global count.
- **D-03:** A new far-ahead real key saved while interpolation is enabled uses compressed source spacing when interpolation is later disabled. Example: a display save that creates a 4-in-between custom span after source key `2` becomes source key `6`, not source key `11`.
- **D-04:** Durable custom overrides are keyed by adjacent source segment, with explicit source endpoints such as `fromSourceFrame`/`toSourceFrame` plus the remembered generated in-between count. Do not persist only a target-key-local flag if that would make shifted/replaced segments ambiguous.

### Save Targeting
- **D-05:** Generated blue frames remain preview-only. `Save current` must not promote or replace a generated frame into a real key during this phase.
- **D-06:** Creating a new real key while interpolation is enabled happens at a far empty display position beyond the current expanded layout, not by saving a selected generated/render-only frame.
- **D-07:** If the selected frame is generated/render-only, painting and `Save current` stay disabled and the workflow strip shows compact guidance that generated frames are render-only and the user should navigate to an empty/custom position for a new key.
- **D-08:** A far empty display save attaches its custom override to the segment from the previous source real key to the new destination real key.
- **D-09:** Far empty display saves may append beyond the last source key, but generated/custom spacing must be clamped by existing safe frame/count limits such as `PHYSIC_PAINT_MAX_APPLY_FRAMES`.

### Key Utilities
- **D-10:** Duplicate and Insert must rebase custom segment overrides with shifted source real keys where possible, so intentional timing follows the real-key sequence.
- **D-11:** Delete removes any custom overrides touching the deleted key. The newly adjacent merged segment falls back to the global in-between count unless the user later creates a custom span intentionally.
- **D-12:** Paste to a far empty display position while interpolation is enabled behaves like `Save current`: it creates/replaces a real key and records previous-segment custom spacing.
- **D-13:** Copy from generated frames remains disallowed. Copy requires a real green source key even when custom spacing is active.

### MVP Feedback and Onion Skinning
- **D-14:** MVP UI should keep the existing workflow strip/timeline layout and add only subtle custom-span feedback, such as a compact badge or selected-segment/connector status. Full timeline polish belongs to Phase 36.14.
- **D-15:** Custom-spacing copy should emphasize the user-facing generated in-between count, e.g. `Custom span: 4 in-betweens`, rather than internal source-gap math.
- **D-16:** Reset/apply-to-all custom spacing UI is deferred. This phase must preserve custom spacing and never silently erase it, but it does not add controls to reset it.
- **D-17:** Invalid generated-frame save attempts, capped distances, and regeneration failures should reuse the compact workflow-strip status line. Do not add modal, toast, tutorial, or blocking overlay feedback for this MVP.
- **D-18:** Generated frames must never become onion-skin anchors. Onion skinning resolves around real source keys only.
- **D-19:** When editing a real green key, previous/next onion references must be neighboring real source keys. Custom distant real keys count as real onion-skin anchors.
- **D-20:** When previewing a generated frame, the generated preview itself may be visible, but previous/next onion references still resolve to the surrounding real source keys for that interpolation segment. Custom segment spacing must not cause onion skinning to step through generated frames as if they were real keys.

## Summary

Phase 36.13 should be planned as a source/display model correction, not as a visual strip patch. The durable truth is the ordered source real-key sequence plus global interpolation settings plus explicit per-source-segment spacing overrides; generated frames are derived render-only cache entries that can be regenerated for preview/export and must never become editable sources. [VERIFIED: CONTEXT.md] [VERIFIED: spec-36.13]

## Common Pitfalls

### Pitfall 1: Treating Display Frame as Source Frame
**What goes wrong:** A far display save creates source key `11` instead of compressed source key `6` in the example `0/1/2/6` with four custom in-betweens. [VERIFIED: CONTEXT.md]
**Why it happens:** Existing `getSourceRotoFrameForDisplayFrame` returns `max(sourceKeys)+1` for new frames when interpolation is enabled and does not know per-segment custom spacing. [VERIFIED: codebase]
**How to avoid:** Add an explicit far-empty display target resolver that computes previous source key, compressed destination source key, and remembered in-between count together. [VERIFIED: CONTEXT.md]
**Warning signs:** Tests assert custom source key `6`, but saved payload/store metadata records `11` or uses display distance directly. [VERIFIED: spec-36.13]

### Pitfall 2: Overwriting Custom Segments on Global Count Change
**What goes wrong:** Changing global `inBetweenCount` from 2 to 1 or 3 rewrites the custom `2 -> 6 = 4` segment. [VERIFIED: spec-36.13]
**Why it happens:** Regeneration uses one global count for every segment. [VERIFIED: codebase]
**How to avoid:** Resolve spacing per adjacent source segment: `override.generatedInBetweenCount ?? global.inBetweenCount`. [VERIFIED: CONTEXT.md]
**Warning signs:** `getExpandedRotoRealKeyFrames([0,1,2,6], { count: 1 })` moves the final real display key as if the last segment were non-custom. [VERIFIED: spec-36.13]

### Pitfall 3: Persisting Generated Frames as Truth
**What goes wrong:** Save/load reconstructs from stale generated metadata rather than real keys + settings + overrides. [VERIFIED: spec-36.13]
**Why it happens:** `loadFromMceOutputs` currently accepts generated metadata into `_rotoGeneratedCacheMetadata` while also regenerating when settings exist. [VERIFIED: codebase]
**How to avoid:** Treat generated metadata as disposable cache; after hydrating settings and real keys, regenerate from the override-aware model. [VERIFIED: codebase]
**Warning signs:** Deleting generated metadata from saved output changes source/display behavior after reload. [VERIFIED: spec-36.13]

### Pitfall 4: Forgetting Key Utility Override Rebasing
**What goes wrong:** Duplicate/Insert shifts real keys but custom spacing remains keyed to old endpoints; Delete leaves orphan overrides. [VERIFIED: CONTEXT.md]
**Why it happens:** `RotoKeyUtilityTransaction` currently returns frame mappings but no override update result. [VERIFIED: codebase]
**How to avoid:** Add pure override rebase/drop helper driven by `frameMappings` and delete target; apply it in store/session transaction handling. [VERIFIED: codebase]
**Warning signs:** Custom timing moves to the wrong segment after Duplicate or survives a Delete touching either endpoint. [VERIFIED: CONTEXT.md]

### Pitfall 5: Generated Frames as Onion Anchors
**What goes wrong:** Onion skin previous/next steps through blue generated frames, making custom spacing visually misleading. [VERIFIED: CONTEXT.md]
**Why it happens:** Onion lookup uses displayed/playback frame numbers rather than real source key metadata. [VERIFIED: CONTEXT.md]
**How to avoid:** Resolve onion previous/next from sorted real source keys only; a generated preview frame may be visible but not an anchor. [VERIFIED: CONTEXT.md]
**Warning signs:** On a generated frame, onion previous/next points to adjacent blue frames instead of surrounding green source keys. [VERIFIED: CONTEXT.md]

### Pitfall 6: Missing Version Invalidations
**What goes wrong:** Preview/export/UI keep stale generated output after override changes. [VERIFIED: codebase]
**Why it happens:** Cache regeneration without `_notifyVisualChange()` does not bump `physicPaintVersion`. [VERIFIED: codebase]
**How to avoid:** Route all settings/source/override changes through store methods that regenerate and notify. [VERIFIED: codebase]
**Warning signs:** UI updates but parent preview/export does not change until another unrelated paint mutation. [VERIFIED: user memory]
