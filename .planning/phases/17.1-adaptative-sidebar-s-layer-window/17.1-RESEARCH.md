# Phase 17.1: Adaptative Sidebar's Layer Window - Research

**Researched:** 2026-03-24
**Domain:** Preact sidebar UI layout restructuring (3-panel to 2-panel adaptive view)
**Confidence:** HIGH

## Summary

This phase replaces the standalone LAYERS panel in the sidebar with an adaptive view inside the SEQUENCES panel. The sidebar changes from a 3-panel layout (Sequences, Layers, Properties) to a 2-panel layout (Sequences/Layers adaptive + Properties). A layer icon button on each sequence header switches the panel from the sequence list to that sequence's layer view, with a back button to return.

The codebase is well-structured for this change. `LayerList` and `AddLayerMenu` are standalone components that can be directly reused. The flex-based panel resize system (`calcFlexResize` in `panelResize.ts`) and its persistence (`setPanelFlex` in `appConfig.ts`) need simplification from 3-panel to 2-panel. The main architectural consideration is that `layerStore.layers` is a computed signal derived from `sequenceStore.activeSequenceId` -- since the adaptive view shows layers for the active sequence, this remains correct and no special routing is needed.

**Primary recommendation:** Implement as a state-driven conditional render in `LeftPanel.tsx` using a new `layerViewSequenceId` signal in `uiStore`. When non-null, the adaptive panel shows layers for that sequence instead of the sequence list. All existing layer components (`LayerList`, `AddLayerMenu`) reuse as-is.

### Pattern 1: State-Driven View Switching
**What:** Use a single signal `layerViewSequenceId: Signal<string | null>` to control which view the adaptive panel shows. When null, show sequence list. When set to a sequence ID, show that sequence's layer view.
**When to use:** Any time a UI panel needs to toggle between two views without routing or DOM manipulation.
**Why it works here:** `layerStore.layers` is already a computed signal derived from `sequenceStore.activeSequenceId`. When the user clicks the layer icon on a sequence, we set that sequence as active (if not already) and set `layerViewSequenceId` to show the layer view. The existing `LayerList` component reads `layerStore.layers` which auto-updates.

```typescript
// In uiStore.ts
const layerViewSequenceId = signal<string | null>(null);

// In SequenceItem -- layer icon click handler
const handleLayerView = (e: MouseEvent) => {
  e.stopPropagation();
  // Ensure this sequence is active so layerStore.layers computes for it
  sequenceStore.setActive(seq.id);
  uiStore.selectSequence(seq.id);
  uiStore.layerViewSequenceId.value = seq.id;
};
```

### Pattern 2: CollapsibleSection Back-Arrow Variant
**What:** Instead of creating a new component, use conditional rendering in `LeftPanel` for the header when in layer view mode. The header shows `ArrowLeft` icon + sequence name on the left, `AddLayerMenu` on the right.
**Why:** CollapsibleSection has a fixed contract (title string, chevron toggle, headerActions). The layer view header has fundamentally different behavior (back navigation instead of collapse toggle). A dedicated inline header JSX in LeftPanel is cleaner than forcing CollapsibleSection into an incompatible variant.

```typescript
// Dedicated header for layer view mode (inline in LeftPanel)
<div class="flex items-center justify-between h-9 px-3 shrink-0">

      {sequenceName}
    </span>
  </div>
  <div onClick={(e) => e.stopPropagation()}>
    <AddLayerMenu />
  </div>
</div>
```

### Pattern 3: Auto-Close on Sequence Switch (D-11)
**What:** When the user clicks a different sequence in SequenceItem's `handleSelect`, reset `layerViewSequenceId` to null.
**Integration point:** In `SequenceItem.handleSelect`, add: `uiStore.layerViewSequenceId.value = null;`
**Important:** This happens in the "switching sequences" branch (not the "toggle collapse" branch for same-sequence clicks). The existing code already has this distinction:

```typescript
// Existing code in SequenceItem.handleSelect:
if (seq.id === wasActive) {
  setKeyPhotoCollapsed(prev => !prev);  // same sequence, toggle collapse
  return;
}
// Switching sequences -- add layerViewSequenceId reset here
uiStore.layerViewSequenceId.value = null;
```

### Anti-Patterns to Avoid
- **Creating a new LayerList variant:** D-07 explicitly says reuse as-is. The existing `LayerList` reads from `layerStore.layers` which is computed from `activeSequenceId`. No props needed.
- **Using CSS transitions for view switching:** The sequence list and layer view have different DOM structures. CSS transitions between them add complexity for minimal UX benefit. Use instant swap (like CollapsibleSection toggle already does).
- **Keeping `layFlex` and migrating at runtime:** The 3-panel flex system should be simplified in code, not wrapped. Remove `layFlex` entirely and change `setPanelFlex` to take 2 values.

## Common Pitfalls

### Pitfall 1: layerStore.layers Coupling to activeSequenceId
**What goes wrong:** The layer view might show stale layers if `activeSequenceId` is not synced with `layerViewSequenceId`.
**Why it happens:** `layerStore.layers` is a computed signal: `sequenceStore.getActiveSequence()?.layers`. If the user opens the layer view for sequence A but `activeSequenceId` points to sequence B, the wrong layers display.
**How to avoid:** When setting `layerViewSequenceId`, always also call `sequenceStore.setActive(seq.id)` to keep them in sync. The existing `handleSelect` in SequenceItem already does this.
**Warning signs:** Layers shown don't match the sequence name in the header.

### Pitfall 2: Persisted 3-Value Flex Breaking 2-Panel Layout
**What goes wrong:** Existing users have `panelFlex: [1, 1, 1]` stored in `app-config.json` (LazyStore). After the update, the code expects `[seqFlex, propFlex]` (2 values) but reads 3.
**Why it happens:** `getPanelFlex` returns `[number, number, number]` and `initSidebarLayout` destructures 3 values.
**How to avoid:** Migration strategy -- `getPanelFlex` returns the stored tuple. If it has 3 values (old format), take indices 0 and 2 (seq and prop), ignoring index 1 (lay). If it has 2 values (new format), use directly. `setPanelFlex` writes 2 values going forward.
**Warning signs:** Panel sizes look wrong on first launch after update, or one panel is invisible (flex=0).

### Pitfall 3: SortableJS DOM Mutation on View Switch
**What goes wrong:** If the user opens the layer view while SortableJS is mid-drag in the sequence list (or vice versa), the DOM swap can crash SortableJS.
**Why it happens:** SortableJS keeps internal references to DOM nodes. Preact unmounting the sequence list while SortableJS holds references causes stale pointers.
**How to avoid:** The existing SortableJS cleanup pattern (useEffect return `() => instance.destroy()`) handles this correctly. When the view switches, Preact unmounts the SequenceList, the cleanup runs, and SortableJS is destroyed cleanly.
**Warning signs:** Console errors about missing DOM nodes after rapid view switching.

### Pitfall 4: Collapse/Expand State Cleanup
**What goes wrong:** `layersSectionCollapsed` signal and `_preCollapseLayFlex` signal remain in uiStore after removing the LAYERS panel, causing dead code and confusing state.
**Why it happens:** Partial removal -- removing the panel UI but leaving the state signals.
**How to avoid:** Remove `layersSectionCollapsed`, `_preCollapseLayFlex`, `layPanelFlex` signals entirely. Remove `collapsePanel('lay')` and `expandPanel('lay')` branches. Update `reset()` to not set these values.
**Warning signs:** Stale signals consuming memory; TypeScript not catching removed references if cleanup is incomplete.

### Pitfall 5: SequenceItem Dual-Function (handleSelect + Layer View)
**What goes wrong:** Clicking the layer icon button also triggers `handleSelect` because of event bubbling from the sequence header row.
**Why it happens:** The layer icon is inside the `.group/row` div that has `onClick={handleSelect}`.
**How to avoid:** Call `e.stopPropagation()` in the layer icon click handler. The existing pattern for the ellipsis button already does this.
**Warning signs:** Clicking the layer icon also triggers key photo collapse/expand or sequence switching.

### Key Integration: Simplified calcFlexResize (2-panel)

The flex resize simplifies to only `seq-prop`:

```typescript
// panelResize.ts -- new 2-panel interface
export interface FlexSizes2 {
  seqFlex: number;
  propFlex: number;
  totalPixelHeight: number;
}

export function calcFlexResize2(
  current: FlexSizes2,
  deltaY: number,
): { seqFlex: number; propFlex: number } {
  const totalFlex = current.seqFlex + current.propFlex;
  if (totalFlex === 0 || current.totalPixelHeight === 0) {
    return { seqFlex: current.seqFlex, propFlex: current.propFlex };
  }
  const pxPerUnit = current.totalPixelHeight / totalFlex;
  const deltaFlex = deltaY / pxPerUnit;
  let seqFlex = current.seqFlex + deltaFlex;
  let propFlex = current.propFlex - deltaFlex;
  // Restore / collapse / clamp logic (same as existing, minus layFlex)
  // ...
  return { seqFlex, propFlex };
}
```

### Key Integration: Panel Flex Persistence Migration

```typescript
// appConfig.ts -- backward-compatible getPanelFlex
export async function getPanelFlex(): Promise<[number, number]> {
  const stored = await store.get<number[]>('panelFlex');
  if (!stored) return [1, 1];
  if (stored.length === 3) {
    // Migration: [seq, lay, prop] -> [seq, prop] (discard lay)
    return [stored[0] || 1, stored[2] || 1];
  }
  return [stored[0] || 1, stored[1] || 1];
}

export async function setPanelFlex(seq: number, prop: number): Promise<void> {
  await store.set('panelFlex', [seq, prop]);
}
```
