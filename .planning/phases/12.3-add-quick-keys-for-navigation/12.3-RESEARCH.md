# Phase 12.3: Add Quick Keys for Navigation - Research

**Researched:** 2026-03-17
**Domain:** Keyboard shortcuts for timeline navigation (Preact + tinykeys)
**Confidence:** HIGH

## Summary

This phase adds keyboard shortcuts for timeline navigation: arrow keys for frame-by-frame scrub (already partially exists), Home/End for go-to-start/end of timeline, and Page Up/Page Down for jumping to the next/previous content sequence boundary. The codebase has a well-established keyboard shortcut infrastructure centered on `Application/src/lib/shortcuts.ts` using tinykeys v3.0.0, with a centralized `mountShortcuts()` function and a ShortcutsOverlay component that documents all bindings.

The critical insight is that **ArrowLeft/ArrowRight already exist** and have dual behavior: they nudge a selected layer when a layer is selected, or step the playback frame when no layer is selected. The phase goal specifies "right/left arrows for frame-by-frame scrub" which is already implemented via the `nudgeOrStep` helper. The real new work is Home/End (seek to frame 0 / last frame), Page Up/Page Down (jump to prev/next content sequence), and updating the ShortcutsOverlay to document these.

**Primary recommendation:** Add Home/End/PageUp/PageDown bindings to the existing `mountShortcuts()` function in `shortcuts.ts`, create a small `sequenceNav.ts` helper for sequence boundary lookup from `trackLayouts`, and update `ShortcutsOverlay.tsx` to document the new shortcuts.

## Architecture Patterns

### Existing Shortcut Registration Pattern
All shortcuts follow this exact pattern in `shortcuts.ts`:
```typescript
'KeyName': (e: KeyboardEvent) => {
  if (shouldSuppressShortcut(e)) return;
  e.preventDefault();
  // action
},
```

The `shouldSuppressShortcut` guard prevents shortcuts from firing when the user is typing in INPUT, TEXTAREA, SELECT, or contentEditable elements.

### Pattern: Sequence Boundary Navigation (Page Up/Page Down)

The `trackLayouts` computed signal provides an ordered array of content sequences with their global frame ranges:
```typescript
interface TrackLayout {
  sequenceId: string;
  sequenceName: string;
  startFrame: number;
  endFrame: number;  // exclusive
  keyPhotoRanges: KeyPhotoRange[];
}
```

To find prev/next sequence from the current frame:
```typescript
// Source: Application/src/lib/frameMap.ts (trackLayouts structure)
function findPrevSequenceStart(currentFrame: number): number | null {
  const layouts = trackLayouts.peek();
  // Find the last sequence whose startFrame is strictly less than currentFrame
  for (let i = layouts.length - 1; i >= 0; i--) {
    if (layouts[i].startFrame < currentFrame) {
      return layouts[i].startFrame;
    }
  }
  return null; // already at or before the first sequence
}

function findNextSequenceStart(currentFrame: number): number | null {
  const layouts = trackLayouts.peek();
  // Find the first sequence whose startFrame is strictly greater than currentFrame
  for (const layout of layouts) {
    if (layout.startFrame > currentFrame) {
      return layout.startFrame;
    }
  }
  return null; // already at or past the last sequence
}
```

### Pattern: Home/End Navigation

Home and End are straightforward seeks:
```typescript
// Home: go to frame 0
playbackEngine.seekToFrame(0);

// End: go to last frame
const lastFrame = timelineStore.totalFrames.peek() - 1;
playbackEngine.seekToFrame(Math.max(0, lastFrame));
```

### Pattern: Dual-Behavior Arrow Keys (already established)

The existing `nudgeOrStep` function demonstrates the project's context-dependent key behavior pattern:
```typescript
// Source: Application/src/lib/shortcuts.ts line 141-149
function nudgeOrStep(axis: 'x', delta: number, _e: KeyboardEvent): void {
  const selectedId = layerStore.selectedLayerId.peek();
  if (selectedId) {
    nudgeIfSelected(axis, delta);
  } else {
    if (delta < 0) playbackEngine.stepBackward();
    else playbackEngine.stepForward();
  }
}
```

Home/End/PageUp/PageDown should NOT have this dual behavior -- they should always navigate the timeline regardless of layer selection, since they have no spatial meaning for layer positioning.

### Anti-Patterns to Avoid
- **Using `.value` in event handlers:** Always use `.peek()` inside tinykeys handlers to avoid creating signal subscriptions outside Preact's effect tracking.
- **Forgetting `shouldSuppressShortcut` guard:** Every handler must check this first.
- **Forgetting `e.preventDefault()`:** Prevents browser default scrolling behavior (especially for Page Up/Down and Home/End which scroll the page by default).
- **Not updating ShortcutsOverlay:** Every new shortcut must be documented in the overlay.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Keyboard binding | Custom keydown listener | tinykeys `mountShortcuts()` | Already handles modifiers, key aliases, cross-browser |
| Frame seeking | Direct `timelineStore.seek()` | `playbackEngine.seekToFrame()` | Handles displayFrame sync, activeSequence sync, player sync |
| Sequence layouts | Manual frame counting | `trackLayouts.peek()` | Already computed and cached reactively |

## Common Pitfalls

### Pitfall 1: Browser Default Behavior for Home/End/PageUp/PageDown
**What goes wrong:** Home/End scroll the page to top/bottom; PageUp/PageDown scroll the viewport.
**Why it happens:** These keys have default browser scroll behavior.
**How to avoid:** Always call `e.preventDefault()` in the handler (already the standard pattern).
**Warning signs:** Page scrolls when pressing these keys.

### Pitfall 2: Page Up/Down When No Sequences Exist
**What goes wrong:** Attempting to navigate sequences when trackLayouts is empty causes no-op but could confuse logic.
**Why it happens:** Empty project has no content sequences.
**How to avoid:** Guard with `if (!targetFrame) return;` or similar null check on the lookup result.
**Warning signs:** Console errors on empty projects.

### Pitfall 3: End Key Off-By-One
**What goes wrong:** Seeking to `totalFrames` instead of `totalFrames - 1` goes past the last valid frame.
**Why it happens:** `totalFrames` is a count (1-based), frames are 0-indexed.
**How to avoid:** `seekToFrame(Math.max(0, totalFrames.peek() - 1))`.
**Warning signs:** Preview shows blank/black on End key.

### Pitfall 4: Shift+Arrow Already Bound for Layer Nudge
**What goes wrong:** Shift+ArrowLeft/Right are already bound for 10px layer nudge.
**Why it happens:** Shift modifier variants are already registered in tinykeys.
**How to avoid:** Do NOT rebind Shift+Arrow keys. The existing Shift+Arrow = 10-frame step (when no layer selected) via `nudgeOrStep('x', -10, e)` already handles 10-frame scrub.
**Warning signs:** Unexpected behavior when pressing Shift+Arrow.

### Pitfall 5: ArrowLeft/Right Behavior Already Correct
**What goes wrong:** Reimplementing arrow key frame stepping that already exists.
**Why it happens:** Phase description says "right/left arrows for frame-by-frame scrub" but this already works.
**How to avoid:** Verify existing behavior first. ArrowLeft/Right already call `nudgeOrStep` which calls `stepBackward()/stepForward()` when no layer is selected.
**Warning signs:** Duplicate bindings in tinykeys cause undefined behavior.

```typescript
'End': (e: KeyboardEvent) => {

```

### Example 2: Page Up/Down for Sequence Jump
```typescript
// Helper (can be inline or in a separate sequenceNav.ts):
import { trackLayouts } from '../lib/frameMap';

function jumpToPrevSequence(): void {
  const currentFrame = timelineStore.currentFrame.peek();
  const layouts = trackLayouts.peek();
  for (let i = layouts.length - 1; i >= 0; i--) {
    if (layouts[i].startFrame < currentFrame) {
      playbackEngine.seekToFrame(layouts[i].startFrame);
      return;
    }
  }
  // Already at or before first sequence -- go to frame 0
  playbackEngine.seekToFrame(0);
}

function jumpToNextSequence(): void {
  const currentFrame = timelineStore.currentFrame.peek();
  const layouts = trackLayouts.peek();
  for (const layout of layouts) {
    if (layout.startFrame > currentFrame) {
      playbackEngine.seekToFrame(layout.startFrame);
      return;
    }
  }
  // Already at or past last sequence -- go to end
  const lastFrame = timelineStore.totalFrames.peek();
  playbackEngine.seekToFrame(Math.max(0, lastFrame - 1));
}

// In mountShortcuts():
'PageUp': (e: KeyboardEvent) => {
  if (shouldSuppressShortcut(e)) return;
  e.preventDefault();
  jumpToPrevSequence();
},
'PageDown': (e: KeyboardEvent) => {
  if (shouldSuppressShortcut(e)) return;
  e.preventDefault();
  jumpToNextSequence();
},
```

## Open Questions

1. **Should Page Up/Down also select the sequence in the sidebar?**
   - What we know: Phase 12.2 auto-selects sidebar when timeline navigates to a new sequence (via `syncActiveSequence()` in playbackEngine).
   - What's unclear: Whether `seekToFrame` already triggers `syncActiveSequence` -- YES it does. `playbackEngine.seekToFrame()` calls `this.syncActiveSequence()` which updates sidebar selection.
   - Recommendation: No extra work needed. `playbackEngine.seekToFrame()` handles sidebar sync.

2. **Meaning of "in-between sequence" in phase goal**
   - What we know: The phrase "jump to next/previous in-between sequence" likely refers to content sequences on the global timeline.
   - What's unclear: Could it mean "tweening/interpolation" sequences or literally "between" sequences?
   - Recommendation: Interpret as "jump to the start frame of the next/previous content sequence." This matches standard NLE behavior (DaVinci Resolve, Premiere Pro use Page Up/Down for edit point navigation).
