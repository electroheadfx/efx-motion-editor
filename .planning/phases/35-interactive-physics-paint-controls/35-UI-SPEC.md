## Interaction Contract

### App Entry Point

- Add or expose a distinct `physic-paint` layer path. It must not replace or overload existing `paint`, perfect-freehand basic paint, or p5.brush FX paint paths.
- In the selected `physic-paint` layer parameters, show a primary button labelled exactly `[open fx paint canvas]`.
- The button must pass current layer/frame context to the standalone workflow. If no valid layer/frame context exists, keep the button visible but disabled and show `Select a physics paint layer and frame first.`
- Opening the standalone must not hide or remove existing Paint/Rotopaint controls elsewhere in the app.

### Standalone Layout

- Keep the live physics canvas as the dominant visual object.
- Controls remain visible as a dense toolbar/control panel. Everything currently present in the demo toolbar stays present; unimplemented controls are disabled, not hidden.
- Add a diagnostics/apply strip adjacent to the toolbar, not as a modal. It must show: current layer id/name if available, current app start frame, engine ready state, canvas mounted state, active tool, active color, brush size, opacity, physics mode, and last error.
- The ready/not-ready state must be visible without opening a disclosure or console.

### Paint and Tool Controls

- Paint and Erase are mutually exclusive active tools. The active tool uses accent blue background/border and white text.
- Color, size, opacity, detail, pickup, erase strength, smoothing, background, paper grain, grain, physics mode, spread, animation frame count/FPS, save/load, undo, and clear remain available in the standalone surface.
- Pointer interactions must use the real engine APIs. Do not display a fake painted preview or app-side replay.

### Apply Actions

- `[apply canvas]` applies the current rendered display canvas to the `physic-paint` layer at the current app frame/opening frame.
- `[apply play canvas]` requires a visible frame-count input labelled `Frames to apply`. Valid range: minimum 1, maximum 600. Default: 120.
- Generated sequence frames apply to app frames starting at the captured app frame: `startFrame + generatedFrameIndex`.
- While generating/applying, disable both apply buttons and show `Applying physics paint output...`.
- On success, show `Applied to frame {frame}` for still output or `Applied {count} frames starting at frame {frame}` for sequence output.

### Diagnostics and Error States

- Readiness is `Ready to apply` only when engine initialized, canvas mounted, launch context exists, app bridge/listener is available, and no apply/play operation is active.
- Not-ready diagnostics must list the missing condition in plain language, for example `Canvas is still mounting` or `No app layer context received`.
- Errors must stay visible until the next successful action or explicit dismissal.
- Console-only errors do not satisfy DIAG-01.

### Import / Export State Files

- Save/load copy must say state, not export/apply, to avoid confusing editable state persistence with rendered-output apply-back.
- `Save state` downloads/saves editable physics paint state JSON.
- `Load state` accepts only valid physics paint state JSON and shows the copy defined above on invalid files.
- Rendered output back to the app is only through `[apply canvas]` and `[apply play canvas]`.

### Accessibility and Input

- All buttons and inputs must have accessible text labels matching visible copy.
- Disabled controls must use `disabled` where possible and include explanatory visible text or `title`.
- Focus rings use accent blue and must be visible against secondary panels.
- Buttons must be keyboard reachable in source order: tools, settings, physics, animation, state, undo/clear, apply actions.
