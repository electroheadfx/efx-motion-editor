In the existing useEffect keyboard handler (from Task 1), add Arrow Up/Down handling ONLY when activeTab === 0:
```typescript
if (e.key === 'ArrowDown') {
  e.preventDefault();
  if (activeTab === 0) {
    setFocusedRow((prev) => (prev + 1) % SHORTCUT_GROUPS.length);
  }
  return;
}
if (e.key === 'ArrowUp') {
  e.preventDefault();
  if (activeTab === 0) {
    setFocusedRow((prev) => (prev - 1 + SHORTCUT_GROUPS.length) % SHORTCUT_GROUPS.length);

```
- Arrow Down: advance focusedRow forward (wrapping from 6 to 0)

- The wrapping uses `% SHORTCUT_GROUPS.length` (which is 7)

**IMPORTANT dependency array note:**
Since the event handler now reads `activeTab` and `focusedRow` for the ArrowUp/Down/Enter logic, the useEffect dependency array must include both:
```typescript
useEffect(() => {
  function handleKeyDown(e: KeyboardEvent) { ... }
  window.addEventListener('keydown', handleKeyDown);
  return () => window.removeEventListener('keydown', handleKeyDown);
}, [activeTab, focusedRow]);
```
However, this means setActiveTab and setFocusedRow calls for ArrowLeft/ArrowRight/Tab/number-keys should continue using functional updaters `(prev) => ...` to avoid stale closures. Only the `activeTab === 0` guard reads the current value, which is safe since it's now in the dependency array.

Alternatively, use refs to avoid the dependency array issue:
```typescript
const activeTabRef = useRef(0);
// Update ref whenever activeTab changes
useEffect(() => { activeTabRef.current = activeTab; }, [activeTab]);
```
Then use `activeTabRef.current` in the event handler while keeping `[]` deps. Choose whichever approach is cleaner -- either is acceptable. The key constraint is: Arrow Up/Down/Enter MUST read the current activeTab value accurately.
