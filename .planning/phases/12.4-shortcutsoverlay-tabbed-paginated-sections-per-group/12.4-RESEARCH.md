# Phase 12.4: ShortcutsOverlay Tabbed/Paginated Sections per Group - Research

**Researched:** 2026-03-17
**Domain:** Preact component UI -- modal overlay with tabbed navigation
**Confidence:** HIGH

## Summary

The current ShortcutsOverlay is a single flat modal that renders all 7 shortcut groups (28 total shortcuts) in a 2-column grid. As the application grows, this will overflow vertically and become hard to scan. Phase 12.4 converts this into a tabbed/paginated overlay where each group becomes a selectable tab, showing only that group's shortcuts at a time.

The implementation is entirely self-contained within `ShortcutsOverlay.tsx` plus minimal additions to `uiStore.ts`. No new libraries are needed -- Preact's `useState` hook (already used throughout the codebase) handles tab selection, and Tailwind CSS v4 utility classes handle all styling. The existing `SHORTCUT_GROUPS` data structure already provides the perfect tab labels (group titles) and per-tab content (group entries).

**Primary recommendation:** Replace the 2-column grid with a horizontal tab bar at the top of the modal and a single-group content area below. Use `useState<number>(0)` for active tab index. Keep all existing CSS variable usage and theme compatibility. This is a single-file refactor with zero new dependencies.

### Pattern 1: Local Tab State with useState
**What:** Use `useState<number>(0)` to track which tab is active. The component re-renders only when the tab changes.
**When to use:** Local UI state that does not need to persist across sessions or be shared with other components.
**Example:**
```typescript
import { useState } from 'preact/hooks';

export function ShortcutsOverlay() {
  const [activeTab, setActiveTab] = useState(0);
  const activeGroup = SHORTCUT_GROUPS[activeTab];

  return (
    <div>
      {/* Tab bar */}
      <div class="flex border-b border-[var(--color-border-subtle)]">
        {SHORTCUT_GROUPS.map((group, i) => (
          <button
            key={group.title}
            class={`px-3 py-2 text-xs font-medium transition-colors cursor-pointer
              ${i === activeTab
                ? 'text-[var(--color-text-heading)] border-b-2 border-[var(--color-accent)]'
                : 'text-[var(--color-text-muted)] hover:text-[var(--color-text-secondary)]'
              }`}
            onClick={() => setActiveTab(i)}
          >
            {group.title}
          </button>
        ))}
      </div>

      {/* Content: single group */}
      <div class="pt-4">
        <div class="flex flex-col gap-1.5">
          {activeGroup.entries.map((entry) => (
            <div key={entry.keys} class="flex items-center justify-between gap-3 py-1">
              <span class="text-sm text-[var(--color-text-button)]">
                {entry.description}
              </span>
              <kbd class="flex-shrink-0 inline-flex items-center px-2 py-0.5 rounded bg-[var(--color-bg-hover-item)] border border-[var(--color-border-kbd)] text-xs font-mono text-[var(--color-text-link)]">
                {entry.keys}
              </kbd>
            </div>
          ))}
        </div>
      </div>
    </div>
  );
}
```

### Pattern 2: Keyboard Navigation Between Tabs
**What:** Allow left/right arrow keys to cycle tabs when focus is in the tab bar, and number keys 1-7 for direct tab selection.
**When to use:** Accessibility and power-user ergonomics in keyboard-heavy applications.
**Example:**
```typescript
function handleTabKeyDown(e: KeyboardEvent) {
  if (e.key === 'ArrowRight') {
    e.preventDefault();
    setActiveTab((prev) => (prev + 1) % SHORTCUT_GROUPS.length);
  } else if (e.key === 'ArrowLeft') {
    e.preventDefault();
    setActiveTab((prev) => (prev - 1 + SHORTCUT_GROUPS.length) % SHORTCUT_GROUPS.length);
  }
}
```

### Anti-Patterns to Avoid
- **Storing tab state in uiStore:** This is ephemeral UI state that resets on overlay close. No persistence needed. Use local useState.
- **Rendering all tabs and hiding with CSS:** With only 7 small groups, conditional rendering is simpler and avoids hidden DOM waste. Do not use `display: none` tabs.
- **Adding a new component file for tabs:** This is a single overlay component with ~150 lines. A separate TabBar component would be over-engineering for one use site.

|---------|-------------|-------------|-----|
| Tab state management | Custom signal store | useState(0) | Local state, no persistence needed |
| Tab styling | Custom CSS file | Tailwind utility classes | Consistent with rest of project |
| Keyboard navigation | Complex focus manager | Simple keydown handler in useEffect | Only 7 tabs, simple left/right cycle |

**Key insight:** The existing `SHORTCUT_GROUPS` array already provides the perfect data structure for tabs. Each group's `title` becomes the tab label, each group's `entries` becomes the tab content. Zero data refactoring needed.

## Common Pitfalls

### Pitfall 1: Tab indicator not respecting theme
**What goes wrong:** Hard-coding the active tab underline color instead of using CSS variables.
**Why it happens:** Copy-pasting from design examples that use literal hex values.
**How to avoid:** Use `var(--color-accent)` for the active tab indicator. All existing theme-variable definitions already include accent colors across dark/medium/light themes.
**Warning signs:** Active tab indicator looks wrong when switching themes.

### Pitfall 2: Escape key interfering with tab navigation
**What goes wrong:** The existing Escape key handler closes the overlay, but if we add arrow key handling for tabs, arrow keys might conflict with the global nudge-or-step shortcuts.
**Why it happens:** The `mountShortcuts()` function in `shortcuts.ts` binds ArrowLeft/ArrowRight globally.
**How to avoid:** The overlay is a focused modal with `e.stopPropagation()` on its container. Arrow key handlers for tab navigation should be added directly on the tab bar element, not globally. The existing `shouldSuppressShortcut` check in shortcuts.ts also helps, since the overlay's button elements would match as interactive targets -- but relying on stopPropagation on the modal container is the more explicit guard.
**Warning signs:** Pressing left/right in the overlay nudges a selected layer instead of switching tabs.

### Pitfall 3: Modal height jump when switching tabs
**What goes wrong:** Different groups have different entry counts (Canvas has 8, Keyframes has 1), causing the modal to resize jarrringly.
**Why it happens:** No minimum height set on the content area.
**How to avoid:** Set a `min-h` on the content area sized to the tallest group (Canvas with 8 entries, roughly 8 * 36px = 288px). Or use a fixed height with scroll.
**Warning signs:** Modal bounces between sizes when clicking tabs.

### Pitfall 4: Forgetting to reset tab state on overlay reopen
**What goes wrong:** User opens overlay, switches to tab 5, closes, reopens -- still on tab 5 instead of tab 0.
**Why it happens:** If tab state is stored in uiStore signal (persists across open/close).
**How to avoid:** Use local `useState(0)` inside the component. Since the component unmounts when `shortcutsOverlayOpen` is false and remounts when true (conditional rendering in EditorShell), useState naturally resets to 0.
**Warning signs:** Overlay remembers last tab across open/close cycles.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Flat grid of all groups | Tabbed per-group view | Phase 12.4 | Better scalability as shortcuts grow |
| Fixed 2-column layout | Single-column per-tab | Phase 12.4 | Cleaner reading flow, consistent height |

**Design rationale:** With 7 groups and 28 shortcuts, the flat 2-column grid is already pushing vertical space limits. Future phases (13-17) will add more shortcuts (audio, export, transitions). Tabs make the overlay future-proof.

## Open Questions

1. **Should "All" be an additional first tab showing the current 2-column view?**
   - What we know: Some apps (VS Code, Figma) include an "All" tab alongside category tabs.
   - What's unclear: Whether the user wants a flat "view all" option or purely per-group tabs.
   - Recommendation: Start with per-group tabs only (simpler). An "All" tab can be trivially added later as index -1 mapping to a 2-column grid render.

2. **Should the modal width change?**
   - What we know: Current modal is `max-w-[600px]`. With single-group content the width is generous.
   - What's unclear: Whether the tab bar with 7 labels fits comfortably at 600px.
   - Recommendation: Keep `max-w-[600px]`. Seven short tab labels (Playback, File, Editing, Navigation, Canvas, Blur, Keyframes) fit within 600px at 11-13px font sizes. If tight, use 10px tracking-wider uppercase labels (same style as current group headers).

3. **Should small groups be merged?**
   - What we know: "Blur" has 2 entries, "Keyframes" has 1 entry. These are very sparse tabs.
   - What's unclear: Whether the user wants to consolidate sparse groups.
   - Recommendation: Keep groups 1:1 with tabs for now. This matches the mental model of one category per tab. Merging would break the conceptual mapping.
