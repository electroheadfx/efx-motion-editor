# Phase 12.8: Fix Timeline Thumb Horizontal Stretching on Zoom - Research

**Researched:** 2026-03-18
**Domain:** Canvas 2D tile/repeat rendering for timeline thumbnails
**Confidence:** HIGH

## Summary

This phase replaces a single stretched `ctx.drawImage(cachedImg, fx, fy, fw, fh)` call (line 266 of TimelineRenderer.ts) with a tile-aware drawing approach that maintains image aspect ratio at any zoom level. The current code stretches each key photo to fill the frame cell width, which distorts thumbnails at non-native zoom levels.

Two viable approaches exist: `ctx.createPattern()` with `CanvasPattern.setTransform()` for GPU-accelerated native tiling, or a manual `drawImage` loop. Both are well-supported. The `createPattern` approach is recommended because it leverages the browser's native pattern engine (fewer JS-side draw calls), can be cached per image, and naturally handles the "repeat" semantic. The pattern object is created once per image, scaled via `setTransform` to match the cell height, then used with `fillRect` to tile across the frame cell.

**Primary recommendation:** Use `ctx.createPattern(img, 'repeat')` with `pattern.setTransform(new DOMMatrix().scale(s, s))` where `s = cellHeight / img.naturalHeight`, clipped per frame cell via `ctx.save()/clip()/restore()`. Fall back to solid placeholder color when `frameWidth < 4px`.

### Locked Decisions
- **Fill strategy:** Repeat/tile approach -- scale image to fill cell height, tile horizontally when cell is wider than one tile. Natural aspect ratio tiles: tile width = `image.naturalWidth * (cellHeight / image.naturalHeight)`. Seamless tiling with no separators between tile repetitions.
- **Range rendering:** Reset per frame -- each frame cell draws its tiles independently (not continuous across the key photo range). Every frame in the same key photo range looks identical.
- **Crop anchor:** Horizontal: left-aligned (tiles start from left edge). Vertical: center-aligned (crop from vertical center when image taller than cell after scaling).
- **Zoom extremes:** Low zoom fallback below a minimum frame width threshold -- skip drawImage and fill with existing placeholder colors (PLACEHOLDER_BG_A / PLACEHOLDER_BG_B alternating by range index). High zoom: unlimited tile repeats.

## Standard Stack

No new libraries are needed. This phase works entirely within existing Canvas 2D APIs already used in the codebase.

### Core APIs Used
| API | Purpose | Why Standard |
|-----|---------|--------------|
| `CanvasRenderingContext2D.createPattern()` | Create repeating tile pattern from image | Native browser API, widely available since 2015 |
| `CanvasPattern.setTransform()` | Scale pattern to match cell height | Native browser API, widely available since Jan 2020 |
| `DOMMatrix` | Build scale+translate transform for pattern | Native browser API, no polyfill needed |
| `ctx.save()/clip()/restore()` | Clip pattern fill to frame cell bounds | Already used extensively in TimelineRenderer |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| `createPattern` + `setTransform` | Manual `drawImage` loop | More JS-side draw calls, manual tile offset math, but avoids `DOMMatrix` construction |
| `createPattern` + `setTransform` | Pre-rendered OffscreenCanvas tile strip | Better for many frames with same image, but added memory and complexity |

**Recommendation:** `createPattern` with `setTransform`. Reasons: (1) one `fillRect` call per frame cell instead of N `drawImage` calls, (2) browser handles tile clipping natively, (3) pattern object can be cached per image and reused across frames in the same range.

### Recommended Modification Pattern

Replace the single `drawImage` at line 266 with a tile-aware pattern fill. The pattern is created once per range (same image), reused across all frames in that range.

```typescript
// Before the frame loop (inside the range loop), create pattern once per range:
if (cachedImg) {
  const cellH = TRACK_HEIGHT - 4; // fh
  const scale = cellH / cachedImg.naturalHeight;
  const pattern = ctx.createPattern(cachedImg, 'repeat');
  if (pattern) {
    // Scale image so its height matches cell height
    // Vertical center-crop: offset pattern so image center aligns with cell center
    const scaledImgH = cachedImg.naturalHeight * scale; // == cellH
    const scaledImgW = cachedImg.naturalWidth * scale;  // natural tile width
    // Center-crop vertical offset (0 when perfectly matching)
    const offsetY = 0; // since we scale to exact cellH, no vertical offset needed at this stage
    pattern.setTransform(new DOMMatrix().scale(scale, scale));
    // Store for use in frame loop
  }
}

// Inside the frame loop:
if (cachedImg && pattern) {
  if (fw < MIN_FRAME_WIDTH_FOR_THUMB) {
    // Low zoom fallback: solid color
    ctx.fillStyle = ri % 2 === 0 ? PLACEHOLDER_BG_A : PLACEHOLDER_BG_B;
    ctx.fillRect(fx, fy, fw, fh);
  } else {
    ctx.save();
    // Clip to frame cell
    ctx.beginPath();
    ctx.rect(fx, fy, fw, fh);
    ctx.clip();
    // Set pattern as fill, offset so tiles start at frame left edge
    ctx.fillStyle = pattern;
    // Translate pattern origin to frame cell position
    // The pattern repeats from (0,0) in pattern space, so we need to
    // translate the context so the pattern origin aligns with (fx, fy)
    ctx.translate(fx, fy);
    ctx.fillRect(0, 0, fw, fh);
    ctx.restore();
  }
}
```

### Vertical Center-Crop Pattern

The user requires vertical center alignment. Since `createPattern` tiles from `(0,0)` of the image, and we scale to match cell height exactly, there is no vertical cropping needed at the scale step. However, if the image aspect ratio means the scaled height exceeds `cellH` (it won't, since we scale to match), we would offset. In our case, `scale = cellH / naturalHeight` means the scaled image height is exactly `cellH` -- no vertical offset is needed.

**Wait -- re-reading the requirement:** "Vertical: center-aligned -- when image is taller than cell (after scaling to tile width)." This means the user envisions a case where we scale to match tile width, not height. Let me reconsider.

Re-reading the decisions: "scale image to fill cell height" -- so the primary scale IS to cell height. The vertical center-crop applies to the scenario described: "when image is taller than cell after scaling to tile width." But the decision also says "tile width = `image.naturalWidth * (cellHeight / image.naturalHeight)`" -- this scales based on cell height, so the tile height equals cell height exactly. The vertical center-crop only matters if we were scaling differently.

**Resolution:** The scale factor is `cellHeight / naturalHeight`. This makes scaled height = cellHeight exactly. The tile width becomes `naturalWidth * scale`. Vertical center-crop is a no-op in this model because the image is scaled to exactly fill the cell height. The horizontal left-alignment is achieved by starting the pattern at the frame cell's left edge (via `ctx.translate(fx, fy)`).

### Anti-Patterns to Avoid
- **Creating pattern inside the frame loop:** Pattern should be created once per range (same image), not per frame. All frames in a range use the same image.
- **Forgetting to translate pattern origin:** `createPattern` tiles from the canvas origin `(0,0)`. Without translating to `(fx, fy)`, the pattern will be misaligned -- tiles won't start at the frame's left edge.
- **Not clipping to frame cell:** The pattern fill will extend beyond the frame cell if not clipped. Must use `ctx.save()/clip()/restore()`.
- **Missing null check on createPattern return:** `createPattern` returns `null` if the image is not fully loaded. The ThumbnailCache already ensures `cachedImg.complete` is true before returning, so this should not happen, but a null guard is still good practice.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Image tiling | Manual drawImage loop with offset math | `ctx.createPattern('repeat')` + `pattern.setTransform()` | Browser handles sub-pixel alignment, clipping, and GPU acceleration natively |
| Transform matrix construction | Manual matrix array `[a, b, c, d, e, f]` | `new DOMMatrix().scale(s, s)` | Readable, less error-prone, handles all browser quirks |
| Pattern caching | Custom Map per image | Reuse pattern within range loop (already scoped correctly) | Pattern creation is cheap; caching across draw calls adds complexity for minimal gain since `draw()` is called fresh each frame |

**Key insight:** The Canvas pattern API was designed for exactly this use case. A manual tile loop would require calculating visible tile count, handling partial tiles at edges, and managing sub-pixel alignment -- all of which `createPattern` + `fillRect` handles automatically.

## Common Pitfalls

### Pitfall 1: Pattern Origin Misalignment
**What goes wrong:** Tiles appear offset or misaligned because `createPattern` anchors at canvas `(0,0)`, not at the frame cell position.
**Why it happens:** The pattern coordinate space is the canvas coordinate space. Without translation, tiles start from the canvas origin.
**How to avoid:** Use `ctx.translate(fx, fy)` before `ctx.fillRect(0, 0, fw, fh)`, or use `pattern.setTransform()` with translation included in the DOMMatrix.
**Warning signs:** Tiles shift position when scrolling the timeline horizontally.

### Pitfall 2: Pattern Not Resetting Between Ranges
**What goes wrong:** A pattern created for one image/range leaks into the next range.
**Why it happens:** `ctx.fillStyle` persists. If a subsequent range has no cached image, the previous pattern might still be active.
**How to avoid:** Only set `ctx.fillStyle = pattern` inside the `if (cachedImg && pattern)` block. The else branch already sets its own `fillStyle`.

### Pitfall 3: DPR (Device Pixel Ratio) Interaction
**What goes wrong:** Tiles appear at wrong scale on Retina displays.
**Why it happens:** The canvas is already scaled by `this.dpr` via `ctx.scale(this.dpr, this.dpr)`. The pattern transform is applied on top of the context transform.
**How to avoid:** The existing DPR scaling in `setupCanvas()` (line 117) applies globally. Pattern `setTransform` operates in the pattern's own coordinate space, which is then transformed by the context. Since the context is already scaled for DPR, the pattern scale factor should use CSS pixel dimensions (which is what `TRACK_HEIGHT` and `naturalWidth`/`naturalHeight` already are). No DPR adjustment needed in the pattern transform itself.
**Warning signs:** Tiles appear 2x too large or 2x too small on Retina displays.

### Pitfall 4: Performance Regression from Excessive save/clip/restore
**What goes wrong:** Adding `ctx.save()/beginPath()/rect()/clip()/restore()` per frame cell adds overhead.
**Why it happens:** Each frame already has a border draw. Adding clip per frame adds 5 canvas state operations per frame.
**How to avoid:** The existing code already operates inside a per-track clip region (line 231-234). The per-frame clip is necessary for pattern containment but is lightweight. If profiling shows issues, consider the manual drawImage loop approach which can handle partial tiles without clipping. For the expected frame count (tens to low hundreds visible), this overhead is negligible.

### Pitfall 5: Low Zoom Threshold Too High or Too Low
**What goes wrong:** Setting threshold too high (e.g., 20px) causes visible pop when switching from solid color to tiled image. Setting too low (e.g., 1px) wastes GPU cycles drawing sub-pixel detail.
**How to avoid:** Use 4px as the threshold. At 4px frame width, individual tiles are imperceptible. The transition from solid color to tiled image is invisible at this scale because detail is too small to distinguish. This matches the `BASE_FRAME_WIDTH * zoom` where zoom approaches ~0.067 (4/60).

## Open Questions

1. **Pattern caching across draw() calls**
   - What we know: `createPattern()` is called each time `draw()` is invoked (on every animation frame during zoom). Creating a pattern object is lightweight (just wraps the image reference), but `setTransform` also runs each time.
   - What's unclear: Whether caching pattern objects in a Map (keyed by imageId + cellHeight) provides measurable performance improvement.
   - Recommendation: Start without caching. The pattern creation is O(1) and involves no image processing. If profiling reveals overhead, add a simple `Map<string, CanvasPattern>` cache keyed by imageId, invalidated when ThumbnailCache reloads images. This is explicitly listed as Claude's discretion.

2. **Interaction with existing per-track clip region**
   - What we know: Lines 231-234 already establish a clip region for the track content area (`TRACK_HEADER_WIDTH` to canvas width). The per-frame clip (for pattern containment) nests inside this existing clip.
   - What's unclear: Whether nested clips cause any visual artifacts or performance issues.
   - Recommendation: Nested clips are fully supported in Canvas 2D. The inner clip (frame cell) is always within the outer clip (track content area), so no issues expected.
