## Implementation Decisions

### Editor Timeline Play Script Markers
- **D-01:** Keep the existing main EFX Motion Physic Paint FX/layer bar as the container. Multiple saved Play scripts appear as nested mini-ranges inside that layer bar, not as separate timeline rows.
- **D-02:** The nested markers must be polished centered graphics inside the layer block height. Do not render literal ASCII/text examples like `|-----|`, `•----•`, or `[4]---[10]` as the UI.
- **D-03:** Markers should communicate range start, range end, and duration visually through endpoint/range graphics. The main timeline ruler already provides frame/time context, so markers should not show frame badges directly by default.
- **D-04:** Marker positions and widths must follow the existing main timeline frame scale, zoom, and horizontal scroll. If the user zooms the timeline, the marker positions and widths must zoom/move with the layer bar.
- **D-05:** The active saved Play script marker uses an accent highlight. Inactive saved Play script markers remain subdued inside the Physic Paint layer bar.
- **D-06:** Saved Play script ranges on the same Physic Paint layer must not overlap. Saving/creating a Play script must respect occupied saved-script ranges.

### Opening the Correct Script
- **D-07:** `[open fx paint canvas]` chooses the saved Play script from the editor scrubber/current frame. If the scrubber is inside exactly one saved Play script range, open that saved script segment.
- **D-08:** When opening a saved Play script from inside its range, the standalone Physics Paint window opens directly to the Play canvas tab for that script at the scrubber's relative frame inside the range.
- **D-09:** If the editor scrubber is not inside any saved Play script range, `[open fx paint canvas]` opens Roto at the current frame.
- **D-10:** From a gap/outside-range Roto open, the user may switch to Play canvas to create a new script, but the standalone must receive/evaluate the maximum allowed Play frame count from the editor based on the available gap before the next saved Play script range.
- **D-11:** If the available gap before the next saved Play script is smaller than the current/default Play duration, clamp the Play duration to the available no-overlap gap and show a clear max/limit message.
- **D-12:** Do not choose the nearest segment when the scrubber is outside a range. Gaps should remain useful for Roto work or starting a new constrained Play script.

### Play Canvas Preview Scrubbing
- **D-13:** Dragging/scrubbing the local Play canvas timeline previews only. It must not save, apply, or publish output until the user clicks `Save play`.
- **D-14:** Local Play canvas scrubbing must not move the main EFX Motion editor scrubber/playhead while the standalone Physics Paint window is open.
- **D-15:** A saved Play script opened at an in-between frame should show the script range with start, end, and current preview position in the standalone Play canvas timeline.
- **D-16:** For an existing saved Play script, local scrubbing previews cached rendered frames by default for faster feedback.
- **D-17:** If the user clears/remakes the animation, old cached frames stop being the preview truth. While remaking before `Save play`, preview comes from the live/current editable script state.
- **D-18:** When the user clicks `Save play`, newly rendered cached frames replace the old saved range frames.

### One-Hand Sequential Playback
- **D-19:** Play script rendering must distribute strokes sequentially in recorded order, like one artist hand painting the drawing, instead of rendering multiple strokes/letters in parallel.
- **D-20:** The current timestamp-ratio/parallel behavior in `AnimationPlayer` is not the desired user-facing Play behavior and should be replaced for Play script rendering.
- **D-21:** Frame allocation within the sequential playback should be based on stroke length: longer strokes receive more frames; short strokes may complete in one frame when the selected duration is tight.
- **D-22:** The chosen Play duration/range length is authoritative. If the duration is too short for smooth drawing, compress all strokes so the complete drawing still finishes within the chosen duration.
- **D-23:** Do not add a user-facing Sequential/Parallel mode toggle in this phase. Sequential playback is the Play animation behavior.

### Claude's Discretion
Planner/researcher may choose the exact storage shape for multiple saved Play script ranges, the exact marker geometry/styling, and the exact short-duration stroke-length allocation formula, but must preserve the no-overlap rule, scrubber-based script selection, cached saved-script preview behavior, and one-hand sequential playback outcome above.

## Summary

Phase 36.1 is a targeted integration phase inside the already-established standalone Physics Paint workflow: the main editor must show multiple saved Play script ranges as nested markers inside the existing `physic-paint` FX/layer bar, and opening the standalone canvas must select the saved script only when the editor scrubber is inside that script's range. [VERIFIED: 36.1-CONTEXT.md; /Users/lmarques/Dev/efx-motion-editor/.planning/ROADMAP.md]

The current code is single-Play-range oriented: `PhysicPaintWorkflowMetadata` contains optional `playStartFrame` and `playFrameCount`, `physicPaintStore` serializes one metadata record per layer, and `createPhysicPaintLaunchContext` uses that single metadata record to choose a Play start frame. [VERIFIED: app/src/types/physicPaint.ts; app/src/stores/physicPaintStore.ts; app/src/lib/physicPaintBridge.ts] Planning should therefore start by defining a multi-range saved script model, then thread that model through validation, persistence, timeline rendering, launch-context selection, standalone preview state, and tests. [VERIFIED: codebase Read]

The animation behavior change belongs in `packages/efx-physic-paint/src/animation/AnimationPlayer.ts`: the current `distributeStrokes` maps strokes by timestamp ratio, which can make strokes overlap in the same animation window; Phase 36.1 requires recorded-order sequential allocation weighted by stroke length/point count. [VERIFIED: packages/efx-physic-paint/src/animation/AnimationPlayer.ts; 36.1-CONTEXT.md]

**Primary recommendation:** Plan this as a no-new-dependency TypeScript refactor/integration across typed contracts, store persistence, launch-context resolution, Canvas 2D marker drawing, standalone Play preview state, and `AnimationPlayer` sequential distribution. [VERIFIED: package manifests + codebase Read]

### Pattern 2: Multi-Range Store Helpers Before UI

**What:** Add typed helper functions such as `getPlayScriptRanges(layerId)`, `findPlayScriptRangeAtFrame(layerId, frame)`, `getMaxPlayFrameCountFromGap(layerId, frame)`, and `upsertPlayScriptRange(...)` before changing UI code. ### Pattern 3: Explicit Version Bumps for Non-Reactive Maps

**What:** Every mutation of physics paint output or range metadata must invalidate serialization, increment `physicPaintVersion`, and mark the project dirty. [VERIFIED: app/src/stores/physicPaintStore.ts]

**When to use:** Use this for `Save play`, clear/remake, range replacement, project load, conversion, and any future range metadata mutation. [VERIFIED: app/src/stores/physicPaintStore.ts; project memory]

### Pattern 4: Sequential Animation Allocation

**What:** Replace timestamp-ratio distribution with recorded-order allocation: sort/use strokes in recorded order, compute each stroke weight from path length or point count, allocate the chosen total frame count across strokes, and ensure the final frame contains the complete drawing. [VERIFIED: packages/efx-physic-paint/src/animation/AnimationPlayer.ts; 36.1-CONTEXT.md]

**When to use:** Use for Play preview and Save play rendering because Phase 36.1 makes sequential playback the only Play behavior. [VERIFIED: 36.1-CONTEXT.md]

### Anti-Patterns to Avoid

- **Separate Play marker timeline rows:** Out of scope because markers must live inside the existing Physic Paint FX/layer bar. [VERIFIED: 36.1-CONTEXT.md]
- **Nearest-range selection from gaps:** Wrong because gaps must open Roto or allow constrained new Play creation. [VERIFIED: 36.1-CONTEXT.md]
- **Editor-side live simulation or headless replay:** Excluded by roadmap/history because prior adapter/headless directions were rejected. [VERIFIED: .planning/REQUIREMENTS.md; .planning/STATE.md]
- **Preview scrub publishes output:** Wrong because local scrubbing is preview-only until `Save play`. [VERIFIED: 36.1-CONTEXT.md]
- **Direct store reads inside `TimelineRenderer`:** Avoid because the renderer currently receives state via `draw(state)` and does not subscribe to stores directly. [VERIFIED: app/src/components/timeline/TimelineRenderer.ts]

## Common Pitfalls

### Pitfall 1: Replacing Single Metadata Too Late
**What goes wrong:** Timeline markers and launch selection cannot distinguish multiple saved scripts if the store still has only one `playStartFrame/playFrameCount`. [VERIFIED: app/src/stores/physicPaintStore.ts]
**Why it happens:** Existing metadata is per-layer and single-range. [VERIFIED: app/src/types/physicPaint.ts]
**How to avoid:** First define a multi-range data model with IDs, start frame, frame count, cached rendered frame references, and editable source state association. [VERIFIED: 36.1-CONTEXT.md]
**Warning signs:** Tests still assert only one `getWorkflowMetadata(layerId)` Play range after saving two separate scripts. [VERIFIED: current app/src/stores/physicPaintStore.test.ts]

### Pitfall 2: Markers Desync From Zoom/Scroll
**What goes wrong:** Nested markers appear at the wrong positions after timeline zoom or horizontal scroll. [VERIFIED: 36.1-CONTEXT.md]
**Why it happens:** Marker x/width calculations use DOM pixels or absolute frame indices instead of the existing `frameWidth` and `scrollX`. [VERIFIED: app/src/components/timeline/TimelineRenderer.ts]
**How to avoid:** Draw markers inside `drawFxTrack` or an adjacent helper using the same `barX`, `frameWidth`, `scrollX`, clip bounds, and `TRACK_HEADER_WIDTH`. [VERIFIED: app/src/components/timeline/TimelineRenderer.ts]
**Warning signs:** Marker endpoints do not align with timeline ruler frames at different zoom levels. [VERIFIED: 36.1-CONTEXT.md]

### Pitfall 3: Local Play Scrub Moves Main Editor Playhead
**What goes wrong:** Previewing an in-between frame in the standalone unexpectedly changes the EFX Motion editor scrubber. [VERIFIED: 36.1-CONTEXT.md]
**Why it happens:** Existing `navigateToSyncedFrame` sends `physic-paint:seek-frame` messages to the opener/main editor. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx; app/src/lib/physicPaintBridge.ts]
**How to avoid:** Add a distinct local Play preview/scrub path that updates standalone preview state without calling `sendPhysicPaintFrameSyncMessage`. [VERIFIED: 36.1-CONTEXT.md]
**Warning signs:** Scrubbing the Play strip triggers `timelineStore.seek(...)` in the main editor. [VERIFIED: app/src/lib/physicPaintBridge.ts]

### Pitfall 4: Cached Saved Frames Stay Truth After Remake
**What goes wrong:** The Play tab shows old saved frames after the user clears/remakes the animation before saving. [VERIFIED: 36.1-CONTEXT.md]
**Why it happens:** Saved cached frames and live editable state are not separated with a dirty/remade flag. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]
**How to avoid:** Track saved range cache status separately from live editable modifications and switch preview source to live state until `Save play` regenerates frames. [VERIFIED: 36.1-CONTEXT.md]
**Warning signs:** `latestPlayFramesRef` or stored range frames are used even after `clearActiveSource()` in Play mode. [VERIFIED: app/src/components/physic-paint/PhysicsPaintStudio.tsx]

### Pitfall 5: Sequential Allocation Drops Strokes Under Tight Durations
**What goes wrong:** A very short Play duration omits late strokes or fails to finish the drawing. [VERIFIED: 36.1-CONTEXT.md]
**Why it happens:** A naive one-frame-per-stroke guarantee cannot fit more strokes than frames. [ASSUMED]
**How to avoid:** Make duration authoritative: compress strokes across available frames and ensure the last frame renders all strokes complete. [VERIFIED: 36.1-CONTEXT.md; packages/efx-physic-paint/src/animation/AnimationPlayer.ts]
**Warning signs:** `frameCount < strokeCount` causes missing letters at the final frame. [ASSUMED]

## Open Questions (RESOLVED)

1. **RESOLVED: Should each saved Play script have its own editable source snapshot?**
   - What we know: Current store keeps one editable state per physics layer. [VERIFIED: app/src/stores/physicPaintStore.ts]
   - Resolution: Each saved Play script should store its own editable source snapshot or source reference when needed by the current single editable layer-state architecture, so reopening two distinct saved scripts can restore the correct script-specific source without requiring a new source-lane model. [VERIFIED: 36.1-CONTEXT.md; planner decision]
   - Final recommendation: Keep the first-wave data-model task that stores per-range editable state or a per-range editable state reference and tests reopening two distinct scripts. [VERIFIED: codebase Read]

2. **RESOLVED: How should active marker highlighting be selected?**
   - What we know: The active marker is defined by the editor scrubber being inside a saved range. [VERIFIED: 36.1-CONTEXT.md]
   - Resolution: Scrubber containment is the source of truth for active marker highlighting and opening behavior. Marker click selection is included only if already planned by an execution plan; otherwise it remains out of scope for Phase 36.1 and should not be added as a new requirement. [VERIFIED: 36.1-CONTEXT.md; planner decision]
   - Final recommendation: Use scrubber containment as the active marker source; do not add marker click selection unless it already falls within an existing planned task without expanding scope. [VERIFIED: 36.1-CONTEXT.md]

3. **RESOLVED: Exact stroke-length formula for sequential allocation**
   - What we know: Longer strokes should receive more frames and short strokes may complete in one frame under tight duration. [VERIFIED: 36.1-CONTEXT.md]
   - Resolution: Use recorded-order sequential allocation with point-count weighting as the first implementation because existing `AnimationPlayer` already exposes `stroke.points.length`; keep duration authoritative and ensure the final frame contains the complete drawing. [VERIFIED: packages/efx-physic-paint/src/animation/AnimationPlayer.ts; planner decision]
   - Final recommendation: Implement point-count weighted allocation first; consider path distance only in a later phase if point-count weighting produces visible unevenness. [VERIFIED: packages/efx-physic-paint/src/animation/AnimationPlayer.ts]
