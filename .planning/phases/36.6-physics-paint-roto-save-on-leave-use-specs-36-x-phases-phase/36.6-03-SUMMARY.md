# Phase 36.6 Plan 03: Save-on-Leave Stale ApplyStatus Gap Closure Summary

Dirty Roto save-on-leave now saves the source frame once and opens the latest requested destination after the matching successful apply result, even when the result handler closure still sees `applyStatus` as `applying`.

## What Changed

- Added a focused source-contract regression for the stale `applyStatus` boundary in `PhysicsPaintStudio.test.ts`.
- Added `openSyncedRotoFrameAfterSave` in `PhysicsPaintStudio.tsx` for already-authorized post-save Roto advances.

- Expanded save-on-leave queueing so rapid clicks during the post-send/pre-result window overwrite `pendingRotoAdvanceRef.current` without starting a duplicate source-frame save.
- Updated existing source-contract expectations so failed save-on-leave behavior remains explicitly checked against the new internal post-save opener.

## Decisions Made

1. Internal post-save advance is intentionally narrower than public navigation: it validates the target frame, stops cached Roto playback, clears cached playback overlay, loads local editable state or cached reference, sends frame sync, and updates `launchContext.startFrame`.
2. The internal opener intentionally does not check `rotoFlushInFlightRef.current` or `applyStatus === 'applying'`, because it is called only after a matching successful apply result has consumed the queued destination.
3. The rapid-navigation no-duplicate-save branch is keyed to `saveOnLeaveSourceFrameRef.current` plus `activeOperationIdRef.current`, preserving the queue across the post-send/pre-result interval after `rotoFlushInFlightRef.current` has settled.
