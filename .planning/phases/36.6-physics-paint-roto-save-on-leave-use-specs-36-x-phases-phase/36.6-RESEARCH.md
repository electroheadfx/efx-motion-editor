### Locked Decisions
## Implementation Decisions

### Navigation Timing
- **D-01:** Painting must only mark the current Roto frame dirty. It must never trigger cache rendering or Save current behavior per stroke.
- **D-02:** Save/render cache happens only at the frame-leave boundary. When the user requests another Roto frame, keep the dirty source frame visible, save that source frame once through the existing **Save current** / cache path, then open the requested destination.
- **D-03:** Rapid frame changes must not trigger multiple saves. Save the original dirty source frame once, keep the source visible with saving feedback, and after the save completes open only the latest requested destination.
- **D-04:** Save-on-leave applies to all existing Roto frame navigation paths: timeline cell clicks and existing previous/next/first/last controls. Do not rebuild navigation and do not add new navigation controls.
- **D-05:** Clean-frame navigation does not call the cache path. It switches immediately.

### Saving Feedback
- **D-06:** While save-on-leave is waiting, show the source frame cell as pending/saving and show one compact status line. Do not use a modal or blocking overlay for the normal saving state.
- **D-07:** Use the status copy `Saving frame N…` for the source frame being cached.
- **D-08:** During saving, allow existing Roto navigation clicks to update the latest queued destination. Disable **Save current** and destructive/convert actions until the save finishes.
- **D-09:** After a successful save-on-leave, do not show a toast or extra success notification. The destination opening and the source cell becoming saved/cached are enough.

### Failure Behavior
- **D-10:** If save-on-leave fails, stay on the dirty source frame, keep unsaved state visible, clear the queued destination, and show an inline error with retry guidance. Do not switch to the destination after a failed save.
- **D-11:** After a failure, the next explicit navigation attempt retries the same boundary save through the existing path. Do not add a dedicated Retry button for this MVP.
- **D-12:** Preserve any previous cached PNG for the source frame if save-on-leave fails. The dirty in-memory state remains the unsaved truth until a retry succeeds.

### Manual Save Role
- **D-13:** Keep **Save current** visible and useful. Save-on-leave adds convenience; it does not replace the explicit manual cache action.
- **D-14:** Manual **Save current** saves the current frame and stays on that frame. It must not navigate or consume any queued destination.
- **D-15:** Add only a subtle hint that dirty frames save when leaving. Keep existing **Save current** wording and avoid tutorials or new controls.
- **D-16:** Phase 36.4 close behavior does not change. Dirty close still uses the explicit close/discard/cancel/save choices; save-on-leave applies only to frame-to-frame Roto navigation.

## Summary

Phase 36.6 is not a new rendering feature; it is a leave-boundary coordinator around the existing Roto durable cache path. The implementation should route every Roto frame navigation entry point through one gate that snapshots the dirty source frame, sends exactly one existing `apply-canvas` / delete payload through `flushRotoFrame`, keeps the source frame visible, and opens only the latest requested destination after the matching apply result succeeds. [CITED: SPECS/36.x-phases/phase-36.6-save-on-leave/spec-36.6-save-on-leave.md] [VERIFIED: codebase grep]

The current code already contains most primitives: `dirtyRotoFramesRef`, `pendingRotoAdvanceRef`, `rotoFlushInFlightRef`, `flushRotoFrame`, `saveRotoFrame`, `navigateToSyncedFrame`, pending Roto cells, and `Saving frame ${frame}...` cell labels. The key planning risk is that existing navigation currently refuses to run while `rotoFlushInFlightRef` or `applyStatus === 'applying'`, which conflicts with the new requirement that rapid Roto navigation clicks update the queued destination while the source save is in flight. [VERIFIED: codebase grep]

Do not plan external libraries, new navigation controls, modal save dialogs, project-format migrations, or final Pencil timeline redesign. The phase should be a narrow TypeScript/Preact state-machine change with targeted tests in existing Physics Paint test files. [CITED: /Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.6-physics-paint-roto-save-on-leave-use-specs-36-x-phases-phase/36.6-CONTEXT.md]

**Primary recommendation:** Implement a single Roto leave-boundary gate in `PhysicsPaintStudio.tsx` that distinguishes clean immediate navigation from dirty save-on-leave, queues only the latest destination during in-flight saves, and reuses `flushRotoFrame` / `saveRotoFrame` plus existing Phase 36.5 pending cell semantics. [VERIFIED: codebase grep]

### Pattern 1: Single Leave-Boundary Gate
**What:** All Roto navigation intents call one function that checks whether the current Roto frame is dirty before switching frames. [VERIFIED: codebase grep]
**When to use:** Use for timeline cell clicks, previous/next/first/last controls, and keyboard frame navigation while in Roto mode. [CITED: 36.6-CONTEXT.md]
**Example:**
```typescript
// Source: existing PhysicsPaintStudio.tsx pattern, adapted for planning only.
// Keep implementation in the existing studio component unless extraction improves tests.
const requestRotoFrameNavigation = useCallback(async (targetFrame: number) => {
  if (!Number.isInteger(targetFrame) || targetFrame < 0) return false;
  if (workflowModeRef.current !== 'roto') return navigateToSyncedFrame(targetFrame);

  snapshotCurrentRotoFrame();
  const sourceFrame = currentFrame;
  const sourceIsDirty = dirtyRotoFramesRef.current.has(sourceFrame);

  if (!sourceIsDirty) return navigateToSyncedFrame(targetFrame);

  // Dirty path: keep source visible, save source once, then advance from apply-result success.
  return saveRotoFrame(targetFrame);
}, [currentFrame, navigateToSyncedFrame, saveRotoFrame, snapshotCurrentRotoFrame]);
```

### Pattern 2: In-Flight Queue Collapses to Latest Destination
**What:** While source save is in flight, navigation clicks update one queued target and return without starting another `flushRotoFrame`. [CITED: 36.6-CONTEXT.md]
**When to use:** During dirty save-on-leave only. Manual `Save current` must save and stay, with no queued destination consumption. [CITED: 36.6-CONTEXT.md]
**Example:**
```typescript
// Source: existing pendingRotoAdvanceRef / rotoFlushInFlightRef seam in PhysicsPaintStudio.tsx.
if (rotoFlushInFlightRef.current && saveOnLeaveSourceRef.current !== null) {
  pendingRotoAdvanceRef.current = targetFrame; // latest wins
  setRotoSaveOnLeaveStatus(`Saving frame ${saveOnLeaveSourceRef.current}…`);
  return false;
}
```

### Pattern 3: Apply-Result Is the Only Success Boundary
**What:** Do not navigate immediately after `sendPhysicPaintApplyPayload`; navigate only after the matching apply result succeeds. Existing code already has `activeOperationIdRef`, `startApplyTimeout`, and `handleApplyResult`. [VERIFIED: codebase grep]
**When to use:** Dirty save-on-leave, close-saving, and manual save. The phase must keep close-saving separate from frame-to-frame navigation. [VERIFIED: codebase grep]
**Example:**
```typescript

  void navigateToSyncedFrame(nextFrame);
```

### Anti-Patterns to Avoid
- **Per-stroke cache rendering:** Painting should mark the frame dirty only; cache writes happen at explicit save or frame-leave boundary. [CITED: 36.6-CONTEXT.md]
- **Second auto-save channel:** A separate save-on-leave payload path risks divergent durability semantics; reuse `flushRotoFrame` / Save current path. [CITED: 36.6-CONTEXT.md] [VERIFIED: codebase grep]
- **Blocking all navigation while applying:** Current `navigateToSyncedFrame` refuses navigation while applying; for this phase, Roto clicks during save must update queued destination instead. [VERIFIED: codebase grep] [CITED: 36.6-CONTEXT.md]
- **Success toast:** The spec explicitly wants no extra success notification after save-on-leave; cell state and destination opening are sufficient. [CITED: 36.6-CONTEXT.md]
- **Changing dirty close:** Phase 36.4 close/discard/cancel/save behavior remains unchanged. [CITED: SPECS/36.x-phases/Done/phase-36.4-explicit-close/spec-36.4-explicit-close.md]

## Common Pitfalls

### Pitfall 1: Saving the Destination Instead of the Source
**What goes wrong:** A rapid click changes `currentFrame` or loads target state before the dirty source is snapshotted, causing the cache payload to use the wrong frame. [ASSUMED]
**Why it happens:** Navigation and save share the current `launchContext.startFrame` and engine state. [VERIFIED: codebase grep]
**How to avoid:** Capture `sourceFrame` before any navigation/load; keep source visible until apply result succeeds. [CITED: 36.6-CONTEXT.md]
**Warning signs:** Tests show `apply-canvas.startFrame` equals the clicked destination instead of the frame being left. [ASSUMED]

### Pitfall 2: Existing Apply Lock Blocks Queued Destination Updates
**What goes wrong:** User clicks frame 10 then 15 while frame 3 saves; code ignores the second click because `applyStatus === 'applying'`. [VERIFIED: codebase grep]
**Why it happens:** Current `navigateToSyncedFrame` returns false when `rotoFlushInFlightRef.current` or `applyStatus === 'applying'`. [VERIFIED: codebase grep]
**How to avoid:** Put the queue update before the generic applying guard for Roto save-on-leave, or create a dedicated request function that handles Roto in-flight saves before calling immediate navigation. [VERIFIED: codebase grep]
**Warning signs:** Rapid-click test opens the first requested target rather than the latest. [ASSUMED]

### Pitfall 3: Manual Save Consumes a Queued Destination
**What goes wrong:** Clicking **Save current** unexpectedly navigates after save. [CITED: 36.6-CONTEXT.md]
**Why it happens:** `saveRotoFrame` already accepts `advanceToFrame`; a careless caller may pass or leave stale queue state. [VERIFIED: codebase grep]
**How to avoid:** Explicit manual save must pass `null`, clear/ignore save-on-leave queue context, and stay on the current frame. [CITED: 36.6-CONTEXT.md]
**Warning signs:** Cmd+S or Save current changes `launchContext.startFrame`. [VERIFIED: codebase grep]

### Pitfall 4: Failure Clears Dirty Truth or Overwrites Previous Cache
**What goes wrong:** A failed save-on-leave removes the unsaved marker or replaces a previous cached PNG with partial output. [CITED: 36.6-CONTEXT.md]
**Why it happens:** Current `flushRotoFrame` deletes dirty state and upserts cache shortly after send succeeds, before apply-result success. [VERIFIED: codebase grep]
**How to avoid:** For save-on-leave failure planning, verify dirty state remains visible on send error, timeout, and failed apply result; avoid destructive cache mutation until success if code changes expose that risk. [VERIFIED: codebase grep]
**Warning signs:** Failed apply result switches frames, removes pending state, or changes saved cell status. [ASSUMED]

### Pitfall 5: Source-Specific Status Copy Is Lost
**What goes wrong:** UI says `Saving current frame…` while the user clicked away, making it unclear which source frame is being saved. [VERIFIED: codebase grep]
**Why it happens:** Existing `getRotoPendingLabel` only knows `hasPending` and `isSaving`, not the saving source frame. [VERIFIED: codebase grep]
**How to avoid:** Add a source-frame-aware status prop/helper for save-on-leave and keep cell label as `Saving frame N...` / displayed copy as `Saving frame N…`. [CITED: 36.6-CONTEXT.md]
**Warning signs:** UAT cannot tell whether frame 3 or clicked frame 10 is being saved. [ASSUMED]
