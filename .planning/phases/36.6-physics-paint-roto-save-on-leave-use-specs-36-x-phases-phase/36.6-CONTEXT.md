# Phase 36.6: Physics Paint Roto Save On Leave - Context

**Gathered:** 2026-06-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 36.6 delivers save-on-leave for existing Physics Paint Roto frame navigation. When the current Roto frame is clean, navigation to another Roto frame remains immediate. When the current Roto frame is dirty and the animator requests another Roto frame, the source frame stays visible, saves once through the existing **Save current** / Roto cache path, and only then opens the requested destination frame.

The source of truth for the behavior contract is `SPECS/36.x-phases/phase-36.6-save-on-leave/spec-36.6-save-on-leave.md`. This context captures implementation decisions only.

This phase must not rebuild Roto navigation, add new navigation controls, change Phase 36.4 explicit close behavior, or expand into key utilities, interpolation, cached autoplay, final Pencil timeline redesign, or broad Physics Paint cleanup.

</domain>

<decisions>
## Implementation Decisions

### Navigation Timing
- **D-01:** Painting must only mark the current Roto frame dirty. It must never trigger cache rendering or Save current behavior per stroke.
- **D-02:** Save/render cache happens only at the frame-leave boundary. When the user requests another Roto frame, keep the dirty source frame visible, save that source frame once through the existing **Save current** / cache path, then open the requested destination.
- **D-03:** Rapid frame changes must not trigger multiple saves. Save the original dirty source frame once, keep the source visible with saving feedback, and after the save completes open only the latest requested destination.
- **D-04:** Save-on-leave applies to all existing Roto frame navigation paths: timeline cell clicks and existing previous/next/first/last controls. Do not rebuild navigation and do not add new navigation controls.
- **D-05:** Clean-frame navigation does not call the cache path. It switches immediately.

### Saving Feedback
- **D-06:** While save-on-leave is waiting, show the source frame cell as pending/saving and show one compact status line. Do not use a modal or blocking overlay for the normal saving state.
- **D-07:** Use the status copy `Saving frame N…` for the source frame being cached.
- **D-08:** During saving, allow existing Roto navigation clicks to update the latest queued destination. Disable **Save current** and destructive/convert actions until the save finishes.
- **D-09:** After a successful save-on-leave, do not show a toast or extra success notification. The destination opening and the source cell becoming saved/cached are enough.

### Failure Behavior
- **D-10:** If save-on-leave fails, stay on the dirty source frame, keep unsaved state visible, clear the queued destination, and show an inline error with retry guidance. Do not switch to the destination after a failed save.
- **D-11:** After a failure, the next explicit navigation attempt retries the same boundary save through the existing path. Do not add a dedicated Retry button for this MVP.
- **D-12:** Preserve any previous cached PNG for the source frame if save-on-leave fails. The dirty in-memory state remains the unsaved truth until a retry succeeds.

### Manual Save Role
- **D-13:** Keep **Save current** visible and useful. Save-on-leave adds convenience; it does not replace the explicit manual cache action.
- **D-14:** Manual **Save current** saves the current frame and stays on that frame. It must not navigate or consume any queued destination.
- **D-15:** Add only a subtle hint that dirty frames save when leaving. Keep existing **Save current** wording and avoid tutorials or new controls.
- **D-16:** Phase 36.4 close behavior does not change. Dirty close still uses the explicit close/discard/cancel/save choices; save-on-leave applies only to frame-to-frame Roto navigation.

### Claude's Discretion
None — the user made concrete decisions for all selected gray areas.

</decisions>

## Existing Code Insights

### Reusable Assets
- `app/src/components/physic-paint/PhysicsPaintStudio.tsx` — Main integration seam. Existing state and callbacks include `dirtyRotoFramesRef`, `pendingRotoAdvanceRef`, `flushRotoFrame`, `saveRotoFrame`, and Roto navigation callbacks. Planning should extend these seams rather than rebuild navigation.
- `app/src/components/physic-paint/PhysicsPaintWorkflowStrip.tsx` — Existing Roto strip UI already accepts saved/cached/editable/pending frame inputs, renders Roto cells, and owns the visible **Save current** action.
- `app/src/components/physic-paint/physicsPaintWorkflowState.ts` — Existing cell-state helpers already model `dirty` and `pending` overlays and produce labels such as `Saving frame N...` / unsaved-frame labels.
- `app/src/stores/physicPaintStore.ts`, `app/src/types/physicPaint.ts`, and `app/src/lib/physicPaintBridge.ts` — Existing store/types/bridge surfaces for Roto cache state and rendered frame payloads.

### Established Patterns
- Reuse the existing **Save current** / Roto cache path; do not create a second save/render/cache channel for auto-save-on-leave.
- Dirty tracking is local to Roto frames. Painting should mark the current frame dirty and update pending cell state, but not render cache output until the user leaves the frame or manually clicks **Save current**.
- The Physics Paint workflow strip owns a compact, physics-paint-specific lane model. Do not import, clone, or redesign the main editor timeline for this phase.
- Prior phases favor minimal visible feedback over modal-heavy flows for normal operation; explicit prompts are reserved for close/discard decisions.

### Integration Points
- Wrap or route all existing Roto frame navigation paths through a single leave-boundary gate: cell clicks plus previous/next/first/last controls.
- Preserve current-frame visibility while `flushRotoFrame` / `saveRotoFrame` performs the source-frame cache write.
- Use pending/saving state to drive the source Roto cell overlay and compact status copy.
- Update the queued destination to the latest requested frame while a save is in flight, without starting another save.
- Keep Phase 36.4 close handler behavior separate from frame navigation behavior.
- Relevant tests to extend: `app/src/components/physic-paint/PhysicsPaintStudio.test.ts`, `app/src/components/physic-paint/PhysicsPaintWorkflowStrip.test.ts`, `app/src/components/physic-paint/physicsPaintWorkflowState.test.ts`, `app/src/lib/physicPaintRotoDurableCore.test.ts`, `app/src/lib/physicPaintBridge.test.ts`, and `app/src/stores/physicPaintStore.test.ts`.
