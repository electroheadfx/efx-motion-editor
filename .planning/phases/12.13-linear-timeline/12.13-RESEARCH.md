# Phase 12.13: Linear-timeline - Research

**Researched:** 2026-03-19
**Domain:** Timeline layout mode (Canvas 2D rendering, Preact signal-driven UI state)
**Confidence:** HIGH

## Summary

This phase adds a "linear timeline" layout mode where all content sequences are drawn on a single horizontal row instead of the current vertically-stacked track layout. The implementation is entirely within the existing Canvas 2D rendering pipeline (`TimelineRenderer.ts`) and its supporting state (`timelineStore.ts`, `frameMap.ts`). No new libraries are needed -- this is a layout toggle that reuses the existing thumbnail tile pattern, frame math, and interaction system.

The core challenge is introducing a `layoutMode` signal (`'stacked' | 'linear'`) that changes how `trackLayouts` maps to canvas drawing coordinates. In stacked mode (current default), each content sequence occupies its own row at `TRACK_HEIGHT` height. In linear mode, all sequences share a single row and are placed side-by-side horizontally. A secondary `displayMode` signal (`'thumb-name' | 'thumb-only'`) controls whether the sequence name overlay is drawn on top of thumbnails.

**Primary recommendation:** Add two new signals to `timelineStore` (layoutMode and displayMode), branch the `TimelineRenderer.draw()` method to use a `drawLinearTracks()` path when in linear mode, and add a toggle UI element in the TimelinePanel toolbar.

## Architecture Patterns

### Recommended Approach

```
Signals (timelineStore)
  layoutMode: Signal<'stacked' | 'linear'>     // NEW
  displayMode: Signal<'thumb-name' | 'thumb-only'>  // NEW

frameMap.ts
  trackLayouts: computed  (unchanged -- always computes per-sequence track data)
  linearTrackLayout: computed  (NEW -- single row with all sequences side-by-side)

  draw() -> branches on layoutMode
    stacked: existing drawTracks code path (unchanged)
    linear:  new drawLinearTrack() method

TimelineInteraction.ts
  trackIndexFromY() / dropIndexFromY() -> in linear mode, all content is index 0
  Click-to-seek, selection -> adapt to single-row coordinate mapping

TimelineCanvas.tsx
  Pass layoutMode into draw state (unchanged subscription pattern)

TimelinePanel.tsx
  Add toggle button(s) in toolbar row
```

### Pattern 1: Layout Mode Signal in timelineStore
**What:** Add `layoutMode` and `displayMode` signals to `timelineStore`, with setter methods.
**When to use:** These belong in timelineStore (not uiStore) because they directly affect timeline rendering and layout calculations (totalContentHeight, maxScrollY, scrolling behavior).
**Rationale:** timelineStore already owns all layout-affecting state: zoom, scrollX/Y, viewportWidth/Height, and layout constants (TRACK_HEIGHT, TRACK_HEADER_WIDTH). Layout mode is a timeline rendering concern, not a UI panel concern.

```typescript
// In timelineStore.ts
const layoutMode = signal<'stacked' | 'linear'>('stacked');
const displayMode = signal<'thumb-name' | 'thumb-only'>('thumb-name');

// totalContentHeight needs to branch on layoutMode
const totalContentHeight = computed(() => {
  const fxCount = fxTrackLayouts.value.length;
  if (layoutMode.value === 'linear') {
    // Linear: only FX tracks + 1 content row
    return RULER_HEIGHT + fxCount * FX_TRACK_HEIGHT + TRACK_HEIGHT;
  }
  const contentCount = trackLayouts.value.length;
  return RULER_HEIGHT + fxCount * FX_TRACK_HEIGHT + contentCount * TRACK_HEIGHT;
});
```

### Pattern 2: Linear Track Layout Computed Signal
**What:** A new computed signal in `frameMap.ts` that produces a single-row layout where all content sequences are placed end-to-end.
**When to use:** Used by `TimelineRenderer.drawLinearTrack()` when layoutMode is 'linear'.

```typescript
// In frameMap.ts
export interface LinearSegment {
  sequenceId: string;
  sequenceName: string;
  startFrame: number;    // global frame where this sequence starts
  endFrame: number;      // global frame where this sequence ends (exclusive)
  keyPhotoRanges: KeyPhotoRange[];
}

// This is effectively the same data as trackLayouts but consumed differently:
// In linear mode, the renderer draws all segments in a single row at the same Y position.
// The existing trackLayouts already contains this information -- each track has
// startFrame/endFrame -- so no new computed is strictly needed. The renderer
// just iterates trackLayouts and draws them all at the same Y coordinate.
```

**Key insight:** The existing `trackLayouts` already positions sequences end-to-end on the global timeline (each track has `startFrame` and `endFrame`). In stacked mode, each track is drawn at a different Y. In linear mode, they are all drawn at the same Y. No new computed signal is needed.

### Pattern 3: Branching the Renderer Draw Path
**What:** TimelineRenderer.draw() checks layoutMode and calls either the existing stacked draw code or a new drawLinearTrack() method.

### Pattern 4: Name Overlay Rendering
**What:** In linear mode, draw sequence name as semi-transparent overlay text on top of thumbnails.
**When to use:** When displayMode is 'thumb-name' (the default).

```typescript
// After drawing thumbnails for a sequence segment in linear mode
if (state.displayMode === 'thumb-name') {
  const segStartX = track.startFrame * frameWidth - scrollX + TRACK_HEADER_WIDTH;
  const segWidth = (track.endFrame - track.startFrame) * frameWidth;

  // Semi-transparent background strip at bottom of track
  ctx.fillStyle = 'rgba(0, 0, 0, 0.6)';
  const labelH = 16;
  ctx.fillRect(segStartX, trackY + TRACK_HEIGHT - 2 - labelH, segWidth, labelH);

  // Sequence name text
  ctx.fillStyle = '#EEEEEE';
  ctx.font = '10px system-ui, sans-serif';
  ctx.textBaseline = 'middle';
  const name = this.truncateText(ctx, track.sequenceName, segWidth - 8);
  ctx.fillText(name, segStartX + 4, trackY + TRACK_HEIGHT - 2 - labelH / 2);
}
```

### Anti-Patterns to Avoid
- **Duplicating frameMap logic:** Do NOT create a separate `linearFrameMap` that recalculates frame positions. The existing `trackLayouts` already has all the frame range data needed. Linear mode just changes the Y coordinate, not the horizontal positioning.
- **Modifying stacked code paths:** Do NOT alter the existing stacked rendering. Add branching early in `draw()` and keep the linear code path entirely separate. This preserves the working stacked mode.
- **Storing layout mode in uiStore:** While uiStore handles UI toggles, this signal directly affects timeline computed signals (totalContentHeight, maxScrollY) which live in timelineStore. Keep it co-located.
- **Hiding track headers in linear mode:** The track header column still shows in linear mode but shows a single header (or no individual names, since names overlay the thumbnails). Do NOT remove the header column as it would shift all frame X coordinates.

## Design Decisions (Claude's Discretion)

### 1. Toggle UI: Segmented icon button pair
**Recommendation:** Two adjacent icon buttons (stacked/linear) in the toolbar, styled like the existing zoom buttons. This matches the existing TimelinePanel toolbar visual language. A third smaller button toggles between thumb-name and thumb-only modes.
**Rationale:** The toolbar already has grouped buttons (zoom -/Fit All/+). A segmented pair is immediately discoverable without needing a dropdown.

### 2. Store location: timelineStore
**Recommendation:** Both `layoutMode` and `displayMode` signals in `timelineStore`.
**Rationale:** They affect `totalContentHeight`, `maxScrollY`, scrollY clamping, and the `DrawState` interface -- all timelineStore concerns. uiStore is for panel/sidebar/editor mode state.

### 3. Name overlay styling
**Recommendation:** Semi-transparent black background bar (`rgba(0,0,0,0.6)`) at the bottom of each sequence segment, 16px tall, with white 10px text. Clipped to the sequence segment boundaries.
**Rationale:** Bottom placement keeps the thumbnail visible. Semi-transparent background ensures readability on any thumbnail. Consistent with NLE conventions (Premiere, Resolve).

### 4. FX/content-overlay in linear mode
**Recommendation:** FX tracks and content-overlay tracks render exactly the same as in stacked mode (above the content area, as range bars). Only the content tracks change layout. Linear mode affects only the `kind === 'content'` sequences.
**Rationale:** FX tracks are already a single row each with range bars -- they don't stack per content track. Changing their layout would be confusing and is not part of the user request.

### 5. Scroll behavior in linear mode
**Recommendation:** Horizontal scroll works identically (same frameWidth, same zoom, same scrollX). Vertical scroll is effectively disabled or clamped to 0 (since there is only one content row, it always fits). Set `maxScrollY` to account for only FX tracks + 1 content row.
**Rationale:** The totalContentHeight computed already handles this when branching on layoutMode. The existing scroll clamping in timelineStore.setScrollY will naturally work.

### 6. Zoom interaction in linear mode
**Recommendation:** Zoom works identically -- it changes `frameWidth` which affects all sequences uniformly. No special handling needed since all sequences are drawn with the same frameWidth in both modes.
**Rationale:** Zoom operates on the frame-to-pixel mapping, which is mode-independent.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Track layout data | New computed for linear positions | Existing `trackLayouts` from frameMap.ts | Already computes startFrame/endFrame per sequence globally |
| Thumbnail tile rendering | New tile pattern logic | Existing createPattern code in drawTrack | Same tile drawing, just at a different Y coordinate |
| Scroll clamping | Manual scroll bounds for linear | Existing totalContentHeight/maxScrollY computeds | Branch on layoutMode in the existing computed |
| Frame hit-testing | New click-to-frame logic | Existing `frameFromX()` + `getFrame()` | Frame X calculations are mode-independent |

**Key insight:** The linear layout is NOT a different data model. It is the same data (`trackLayouts`) drawn at a different Y position. The only real new code is the Canvas 2D drawing path and the name overlay rendering.

## Common Pitfalls

### Pitfall 1: Track Header Column in Linear Mode
**What goes wrong:** Forgetting to handle the track header column in linear mode -- either leaving per-sequence labels (wrong for single row) or removing the column entirely (shifts all X coordinates).
**Why it happens:** The stacked mode draws sequence names in the header. Linear mode doesn't need per-sequence headers since names overlay the thumbnails.
**How to avoid:** In linear mode, draw the header column with a single label (e.g., "Timeline" or empty) and keep TRACK_HEADER_WIDTH unchanged. This preserves all frame X coordinate math.
**Warning signs:** Thumbnails shifted left or right compared to ruler markings.

### Pitfall 2: Sequence Selection in Linear Mode
**What goes wrong:** Click-to-select-sequence logic in `TimelineInteraction.onPointerDown` uses `trackIndexFromY()` which assumes stacked layout. In linear mode, all sequences are at trackIndex 0 and must be differentiated by X position.
**Why it happens:** The existing code maps Y to track index. Linear mode needs X-to-sequence mapping.
**How to avoid:** In linear mode, use the clicked frame number to determine which sequence was clicked (compare against each track's startFrame/endFrame range).
**Warning signs:** Clicking on any sequence always selects the first sequence, or clicking anywhere selects no sequence.

### Pitfall 3: Drag Reorder in Linear Mode
**What goes wrong:** Track reorder drag uses vertical mouse movement to determine drop position. In linear mode, reorder should be disabled or adapted to horizontal reorder.
**Why it happens:** The drag logic assumes vertical track stacking.
**How to avoid:** Disable content track drag reorder in linear mode (it doesn't make visual sense for a single row). Keep FX track drag reorder unchanged (still vertical).
**Warning signs:** Ghost track rendering at wrong position, reorder completing with incorrect indices.

### Pitfall 4: Sequence Boundary Visual Separators
**What goes wrong:** In linear mode, adjacent sequences blend together visually because there are no vertical separators between them.
**Why it happens:** In stacked mode, each sequence is a separate row with natural visual separation. In linear mode they are adjacent horizontally.
**How to avoid:** Draw a visible vertical separator line (2px, contrasting color) at sequence boundaries in linear mode. Similar to the existing key photo separator logic.
**Warning signs:** User cannot tell where one sequence ends and another begins.

### Pitfall 5: DrawState Interface Extension
**What goes wrong:** Adding layoutMode and displayMode to DrawState but forgetting to pass them from TimelineCanvas.tsx effect subscription, causing linear mode to never trigger.
**Why it happens:** The signal subscription in TimelineCanvas must read the new signals to include them in the draw call.
**How to avoid:** Add `timelineStore.layoutMode.value` and `timelineStore.displayMode.value` reads in the TimelineCanvas effect, and pass them in the `renderer.draw({...})` call.
**Warning signs:** Toggle button updates but canvas does not redraw.

### Pitfall 6: Keyframe Diamonds in Linear Mode
**What goes wrong:** Keyframe diamond Y coordinates are calculated using `trackIndex * TRACK_HEIGHT`. In linear mode, all content tracks share index 0.
**Why it happens:** `drawKeyframeDiamonds()` uses `trackIndex` for Y calculation.
**How to avoid:** In linear mode, all content track keyframe diamonds are drawn at the single content row Y. The X coordinates remain correct since they use global frames.
**Warning signs:** Keyframe diamonds drawn at wrong vertical position or missing in linear mode.

### Example 2: Linear Mode Drawing (Core Logic)
```typescript
// New method in TimelineRenderer
private drawLinearTrack(
  ctx: CanvasRenderingContext2D,
  state: DrawState,
  tracks: TrackLayout[],
  frameWidth: number,
  scrollX: number,
  w: number,
  fxOffset: number,
  colors: Record<string, string>,
): void {
  const trackY = RULER_HEIGHT + fxOffset;

  // Single row background
  ctx.fillStyle = colors.trackBg;
  ctx.fillRect(0, trackY, w, TRACK_HEIGHT);

  // Header column (empty or "Timeline" label)
  ctx.fillStyle = colors.headerBg;
  ctx.fillRect(0, trackY, TRACK_HEADER_WIDTH, TRACK_HEIGHT);

  // Clip content area
  ctx.save();
  ctx.beginPath();
  ctx.rect(TRACK_HEADER_WIDTH, trackY, w - TRACK_HEADER_WIDTH, TRACK_HEIGHT);
  ctx.clip();

  // Draw each sequence's key photo ranges (same X logic as stacked mode)
  for (let ti = 0; ti < tracks.length; ti++) {
    const track = tracks[ti];
    const isSelected = track.sequenceId === this.selectedContentSequenceId;

    // Selection highlight for the sequence's frame range
    if (isSelected) {
      const selX = track.startFrame * frameWidth - scrollX + TRACK_HEADER_WIDTH;
      const selW = (track.endFrame - track.startFrame) * frameWidth;
      ctx.fillStyle = '#151A20';
      ctx.fillRect(selX, trackY, selW, TRACK_HEIGHT);
    }

    // Draw key photo ranges (reuse existing tile pattern logic)
    for (const range of track.keyPhotoRanges) {
      // ... same thumbnail rendering as stacked mode ...
    }

    // Sequence boundary separator
    if (ti > 0) {
      const sepX = track.startFrame * frameWidth - scrollX + TRACK_HEADER_WIDTH;
      if (sepX >= TRACK_HEADER_WIDTH && sepX <= w) {
        ctx.strokeStyle = '#555555';
        ctx.lineWidth = 2;
        ctx.beginPath();
        ctx.moveTo(sepX, trackY + 1);
        ctx.lineTo(sepX, trackY + TRACK_HEIGHT - 1);
        ctx.stroke();
      }
    }

    // Name overlay (if displayMode is 'thumb-name')
    if (state.displayMode === 'thumb-name') {
      const segX = track.startFrame * frameWidth - scrollX + TRACK_HEADER_WIDTH;
      const segW = (track.endFrame - track.startFrame) * frameWidth;
      // Only draw if segment is visible and wide enough
      if (segX + segW > TRACK_HEADER_WIDTH && segX < w && segW > 20) {
        const labelH = 16;
        const labelY = trackY + TRACK_HEIGHT - 2 - labelH;
        ctx.fillStyle = 'rgba(0, 0, 0, 0.6)';
        ctx.fillRect(Math.max(segX, TRACK_HEADER_WIDTH), labelY, segW, labelH);
        ctx.fillStyle = '#EEEEEE';
        ctx.font = '10px system-ui, sans-serif';
        ctx.textBaseline = 'middle';
        const name = this.truncateText(ctx, track.sequenceName, segW - 8);
        ctx.fillText(name, Math.max(segX, TRACK_HEADER_WIDTH) + 4, labelY + labelH / 2);
      }
    }
  }

  ctx.restore();
}
```

// In TimelineInteraction.ts -- adapt click handling for linear mode
private sequenceFromFrame(frame: number): string | null {
  const tracks = trackLayouts.peek();
  for (const track of tracks) {
    if (frame >= track.startFrame && frame < track.endFrame) {
      return track.sequenceId;
    }
  }
  return null;
}

## Open Questions

1. **Track height in linear mode**
   - What we know: The current `TRACK_HEIGHT` is 52px. In linear mode with a single content row, this same height works fine.
   - What's unclear: Should the single row be taller than 52px to give more vertical space to thumbnails? Or should it stay consistent with stacked mode for visual continuity?
   - Recommendation: Keep at 52px for now. Users can always zoom to see more detail. Consistency is more important than marginal thumbnail improvement.

2. **Keyboard shortcut for toggle**
   - What we know: The toggle is primarily a toolbar UI element.
   - What's unclear: Should there be a keyboard shortcut (e.g., `L` for layout toggle)?
   - Recommendation: Defer keyboard shortcut to a quick-task if users request it. The toggle is not a frequent operation -- it is a preference setting.
