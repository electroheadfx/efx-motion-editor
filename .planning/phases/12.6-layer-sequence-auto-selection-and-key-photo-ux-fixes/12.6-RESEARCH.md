# Phase 12.6: Layer Sequence Auto-Selection and Key Photo UX Fixes - Research

**Researched:** 2026-03-18
**Domain:** Preact signals-based UI state orchestration, multi-step UX flows, DOM scroll sync
**Confidence:** HIGH

## Summary

Phase 12.6 modifies existing Preact signal-driven state management to add auto-selection behaviors and UX flow orchestration across the sidebar (SequenceList, KeyPhotoStrip, LeftPanel), ImportedView, and playback engine. The core challenge is state choreography: chaining multiple signal mutations in the correct order so that UI animations, layer selection, and playhead seeking all fire deterministically.

The codebase already has all the building blocks -- `sequenceStore.createSequence()`, `uiStore.setEditorMode('imported')`, bidirectional layer selection (`layerStore.setSelected` + `uiStore.selectLayer`), `playbackEngine.seekToFrame()`, and `trackLayouts` for frame-position lookups. The work is primarily wiring these together in new sequences and adding a "pending new sequence" transactional pattern for cancel-delete on importer close.

**Primary recommendation:** Implement as 3-4 focused plans: (1) new sequence creation flow with transactional cancel-delete, (2) layer auto-selection on sequence switch + key photo click enhancements, (3) key photo strip auto-scroll sync with timeline, (4) empty sequence "Add photos" prompt.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- Clicking "+ Add" creates the sequence, auto-selects it in the sidebar (key photos section expands with slide animation), then opens the Imported view (`setEditorMode('imported')`)
- Importer opens in **multi-select mode** with a prominent **Confirm button** -- user selects multiple photos, clicks Confirm to add all as key photos
- **Cancel/close behavior:** If the user closes the importer or clicks Cancel without confirming, the newly created sequence is **deleted entirely** (treated as uncommitted)
- This flow only applies to the "+ Add" button -- duplicate and other creation paths do NOT trigger the importer flow
- When switching sequences in the sidebar, **auto-select the top-most layer** (highest index in the layer stack)
- If the sequence has only the base layer, select the base layer
- Auto-selection happens on **every** sequence switch, regardless of layer count
- Auto-selecting the layer also **shows transform overlay handles** on the canvas (full visual feedback, same as manual click)
- Both `layerStore.setSelected()` and `uiStore.selectLayer()` must be called (bidirectional sync pattern from Phase 11-04)
- For **existing** sequences with no key photos, replace passive "No key photos yet" text with a **clickable "Add photos" prompt** that opens the Imported view
- This is a normal importer open -- closing without adding photos does NOT delete the sequence (cancel-delete only applies to brand new sequences from "+ Add")
- Clicking a key photo in the strip **selects the top-most layer** (or base layer if only one exists) -- shows PROPERTIES panel with controls
- Clicking a key photo **auto-seeks the playhead** to that key photo's frame position in the timeline
- **Single highlight style only** -- the key photo under the playhead gets the thick ring; clicking a key photo seeks the playhead there so the same highlight applies
- **Auto-scroll key photo strip** on timeline interactions (scrub, frame navigation, keyframe nav) to keep the active key photo visible -- but **NOT during playback**
- When **playback stops**, snap the strip to show the active key photo

### Pattern 1: Pending Sequence Transactional State
**What:** Track that a newly created sequence is "pending" (not yet committed) via a signal. If the importer is closed without confirming photos, delete the pending sequence.
**When to use:** Only for the "+ Add" button creation flow.
**Example:**
```typescript
// In sequenceStore.ts or uiStore.ts
const pendingNewSequenceId = signal<string | null>(null);

// In LeftPanel.tsx "+ Add" onClick:
const seq = sequenceStore.createSequence(`Sequence ${sequences.length + 1}`);
uiStore.selectSequence(seq.id);
sequenceStore.setActive(seq.id);
pendingNewSequenceId.value = seq.id;
uiStore.setEditorMode('imported');

// In ImportedView.tsx on confirm:
// Add all selected photos, then:
pendingNewSequenceId.value = null;
uiStore.setEditorMode('editor');

// In ImportedView.tsx on close/cancel:
if (pendingNewSequenceId.peek()) {
  sequenceStore.remove(pendingNewSequenceId.peek()!);
  pendingNewSequenceId.value = null;
}
uiStore.setEditorMode('editor');
```

### Pattern 2: Multi-Select Confirm in ImportedView
**What:** When opened in multi-select mode, ImportGrid tracks selected image IDs in local state. A Confirm button adds all selected photos at once.
**When to use:** When pendingNewSequenceId is set (new sequence flow).
**Example:**
```typescript
// In ImportedView.tsx
const [selectedIds, setSelectedIds] = useState<string[]>([]);
const isMultiSelect = !!pendingNewSequenceId.value;

// ImportGrid gets onToggleSelect instead of onSelect
const handleToggleSelect = (imageId: string) => {
  setSelectedIds(prev =>
    prev.includes(imageId)
      ? prev.filter(id => id !== imageId)
      : [...prev, imageId]
  );
};

const handleConfirm = () => {
  const seqId = pendingNewSequenceId.peek() ?? sequenceStore.activeSequenceId.peek();
  if (seqId) {
    for (const imageId of selectedIds) {
      sequenceStore.addKeyPhoto(seqId, imageId);
    }
  }
  pendingNewSequenceId.value = null;
  uiStore.setEditorMode('editor');
};
```

### Pattern 3: Bidirectional Layer Selection (Established Phase 11-04 Pattern)
**What:** Always call both `layerStore.setSelected(id)` and `uiStore.selectLayer(id)` to keep stores in sync. This triggers TransformOverlay to show handles.
**When to use:** Every time a layer is selected -- sequence switch auto-select, key photo click.
**Example:**
```typescript
// Get top-most layer (last in array = highest in visual stack)
const seq = sequenceStore.getById(seqId);
if (seq && seq.layers.length > 0) {
  const topLayer = seq.layers[seq.layers.length - 1];
  layerStore.setSelected(topLayer.id);
  uiStore.selectLayer(topLayer.id);
}
```

### Pattern 4: Key Photo Frame Position Lookup
**What:** Use `trackLayouts` to find the global frame position of a key photo for playhead seeking.
**When to use:** Key photo click-to-seek, strip auto-scroll calculation.
**Example:**
```typescript
// Find the start frame of a key photo
const layouts = trackLayouts.peek();
const track = layouts.find(t => t.sequenceId === sequenceId);
if (track) {
  const range = track.keyPhotoRanges.find(r => r.keyPhotoId === keyPhotoId);
  if (range) {
    playbackEngine.seekToFrame(range.startFrame);
  }
}
```

### Pattern 5: Key Photo Strip Auto-Scroll
**What:** Programmatically scroll the key photo strip container to keep the active key photo visible. Subscribe to frame changes (but not during playback).
**When to use:** On timeline scrub, frame navigation, playback stop.
**Example:**
```typescript
// In KeyPhotoStripInline, subscribe to displayFrame changes
useEffect(() => {
  // Only auto-scroll when NOT playing
  if (timelineStore.isPlaying.peek()) return;

  const frame = timelineStore.displayFrame.value;
  const layouts = trackLayouts.peek();
  const track = layouts.find(t => t.sequenceId === sequenceId);
  if (!track || !stripRef.current) return;

  // Find which key photo index contains this frame
  const kpIndex = track.keyPhotoRanges.findIndex(
    r => frame >= r.startFrame && frame < r.endFrame
  );
  if (kpIndex < 0) return;

  // Scroll to make that thumbnail visible
  const children = stripRef.current.children;
  const child = children[kpIndex] as HTMLElement;
  if (child) {
    child.scrollIntoView({ behavior: 'smooth', block: 'nearest', inline: 'nearest' });
  }
}, [timelineStore.displayFrame.value]);
```

### Anti-Patterns to Avoid
- **Signal subscription in rAF loop:** Never use `.value` inside requestAnimationFrame -- always use `.peek()`. The playback tick already follows this.
- **Circular signal updates:** When auto-selecting a layer triggers a re-render that re-selects the layer -- use `.peek()` guards to break cycles.
- **Undo history pollution:** The cancel-delete flow should ideally remove the sequence AND its undo entry, or batch the create+delete as a no-op. Consider using `sequenceStore.remove()` which pushes its own undo action, accepting that "undo" after cancel would re-create the empty sequence.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Frame position lookup | Manual frame counting | `trackLayouts` computed signal | Already computes `keyPhotoRanges` with start/end frames per key photo |
| Playhead seeking | Direct `timelineStore.seek()` | `playbackEngine.seekToFrame()` | Handles seek + displayFrame sync + ensureFrameVisible + player sync |
| Layer selection | Just `layerStore.setSelected()` | Bidirectional: `layerStore.setSelected()` + `uiStore.selectLayer()` | Missing either breaks TransformOverlay or sidebar highlighting |
| Scroll-into-view | Manual scrollLeft math | `Element.scrollIntoView({ inline: 'nearest' })` | Handles edge detection automatically, supports smooth scrolling |

## Common Pitfalls

### Pitfall 1: Key Photo Strip Not Expanding for New Empty Sequences
**What goes wrong:** The current maxHeight guard is `isActive && seq.keyPhotos.length > 0 ? '72px' : '0px'`. A newly created sequence has 0 key photos, so the strip stays collapsed even though we want it visible.
**Why it happens:** The expand condition was designed for showing existing key photos, not for empty active sequences.
**How to avoid:** Change the condition to `isActive ? '72px' : '0px'` (always show strip area for active sequence), and render the "Add photos" prompt inside the strip area when there are no key photos.
**Warning signs:** Creating a new sequence shows no key photo area.

### Pitfall 2: Undo After Cancel-Delete Creates Ghost Sequence
**What goes wrong:** User clicks "+ Add", cancels the importer, the sequence is deleted. User presses Ctrl+Z -- the undo stack has both "Create sequence" and "Delete sequence" actions, so undoing brings back the empty sequence.
**Why it happens:** `createSequence()` and `remove()` each push their own undo actions.
**How to avoid:** Either (a) batch the create+delete as a single no-op by popping the last two undo actions, or (b) accept this behavior as "expected" (undo restores the cancelled sequence), or (c) use a non-undoable create path for the pending sequence and only push undo when photos are confirmed.
**Warning signs:** Ctrl+Z after cancel produces empty sequence in sidebar.

### Pitfall 3: Race Between Sequence Selection and Layer Auto-Select
**What goes wrong:** `handleSelect` calls `sequenceStore.setActive(seq.id)` which changes `activeSequenceId`, triggering `layerStore.layers` recomputation. If layer auto-selection happens before `layers` recomputes, it reads the previous sequence's layers.
**Why it happens:** Signal batching and subscription ordering.
**How to avoid:** Use `batch()` from `@preact/signals` to group the sequence activation and layer selection, OR read layers directly from the sequence object (`seq.layers`) instead of from the `layerStore.layers` computed.
**Warning signs:** Wrong layer selected after switching sequences, or selection shows stale data.

### Pitfall 4: Auto-Scroll During Playback Causes Jank
**What goes wrong:** If `displayFrame` subscription triggers scroll updates during playback, the strip scrolls rapidly every frame.
**Why it happens:** During playback, `displayFrame` is frozen (gated by the dual-signal pattern from quick-2/quick-24), but `currentFrame` still changes. If subscribing to the wrong signal, or if `syncDisplayFrame` on stop triggers a cascade of scroll updates.
**How to avoid:** Gate auto-scroll on `!timelineStore.isPlaying.peek()`. Subscribe to `displayFrame` (not `currentFrame`) so updates only happen when the display is actually synced (on stop, on manual seek, on step).
**Warning signs:** Strip scrolling rapidly during playback, UI jank.

### Pitfall 5: Multi-Select State Leak Between Importer Opens
**What goes wrong:** User opens importer via "+ Add", selects 3 photos, cancels. Next time they open importer normally (via AddKeyPhotoButton), the 3 selections are still checked.
**Why it happens:** Multi-select state persists across importer open/close.
**How to avoid:** Clear selection state on importer close AND on importer open. Reset `selectedIds` to `[]` in the close/cancel handler and when the component mounts.
**Warning signs:** Stale checkmarks in the import grid.

### Pitfall 6: Key Photo Highlight Desync
**What goes wrong:** Clicking a key photo selects it (thick ring), but the playhead doesn't move, or the playhead moves but the ring doesn't update.
**Why it happens:** Current `KeyPhotoCard` uses `sequenceStore.selectedKeyPhotoId` for the ring highlight, but the new behavior should use playhead position (which key photo the current frame falls within).
**How to avoid:** Change the highlight logic to derive from `displayFrame` + `trackLayouts` (which key photo range contains the current frame) instead of a separate `selectedKeyPhotoId`. Clicking a key photo seeks the playhead, which naturally updates the highlight.
**Warning signs:** Ring highlights on wrong photo, or two photos highlighted simultaneously.

## Code Examples

### Top-Most Layer Selection Helper
```typescript
// Pure function: get the top-most layer ID for a sequence
function getTopLayerId(seq: Sequence): string | null {
  if (!seq || seq.layers.length === 0) return null;
  // Last layer in array = top of visual stack
  return seq.layers[seq.layers.length - 1].id;
}
```

### Enhanced handleSelect with Layer Auto-Selection
```typescript
const handleSelect = useCallback(() => {
  if (editing) return;
  const wasActive = sequenceStore.activeSequenceId.peek();

  // Switch sequence
  uiStore.selectSequence(seq.id);
  sequenceStore.setActive(seq.id);
  sequenceStore.clearKeyPhotoSelection();

  // Auto-select top-most layer (bidirectional sync)
  const topLayerId = getTopLayerId(seq);
  if (topLayerId) {
    layerStore.setSelected(topLayerId);
    uiStore.selectLayer(topLayerId);
  } else {
    layerStore.setSelected(null);
    uiStore.selectLayer(null);
  }

  // Auto-seek timeline playhead to sequence start (only when switching)
  if (seq.id !== wasActive) {
    const layouts = trackLayouts.peek();
    const track = layouts.find(t => t.sequenceId === seq.id);
    if (track) {
      playbackEngine.seekToFrame(track.startFrame);
    }
  }
}, [seq.id, seq, editing]);
```

### Key Photo Click with Seek and Layer Selection
```typescript
// In KeyPhotoCard onClick
const handleKeyPhotoClick = useCallback(() => {
  // Select the key photo
  sequenceStore.selectKeyPhoto(keyPhotoId);

  // Auto-select top-most layer
  const seq = sequenceStore.getById(sequenceId);
  if (seq) {
    const topLayerId = getTopLayerId(seq);
    if (topLayerId) {
      layerStore.setSelected(topLayerId);
      uiStore.selectLayer(topLayerId);
    }
  }

  // Seek playhead to key photo start frame
  const layouts = trackLayouts.peek();
  const track = layouts.find(t => t.sequenceId === sequenceId);
  if (track) {
    const range = track.keyPhotoRanges.find(r => r.keyPhotoId === keyPhotoId);
    if (range) {
      playbackEngine.seekToFrame(range.startFrame);
    }
  }
}, [keyPhotoId, sequenceId]);
```

## Open Questions

1. **Undo behavior for cancel-delete**
   - What we know: `createSequence()` and `remove()` each push undo actions. Cancelling a new sequence will leave two entries in the undo stack.
   - What's unclear: Should we clean up the undo stack to make cancel truly invisible, or is it acceptable that Ctrl+Z after cancel restores the empty sequence?
   - Recommendation: Accept that undo restores the empty sequence. It's a minor edge case and cleaning up the undo stack adds complexity (would need an `undoAction.pop()` or similar). Document this as known behavior.

2. **Multi-select for non-creation importer opens**
   - What we know: The AddKeyPhotoButton on the key photo strip currently opens the importer in single-select mode. The CONTEXT says multi-select + confirm is for the "+ Add" creation flow.
   - What's unclear: Should the existing AddKeyPhotoButton also use multi-select, or stay single-click?
   - Recommendation: Keep AddKeyPhotoButton as single-click-to-add (current behavior). Multi-select confirm mode is only for the "+ Add" new sequence creation flow, as stated in the locked decisions.

3. **Key photo highlight: selectedKeyPhotoId vs. frame-derived**
   - What we know: The CONTEXT says "single highlight style only -- the key photo under the playhead gets the thick ring." This implies deriving highlight from playhead position.
   - What's unclear: Whether `selectedKeyPhotoId` should be repurposed, removed, or kept alongside.
   - Recommendation: Derive the visual highlight from `displayFrame` position within `trackLayouts.keyPhotoRanges`. Keep `selectedKeyPhotoId` for internal bookkeeping (e.g., Delete key target) but do not use it for the ring visual. This unifies the highlight source.
