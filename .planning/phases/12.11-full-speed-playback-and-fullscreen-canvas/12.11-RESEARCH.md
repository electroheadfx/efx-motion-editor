# Phase 12.11: Full-Speed Playback and Fullscreen Canvas - Research

**Researched:** 2026-03-19
**Domain:** Playback performance optimization, Fullscreen API, Tauri v2 window management
**Confidence:** HIGH

## Summary

This phase adds two independent but overlapping features: (1) a full-speed playback mode triggered by Shift+Space that skips all UI feedback (timeline scroll, playhead, timecode, sidebar) and only renders canvas frames, and (2) a fullscreen canvas mode that hides all editor chrome and presents the canvas filling the screen with auto-hiding controls.

The codebase is well-prepared for both features. The PlaybackEngine tick loop has clear, identified lines where UI sync calls can be conditionally gated behind a `fullSpeed` signal. The SpeedBadge component provides an exact pattern for the full-speed indicator badge. EditorShell's mode-switching architecture supports rendering a fullscreen overlay above all chrome. The main complexity is the Tauri-specific fullscreen bridging -- the standard Fullscreen API (`requestFullscreen`) does not properly hide the title bar in Tauri v2 WebViews, requiring a bridge pattern that combines the DOM Fullscreen API with Tauri's `getCurrentWindow().setFullscreen()`.

**Primary recommendation:** Implement full-speed as a signal-gated conditional in PlaybackEngine.tick(), and fullscreen as a portal-style overlay component in EditorShell that bridges DOM fullscreen events to Tauri's window API.

### Pattern 1: Signal-Gated UI Feedback in PlaybackEngine.tick()

**What:** Add a `fullSpeed` signal to PlaybackEngine. When true, the tick() method skips ALL UI sync calls (ensureFrameVisiblePaged, ensureTrackVisible) but still advances frames and calls syncPlayer(). On stop(), the existing sync logic (syncDisplayFrame, syncActiveSequence, resetShuttle) already handles re-sync.

**When to use:** During full-speed playback (Shift+Space) or any playback in fullscreen mode.

**Example:**
```typescript
// In playbackEngine.ts
import {signal} from '@preact/signals';

export const isFullSpeed = signal(false);

// Inside PlaybackEngine class:
private tick = (now: number) => {
  // ... frame advancing logic unchanged ...

  // Gate UI feedback behind fullSpeed signal
  if (!isFullSpeed.peek()) {
    timelineStore.ensureFrameVisiblePaged(timelineStore.currentFrame.peek());
    const cf = timelineStore.currentFrame.peek();
    const entry = frameMap.peek()[cf];
    if (entry) {
      timelineStore.ensureTrackVisible(entry.sequenceId);
    }
  }

  this.syncPlayer(); // Always sync canvas preview
  // ...
};

toggleFullSpeed() {
  if (timelineStore.isPlaying.peek()) {
    isFullSpeed.value = !isFullSpeed.value;
  } else {
    isFullSpeed.value = true;
    this.start();
  }
}

// In stop():
stop() {
  // ... existing stop logic ...
  isFullSpeed.value = false; // Always clear on stop
}
```

**Key insight:** The displayFrame signal already gates sidebar re-renders during playback (quick-24). Full-speed mode takes this further by also gating timeline horizontal scroll and vertical track scroll. The currentFrame signal still advances (so the canvas preview keeps rendering), and syncDisplayFrame() on stop brings everything back in sync.

### Pattern 2: Tauri Fullscreen Bridge

**What:** The standard DOM Fullscreen API (`document.documentElement.requestFullscreen()`) does not properly hide the Tauri window title bar. The solution is to bridge DOM fullscreen events with Tauri's window API.

**When to use:** Any time the app enters/exits fullscreen.

**Example:**
```typescript
// In fullscreenManager.ts
import {getCurrentWindow} from '@tauri-apps/api/window';
import {signal} from '@preact/signals';

export const isFullscreen = signal(false);

export async function enterFullscreen(): Promise<void> {
  try {
    await document.documentElement.requestFullscreen();
    await getCurrentWindow().setFullscreen(true);
    isFullscreen.value = true;
  } catch (err) {
    console.error('Failed to enter fullscreen:', err);
  }
}

export async function exitFullscreen(): Promise<void> {
  if (document.fullscreenElement) {
    await document.exitFullscreen();
  }
  await getCurrentWindow().setFullscreen(false);
  isFullscreen.value = false;
}

// Listen for external fullscreen exit (e.g., ESC handled by browser)
document.addEventListener('fullscreenchange', () => {
  if (!document.fullscreenElement && isFullscreen.peek()) {
    getCurrentWindow().setFullscreen(false);
    isFullscreen.value = false;
  }
});
```

**Critical caveat:** The `core:window:allow-set-fullscreen` permission is NOT included in `core:window:default`. It must be explicitly added to `capabilities/default.json`.

### Pattern 3: Auto-Hiding Controls Overlay

**What:** A controls bar that appears on mouse movement and fades out after ~2 seconds of inactivity, using a timer-based visibility pattern.

**When to use:** In fullscreen mode only.

**Example:**
```typescript
// In FullscreenOverlay.tsx
const controlsVisible = useSignal(true);
const hideTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);

const showControls = useCallback(() => {
  controlsVisible.value = true;
  if (hideTimerRef.current) clearTimeout(hideTimerRef.current);
  hideTimerRef.current = setTimeout(() => {
    controlsVisible.value = false;
  }, 2000);
}, []);

// On mouse move over the fullscreen container:
onMouseMove={showControls}
```

### Pattern 4: Fullscreen Shortcut Scoping

**What:** In fullscreen mode, only a subset of shortcuts should be active (Space, Left/Right arrows, Shift+Space, ESC). All other shortcuts (file ops, editing, zoom, etc.) must be suppressed.

**When to use:** When isFullscreen is true.

**Implementation approach:** Add `isFullscreen.peek()` guard at the top of tinykeys handlers that should be suppressed in fullscreen. For ESC specifically, the fullscreen handler should take priority over the existing ESC deselect behavior. The Space keydown/keyup handler in CanvasArea already handles play/pause -- just need to add Shift+Space branch for full-speed toggle.

```typescript
// In shortcuts.ts handlers that should be suppressed:
'$mod+KeyS': (e: KeyboardEvent) => {
  if (shouldSuppressShortcut(e)) return;
  if (isFullscreen.peek()) return; // Suppress in fullscreen
  e.preventDefault();
  handleSave();
},

// ESC gets fullscreen exit priority:
'Escape': (e: KeyboardEvent) => {
  if (shouldSuppressShortcut(e)) return;
  if (isFullscreen.peek()) {
    e.preventDefault();
    exitFullscreen();
    return;
  }
  // Normal deselect behavior
  e.preventDefault();
  layerStore.setSelected(null);
  uiStore.selectLayer(null);
},
```

### Pattern 5: Fullscreen Canvas Scaling (Letterbox)

**What:** In fullscreen mode, the canvas should fill the screen while maintaining aspect ratio (letterbox). Black bars appear on sides where needed.

**Implementation:** Use the existing `canvasStore.fitToWindow()` pattern but with screen dimensions instead of container dimensions. The FullscreenOverlay has a ResizeObserver on its container (which is window.innerWidth x window.innerHeight in fullscreen) and calls `fitToWindow()` on mount.

**Recommendation:** Letterbox (fit, not fill) is the correct cinema-preview approach. It preserves the full frame without cropping. The background should be black (#000000) for a cinema feel.

### Anti-Patterns to Avoid

- **Do NOT add a new EditorMode for fullscreen:** Fullscreen is an overlay on top of the editor, not a replacement mode. The editor state (CanvasArea, Timeline, etc.) must remain mounted beneath the overlay so that stop() re-sync works correctly. Using `editorMode` would unmount the timeline and break state.
- **Do NOT duplicate the Preview component for fullscreen:** Reuse the same Preview canvas. The FullscreenOverlay should contain a canvas that mirrors or reuses the existing preview, or better yet, the overlay positions itself to show only the canvas (the existing Preview continues rendering to its canvas, and the fullscreen overlay simply repositions/rescales it visually).
- **Do NOT use CSS-only fullscreen without Tauri bridge:** The title bar will remain visible, creating a broken experience.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Fullscreen window management | Custom window resizing/repositioning | Fullscreen API + Tauri `setFullscreen()` bridge | OS-level fullscreen handles taskbar, menu bar, multi-monitor correctly |
| Auto-hide timer | Manual setInterval with visibility tracking | setTimeout with reset on mousemove | Standard debounce pattern, simpler and no cleanup issues |
| Canvas aspect-ratio scaling | Manual dimension math | Existing `canvasStore.fitToWindow()` with updated container size | Already handles all the edge cases |

## Common Pitfalls

### Pitfall 1: Tauri Fullscreen Title Bar Leak
**What goes wrong:** Using `document.documentElement.requestFullscreen()` alone leaves the Tauri title bar visible.
**Why it happens:** Tauri's WebView does not automatically bridge DOM fullscreen events to native window fullscreen.
**How to avoid:** Always pair DOM `requestFullscreen()` with `getCurrentWindow().setFullscreen(true)` via Tauri's API. Listen for `fullscreenchange` to handle external exits (browser ESC).
**Warning signs:** Title bar visible in fullscreen mode during testing.

### Pitfall 2: Missing Tauri Capability Permission
**What goes wrong:** `getCurrentWindow().setFullscreen(true)` silently fails or throws.
**Why it happens:** `core:window:default` does NOT include `allow-set-fullscreen`. Only `allow-is-fullscreen` (read) is included by default.
**How to avoid:** Add `"core:window:allow-set-fullscreen"` to `capabilities/default.json` before implementing.
**Warning signs:** Fullscreen toggle does nothing; no error in console (Tauri permission errors can be silent).

### Pitfall 3: ESC Key Conflict
**What goes wrong:** Pressing ESC in fullscreen deselects layers instead of exiting fullscreen, or exits fullscreen but also deselects.
**Why it happens:** The existing ESC handler in `shortcuts.ts` deselects layers. tinykeys and the browser's built-in fullscreen ESC handling can conflict.
**How to avoid:** Check `isFullscreen` first in the ESC handler and return early after triggering exit. The browser may also auto-exit fullscreen on ESC (via the Fullscreen API), so listen for `fullscreenchange` to detect this and clean up state.
**Warning signs:** Double-action on ESC press; fullscreen exits but layer also deselects.

### Pitfall 4: Shift+Space Conflict with Space Handler
**What goes wrong:** Shift+Space triggers both full-speed toggle AND play/pause toggle.
**Why it happens:** Space is handled in CanvasArea via keydown/keyup events (not tinykeys) for Space+drag support. The keyup handler checks `e.code === 'Space'` without checking modifier keys.
**How to avoid:** In the CanvasArea keyup handler, check `e.shiftKey` before toggling play/pause. If Shift is held, call `playbackEngine.toggleFullSpeed()` instead of `playbackEngine.toggle()`.
**Warning signs:** Full-speed activates but playback immediately toggles off.

### Pitfall 5: Fullscreen Overlay Blocks Canvas Events
**What goes wrong:** The fullscreen overlay renders on top of the existing CanvasArea, blocking pointer events for the Preview component.
**Why it happens:** The overlay has its own event handlers and z-index above everything.
**How to avoid:** The fullscreen overlay should contain its OWN reference to the Preview canvas or reposition the existing one. Since Preview renders to an HTML canvas via rAF, the simplest approach is to have the FullscreenOverlay host its own canvas element that shares the same PreviewRenderer, or simply layer the overlay to show the existing canvas through a "window" (CSS positioning).
**Warning signs:** Canvas shows black or stale frame in fullscreen.

### Pitfall 6: Stop Not Re-syncing After Fullscreen Exit
**What goes wrong:** After exiting fullscreen, the timeline playhead, timecode, and sidebar are out of sync.
**Why it happens:** Full-speed mode skips all UI sync during playback, and the exit-fullscreen handler might not call stop().
**How to avoid:** `exitFullscreen()` must call `playbackEngine.stop()` if playing. The existing `stop()` already calls `syncDisplayFrame()`, `syncActiveSequence()`, and `resetShuttle()`, which handles everything.
**Warning signs:** After fullscreen exit, playhead is at frame 0 but timecode shows different value.

### Shift+Space in CanvasArea keyup handler

```typescript
// Modified onKeyUp in CanvasArea.tsx useEffect:
const onKeyUp = (e: KeyboardEvent) => {
  if (e.code === 'Space') {
    const wasHeld = isSpaceHeld.current;
    isSpaceHeld.current = false;
    if (!wasHeld) return;

    const target = e.target as HTMLElement;
    if (target.tagName === 'INPUT' || target.tagName === 'TEXTAREA' || target.isContentEditable) return;

    if (!spaceDragOccurred.current) {
      if (e.shiftKey) {
        // Shift+Space: toggle full-speed playback
        playbackEngine.toggleFullSpeed();
      } else {
        // Plain Space: toggle play/pause
        playbackEngine.toggle();
      }
    }
    spaceDragOccurred.current = false;
  }
};
```

## Open Questions

1. **Canvas reuse in fullscreen**
   - What we know: The Preview component renders to a canvas via a rAF loop. It mounts inside CanvasArea.
   - What's unclear: Whether to reuse the same canvas element (reparent it into the fullscreen overlay) or create a second canvas that shares the same render output.
   - Recommendation: The simplest approach is to NOT duplicate the canvas. Instead, have the FullscreenOverlay be a fixed-position overlay that visually covers everything, with the existing CanvasArea continuing to render beneath it. The overlay simply shows/hides editor chrome. The CanvasArea's canvas remains in the DOM and continues its rAF loop. The overlay positions itself with black background and centers the canvas area visually using CSS. This avoids any renderer duplication.

2. **Fullscreen on secondary monitors**
   - What we know: requestFullscreen() can accept a `screen` option for multi-monitor.
   - What's unclear: How Tauri's `setFullscreen(true)` behaves with multi-monitor setups.
   - Recommendation: Use default behavior (fullscreen on current monitor). Multi-monitor is out of scope for this phase.
