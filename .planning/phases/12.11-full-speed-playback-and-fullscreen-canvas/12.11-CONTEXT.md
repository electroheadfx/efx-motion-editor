# Phase 12.11: Full-Speed Playback and Fullscreen Canvas - Context

**Gathered:** 2026-03-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Two related features: (1) a full-speed playback mode that skips all UI feedback for maximum frame rate, and (2) a fullscreen canvas mode that hides all editor chrome and shows the preview filling the screen with auto-hiding controls. Fullscreen always uses full-speed playback. Both features also work independently.

</domain>

<decisions>
## Implementation Decisions

### Full-speed playback
- Skip ALL UI feedback during full-speed: no timeline horizontal/vertical scroll, no playhead movement, no timecode update, no speed badge — only the canvas preview renders frames
- Triggered via Shift+Space shortcut (no button needed — power-user feature)
- Show a subtle badge on the canvas (like existing SpeedBadge) indicating full-speed mode is active (e.g. ⚡ FULL SPEED)
- On stop: sync everything — timeline playhead jumps to current frame, timecode updates, sidebar re-syncs (same as current stop() behavior)
- Works independently in normal editor view (without fullscreen)

### Fullscreen canvas mode
- Enter via Cmd+Shift+F keyboard shortcut AND a small expand icon button in the canvas controls bar
- Exit via ESC key
- Uses Fullscreen API (document.documentElement.requestFullscreen), not native macOS fullscreen
- Canvas fills the screen with auto-hiding controls bar: play/pause, step forward/backward, timecode display — fades in on mouse move, auto-hides after ~2 seconds of inactivity
- Active shortcuts in fullscreen: Space (play/pause), Left/Right arrows (step frame), Shift+Space (explicit full-speed toggle — though fullscreen always uses full-speed), ESC (exit). No file ops, no layer editing, no zoom shortcuts.

### Interaction between modes
- Full-speed and fullscreen are independent features that overlap: Shift+Space works in normal editor, Cmd+Shift+F enters fullscreen
- Fullscreen always activates full-speed playback mode (playback in fullscreen never updates timeline/sidebar)
- Entering fullscreen does NOT auto-start playback — shows current frame, user presses Space to begin
- Exiting fullscreen (ESC) stops playback if running — clean return to normal editor with everything synced

### Claude's Discretion
- Badge styling and positioning for full-speed indicator
- Auto-hide timing for fullscreen controls (suggested ~2 seconds)
- Fullscreen background color (likely black to match cinema feel)
- Canvas scaling strategy in fullscreen (fit vs fill vs letterbox)
- Whether to show zoom controls in fullscreen controls bar

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `SpeedBadge` component: existing overlay badge for JKL shuttle speed — same pattern can be used for full-speed indicator
- `playbackEngine.start()/stop()/toggle()`: core playback lifecycle — full-speed mode adds a conditional branch to skip UI sync calls in tick()
- `timelineStore.syncDisplayFrame()`: called on stop — reusable for re-syncing after full-speed playback ends
- `canvasStore.fitToWindow()`: can be reused for fullscreen canvas scaling

### Established Patterns
- Dual-signal pattern (currentFrame vs displayFrame): sidebar already frozen during playback via displayFrame gating
- `uiStore.editorMode` string union: controls which view renders in the right area (editor/imported/settings) — could add a fullscreen mode or use separate signal
- `shouldSuppressShortcut()`: existing guard for form elements — fullscreen mode needs its own shortcut scope
- Space key handled in CanvasArea via keydown/keyup events (not tinykeys) for Space+drag pan support

### Integration Points
- PlaybackEngine.tick(): the main place to gate UI feedback calls behind a full-speed signal
- CanvasArea controls bar: where the fullscreen button goes
- EditorShell: where fullscreen overlay component would mount (above all other chrome)
- CanvasArea Space keyup handler: needs Shift+Space branch for full-speed toggle

</code_context>
