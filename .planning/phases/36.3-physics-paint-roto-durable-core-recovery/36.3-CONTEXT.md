# Phase 36.3: Physics Paint Roto Durable Core Recovery - Context

**Gathered:** 2026-06-19T00:00:00.000Z
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 36.3 is the first MVP/TDD recovery slice after the failed/superseded Phase 36.2 work. It must prove one narrow durable-cache walking skeleton: a user opens Physics Paint in Roto mode for a physics-paint layer, paints one Roto frame, explicitly clicks **Save current**, the parent EFX Motion editor receives the rendered PNG for the correct layer/frame, the EFX Motion preview draws that cached PNG, project save/load preserves it, and reopening Physics Paint shows the cached PNG as a visual reference.

This phase is intentionally not a continuation of broad Phase 36.2 gap closure. It must not implement save-on-leave, key utilities, interpolation, generated frames, cached Roto playback, missing-background behavior, Play/Roto conversion, final Pencil timeline UI, or Tauri asset protocol hardening. The phase is complete only after the durable-cache skeleton and unsaved-discard semantics pass behavior-oriented automated proof plus user-run manual UAT.

</domain>

<decisions>
## Implementation Decisions

### Narrow UI Surface
- **D-01:** Phase 36.3 must use a strict minimal Roto UI surface. Hide or do not render excluded 36.2-era controls: interpolation, key utilities, cached playback, Play/Roto conversion, generated frames, missing backgrounds, and final timeline UI.
- **D-02:** The visible UI should support only the durable-cache walking skeleton: open Roto, paint one frame, **Save current**, show saved/reference status, close, and reopen.
- **D-03:** Use the existing Physics Paint workflow strip only in a minimal mode: current frame cell, **Save current** button, and compact saved/reference status. Do not implement the final Pencil timeline UI and do not create a separate temporary UI pattern.
- **D-04:** Gate broad 36.2-era controls out of the UI only. Do not delete/refactor broad 36.2-era internals in Phase 36.3 unless absolutely required for the durable-cache slice. Deeper cleanup belongs to later recovery phases.

### Save Current UX
- **D-05:** The primary Roto save action is labeled **Save current**. It means only the current Roto frame is saved into the parent EFX Motion cache.
- **D-06:** Do not introduce **Save pending**, **Save all**, save-on-leave, or auto-save language in Phase 36.3.
- **D-07:** After **Save current** succeeds, keep the user on the same frame. Do not auto-advance in this MVP slice.
- **D-08:** After save, show clear saved/reference status so the user can visually confirm the cached result.
- **D-09:** Reopened cached frames are reference-only. On reopen, show the cached PNG as a visual reference/background with explicit copy such as **Cached reference**.
- **D-10:** Do not imply editable stroke persistence and do not implement per-frame editable restore in Phase 36.3. Cached PNG is the durable truth; editable strokes are not guaranteed after reopen.

### Close Unsaved Behavior
- **D-11:** Closing Physics Paint in Phase 36.3 must never auto-save and must never block. Unsaved edits after the last **Save current** are lost.
- **D-12:** Previously saved cached frames must remain intact when closing with later unsaved edits.
- **D-13:** Show compact unsaved status copy such as **Unsaved changes — click Save current to cache**. Closing still discards unsaved edits without prompt, auto-save, or blocking.
- **D-14:** Reopening after discard must show only the last successfully saved cached PNG as a reference. Unsaved edits made after that save and discarded by close must not appear.
- **D-15:** Close prompt / save-on-close behavior belongs to a later MVP slice after the explicit **Save current** durable core works.

### Proof Boundary
- **D-16:** The primary automated proof must be one vertical RED test. It must fail before implementation and then prove the full durable-cache skeleton: rendered Roto payload reaches the parent store/cache, preview can read it, project save/load preserves it, and reopen launch context exposes it as cached reference.
- **D-17:** Focused behavior tests are allowed only as secondary support. Source-string tests must not be primary evidence.
- **D-18:** Require a separate RED test commit before implementation starts. The RED commit must touch behavior tests and record that the test failed before the GREEN implementation commit.
- **D-19:** Manual UAT must prove the full walking skeleton plus discard semantics: open Roto, paint one frame, **Save current**, preview shows cache, project save/load preserves it, reopen shows **Cached reference**, then make unsaved edits after the save, close without saving, and confirm reopen shows only the last saved cache reference.
- **D-20:** The user runs the server/app for manual UAT. Agents must not start the server/app.

### Claude's Discretion
Planner/researcher may choose exact prop names, helper boundaries, test file placement, minimal status copy details, and whether existing broad internals need tiny compatibility adjustments, but must preserve the strict UI gate, explicit **Save current** semantics, no auto-save/auto-advance, reference-only reopen, separate RED commit, and user-run UAT gate above.

</decisions>

## Existing Code Insights

### Reusable Assets
- `app/src/components/physic-paint/PhysicsPaintStudio.tsx` — Owns Roto/Play standalone behavior, `saveRotoFrame`, `saveRotoFrameAndAdvance`, `flushRotoFrame`, cached Roto reference display, dirty Roto tracking, launch-context updates, and apply payload dispatch. Phase 36.3 should narrow the explicit **Save current** path and avoid save-on-leave/auto-advance behavior.
- `app/src/components/physic-paint/PhysicsPaintWorkflowStrip.tsx` — Owns the current Roto/Play workflow strip. Current main includes broad 36.2-era controls for interpolation, key utilities, and cached playback; Phase 36.3 should gate those out and expose only minimal current-frame cell, **Save current**, and compact saved/reference status.
- `app/src/stores/physicPaintStore.ts` — Stores rendered PNG frames in `_frames`, Roto cache metadata in `_rotoCacheMetadata`, MCE serialization via `toMceOutputs`, hydration via `loadFromMceOutputs`, and explicit `physicPaintVersion` invalidation. `applyCanvas` and `upsertRealRotoKeyFrame` are key paths for one-frame durable cache.
- `app/src/types/physicPaint.ts` — Defines launch/apply payloads, `PhysicPaintRenderedFrame`, `PhysicPaintRotoCacheFrame`, workflow metadata, and validation. Use existing rendered-output contracts; do not transport engine internals.
- `app/src/lib/physicPaintBridge.ts` — Creates `PhysicPaintLaunchContext`, sends cached Roto frames into standalone reopen context, and applies Physics Paint payloads back to the parent store.
- `app/src/lib/previewRenderer.ts` and `app/src/components/Preview.tsx` — Preview renderer reads `physicPaintStore.getFrame(...)` for `physic-paint` layers and subscribes to `physicPaintVersion`; this is the parent preview proof path.
- `app/src/stores/projectStore.ts` and `app/src/lib/physicPaintPersistence.ts` — Project save/load includes `physic_paint_outputs`; PNG data URLs are persisted as cache files and hydrated back into runtime outputs.
- `app/src/components/sidebar/PhysicPaintProperties.tsx` — Existing editor-side open-canvas entry point from the selected physics-paint layer/current frame.

### Established Patterns
- Physics paint is additive and must not replace perfect-freehand Basic paint or p5.brush FX paint.
- The rejected headless/batch adapter/editor-driven replay direction must not return. Preserve interactive standalone behavior and rendered-frame/cache output.
- Rendered PNG output crosses the app/standalone bridge; editable engine internals should not become the durable truth.
- Non-reactive Map storage must bump explicit version signals (`physicPaintVersion`) and mark the project dirty on mutations.
- Current main still contains broad 36.2-era Roto hooks. Phase 36.3 should narrow visibility and proof to the walking skeleton instead of trying to clean up or finish every broad hook.
- Use pnpm and existing test setup. Do not add one-off test config hacks.

### Integration Points
- The explicit **Save current** action should reuse the existing apply-canvas/rendered-frame path where possible, but it must stay on the same frame and must not use save-and-advance behavior in this phase.
- Parent preview proof should use existing `physicPaintStore.getFrame(layerId, appFrame)` and `physicPaintVersion` subscriptions rather than a new rendering channel.
- Project persistence proof should exercise `physicPaintStore.toMceOutputs()`, `projectStore` save/load paths or equivalent behavior-level helpers, and `physicPaintStore.loadFromMceOutputs(...)`.
- Reopen proof should exercise `createPhysicPaintLaunchContext(...)` / bridge launch context so the cached PNG appears in `cachedRotoFrames` and standalone can label it **Cached reference**.

<specifics>
## Specific Ideas

- The walking skeleton is: `open Roto → paint one frame → Save current → parent preview shows cached PNG → project save/load keeps it → reopen shows Cached reference`.
- UI copy examples: **Save current**, **Cached reference**, **Unsaved changes — click Save current to cache**.
- The existing workflow strip may be reused only as a minimal current-frame/saved-reference surface; do not create a separate temporary marker UI.
- Cached-only/reopened frame means visual reference/background only. If the user paints after reopen, they are repainting from a clean editable state over/with the reference, not editing restored strokes.
- Manual UAT must include unsaved-discard proof: after a successful save, make unsaved edits, close without saving, reopen, and verify only the last saved cached reference appears.

</specifics>
