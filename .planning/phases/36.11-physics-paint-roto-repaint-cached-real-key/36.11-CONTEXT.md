# Phase 36.11: Physics Paint Roto Repaint Cached Real Key - Context

**Gathered:** 2026-06-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 36.11 delivers additive raster repaint for existing cached real Physics Paint Roto keys. When an animator reopens a real cached Roto key, the previously saved alpha paint result must appear as a normal-looking non-editable base under newly painted live strokes. On save, the real-key cache is replaced by an alpha-only merge of the previous cached paint alpha plus newly rendered live paint alpha.

This phase is not old stroke restoration. The cached base is flattened raster paint, not editable stroke/script state. It must not add destructive raster-base erase, generated interpolation, final Pencil timeline UI, Play/Roto conversion, XState/state-machine work, or full per-frame editable script persistence.

</domain>

<decisions>
## Implementation Decisions

### Base Display
- **D-01:** Reopened cached real Roto keys must show the existing cached alpha at full strength as normal paint, not as a tinted or special cached-reference overlay.
- **D-02:** The cached alpha base remains outside the live physics-paint engine's editable stroke state. It is a visual paint underlay while editing and is merged with live paint only on save.
- **D-03:** Edit-time preview should match the final save concept: paper/background preview + previous cached paint alpha + live engine strokes. This is required for confident live UAT of the repaint behavior.
- **D-04:** Opening a cached real key without new live paint keeps the frame classified as a clean cached real key. Reopening alone must not imply editable stroke restoration or mark the frame dirty.

### Erase and Clear Rules
- **D-05:** Erase remains available only for newly painted live strokes. It must not erase, mask, mutate, or corrupt the previous cached alpha base.
- **D-06:** Clear on a reopened cached key removes only unsaved live repaint strokes and leaves the cached base/real key intact. Removing the whole real key remains the explicit Delete key utility behavior from Phase 36.7.
- **D-07:** Undo/Redo during a cached-key repaint session covers only new live actions since reopen. The old cached base has no editable undo history in this phase.
- **D-08:** Destructive removal or replacement of old cached pixels is explicitly out of scope for Phase 36.11. Do not add masking/compositing erase over the cached base in this phase.

### No-Change Saves
- **D-09:** If a cached real key is reopened and the animator navigates away without adding live paint, preserve the existing cached key with no save/rewrite.
- **D-10:** If the animator explicitly presses Save current without adding new paint, preserve the existing cache and show compact feedback that there was no new paint to save.
- **D-11:** A cached-key repaint session becomes dirty only on the first live edit: adding live paint, live erase, or clearing live repaint content. Loading/displaying the cached base is not dirty.
- **D-12:** If the animator adds live paint and then Undo/Clear returns the live overlay to empty while the cached base is unchanged, the frame returns to clean state and save-on-leave should no-op preserve.

### Status Feedback
- **D-13:** Reuse the existing Physics Paint workflow strip/status area for cached-base feedback. Do not add a modal, toast, new persistent control, or timeline redesign.
- **D-14:** When a cached base opens, use compact copy such as “Cached key base loaded — add paint to update.” Exact wording can vary, but it must explain that the base is visible and non-editable.
- **D-15:** Successful repaint save feedback should mention the merge, such as “Merged new paint into frame N,” so UAT can distinguish additive repaint from the existing generic save path.
- **D-16:** If save/merge fails after live repaint, keep the cached base visible and preserve live dirty edits in memory. Show retry-oriented compact copy such as “Could not merge frame N — edits are still open.” This follows Phase 36.6 failed-save semantics.
- **D-17:** Phase 36.11 should introduce no new UI surfaces. Final timeline layout/visual polish remains Phase 36.13.

### Claude's Discretion
No user decisions were delegated to Claude. Exact status copy may be refined by downstream planning if it preserves the locked behavior and remains compact in the existing workflow/status surface.

</decisions>

## Existing Code Insights

### Reusable Assets
- `app/src/components/physic-paint/PhysicsPaintStudio.tsx` — Main repaint integration seam. Current Roto save output uses `buildRotoOutputFrame(engine, frame, canvasWidth, canvasHeight)`, while cached real keys currently load through `loadCachedRotoReferenceFrame`. Phase 36.11 likely needs to turn that reference into a normal visual underlay and merge it with live transparent stroke output on save.
- `app/src/components/physic-paint/physicsPaintRotoSession.ts` — Existing compact Roto session boundary for current frame, dirty state, real keys, cached/generated/background-only frame numbers, save continuations, restore intent, and effects. Extend this boundary incrementally; do not reintroduce scattered broad Roto orchestration in `PhysicsPaintStudio`.
- `app/src/stores/physicPaintStore.ts` — Owns real Roto key cache metadata and `upsertRealRotoKeyFrame`. The saved merged result must remain `source: 'real-key'` alpha paint data and must not be marked background-only support.
- `app/src/lib/previewRenderer.ts` — Already composes real-key paper/background separately before drawing cached paint alpha. Preserve this separation; do not bake paper/background into the saved real-key alpha cache.
- `app/src/lib/rotoFrameDraw.ts` — Shared missing-frame background resolver/draw helper from Phase 36.10. Phase 36.11 should not bypass the established background/paper composition rules.
- `app/src/types/physicPaint.ts` — Typed payload/cache contracts for `PhysicPaintRotoCacheFrame`, `PhysicPaintApplyCanvasPayload`, and Roto background metadata. Use these types to keep the merged real-key output alpha-only.

### Established Patterns
- Cached Roto PNG is durable visual truth, not restored editable stroke state.
- Real Roto keys are the only editable authoritative frames. Generated/background-only support frames remain render support and must not become editable sources.
- Roto cache mutations must update launch context/cache metadata, timeline/cell state, preview state, and `physicPaintVersion`/visual invalidation through existing store paths.
- Failed save-on-leave/save-before-action preserves dirty in-memory state and cancels the queued navigation/action until retry.
- Preact/Signals boundary work should stay small and state-machine-shaped, following Phase 36.8, without adding XState or broad new `useEffect` orchestration.
- UI feedback should stay in the existing Physics Paint workflow strip/status surfaces unless a source spec explicitly requires otherwise.

### Integration Points
- Repaint base loading connects where `navigateToSyncedFrame`, `openSyncedRotoFrameAfterSave`, `restoreRotoFrameFromSessionEffect`, and `loadCachedRotoReferenceFrame` currently restore cached real keys.
- Alpha merge connects where `snapshotCurrentRotoFrame` and `flushRotoFrame` currently capture `exportTransparentStrokeCanvas(engine)` as the Roto output frame.
- Dirty/no-change behavior connects to `beginRotoFrameEdit`, `markCurrentRotoFrameDirty`, `snapshotCurrentRotoFrame`, `dirtyRotoFramesRef`, and the Roto session `markDirty`/save continuation flow.
- Clear/erase behavior connects to the existing engine `clear`, undo/redo, erase tool, and `clearActiveSource` behavior; Phase 36.11 must keep these live-overlay-only when a cached base exists.
- Automated proof should cover helper-level alpha merge, Studio/session dirty transitions, no-change save/no-op behavior, failed-merge preservation, and PreviewRenderer/export alpha/background separation.

<specifics>
## Specific Ideas

- Think of the edit stack as: `paper/background preview + previous cached paint alpha + live engine strokes`.
- Think of save as: `new real-key paint alpha = previous cached paint alpha + newly rendered live paint alpha`.
- The cached base should look like real paint to the animator, but remain technically immutable/non-editable during this phase.
- No-change paths are important: open-only and save-without-new-paint must preserve the existing cache without rewriting it.
- Existing key Delete remains the destructive path for removing a real key. Do not overload Clear or Erase to delete old cached pixels.

</specifics>

<deferred>
## Deferred Ideas

- Destructive raster erase/masking over the cached base belongs in a future phase with explicit masking/compositing rules. It is not part of Phase 36.11.

</deferred>
