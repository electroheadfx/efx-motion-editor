## User Constraints (from CONTEXT.md)

### Locked Decisions
## Implementation Decisions

### Base Display
- **D-01:** Reopened cached real Roto keys must show the existing cached alpha at full strength as normal paint, not as a tinted or special cached-reference overlay.
- **D-02:** The cached alpha base remains outside the live physics-paint engine's editable stroke state. It is a visual paint underlay while editing and is merged with live paint only on save.
- **D-03:** Edit-time preview should match the final save concept: paper/background preview + previous cached paint alpha + live engine strokes. This is required for confident live UAT of the repaint behavior.
- **D-04:** Opening a cached real key without new live paint keeps the frame classified as a clean cached real key. Reopening alone must not imply editable stroke restoration or mark the frame dirty.

### Erase and Clear Rules
- **D-05:** Erase remains available only for newly painted live strokes. It must not erase, mask, mutate, or corrupt the previous cached alpha base.
- **D-06:** Clear on a reopened cached key removes only unsaved live repaint strokes and leaves the cached base/real key intact. Removing the whole real key remains the explicit Delete key utility behavior from Phase 36.7.
- **D-07:** Undo/Redo during a cached-key repaint session covers only new live actions since reopen. The old cached base has no editable undo history in this phase.
- **D-08:** Destructive removal or replacement of old cached pixels is explicitly out of scope for Phase 36.11. Do not add masking/compositing erase over the cached base in this phase.

### No-Change Saves
- **D-09:** If a cached real key is reopened and the animator navigates away without adding live paint, preserve the existing cached key with no save/rewrite.
- **D-10:** If the animator explicitly presses Save current without adding new paint, preserve the existing cache and show compact feedback that there was no new paint to save.
- **D-11:** A cached-key repaint session becomes dirty only on the first live edit: adding live paint, live erase, or clearing live repaint content. Loading/displaying the cached base is not dirty.
- **D-12:** If the animator adds live paint and then Undo/Clear returns the live overlay to empty while the cached base is unchanged, the frame returns to clean state and save-on-leave should no-op preserve.

### Status Feedback
- **D-13:** Reuse the existing Physics Paint workflow strip/status area for cached-base feedback. Do not add a modal, toast, new persistent control, or timeline redesign.
- **D-14:** When a cached base opens, use compact copy such as “Cached key base loaded — add paint to update.” Exact wording can vary, but it must explain that the base is visible and non-editable.
- **D-15:** Successful repaint save feedback should mention the merge, such as “Merged new paint into frame N,” so UAT can distinguish additive repaint from the existing generic save path.
- **D-16:** If save/merge fails after live repaint, keep the cached base visible and preserve live dirty edits in memory. Show retry-oriented compact copy such as “Could not merge frame N — edits are still open.” This follows Phase 36.6 failed-save semantics.
- **D-17:** Phase 36.11 should introduce no new UI surfaces. Final timeline layout/visual polish remains Phase 36.13.

```typescript
function exportTransparentStrokeCanvas(engine: EfxPaintEngine): HTMLCanvasElement {
  const state = engine.save();
  const background = state.settings.bgMode as BgMode;
  try {
    engine.setBgMode('transparent');
    return engine.exportCompositeCanvas();
```

```typescript
async function mergeCachedBaseWithLiveAlpha(baseDataUrl: string, liveCanvas: HTMLCanvasElement, appFrame: number, size: { width: number; height: number }) {
  const output = document.createElement('canvas');
  output.width = size.width;
  output.height = size.height;
  const ctx = output.getContext('2d');
  if (!ctx) throw new Error('Unable to create alpha merge canvas');
  const base = await loadImage(baseDataUrl);
  ctx.clearRect(0, 0, size.width, size.height);
  ctx.drawImage(base, 0, 0, size.width, size.height);
  ctx.drawImage(liveCanvas, 0, 0, size.width, size.height);
  return buildRotoFrameFromCanvas(output, appFrame);
}
```

**What goes wrong:** `markCurrentRotoFrameDirty` currently calls `setCachedRotoReferenceUrl(null)`, so the old cached paint disappears as soon as the user starts painting. [VERIFIED: codebase `/Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/PhysicsPaintStudio.tsx:1163-1171`]
**Why it happens:** The earlier cached-reference model treated the cached PNG as a reference that should disappear once the frame becomes live/dirty. [VERIFIED: codebase `/Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/PhysicsPaintStudio.tsx:1018-1032,1163-1171`]
**How to avoid:** Distinguish `cachedRotoReferenceUrl` or successor state as a non-editable repaint base that can remain visible during dirty live edits for real-key repaint sessions. [VERIFIED: phase context `/Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.11-physics-paint-roto-repaint-cached-real-key/36.11-CONTEXT.md:18-22`]
**Warning signs:** Tests or UAT show the old paint vanishing after the first pointer stroke.

### Pitfall 2: Save overwrites old cached alpha with only new strokes
**What goes wrong:** `flushRotoFrame` currently builds `renderedFrame` from `capturedFrame` or `buildRotoOutputFrame(engine, ...)`; this path does not merge a previous cached real-key PNG. [VERIFIED: codebase `/Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/PhysicsPaintStudio.tsx:1457-1505`]
**Why it happens:** Durable-core behavior assumed a replacement save, not additive repaint over an immutable cached base. [VERIFIED: roadmap `/Users/lmarques/Dev/efx-motion-editor/.planning/ROADMAP.md:320-336`]
**How to avoid:** Store the prior cached base identity/data URL at open time and feed it into the merge helper only when live edits exist. [VERIFIED: phase context `/Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.11-physics-paint-roto-repaint-cached-real-key/36.11-CONTEXT.md:30-35`]
**Warning signs:** Reopening after save shows only newly painted strokes.

### Pitfall 3: No-change save deletes or rewrites the cached real key
**What goes wrong:** For a clean cached reference, `snapshotCurrentRotoFrame` returns `false`; forced save paths could still enter delete/no-persist logic if not gated correctly. [VERIFIED: codebase `/Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/PhysicsPaintStudio.tsx:1207-1225,1400-1455`]
**Why it happens:** Existing empty-frame behavior maps no persistent engine content to `delete-roto-frame`; cached-base-only reopen has no engine strokes by design. [VERIFIED: codebase `/Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/PhysicsPaintStudio.tsx:394-400,1407-1455`]
**How to avoid:** Add explicit no-change preserve logic for cached-base-only frames before delete/no-persist save behavior. [VERIFIED: phase context `/Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.11-physics-paint-roto-repaint-cached-real-key/36.11-CONTEXT.md:30-35`]
**Warning signs:** Pressing Save current on a reopened cached key removes the key or changes its data URL with no new paint.

### Pitfall 4: Clear removes the real key instead of live overlay
**What goes wrong:** `clearActiveSource` currently clears engine state and removes current frame from local Roto preview/saved lists; on a cached-base repaint this would violate the locked Clear rule if applied to the base. [VERIFIED: codebase `/Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/PhysicsPaintStudio.tsx:1178-1192`]
**Why it happens:** Previous Clear semantics were aligned with clearing an editable current Roto frame, not a live overlay over immutable cached base. [VERIFIED: phase context `/Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.11-physics-paint-roto-repaint-cached-real-key/36.11-CONTEXT.md:24-28`]
**How to avoid:** In cached-base repaint sessions, Clear should call `engine.clear()` for live overlay only, preserve cached-base state and saved real-key lists, and recalculate dirty state based on whether live overlay content remains. [VERIFIED: phase context `/Users/lmarques/Dev/efx-motion-editor/.planning/phases/36.11-physics-paint-roto-repaint-cached-real-key/36.11-CONTEXT.md:24-35`]
**Warning signs:** Clear makes the real key cell empty or removes the cached base from preview.

### Pitfall 5: Paper/background leaks into merged real-key PNG
**What goes wrong:** If merge uses a canvas that includes the paper/background or an engine background image, the saved real-key cache will no longer be alpha-only. [VERIFIED: phase spec `/Users/lmarques/Dev/efx-motion-editor/SPECS/36.x-phases/phase-36.11-repaint-cached-real-key/spec-36.11-repaint-cached-real-key.md:21-28,71-77`]
**Why it happens:** The user-visible edit stack includes paper/background preview, but the persisted cache must include paint alpha only. [VERIFIED: phase spec `/Users/lmarques/Dev/efx-motion-editor/SPECS/36.x-phases/phase-36.11-repaint-cached-real-key/spec-36.11-repaint-cached-real-key.md:63-77`]
**How to avoid:** Merge only prior cached real-key PNG and `exportTransparentStrokeCanvas(engine)` output; leave `PreviewRenderer` responsible for paper/background. [VERIFIED: codebase `/Users/lmarques/Dev/efx-motion-editor/app/src/components/physic-paint/PhysicsPaintStudio.tsx:414-423`; `/Users/lmarques/Dev/efx-motion-editor/app/src/lib/previewRenderer.ts:401-423`]
