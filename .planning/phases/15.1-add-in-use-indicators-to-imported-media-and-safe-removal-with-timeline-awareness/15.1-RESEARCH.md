# Phase 15.1: Media In-Use Indicators & Safe Removal - Research

**Researched:** 2026-03-22
**Domain:** Preact + Preact Signals store architecture, asset reference tracking, UI badges, cascade removal
**Confidence:** HIGH

## Summary

This phase adds visual in-use indicators (color-coded badges with usage counts) to every asset in ImportGrid and implements safe cascade removal with timeline awareness. The work spans three domains: (1) a unified asset usage scanner that counts references across all sequences for images, videos, and audio; (2) UI components -- a clickable badge overlay on each asset and a popover listing exact usage locations; (3) cascade removal logic that handles key photo placeholders, layer removal, and audio track removal with undo support.

The most architecturally significant sub-task is the asset reference refactor (D-15, D-16, D-17): changing video layers from `videoPath` string references to `videoAssetId` ID-based references, and audio tracks from `filePath`/`relativePath` to `audioAssetId`. This creates consistency with the existing image pattern (`imageId`) and is prerequisite for reliable usage scanning. This refactor touches types, stores, serialization (projectStore build/hydrate), and the renderer -- requiring a project format version bump to v9 with v8 backward compatibility.

**Primary recommendation:** Implement in three waves: (1) asset reference refactor + unified usage scanner, (2) badge UI + popover, (3) cascade removal with undo. The refactor must come first because usage scanning depends on ID-based references.

## Architecture Patterns

### Current Asset Reference Architecture

**Images** -- ID-based (correct pattern):
```
ImportedImage.id  <--referenced-by--  KeyPhoto.imageId
ImportedImage.id  <--referenced-by--  LayerSourceData.imageId (static-image)
ImportedImage.id  <--referenced-by--  LayerSourceData.imageIds[] (image-sequence)
```

**Videos** -- path-based (needs refactor per D-15):
```
VideoAsset.path  <--coupled-to--  LayerSourceData.videoPath (string path)
VideoAsset.id    <--set-to--      layer.id during hydration (NOT a proper asset ID)
```
Critical issue: Video asset IDs are currently set to `layer.id` during hydration (projectStore.ts:362), making them non-stable. When a user creates a video layer, a new `layerId` is generated and the video asset is stored with a separate `crypto.randomUUID()` ID (ImportedView.tsx:347). The path is the only consistent link.

**Audio** -- path-based (needs refactor per D-16):
```
AudioAsset.path  <--coupled-to--  AudioTrack.filePath (absolute path)
AudioAsset.id    <--set-to--      AudioTrack.id during hydration (linked but fragile)
```
Slightly better: Audio asset IDs are set to `track.id` during hydration (projectStore.ts:404), and the audio import flow also uses `track.id` as the asset ID. But the AudioTrack type uses `filePath`/`relativePath` strings, not an asset ID reference.

### Target Architecture (after refactor)

```
VideoAsset.id  <--referenced-by--  LayerSourceData.videoAssetId (new field)
AudioAsset.id  <--referenced-by--  AudioTrack.audioAssetId (new field)
```

### Pattern: Cascade Removal

The removal flow for an in-use asset follows this sequence:

```
1. User triggers removal (badge popover or context menu)
2. getAssetUsage() returns usage locations
3. If count > 0:
   a. Show detailed confirm() with location list
   b. If confirmed:
      - Snapshot all affected stores (sequenceStore + audioStore + imageStore)
      - For images in keyPhotos: replace imageId with PLACEHOLDER_ID (D-09)
      - For images/videos in layers: remove the layer entirely (D-10)
      - For audio: remove the audio track entirely (D-11)
      - Remove asset from imageStore
      - Push single undo action that restores all snapshots
   c. If "Delete File" was chosen: second confirm(), then removeFile()
4. If count === 0:
   a. Single confirm() (D-13)
   b. Remove asset from imageStore
   c. If "Delete File": removeFile()
```

### Pattern: Key Photo Placeholder (D-09)

When an image is removed but key photos reference it, the key photo should retain its structure (hold frames, position in sequence) but display a blank/placeholder frame. Two approaches:

**Approach A -- Sentinel imageId:** Set `keyPhoto.imageId` to a well-known placeholder ID (e.g., `'__placeholder__'`). The renderer checks for this and draws a gray frame. The timeline thumbnail cache also checks.
- Pro: Simple, no structural changes
- Con: Needs special-case in renderer + timeline

**Approach B -- Nullable imageId:** Make `imageId` optional (`string | null`). Null means placeholder.
- Pro: Type-safe, explicit
- Con: Requires updating KeyPhoto type and all consumers

**Recommendation:** Approach A (sentinel ID) is simpler and matches the codebase style. The PreviewRenderer already handles "image not found" gracefully by returning null from `resolveImageSource()`, which results in nothing being drawn for that layer. We just need to ensure the timeline renderer shows a distinct placeholder visual rather than nothing.

### Anti-Patterns to Avoid

- **Computing usage on every render:** Usage scanning iterates all sequences. Cache results in a computed signal or memoize, do not call inside render loops.
- **Separate undo entries per affected store:** A cascade removal that touches sequenceStore, audioStore, and imageStore must be a SINGLE undo entry. Capture all snapshots before, restore all on undo.
- **Direct store-to-store imports for cascade:** Use the existing callback pattern (like `_markDirty`) or keep cascade logic in a coordination module (lib/assetUsage.ts) rather than having imageStore import sequenceStore.

## Common Pitfalls

### Pitfall 1: Video Asset ID Instability
**What goes wrong:** Video assets currently get `layer.id` as their asset ID during hydration, but `crypto.randomUUID()` during import. After save/reload, the video asset ID changes (from the random UUID to the layer ID), breaking any usage tracking.
**Why it happens:** The hydration code in projectStore.ts:361-365 re-discovers videos from layers and uses `layer.id` as the video asset ID.
**How to avoid:** The refactor (D-15) must persist `videoAssetId` in the MceLayerSource and use it consistently. During hydration, the video asset ID comes from the saved data, not from the layer ID.
**Warning signs:** Video usage counts change after save/reload.

### Pitfall 2: Circular Import Between Stores
**What goes wrong:** Cascade removal needs to modify sequenceStore, audioStore, and imageStore. If imageStore imports sequenceStore (or vice versa), circular dependency.
**Why it happens:** Stores are designed as independent modules with callback wiring (see `_setMarkDirtyCallback` pattern).
**How to avoid:** Put the cascade removal coordination in a separate module (e.g., `lib/assetRemoval.ts`) that imports all three stores. Or keep it in the component (ImportGrid) which already imports all needed stores.
**Warning signs:** Import errors, module initialization order issues.

### Pitfall 3: Stale Usage Cache
**What goes wrong:** If usage counts are cached (as they should be for performance), they become stale when the user adds/removes key photos, layers, or audio tracks elsewhere in the app.
**Why it happens:** Computed signals only re-compute when their dependencies change. If the computed reads `sequenceStore.sequences.value`, it will auto-update.
**How to avoid:** Use `computed()` from @preact/signals that depends on `sequenceStore.sequences`, `audioStore.tracks`, etc. The signal dependency tracking handles cache invalidation automatically.
**Warning signs:** Badge shows wrong count after adding a new key photo.

### Pitfall 4: Project Format Version Compatibility
**What goes wrong:** Adding `videoAssetId` to MceLayerSource and `audioAssetId` to MceAudioTrack changes the project format. Old v8 files won't have these fields.
**Why it happens:** Format evolution without backward compatibility.
**How to avoid:** Use optional fields with fallback during hydration. When `video_asset_id` is missing from a v8 file, fall back to current behavior (derive from layer/path). Version bump to v9 for new saves. The project already handles v7->v8 migration pattern (audio_tracks is optional).
**Warning signs:** Old projects fail to open after the refactor.

### Pitfall 5: Composite Undo Across Multiple Stores
**What goes wrong:** Removing an in-use image touches sequenceStore (key photo placeholders + layer removal), potentially audioStore (if future), and imageStore (asset removal). If each store pushes its own undo action, undoing creates partial states.
**Why it happens:** Each store's mutation methods call `pushAction` internally.
**How to avoid:** For cascade removal, bypass individual store's `pushAction` calls. Instead: capture snapshots of all stores BEFORE, perform all mutations without individual undo entries, then push a SINGLE composite undo action that restores all snapshots. This requires adding "silent" mutation methods or directly modifying signal values with `batch()`.
**Warning signs:** Ctrl+Z after cascade removal only undoes part of the changes.

### Pitfall 6: Audio Asset ID Mapping
**What goes wrong:** Audio assets are currently linked by having `AudioAsset.id === AudioTrack.id` (set during hydration at projectStore.ts:404). The refactor adds `audioAssetId` to AudioTrack, but existing audio tracks won't have this field.
**Why it happens:** The current system implicitly links audio assets to tracks by using the same ID.
**How to avoid:** During hydration of v8 files, derive `audioAssetId` from the existing `track.id` (which equals the audio asset ID). For v9 files, read it from the saved data.
**Warning signs:** Audio usage counts show 0 for tracks that should show 1.

## Code Examples

### Usage Scanner (lib/assetUsage.ts)

```typescript
// Source: Derived from existing isImageInUse() pattern in imageStore.ts:174-183
import type {Sequence} from '../types/sequence';
import type {AudioTrack} from '../types/audio';

export interface UsageLocation {
  sequenceId: string;
  sequenceName: string;
  type: 'key-photo' | 'layer' | 'audio-track';
  detail: string;
}

export interface AssetUsage {
  assetId: string;
  assetType: 'image' | 'video' | 'audio';
  locations: UsageLocation[];
  count: number;
}

export function getImageUsage(imageId: string, sequences: Sequence[]): AssetUsage {
  const locations: UsageLocation[] = [];

  for (const seq of sequences) {
    // Check key photos
    seq.keyPhotos.forEach((kp, idx) => {
      if (kp.imageId === imageId) {
        locations.push({
          sequenceId: seq.id,
          sequenceName: seq.name,
          type: 'key-photo',
          detail: `Key Photo #${idx + 1}`,
        });
      }
    });

    // Check layers
    for (const layer of seq.layers) {
      if (layer.source.type === 'static-image' && layer.source.imageId === imageId) {
        locations.push({
          sequenceId: seq.id,
          sequenceName: seq.name,
          type: 'layer',
          detail: `Layer '${layer.name}'`,
        });
      }
      if (layer.source.type === 'image-sequence' && layer.source.imageIds?.includes(imageId)) {
        locations.push({
          sequenceId: seq.id,
          sequenceName: seq.name,
          type: 'layer',
          detail: `Layer '${layer.name}'`,
        });
      }
    }
  }

  return { assetId: imageId, assetType: 'image', locations, count: locations.length };
}

export function getVideoUsage(videoAssetId: string, sequences: Sequence[]): AssetUsage {
  const locations: UsageLocation[] = [];

  for (const seq of sequences) {
    for (const layer of seq.layers) {
      if (layer.source.type === 'video' && layer.source.videoAssetId === videoAssetId) {
        locations.push({
          sequenceId: seq.id,
          sequenceName: seq.name,
          type: 'layer',
          detail: `Layer '${layer.name}'`,
        });
      }
    }
  }

  return { assetId: videoAssetId, assetType: 'video', locations, count: locations.length };
}

export function getAudioUsage(audioAssetId: string, tracks: AudioTrack[]): AssetUsage {
  const locations: UsageLocation[] = [];

  for (const track of tracks) {
    if (track.audioAssetId === audioAssetId || track.id === audioAssetId) {
      locations.push({
        sequenceId: '', // audio tracks are project-level
        sequenceName: 'Timeline',
        type: 'audio-track',
        detail: `Track '${track.name}'`,
      });
    }
  }

  return { assetId: audioAssetId, assetType: 'audio', locations, count: locations.length };
}
```

## Key Architectural Decisions (Claude's Discretion)

### 1. Project Format Version Bump: YES, v8 -> v9

**Rationale:** The `video_asset_id` field is a new serialization field in MceLayerSource, and `audio_asset_id` is a new field in MceAudioTrack. While both can be optional for backward compat reading, the semantics change (layers now reference asset IDs, not paths). A version bump clearly signals this. The existing v7->v8 migration pattern (audio_tracks optional) already established the precedent.

**Backward compatibility:** v9 reader handles v8 files by falling back to path-based lookup. v8 reader ignores unknown fields (standard JSON behavior).

### 2. Undo Scope: Full cascade undo

**Rationale:** The snapshot/restore pattern already captures full store state via `structuredClone`. Capturing 3 store snapshots is trivially cheap (these are small arrays). Full undo means the user gets everything back: removed key photos are restored, layers reappear, audio tracks return. The only thing NOT undoable is file deletion from disk (which happens after a second confirmation).

### 3. Badge Rendering: Positioned overlay on all asset types

**Rationale:** Images and videos use thumbnail grids (aspect-[4/3] containers). The badge goes `absolute top-1 right-1`. Audio uses a list layout. The badge goes inline or absolute right-aligned. Same component, slightly different positioning via props.

### 4. Video Asset Registry in imageStore

**Current state:** Video assets exist in `imageStore.videoAssets` (a signal array), but they are loosely linked to layers. A video asset ID is a random UUID set during import, but during hydration it's overwritten with `layer.id`.

**After refactor:** Video assets in `imageStore.videoAssets` have stable IDs. Each VideoAsset also stores `relativePath` for serialization. Layers reference `videoAssetId`. During import, a VideoAsset is created first, then the layer references its ID. During hydration, the video_asset_id is read from the saved layer source.

### 5. VideoAsset Type Enhancement

VideoAsset currently has `{ id, name, path }`. After refactor, it should also include `relativePath` for serialization, mirroring how images work. Video assets should be persisted in the .mce file as a top-level array (like `images`), not just reconstructed from layers during hydration.

Alternatively, to minimize format changes, video assets can continue to be derived from layers (their source data includes the path), with the key change being that `videoAssetId` in the layer source maps to the stable asset ID. The planner should decide which approach minimizes risk.

## Open Questions

1. **Should video assets be persisted as a top-level array in .mce?**
   - What we know: Currently, video assets are re-discovered from video layers during hydration. Images have a dedicated `images[]` array in MceProject.
   - What's unclear: Whether adding `video_assets[]` and `audio_assets[]` top-level arrays is worth the format complexity vs keeping them derived from layers/tracks.
   - Recommendation: Keep derived approach for now (less format change). The `videoAssetId` in the layer source is sufficient. Audio assets are already implicitly stored via `audio_tracks[]`. Adding top-level arrays can be deferred.

2. **Placeholder image rendering in timeline thumbnails**
   - What we know: PreviewRenderer returns null when image not found (draws nothing). Timeline renderer draws colored alternating blocks for frames without thumbnails.
   - What's unclear: Should placeholders show a distinct "removed image" icon/pattern vs the existing "loading" appearance?
   - Recommendation: Use a distinct visual (e.g., a small X icon or hatched pattern) so users can distinguish "image removed" from "image loading". Keep it simple -- a gray frame with a centered X text.
