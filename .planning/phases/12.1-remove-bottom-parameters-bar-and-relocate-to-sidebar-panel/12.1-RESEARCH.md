# Phase 12.1: Remove Bottom Parameters Bar and Relocate to Sidebar Panel - Research

**Researched:** 2026-03-15
**Domain:** UI layout refactoring -- Preact + Preact Signals + Tailwind CSS
**Confidence:** HIGH

## Summary

This phase is a significant UI layout refactoring within an existing Preact/Signals/Tailwind application. The core task is dismantling the bottom `PropertiesPanel` bar (a horizontal strip of property controls) and reconstructing those controls vertically in the left sidebar, while also extracting the ImportGrid and Settings into full-window overlay views. No new libraries are needed; this is entirely a component architecture and layout restructuring task.

The codebase is well-structured with clear separation between stores (Preact Signals), components (Preact functional components), and styling (Tailwind CSS with CSS custom properties for theming). The `PropertiesPanel.tsx` at 822 lines is the largest file and contains all sub-components (`NumericInput`, `SectionLabel`, `TransformSection`, `CropSection`, `BlurSection`, `FxSection`, `KeyframeButton`, etc.) that must be extracted and reused in the new sidebar layout.

**Primary recommendation:** Extract reusable controls from `PropertiesPanel.tsx` into shared modules, build the new sidebar properties section as a vertical two-column grid layout, add editor mode switching (editor/imported/settings) to `uiStore` and `EditorShell`, and remove the bottom bar last after all functionality is verified in the sidebar.

### Pattern 1: Collapsible Section Component
**What:** Reusable collapsible section with header, chevron toggle, and content slot.
**When to use:** SEQUENCES, LAYERS, PROPERTIES sections in sidebar.
**Example:**
```typescript
// CollapsibleSection.tsx
import {Signal} from '@preact/signals';

interface CollapsibleSectionProps {
  title: string;
  collapsed: Signal<boolean>;
  headerActions?: preact.ComponentChildren;
  children: preact.ComponentChildren;
}

export function CollapsibleSection({title, collapsed, headerActions, children}: CollapsibleSectionProps) {
  return (
    <>
      <div
        class="flex items-center justify-between h-9 px-3 bg-[var(--color-bg-section-header)] cursor-pointer select-none"
        onClick={() => { collapsed.value = !collapsed.value; }}
      >
        <div class="flex items-center gap-1.5">
          <span class="text-[9px] text-[var(--color-text-dimmer)]">
            {collapsed.value ? '\u25B6' : '\u25BC'}
          </span>
          <span class="text-[10px] font-semibold text-[var(--color-text-muted)]">
            {title}
          </span>
        </div>
        {headerActions && <div onClick={(e) => e.stopPropagation()}>{headerActions}</div>}
      </div>
      {!collapsed.value && children}
    </>
  );
}
```

### Pattern 3: Uniform Scale Input
**What:** A single Scale input that adjusts both `scaleX` and `scaleY` together.
**When to use:** Transform section in sidebar properties.
**Example:**
```typescript
// Uniform scale -- use average of SX/SY as display value,
// set both to the new value on change
const uniformScale = (vals ? (vals.scaleX + vals.scaleY) / 2 : (layer.transform.scaleX + layer.transform.scaleY) / 2);

<NumericInput
  label="Scale"
  value={uniformScale}
  step={0.01}
  min={0.01}
  onChange={(val) => {
    updateTransform('scaleX', val);
    updateTransform('scaleY', val);
  }}

```

### Pattern 4: Key Photos Under Sequences
**What:** Key photos appear inline under the expanded active sequence in the SEQUENCES section, rather than as a separate section.
**When to use:** SEQUENCES collapsible section.
**Key insight:** Currently `SequenceList` renders sequence rows and `KeyPhotoStrip` is rendered separately in `LeftPanel`. Refactor so each `SequenceItem` conditionally renders its own `KeyPhotoStrip` when active and expanded.

### Anti-Patterns to Avoid
- **Duplicating property edit logic:** The `PropertiesPanel` has keyframe-aware edit routing (`handleKeyframeEdit`) that must be preserved exactly when moving to sidebar. Extract this logic into a shared hook or helper, not duplicated inline.
- **Breaking the transient overrides pattern:** `keyframeStore.displayValues` and `keyframeStore.setTransientValue` are used for edits between keyframes. The sidebar properties must use the same pattern.
- **Bypassing the coalescing pattern:** `startCoalescing()`/`stopCoalescing()` calls in `NumericInput` and range sliders are critical for undo/redo grouping. Ensure they survive extraction.
- **Mixing per-sequence and per-project FPS:** The context says to remove per-sequence FPS/Res duplication. But `sequenceStore.setSequenceFps` is still called in `LeftPanel.tsx`. The Settings view should only use `projectStore.setFps` and `projectStore.setResolution`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Collapsible sections | Custom expand/collapse with useRef + height animation | Signal<boolean> + conditional render | Simpler, no layout thrashing, matches Preact Signals pattern |
| Drag-and-drop between views | Custom drag handlers | SortableJS `forceFallback:true` | Already proven in codebase, handles edge cases |
| Theme switching | New theme selector | Existing `ThemeSwitcher` component | Already styled and functional, just relocate |
| Sidebar collapse animation | CSS keyframes | Tailwind `transition-all duration-200` with width toggle | Minimal code, GPU-composited |

## Common Pitfalls

### Pitfall 1: Breaking Keyframe-Aware Property Editing
**What goes wrong:** Moving property controls to the sidebar without preserving the `handleKeyframeEdit` routing causes edits between keyframes to modify layer state instead of transient overrides, corrupting interpolation.
**Why it happens:** The keyframe routing logic in `PropertiesPanel` is 20 lines of conditional logic that's easy to overlook.
**How to avoid:** Extract `handleKeyframeEdit` as a reusable function/hook before building sidebar properties. Test with: select layer with keyframes, scrub to between-keyframe position, edit a property, verify it doesn't persist after scrubbing away.
**Warning signs:** Properties "jumping" when scrubbing between keyframes.

### Pitfall 2: FX Layer Selection Replacing LAYERS Section
**What goes wrong:** When an FX layer is selected from the timeline, the sidebar must replace the LAYERS section with "FX: [Layer Name]" but restore it when deselected. If the selection/deselection signal flow is wrong, the LAYERS section can disappear permanently.
**Why it happens:** FX layers live in FX sequences (not the active content sequence), so `layerStore.selectedLayerId` may point to a layer that isn't in `layerStore.layers`.
**How to avoid:** Use the same lookup pattern as `PropertiesPanel` (lines 546-557): iterate all sequences to find the selected layer. Check if it's in an FX sequence to decide rendering mode.
**Warning signs:** Clicking an FX layer in the timeline shows nothing in the sidebar, or clicking away doesn't restore the LAYERS list.

### Pitfall 3: Import Grid Drag to Sidebar
**What goes wrong:** The ImportedView is a full-window replacement of canvas+timeline, but the sidebar remains visible. Users should be able to drag thumbnails from ImportedView to KeyPhotoStrip or LayerList in the sidebar, but SortableJS instances in different DOM trees may not communicate.
**Why it happens:** SortableJS `group` configuration is needed for cross-container drag. The existing KeyPhotoStrip and ImportGrid don't use groups.
**How to avoid:** For the initial implementation, keep the existing drag-from-ImportGrid-to-sidebar working as-is (it currently works because they're in the same sidebar). The ImportedView will need a different approach -- consider a simple "click to add" or a separate drag group configuration. This is a discretion area and can use a simpler approach first.
**Warning signs:** Drag operations fail silently or drop events fire on wrong targets.

### Pitfall 4: Sidebar Scroll Behavior
**What goes wrong:** When SEQUENCES, LAYERS, and PROPERTIES are all expanded with many items, the sidebar becomes taller than the viewport. Without proper scroll containment, sections overlap or the sidebar becomes unusable.
**Why it happens:** Each section currently has its own `overflow-y-auto` but the parent `LeftPanel` uses fixed heights.
**How to avoid:** The overall sidebar should be a flex column with `overflow-y-auto` on the entire panel, OR each section should have a max-height with individual scroll. Recommendation: make the PROPERTIES section scrollable independently since it can be long, while SEQUENCES and LAYERS use the remaining space with their own overflow.
**Warning signs:** Content clipped without scrollbar, or entire sidebar scrolls as one unit causing header/chevrons to scroll off-screen.

### Pitfall 5: Per-Sequence FPS/Res Data Still in Store
**What goes wrong:** Removing per-sequence FPS/Res from the UI but not from `sequenceStore` means the data model still has `seq.fps` and `seq.width/height` that could diverge from project-level values.
**Why it happens:** The save/load format (`MceSequence`) includes `fps`, `width`, `height` per sequence.
**How to avoid:** Keep the per-sequence fields in the data model for backward compatibility but stop exposing separate UI controls. The `SequenceSettings` component in `LeftPanel.tsx` should be removed. Project-level FPS/Res in Settings view should be the only way to set these values. Consider syncing: when project FPS changes, update all sequences.
**Warning signs:** Sequences saved with old FPS values that don't match the project FPS in the Settings view.

### Pitfall 6: Blur Bypass During Drag-to-Scrub
**What goes wrong:** The `NumericInput` component has performance optimization that bypasses blur rendering during label drag-scrub. When extracting to a shared module, the `blurStore` import may be forgotten.
**Why it happens:** The blur bypass is a non-obvious side effect buried in `handleLabelPointerDown`.
**How to avoid:** Ensure extracted `NumericInput` retains all imports: `blurStore`, `startCoalescing`, `stopCoalescing`.

## Open Questions

1. **Per-sequence FPS/Res backward compatibility**
   - What we know: `MceSequence` format has `fps`, `width`, `height` per sequence. `projectStore` has project-level `fps`, `width`, `height`.
   - What's unclear: Should saving a project now force all sequences to project-level values, or should the per-sequence fields simply be hidden from the UI but preserved?
   - Recommendation: Hide from UI, but on project save, sync all sequences to project-level values. On load, use project-level values (existing migration handles this). This is the simplest approach that avoids data divergence.

2. **ImportedView drag to sidebar KeyPhotoStrip**
   - What we know: Currently ImportGrid is IN the sidebar so SortableJS drag works naturally.
   - What's unclear: When ImportedView is full-window (separate from sidebar), cross-container drag needs SortableJS groups or a different UX.
   - Recommendation: For initial implementation, use "click to add to selected sequence" in ImportedView rather than cross-container drag. This avoids SortableJS group complexity and can be enhanced later.

3. **Sidebar width when Properties section is showing**
   - What we know: Sidebar is currently fixed at 268px.
   - What's unclear: 268px may feel cramped with two-column numeric inputs.
   - Recommendation: Keep 268px. NumericInput is currently `w-16` (64px). Two inputs + labels + gap fits in ~252px usable width. Test and adjust if needed.
