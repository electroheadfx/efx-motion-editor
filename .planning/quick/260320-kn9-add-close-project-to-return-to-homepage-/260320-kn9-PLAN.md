Add a "File" submenu to the macOS native menu bar, positioned between the app submenu and the Edit submenu. Create four MenuItem::with_id items:

1. `MenuItem::with_id(app, "new-project", "New Project", true, Some("CmdOrCtrl+N"))`
2. `MenuItem::with_id(app, "open-project", "Open Project...", true, Some("CmdOrCtrl+O"))`
3. `MenuItem::with_id(app, "save-project", "Save", true, Some("CmdOrCtrl+S"))`
4. `MenuItem::with_id(app, "close-project", "Close Project", true, Some("CmdOrCtrl+W"))`

Build the file_submenu using SubmenuBuilder::new(app, "File") with items in order: new, open, separator, save, separator, close-project. Add `.item(&file_submenu)` to the MenuBuilder between app_submenu and edit_submenu.

In the on_menu_event closure, add four new event ID matches that emit corresponding events to the frontend:
- "new-project" -> handle.emit("menu:new-project", ())
- "open-project" -> handle.emit("menu:open-project", ())
- "save-project" -> handle.emit("menu:save-project", ())
- "close-project" -> handle.emit("menu:close-project", ())

IMPORTANT: Because these menu items use native accelerators (CmdOrCtrl+N/O/S/W), the native Cocoa menu will intercept the keydown events BEFORE they reach the webview. This means the tinykeys shortcuts for Cmd+N/O/S in shortcuts.ts will NOT fire when these menu accelerators are active. This is the CORRECT behavior — the menu event listeners in main.tsx will handle these instead (same pattern as Edit > Undo/Redo with Cmd+Z).

Remove the `$mod+KeyN`, `$mod+KeyO`, and `$mod+KeyS` bindings from shortcuts.ts in the next task since the native menu accelerators will handle those keys. The tinykeys bindings would be dead code.
